0Why spec-driven development, and why with care

E1–E7 built and measured the harness. E8 is about the workflow that feeds it: spec-driven development (SDD) turns intent into an explicit, reviewable specification that becomes the primary quality gate before code exists. It is the engineering-side counterpart to the handoff contract (M3) — the discipline that decides whether an agent builds the right thing or faithfully builds the wrong one.

But E8 comes with a deliberate double message. SDD is genuinely powerful for the cases it fits, and actively counter-productive for the cases it doesn't. Our job in the engagement is not to sell "adopt SDD wholesale" — it is to help the client apply it where the leverage is real (greenfield, legacy modernization, multi-task features) and avoid the sledgehammer-on-a-thumbtack failure mode Böckeler warns about.

The one idea to internalise

"Intent is the source of truth." SDD's core move is to make the specification — not the code — the thing you review, version, and hold agents accountable to. That inverts the classic gate: spec review precedes (or replaces) code review as the primary quality control. The prize is catching a wrong intent on paper; the risk is drowning a small task in ceremony.

1What SDD actually is

SDD is a workflow in which a structured specification is authored first, reviewed and agreed, and then drives implementation — with the agent building toward the spec rather than improvising intent from a terse ticket. The spec is executable-adjacent: precise enough that acceptance criteria (EARS/Gherkin, M3) can be derived and checked.

The contrast with "just prompt the agent" is that intent becomes a durable artifact (M4) with an owner and a review, rather than ephemeral text in a chat. For an L3 delegated workflow (E6), that artifact is what the agent is held accountable to and what the human reviews.

2The tools — three points on a spectrum

We care about methodology over tools, but the three reference implementations usefully mark out the design space, from lightweight-and-portable to full-team simulation.

A

GitHub Spec Kit

lightweight, portable

A slash-command workflow: five core commands, /speckit.constitution → /speckit.specify → /speckit.plan → /speckit.tasks → /speckit.implement (with /speckit.clarify and /speckit.analyze as optional refinement steps), unchanged across releases. The constitution.md holds non-negotiable project principles; the philosophy is "intent is the source of truth." Strongest on greenfield and legacy modernization; deliberately minimal and tool-agnostic. As of v0.11.0 (June 2026) it supports 30+ agents and ships extensions, presets, a skills-installation mode, and — most telling for E8 — upgrade detection with version pinning, so a team can manage the workflow itself like a dependency: the spec-as-artifact argument (M4) made concrete, applied to the very process that produces the specs. Point-in-time facts; re-verify before relying on them.

B

Amazon Kiro

agentic IDE, gated phases

Three phases with approval gates: requirements.md (user stories with EARS acceptance criteria) → design.md → tasks.md (numbered checklist). Bakes EARS directly into requirements and runs automated-reasoning checks for contradictions before code — the tightest coupling of spec to machine-verifiability. General availability came on Nov 17, 2025, adding property-based testing for spec correctness — generated properties measure whether the implementation actually matches the spec, the strongest bridge yet between SDD and verification-first (E4): the spec stops being prose the agent aims at and becomes something the harness can check the code against — plus checkpointing, a Kiro CLI, team plans and Pro/Pro+/Power tiers. Point-in-time facts; re-verify before relying on them.

C

BMAD-METHOD

full-team simulation

An MIT-licensed, bring-your-own-model framework distributed as files (agent personas in Markdown/YAML + workflows, templates, a CLI installer) that drop into your existing AI IDE — not a product or service. It simulates 12+ specialized agent roles (analyst, PM, architect, dev, QA, scrum master…), each a self-contained "Agent-as-Code" definition, running a four-phase cycle (Analysis → Planning → Solutioning → Implementation) with explicit artifact handoffs and document sharding (large PRD/architecture docs are split into small "story files" that pack exactly what one agent needs, fighting context loss). Its defining stance: documentation is the source of truth, code is a downstream derivative ("docs-as-code") — the aggressive spec-as-source end of the §4 spectrum. The heaviest end: powerful for complex work, but the most ceremony and the easiest to over-apply. The v6 line is current (v6.8, May 2026 — Skills Architecture, sub-agents, Dev Loop Automation) and evolving fast — verify specifics before relying on them.

What we extract

The methodology is portable even if the tool is not: a constitution of non-negotiables, an explicit spec authored and reviewed first, EARS/Gherkin acceptance criteria, and a decomposition into agent-ready tasks. We can assemble that lightweight pipeline for a client without adopting any single vendor tool — which is also how we avoid lock-in.

BMAD on an existing system — the brownfield path

Because our engagement targets an existing codebase, BMAD's brownfield workflow is the part most relevant to us — and it is more thought-through than the other SDD tools. It is worth knowing in detail even if we only borrow pieces of it.

How BMAD handles brownfield
  • Document-first is mandatory. The *document-project workflow scans the whole codebase and writes AI-optimised docs (overview, architecture, source-tree) into docs/ — the required first step when docs are missing or stale.
  • Two routes by size. PRD-First (large codebases: write the PRD, then document only the areas it touches) vs Document-First (smaller: document everything, then PRD).
  • Respects conventions. Auto-detects the stack and explicitly asks whether to follow existing patterns or set new ones.
  • Regression safety. A Test Architect (QA) role is mandatory when a change touches legacy code: *risk → regression potential, *design → test coverage, *review → verify nothing broke.
  • Quick Flow for small work. Bug fixes / small increments skip the ceremony: short spec, implement, self-review against the codebase.
How it maps to our framework
  • *document-project = our brownfield bootstrap (Report 4: read-only → characterize) and durable artifacts (M4)
  • "respect existing conventions" = the agent adapts to the estate rather than rewriting it (our brownfield principle)
  • Test Architect *risk/*design/*review = verification-first (E4) + blast-radius thinking, wrapped as a role
  • doc sharding + resumable state = context-rot management (E2) + progress/clean-state (E3)
  • Quick Flow = the answer to Böckeler's "sledgehammer on a thumbtack" (§4)
What we borrow for our engagement

Don't adopt BMAD wholesale — it is the most ceremony-heavy option, it pushes docs-as-source (we recommend spec-anchored, not spec-as-source), and it moves fast (V6, high churn). Instead borrow the mechanics that map cleanly onto our framework: the *document-project step as a brownfield bootstrap that produces durable, reviewable docs; brownfield PRD/architecture as durable artifacts (M4); and a mandatory regression gate on any legacy touch (Test Architect's *risk/*design/*review = our E4). That gives a client BMAD's real brownfield value — structure and regression safety when modifying an existing system — without lock-in to a single fast-moving tool.

3Two plugin approaches: Superpowers & Compound Engineering

Spec Kit, Kiro and BMAD are spec authoring tools. But two of the most-adopted 2025–2026 Claude Code plugins go further — they package a whole methodology as auto-triggering skills, and they differ most instructively in one dimension the earlier tools barely touch: whether the system learns between tasks. Both are worth knowing because a client may already be using one, and they embody two distinct philosophies of the same loop.

SP

Superpowers

Jesse Vincent — discipline enforcement

A skills framework — v5.x as of Mar 2026, now multi-platform (Claude Code, Cursor, Codex, Copilot CLI) — that forces a coding agent through disciplined phases instead of its default "jump straight to code" instinct. A master dispatcher skill auto-activates at session start and routes the task through the right skills; nothing needs to be invoked manually. Notably, v5 reversed an early design choice: the subagent review loops were removed in favour of a Spec Self-Review checklist — even the methodology tools iterate on their own verification story. (An earlier claim that it was accepted into Anthropic's official plugin marketplace in Jan 2026 could not be verified — treat as unconfirmed. Point-in-time facts; re-verify.)

CE

Compound Engineering

Kieran Klaassen / Every — the system learns

A plugin (from Every; the workflow they use to run five products with near single-person engineering teams) built on one thesis: "each unit of engineering work should make subsequent units easier, not harder." It ships the plan→work→review loop like Superpowers, but adds a step the others don't have — an explicit compound step that writes learnings back into the system so the next task starts smarter.

The same loop, seen through three lenses

Comparing them on planning, execution and feedback-learning is the useful exercise — it isolates what actually differs.

DimensionSuperpowersCompound Engineering
PlanningSocratic brainstorming → design sign-off → a plan of bite-sized tasks (2–5 min each) with exact file paths and complete code, written for "an enthusiastic junior with no context"brainstorm → plan where sub-agents research the codebase, edge cases and past learnings in parallel; the ratio is stated as ~80% planning+review, 20% execution
Executionsubagent-driven development: a fresh subagent per task; two-stage subagent review (spec compliance, then code quality) up to v4, replaced in v5 by a Spec Self-Review checklist; enforced red-green TDD, YAGNI, DRY/ce-work implements the plan (asks clarifying questions first, writes tests), then a simplify pass; specialist reviewer agents fan out (security, architecture, performance…)
Feedback learninglargely within a run (reviews, checkpoints); learning across tasks is not the core loop — the discipline resets each sessionthe defining feature: /ce-compound codifies what was learned into CLAUDE.md / docs (ADRs, solutions, new agents) so future brainstorms and plans read it as grounding
The one difference that matters most

Superpowers makes a single task go well by enforcing discipline; Compound Engineering makes the next task go better by turning each finished task into durable, machine-readable institutional knowledge. Superpowers is a within-task quality system; Compound Engineering adds a between-task learning loop. That "compound" step — capture the lesson, make it findable (tags/frontmatter), update CLAUDE.md, verify the system would now catch it automatically — is exactly the feedback dimension the classical SDD tools (Spec Kit/Kiro/BMAD) leave out.

How they map onto our framework

Neither is a new idea so much as a packaging of components we already have — which is exactly why they are useful reference points, not dependencies:

What they operationalise
  • plan-before-code = the SDD spec-review gate (this topic)
  • enforced TDD + two-stage review = verification (E4)
  • subagent-per-task, single-threaded = harness patterns (E3)
  • auto-triggering skills = progressive disclosure (M4/I5)
Where Compound's "compound" step lands
  • it is the feedback loop of information architecture (M4) made routine
  • learnings → CLAUDE.md/ADRs = durable artifacts with provenance
  • caution: writing to CLAUDE.md/memory is also a poisoning surface (M7) — the learnings need review, not blind append

For our rollout: we don't have to adopt either wholesale (both are opinionated, single-vendor, fast-moving — v-numbers change weekly). The lesson to carry to a client is the compound principle itself: build the between-task learning loop deliberately (codify review findings and fixes back into CLAUDE.md/ADRs/skills), because that is what turns a one-off productivity bump into a system that compounds — while keeping a human gate on what gets written back, per M7. Note also where the learnings land: CLAUDE.md is the advisory layer (E5), so compounded lessons inherit its failure modes — they can be silently ignored, drift stale, or be poisoned — which is exactly why the write-back needs a human gate rather than a blind append. Superpowers teaches the within-task discipline; Compound Engineering teaches that the discipline should also make the system smarter each time.

4The critique — and when SDD is overkill

Part of E8's value is knowing the failure mode. Birgitta Böckeler (ThoughtWorks, writing on martinfowler.com) and others have documented SDD's downsides candidly, and we should carry the critique openly rather than as evangelists.

The critique
  • generates piles of repetitive markdown that review worse than code
  • a sledgehammer for small tasks (one documented Kiro session turned a small bug into 4 stories + 16 acceptance criteria)
  • brownfield enterprise is the weak spot — specs assume clean intent
  • ceremony can relocate review burden rather than remove it
When SDD genuinely fits
  • greenfield features spanning multiple agent tasks
  • legacy modernization (capture business logic into a spec)
  • anything where the spec is the durable contract & review artifact
  • work where getting intent wrong is expensive

The heuristic: a task under ~15 minutes needs no spec; a one-line bugfix drowned in 16 acceptance criteria is the anti-pattern, not the ideal. Keep specs minimal (the M3 sizing: ~10–30 lines) and portable. SDD is a tool you reach for when intent is non-trivial and the work is delegated — not a universal process to impose on every ticket.

What ~15 lines of spec actually looks like — a brownfield micro-feature

We legislate "10–30 lines" above, so here is one, for a realistic task: throttling a legacy password-reset endpoint that support reports being abused.

Constitution excerpt (2 lines):
Never weaken existing auth flows.
New behaviour ships behind a config flag, off by default.

Spec: rate-limit POST /password-reset per account. Non-goals: no CAPTCHA, no per-IP limits (separate ticket), no change to token expiry.

Acceptance criterion (EARS, one of three):
WHEN a client submits more than 5 reset requests for the same account within 15 minutes, THE SYSTEM SHALL respond 429 and SHALL NOT send a reset email.

Task slice:
T1 sliding-window counter on the existing Redis · T2 gate the handler + 429 path behind the flag · T3 tests: 6th request throttled, 5th passes, flag off = old behaviour byte-for-byte.

Fifteen lines in the repo, one review comment ("make the window configurable"), a wrong assumption (per-IP vs per-account) caught before any code existed. That is the entire ceremony a task this size earns — anything more is the sledgehammer.

5How SDD connects the whole framework

SDD is not a standalone practice; it is the connective tissue between several components we have already studied. Seeing those links is what makes it usable rather than a buzzword.

SDD elementConnects toHow
Constitutiongovernance (E5)non-negotiables encode advisory principles; the deterministic ones become hooks
Specify / acceptance criteriahandoff contract (M3)the spec is the engineering face of the contract; EARS/Gherkin make it verifiable
Spec review gateautonomy (E6)a reviewed spec is what makes L3 delegation accountable
Spec as artifactinformation architecture (M4)durable, versioned, owned — not ephemeral chat text
Tasks → implementverification (E4)acceptance criteria become the failing tests the agent builds toward
The spec-anchored recommendation

Böckeler distinguishes spec-first (spec guides, code stays primary), spec-anchored (spec is a living governing contract), and spec-as-source (code is a disposable byproduct). For a brownfield client we recommend spec-anchored: it works with human review and imperfect specs, whereas spec-as-source presumes complete, always-current specs and fully reliable regeneration — which today's reality does not support — Beck (Dec 2025): whole-spec-first "encodes the (to me bizarre) assumption that you aren't going to learn anything during implementation that would change the specification".

SDD's payoff is also measurable, which ties it to E7: track the spec-review catch rate (wrong intent caught at the spec gate rather than in code review or production) and compare rework rate on specced vs unspecced tasks. The latter now has an industry hook — DORA introduced Rework Rate as an official fifth delivery metric in its 2024 report — so the baseline half of that comparison likely already exists in the client's delivery dashboards.

7Takeaways for our engagement

Assess
  • where is intent non-trivial and delegated (SDD pays off)?
  • where are tasks tiny (SDD is overkill)?
  • is there appetite to review specs, not just code?
  • can we extract a lightweight pipeline without tool lock-in?
Build (feeds E7 rollout)
  • a minimal constitution of non-negotiables
  • spec-anchored workflow; specs versioned in the repo (M4)
  • EARS/Gherkin acceptance criteria feeding tests (E4)
  • spec review as the gate for L3 delegation (E6)
Three sentences to remember E8 by

SDD makes intent the source of truth: an explicit spec is authored and reviewed first, so spec review becomes the primary quality gate before code exists. It is powerful for greenfield, legacy modernization and multi-task delegated features — but overkill for small tasks, so keep specs minimal (10–30 lines) and reach for it only when getting intent wrong is expensive. Recommend spec-anchored (a living contract with human review), not spec-as-source, and wire the constitution to governance, the acceptance criteria to verification, and the spec review to L3 autonomy.
