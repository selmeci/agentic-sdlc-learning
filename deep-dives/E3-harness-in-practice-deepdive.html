0Why the harness is the thing we actually build

E1 named the equation (Agent = Model + Harness) and E2 dug into context. E3 is where it gets concrete: the harness is the software we design, own, and maintain. It is the deliverable of an agentic-engineering engagement — not the model (a commodity we rent) and not the prompt (advisory text). If a client asks "what are we paying you to build?", the honest answer is: this.

This document turns the abstract "orchestrating scaffold" into named patterns with inputs and outputs: how a memoryless model is bootstrapped into a session, how state survives between sessions, how work stays merge-ready, and why the right long-term move is to delete parts of the harness as models improve. Everything here is drawn from Anthropic's own harness-engineering practice and generalised for a brownfield client.

The one idea to internalise

Every harness component encodes an assumption about what the model cannot do yet. The harness is therefore a liability register of model weaknesses, written in code. When a weakness disappears with a better model, the matching scaffold should be removed — a good harness shrinks over time, it does not accrete.

1What the harness is responsible for

Between "a model that can reason" and "an agent that ships merge-ready code" sits a set of responsibilities nothing else owns. The harness is the software that discharges them:

State & continuity
  • bootstrap a cold, memoryless model into a working session
  • carry decisions and progress across sessions
  • leave the repo in a clean, resumable state
Orchestration & control
  • assemble context each turn (E2) and invoke tools
  • run the loop; detect stuck/blocked states
  • enforce deterministic constraints (E5 governance)
Why it can't live elsewhere
  • The model is stateless and unconstrained; the prompt is advisory; CI is downstream. Only the harness sits around the model at runtime, so only it can guarantee state, sequencing and hard limits.

2The core patterns — inputs & outputs

Read the diagram first: a harness runs sessions like relay legs. Each session is bootstrapped from durable state, does bounded work, and hands a clean baton to the next — the "engineers working in shifts" model.

A

The initializer agent

solving the cold start

A memoryless model dropped into a repo has no idea how to build, test, or where it left off. The initializer pattern uses a first, special session to lay down the scaffolding every later session relies on: a runnable setup script, a progress file, and an initial commit as a known-good baseline.

Requires on input
  • repo access + a way to run commands
  • an initializer prompt describing the setup contract
  • enough repo signal to infer build/test commands
Produces on output
  • init.sh (deterministic environment setup)
  • claude-progress.txt seeded with current state
  • an initial git commit — the resumable baseline
Where it sits
  • Phase-1 setup / onboarding — run once per repo; also the moment we discover whether a repo is agent-ready at all (feeds E6).

For our rollout: the initializer is the cheapest early diagnostic. If a model can't figure out how to build and test the repo, neither can a delegated agent — that's an agent-readiness red flag, not a prompt problem.

B

The progress file & cross-session state

continuity for a stateless model

The model forgets everything between sessions; the harness must not. A progress file plus git history is the "shift handover notes" that let session N+1 resume session N's work without re-deriving intent from scratch.

Requires on input
  • a convention for what belongs in the file (E1 memory component / M4)
  • a protocol: read state + git log at session start
  • a discipline: write continuously (assume interruption)
Produces on output
  • what this session finished, WIP + next step, blockers
  • file paths (not contents) pointing at the work
  • a resumable thread across many sessions
Where it sits
  • Implementation runtime; the boundary where the harness (E3) leans on memory (E1's memory component, module M4). Architectural decisions belong in ADRs, not here.

For our rollout: keep it disposable and pruned — a progress file over ~200 lines is a smell; move durable facts to ADRs/CLAUDE.md and detail to topic memory. It records where we are, not what is true.

C

The clean-state principle

always merge-ready

Every session ends with the repo in a coherent, mergeable state — never mid-refactor with broken tests. This is what makes the relay safe: any session can be the last one, and any human can pick up cleanly.

Requires on input
  • a fast verification loop to confirm "green" (E4 / M1)
  • a stop protocol: finish or cleanly park work
Produces on output
  • a committed, test-passing state at every boundary
  • no half-applied changes to reverse-engineer later
Where it sits
  • Every session boundary; the precondition that makes review/merge (governance) tractable.

For our rollout: clean-state is only achievable if verification is fast and trustworthy — another reason a weak test base caps autonomy at L2. The principle and the prerequisite are inseparable.

D

The generator–evaluator loop & the sprint contract

agree on "done" before coding

Rather than one model both writing and judging its own work, the harness can separate generation from evaluation, and force an explicit agreement on the definition of done before code is written — a "sprint contract" that turns a vague task into a checkable target.

Requires on input
  • a scoped task with acceptance criteria (M3 handoff)
  • an evaluator signal: tests, or a second agent/human
Produces on output
  • an agreed "done" the work is checked against
  • less rework from mid-task goal drift
Where it sits
  • Task kickoff + implementation; the runtime face of verification-first (E4) and the handoff contract (M3).

For our rollout: the sprint contract is where under-specification gets caught early. If the agent can't restate a checkable "done", the task isn't ready — send it back to refinement rather than letting the loop churn.

E

Controlled write-surfaces

JSON over prose, status-only edits

What the agent may write, and in what shape, is itself a harness decision. Anthropic's pattern: keep a machine-state file (e.g. feature_list.json) that the agent may update only in a narrow way — flipping a passes field — with tests explicitly off-limits. JSON is chosen over Markdown because the model treats it as code and is less likely to "improve" it.

Requires on input
  • a durable/derived/disposable classification (M4)
  • write-permissions enforced by hooks, not trust (E5)
Produces on output
  • agent edits confined to safe fields/paths
  • protected artifacts (tests, specs) the agent can't rewrite
Where it sits
  • Cross-cutting; the concrete meeting point of the harness (E3), information architecture (M4) and governance (E5).

For our rollout: this is where the durable/derived/disposable taxonomy stops being theory. Decide per artifact who may write it and enforce it with a hook — "the agent won't touch tests" is advisory; a PreToolUse deny is deterministic.

3The pruning principle: a harness that shrinks

The most counter-intuitive practice, and the one that keeps a harness healthy. Anthropic saw it directly: moving from one model generation to the next let them remove scaffolding they previously needed — sprint decomposition, certain context resets — because the newer model no longer needed the crutch.

Operating rule

Treat every component as a hypothesis: "the model can't do X unaided, so the harness does it." Re-test that hypothesis each model upgrade. If it no longer holds, delete the component — the space of what the harness must cover shifts and shrinks, it does not only grow. A harness that only accretes is accumulating tech debt against a moving model.

This has a commercial implication for the engagement: we are not selling a fixed artifact but a maintained one, with a cadence of re-evaluation tied to model releases. It also disciplines scope — we build the minimum scaffold the current model needs, not the maximum we can imagine.

5Takeaways for our engagement

Assess (feeds E6)
  • Can a model bootstrap the repo (build/test) unaided?
  • Is there a fast verification loop to make clean-state real?
  • Are artifacts classifiable as durable/derived/disposable?
  • Do we have hook enforcement points for controlled writes?
Build (feeds E7)
  • initializer → progress-file convention → clean-state protocol
  • sprint contract at task kickoff; evaluator signal wired in
  • controlled write-surfaces (JSON state, protected tests)
  • a re-evaluation cadence to prune per model release
Three sentences to remember E3 by

The harness is the software we build and own — it supplies the state, sequencing and hard limits a stateless, unconstrained model lacks. Its patterns (initializer, progress file, clean-state, sprint contract, controlled writes) all exist to make a memoryless model behave like disciplined engineers working in shifts. Every component encodes a model weakness, so a healthy harness is pruned as the model grows — it shrinks, not just grows.
