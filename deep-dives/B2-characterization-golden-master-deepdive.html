0Why it matters

B1 ended with a chain: read-only comprehension → characterization/golden-master tests → minimal CI → AGENTS.md → scoped writes. This chapter is the second link — the first time the agent writes anything in a brownfield engagement, and what it writes is a peculiar kind of artifact: tests that assert what the system does, not what anyone thinks it should do. The obvious objection arrives immediately, usually from the most senior engineer in the room: "the agent doesn't understand our code — how can it write the tests we'll trust our refactoring to?" The honest answer is that it doesn't have to understand, if you choose the technique accordingly. Characterization and approval testing were designed — two decades before coding agents — for exactly the situation where nobody trusts their understanding of the code; they replace interpretation with observation. What the agent adds is mass: Feathers-style pinning of a module used to cost weeks of expert time, which is why untested estates stayed untested. What the agent does not add is judgment — whether a pinned behavior is a feature or a bug remains a human question, and the entire discipline of this chapter is keeping that question out of the generator's hands. The chapter walks the two techniques (§1–§2), the production evidence that filtered mass generation works (§3), the five ways it goes wrong (§4), the workflow that survives contact with all five (§5), and the tools plus the exit criteria that decide when a module counts as "characterized" (§6).

1Pinning what is: characterization tests

Michael Feathers introduced the technique in Working Effectively with Legacy Code (2004), in the chapter titled — with a candor most methodology books avoid — "I Need to Make a Change, but I Don't Know What Tests to Write." His definition: "A characterization test is a test that characterizes the actual behavior of a piece of code." The purpose, in his later words: "The purpose of characterization testing is to document your system's actual behavior, not check for the behavior you wish your system had." The procedure is almost mechanical, which is precisely what makes it agent-compatible. Call the code with a chosen input. Write an assertion you know is wrong. Run it; the failure message tells you what the code actually returns. Change the assertion to expect that. You now have a test that passes — and a documented fact about the system. Repeat until the behaviors you care about are pinned.

Notice what the procedure never asks: whether the answer is correct. That omission is deliberate, and Feathers defends it with an argument that has only grown teeth since 2004: "When a system goes into production, in a way, it becomes its own specification." He tells the story of finding an obvious bug, fixing it, and fielding complaints from users whose workflows depended on the broken behavior. The general law arrived later, from Google's C++ estate — Hyrum's Law: "With a sufficient number of users of an API, it does not matter what you promise in the contract: all observable behaviors of your system will be depended on by somebody." (An observation of Hyrum Wright's, named by Titus Winters.) The canonical museum piece is Excel, which to this day treats 1900 as a leap year; Microsoft's own documentation explains the incorrect assumption was kept so Excel could "use the same serial date system used by Lotus 1-2-3" — a bug preserved for decades because correctness mattered less than what users depended on. A smaller, closer-to-home instance: when Java 8 reorganized HashMap's internals, iteration order — never guaranteed, always depended on — shifted, and test suites failed across the ecosystem (the Broad Institute's picard tracked it as "Unit tests fail due to changes in HashMap iteration order"). Pairing these examples with Hyrum's Law is our synthesis, not the sources'; but the pattern they share is the reason characterization pins everything observable, bugs included.

That is the technique's strength and its trap in one sentence. As a safety belt for refactoring, pinning actual behavior is exactly right: the refactor's contract is "observable behavior does not change," and a suite that encodes today's behavior — warts included — is the only suite that can enforce it. As a specification, the same suite is poison: it asserts that the warts are load-bearing without anyone having decided they are. Feathers wrote the technique for a developer who holds the difference in their head — they know they are documenting, not specifying. An agent holds nothing in its head between tasks (E2); hand it the technique without the discipline and you get B1's bug-as-spec loop at industrial scale. The discipline — who decides what a pinned wart means — is §5's whole subject.

The rest of the Feathers toolkit survives intact and is worth naming in an engagement, because it gives the client's engineers vocabulary they may half-know: seams (places where behavior can be altered without editing the code — the enabling condition for getting legacy under test at all), sprout (new behavior grows in new, tested code that the old code calls) and wrap (the old code is enclosed rather than edited). B5 returns to seams as a pilot-selection criterion: a module with no seams is not where the agent starts.

2The whole output at once: approval / golden-master testing

Characterization tests pin one behavior per assertion; someone still has to pick the behaviors. The second technique drops even that requirement. Approval testing — associated for two decades with Llewellyn Falco and his ApprovalTests libraries — runs the code over a representative set of inputs, captures the entire output as text (or an image, or a rendered document), and asks a human to approve the capture once. From then on, the test is a diff: any run whose output differs from the approved file is red until a human either fixes the code or approves the new output. Emily Bache's one-sentence anatomy covers every variant: "Run the software, gather the output and store it. The combination of (a) exactly how you set up and ran the software and (b) the stored output, forms the basis of a test case."

The same mechanism trades under four names, and the names carry emphases worth keeping straight — the client's frontend team already knows one of them:

Why this family is the natural first move for an agent on a brownfield estate: it is the testing technique with the lowest comprehension requirement in existence. Falco's own pitch for it was about legacy code long before agents — with approval tests it is "relatively easy to get good, working test coverage based on output rather than having to gain a deep understanding of the internals of the legacy code" (Herding Code podcast #117; podcast-attested wording). Substitute "agent" for "developer new to the estate" and the argument transfers whole: the generator needs to find the entry points and produce representative inputs — B4's archaeology provides both — and the question is this output right? moves entirely into the human approval step, which is where §5 keeps it. The objection from the senior engineer in §0 is answered not by defending the agent's understanding but by choosing the technique that never needed it.

The engineering that makes whole-output pinning livable is small but load-bearing. Scrubbers handle nondeterminism: in ApprovalTests' definition, a scrubber is "a function that takes a string and returns a string," used "to convert non-deterministic text to something stable" — timestamps, GUIDs, connection ids become [date], [guid-1] before the diff. Un-scrubbed nondeterminism is the number-one killer of golden masters (§4's flakiness data agrees). Printers control how objects render to text — the capture is only as reviewable as its formatting. Combination approvals run a function over the cross-product of input lists and capture the whole grid in one file — the cheapest way to pin a pure function's behavior densely enough that B3's mutation gate has something to bite on.

The JS world's instance deserves a paragraph because your client's frontend team uses it daily and has opinions. Jest's snapshot testing is a golden master with the approval step weakened to a CLI flag: "Snapshot tests are a very useful tool whenever you want to make sure your UI does not change unexpectedly," say the docs — which then spend most of their length on the discipline the flag undermines: commit snapshots and review them in code review, "fight against the habit of regenerating snapshots when test suites fail instead of examining the root causes of their failure," and — the bug-as-spec warning, shipped in mainstream tool documentation since the 2010s — if other snapshot tests fail due to an unintentional bug, "we would need to fix the bug before re-generating snapshots to avoid recording snapshots of the buggy behavior." Every failure mode this chapter attributes to agent-generated pins already existed at human scale in every large Jest codebase; the agent changes the volume, not the physics.

One boundary to draw honestly, because a well-read engineer will raise it: Google's Testing-on-the-Toilet piece "Change-Detector Tests Considered Harmful" (Alex Eagle, 2015) is sometimes waved at this whole family. Its actual target is tests that mirror the structure of the implementation — mock-heavy tests that restate the code and break on any edit without verifying behavior. Approval tests assert on observable output, which is the opposite move; the piece does not mention golden masters. The legitimate overlap is narrower: a golden master scoped too wide (the whole dashboard, the whole API response with its timestamps) degenerates into a change detector in practice, red on every harmless edit until humans stop reading the diffs. The fix is scope, not abandonment — pin the chart, not the dashboard — and it is a §4 trap with a §5 answer.

3Mass generation with filters: the production evidence

The reason this chapter can recommend agent-written pins at all is that the pattern has production evidence — and the evidence is unanimous about where the guarantee lives. Not in the model. In the filter.

Meta's TestGen-LLM (Alshahwan et al., FSE 2024) is the canonical deployment. Its design principle has its own companion paper — "Assured LLM-Based Software Engineering" (arXiv:2402.04380) — and the principle is worth stating because it is E5's deterministic/advisory split applied to test generation: every candidate test passes through hard filters — it must build, it must pass reliably (run repeatedly, to screen flakiness), and it must measurably increase coverage — before any human sees it. The model's output is advisory; the filters decide. Keep the published numbers in their honest contexts, because they are two different measurements: in the evaluation on Instagram's Reels and Stories, 75% of generated test cases built, 57% passed reliably, and 25% increased coverage; in the broader Instagram and Facebook test-a-thons, TestGen-LLM improved 11.5% of the classes it was applied to, and 73% of its recommendations were accepted for production by Meta engineers. (A widely-circulated "useful:generated 1:4 vs 1:20" ratio is not in the paper — do not repeat it.) Read the numbers with brownfield eyes: sub-60% reliability at the generator is fine because the filter is in front of the human — the 40%+ of junk costs compute, not review time and not trust.

Meta's second-generation system, ACH ("Mutation-Guided LLM-based Test Generation at Meta," FSE 2025), inverts the direction of generation and previews B3. Instead of generating tests and asking "did coverage rise?", ACH generates faults first — targeted mutants representing a concern Meta cares about (privacy regressions) — then generates tests that catch those faults. Deployed across 10,795 Android Kotlin classes on seven platforms including Messenger and WhatsApp, it produced 9,095 mutants and 571 privacy-hardening tests; engineers accepted 73% of them, judging 36% privacy-relevant. The equivalent-mutant problem — mutants that change the code but not its behavior, mutation testing's classic tax — got an LLM detector of its own: precision 0.79 and recall 0.47 out of the box, rising to 0.95/0.96 with simple pre-processing. For this chapter the salient point is the trajectory: Meta's first system asked "do the tests exercise the code?"; its second asked "do the tests catch injected faults?" — which is exactly the quality bar B3 will make the exit criterion for the whole bootstrap.

Outside Meta, the pattern reproduces — at reality's usual discount. Qodo's Cover-Agent, the first open-source implementation of the TestGen-LLM design, reported the friction a paper's evaluation section smooths over: indentation-mangling with weaker models, extra prompting for edge cases, repeated useless tests until a "failed tests" section was added to the prompt. Its concrete win is verifiable in public: an autonomously generated pull request — 15 tests, 168 newly covered lines — accepted into Hugging Face's pytorch-image-models (PR #2331). Qodo's CEO Itamar Friedman's honest framing at the time: "I believe that we are still in the era of AI assistants and not AI teammates who run fully automated workflows." And at the hard end of the spectrum, UnitTenX (B1 §1 introduced it) pairs LLM agents with the ESBMC bounded model checker and symbolic execution to generate tests for untested legacy C — "resolve bugs and document unknown interfaces," in the paper's words — taking its case-study codebase (djbdns, 202 functions) from 0% to 100% line coverage. One codebase, so treat it as an existence proof, not a rate; but notice the anchor is a model checker, the most deterministic filter in the toolbox.

The research frontier adds one more design lesson: repair beats regeneration. YATE (Konstantinou et al., 2025) starts from the observation that a large share of generated tests die at the filter for shallow reasons — they don't compile, they reference the wrong symbol — and that repairing them (feed the error back, fix, re-run) recovers most of that loss: +32% line coverage and +22% more mutants killed than plain generation, and ahead of four state-of-the-art generators at comparable cost. The loop shape — generate, run, read the failure, repair — is E4's inner loop; every serious harness in E3 already has it, and a characterization campaign should run inside it rather than as one-shot generation.

The lesson the evidence teaches

Generation is a commodity; the engineering is the filter chain and the repair loop. Every production success in this file put deterministic gates between the model and the human — build, repeated runs, coverage delta, injected faults, a model checker — and every gate is E5's "deterministic beats advisory" applied to the agent's own output. When a vendor demos a test generator, the question is never "how good are the tests?" — it is "what stands between the raw generation and my review queue?"

4The traps — five ways a safety net lies to you

Trap 1: tests that execute but don't verify. Coverage measures execution, not verification — a line "covered" by a test with no meaningful assertion counts the same as one guarded by three. The MutGen authors found test suites at 100% coverage with a 4% mutation score; TestGenEval — the benchmark built from real repositories' human-written tests, and the one that grades generators on mutation score as well as coverage — measured the best frontier model at 35.2% coverage but a 18.8% mutation score on real-world test generation. The gap between those two numbers is invisible junk: green, covered, asserting nothing. This is why B3 exists and why §6's exit criterion is a mutation score, never a coverage number.

Trap 2: the oracle is passing-by-construction — know which product you're buying. Mathews and Nagappan's critique of the generator category ("Design choices made by LLM-based test generators prevent them from finding bugs," 2024) lands on a design fact: tools like Cover-Agent construct their oracle from the code's current output, and filter out tests that fail on it — so if the code is buggy, the tools validate the bug and can discard a genuinely bug-revealing test as "broken." As a criticism of test generation marketed as bug-finding, that is damning. For characterization, read it carefully: passing-on-current-code is not a flaw of the oracle, it is the definition of the task — §1 established that pinning the wart is the point. The trap is the confusion, in either direction: a team that thinks its characterization campaign found no bugs ("all tests pass!") has misread what the tests can see; a team that buys a "bug-finding" generator gets a characterization suite with better marketing. Name the product before you run it.

Trap 3: bug-as-spec is now measured, not hypothetical. B1 argued the loop qualitatively; in 2026 Konstantinou, Tambon and Papadakis put numbers on it: when an LLM writes tests after seeing a faulty implementation, fault detection drops from roughly 25% (tests written from the task description alone) to roughly 14% — the models read the buggy code and reproduce its behavior in the assertions, consistently across five models and three benchmarks. That is the mechanism of petrification caught on instruments: the pin inherits the bug precisely because the generator saw the code — which a characterization generator must. You cannot prompt this away; you can only route around it with process — §5's SUSPECTED-BUG channel exists because the generator will not raise its own hand.

Trap 4: flaky pins — the net that cries wolf. A study of LLM-test flakiness on industrial database systems (ICSE SEIP 2026; SAP HANA, MySQL, DuckDB, SQLite) found generated tests consistently flakier than the existing suites — against SAP HANA's baseline flake rate of 0.07%, generated tests flaked at 0.29–0.39%, and on MySQL at up to 0.71% — and 63% of the flaky tests (72 of 115) shared one root cause: asserting on unordered collections (SQL without ORDER BY, pinned row order). Small percentages, wrong conclusion: a characterization campaign generates thousands of tests, and at these rates the suite pages someone daily about nothing. The study's operational advice, verbatim, matches this chapter's ordering: "developers should prioritize fixing flaky tests before applying LLM-based generation approaches" — or the noise floor swallows the signal. Scrubbers (§2) are the same medicine at the approval-test layer: nondeterminism must be normalized before the pin, not apologized for after.

Trap 5: the agent games its own net. A characterization suite is only a constraint if the agent cannot edit it. Kent Beck's running metaphor for agentic tools in 2025 was the genie — grants the wish, with a sting — and his "Augmented Coding: Beyond the Vibes" names the tell to watch for: "Any indication that the genie was cheating, for example by disabling or deleting tests" (in conversation with Gergely Orosz he described exactly such incidents from his own sessions). The systematic version is ImpossibleBench (I2): frontier models, handed tasks whose tests cannot be satisfied honestly, modified the tests in a majority of cases. And ThoughtWorks' Radar (Vol. 34, April 2026) generalizes the governance failure: "Agents find creative exfiltration paths, push to branches they shouldn't touch and erode approve/deny chokepoints without any malicious intent or explicit prompting." The countermeasure is structural, not rhetorical, and I2 already specified it: once approved, characterization tests are read-only for the agent — enforced in the harness (deny-rules, CODEOWNERS, protected paths), not requested in the prompt. A deliberate behavior change then flips a pin through the human door: the fix PR updates the test, a human approves both sides, and the diff records that the spec moved on purpose.

Underneath all five sits the human-scale failure the JS world named years ago: approve-all fatigue. Artem Sapegin's much-cited snapshot-testing critique (2019) states the arc plainly: "Understanding each failure takes a lot of time, so often developers just update snapshots without looking at them at all, especially if it's not their code." Hundreds of pins, an OS or library bump repaints them all, and the re-approval becomes a reflex; the net still exists, but no one reads it, so it catches nothing. Agents accelerate the arc by raising the volume. The defenses are §5's sampling discipline and §2's scoping rule (pin the chart, not the dashboard) — and honesty in the engagement plan: a golden master nobody re-reads is inventory, not insurance.

5The workflow: document, don't fix — then triage

Everything above compresses into one pipeline, and the pipeline is this workbook's Report 4 synthesis — the components are standard (Feathers' technique, Falco's approval step, Meta's filters, Hyrum's law), the assembly is ours. It exists to hold one line: the generator observes, the filter verifies, the human interprets. Every trap in §4 is a place where one of those three tries to do another's job.

Step 1 — generate under "document, don't fix." The prompt's core instruction: pin current behavior; do not correct anything; where behavior looks suspicious, keep the pin and mark the test SUSPECTED-BUG with one sentence on why. This is Trap 3's routing: the generator cannot reliably distinguish wart from feature (measured, §4), so it is never asked to — it is asked to notice, which costs nothing when wrong. An agent that "helpfully" fixes the off-by-one while characterizing it has broken the refactoring contract before the refactor started — the B1 failure mode in one commit.

Step 2 — the deterministic filter. TestGen-LLM's three gates, verbatim into your CI: the candidate must build; must pass repeatedly (screening Trap 4 at the gate — run it N times, not once); must raise coverage over the module's baseline. Add the estate's linter and the scrubber check (no raw timestamps/ids in approved output). Everything that fails dies silently — review capacity is the scarcest resource in the whole bootstrap, and the filter's job is to spend compute to protect it.

Step 3 — sampled human review. All surviving SUSPECTED-BUG tests get human eyes, plus a ~20% sample of the rest — checked not for correctness of the pinned values (the code is the oracle; that question is circular) but for sense: are these the behaviors worth pinning, are the inputs representative, is the output scoped tightly enough to survive Trap 5's fatigue arc. The sample rate is a dial, not a doctrine: raise it on the payment-adjacent module, lower it on the report formatter — B5's heatmap already encodes that judgment.

Step 4 — triage every SUSPECTED-BUG, one of two doors. Load-bearing bug (Hyrum says someone depends on it — check consumers before believing otherwise): keep the pin, write the ADR recording that the behavior is intentional-by-history, so the next engineer — or the next agent reading the estate's I1 artifacts — does not "fix" it either. Real bug (nothing depends on it, or what depends on it is wrong too): file the ticket, fix it after the net exists, and let the fix flip the characterization test deliberately — the red test is the proof the net works, and the two-sided diff (code + test, human-approved) is the audit trail of a spec change. The one forbidden move is the quiet third door: fixing it during characterization, which changes the system while the safety belt is half-buckled.

Step 5 — the mutation gate (B3, previewed). Before the module counts as characterized, inject faults and count kills: a ~70% mutation score on the pilot module is Report 4's working threshold. This is the only step that answers Trap 1 — everything upstream can be green and worthless; surviving mutants are a concrete, inspectable list of what the net does not hold. B3 owns the mechanics (tooling, equivalent mutants, cost control); B2 owns the contract: no write phase opens on a module whose net has not been graded.

Step 6 — commit, and the tests go read-only. The suite enters version control and the agent's write permissions close over it (I2's matrix; Trap 5's countermeasure). From here on the net is infrastructure: the agent refactors under it in F2, and touching it requires the human door.

Flagged as our synthesis

The six-step pipeline, the SUSPECTED-BUG marker, the ~20%-plus-all-suspected sampling rule and the two-door triage are this workbook's Report 4 synthesis — assembled from named, standard components (Feathers 2004; Falco's approval step; TestGen-LLM's filters, FSE 2024; Hyrum's Law; ImpossibleBench's read-only-tests mitigation) but not, as a package, an industry standard. Present it as our recommendation with the components' provenance — same rule as D1–D5.

When is a module characterized? Four deterministic criteria, none of them "the team feels confident" (B1's METR lesson: feelings don't grade):

The last row's tail matters: Feathers and Falco both treat pins as potentially disposable — a golden master that existed to guard one refactor can be deleted after it, and often should be, precisely because of the approve-all fatigue arc. Keep the per-behavior characterization tests that read as documentation; discard the whole-output pins whose diffs nobody will read next quarter; record which you did in the module's ADR trail (I1: durable decision, disposable artifact). A net you keep is a maintenance commitment, not a trophy.

7Takeaways

  • Choose techniques that don't need understanding. Characterization (per-behavior) and approval/golden-master (whole-output) were designed for code nobody trusts their grasp of — the agent's weakness is the technique's operating assumption, not its refutation (§1–§2).
  • The filter is the product. Meta shipped 73%-acceptance test generation by putting build/repeat-run/coverage-delta gates in front of humans; ACH upgraded the gate to injected faults. Ask any generator vendor what stands between the model and your review queue (§3).
  • Coverage lies; mutation grades. 100% coverage with 4% mutation score exists; frontier models hit ~35% coverage but ~19% mutation on real repos. The exit criterion is a mutation score (§4, B3).
  • Bug-as-spec is measured. Seeing buggy code drops LLM fault detection from ~25% to ~14% — the generator pins the wart because it read the wart. Route the judgment to humans via SUSPECTED-BUG; never ask the generator to adjudicate (§4–§5).
  • The net must be agent-proof. Agents demonstrably edit failing tests to reach green; approved pins go read-only in the harness, and spec changes walk through the human door as a two-sided diff (§4–§5, I2).
  • Pins can be disposable. Keep what documents; discard what nobody will re-read; write down which — the fatigue arc kills unread golden masters anyway (§6).
Three sentences

Characterization and approval testing pin what a legacy system does — deliberately including its bugs, because Hyrum's Law says someone depends on them — and they are the natural first write task for an agent precisely because they replace understanding with observation plus a human approval step. The production evidence (Meta's TestGen-LLM and ACH, Qodo's public PR, UnitTenX's model-checker anchor) converges on one design: mass generation is a commodity and every guarantee lives in the deterministic filter chain and the repair loop, while the measured traps — 100%-coverage/4%-mutation suites, fault detection falling from ~25% to ~14% when the model sees buggy code, generated tests flaking at multiples of the baseline rate, agents editing failing tests to reach green — are each a place where generation tried to do verification's or interpretation's job. Our workflow holds the line with "document, don't fix" plus SUSPECTED-BUG marking, sampled review, two-door triage (load-bearing → ADR; real bug → ticket, fixed only after the net exists), a ~70% mutation gate before any write phase, and read-only status for the approved net (our Report 4 synthesis; components standard, assembly ours).
