0Why this runbook exists

PB2 solves the bootstrap paradox. An agent working on legacy code needs a verification loop — tests, CI, machine-readable red/green — that legacy code by definition does not have; and the agent itself is the cheapest tool for building that loop. Left unanchored, the circle closes into bug-as-spec: the generator reads the faulty code, reproduces the fault in its own assertions, and the net certifies the bug as intended behavior. That failure mode is measured, not hypothetical — fault detection drops from roughly 25% to roughly 14% when the LLM sees the faulty implementation before writing tests (Konstantinou, Tambon & Papadakis, 2026). PB2 is the forced ordering that keeps the circle open (B1 Safe ordering).

There are two ways in. Posture 1 — PB1 said GO (score ≥ 17, 4/4 gates): bootstrap the pilot module chosen from PB1's shortlist, with the PB1 report's §2 numbers — suite runtime, pass rate, flaky list — as the bootstrap baseline. Posture 2 — PB1 scored below 10: the client enters at L1 only and PB2 is the engagement deliverable — building the verification base is the work, not a prerequisite for it. Either way PB2 produces the same artifact: one characterized pilot module with a mutation-audited safety net and a CI loop the agent can read — the evidence that carries the module from F0 to the F2 boundary.

What this is not

A characterization net never "finds bugs" — it is passing-by-construction, because the current code is the oracle (B2). Do not sell it or judge it as a bug-finder; its job is to make every future change visible. Bugs discovered along the way exit through the SUSPECTED-BUG door in Phase B, never through a quiet fix.

The theory this runbook distils lives in three deep dives: B1 The bootstrap paradox for the ordering, B2 Characterization & golden master for the net, and B3 The mutation-testing gate for how the net is graded.

1How to run it

Run PB2 with one person plus the agent, scoped to a single pilot module, agent write access limited to test and CI files, and the PB1 report at hand. The steps are sequential by design — each is a precondition of the next, and you never verify a step with a later step; that is the whole point of B1's ordering. Size each step at two days or less: if a step will not fit, the module is too big — shrink the target, not the step. As in PB1, every step is written as ČO (the action), PREČO (the theory link), and DONE (the exit criterion).

In the workbook version of this runbook, progress is saved locally (same store as topic progress). This standalone copy does not persist checkbox state.

2Phase A — Read-only comprehension of the pilot module

PB1 shortlisted pilot candidates at codebase level; Phase A commits to one of them and builds module-level comprehension against it. This is the F0 stage of B1's chain scoped down to one module: zero write risk, and the output is the comprehension layer every later phase reads from (B4 Agent archaeology).

Pick the single module PB2 will bootstrap and write down why: seams the tests can attach to, low external risk, a reachable owner, a size that fits the two-day step budget. Why: the heatmap logic picks low-risk, high-learning targets — a wrong first module burns trust a right one would have earned (B5 Strangler heatmap, B1 Safe ordering).

DONE = one named module plus a written rationale (seams, size, risk, owner) in report §1.

Let the agent map the module read-only: entry points, public surface, side effects, existing tests and their status, the current coverage number if one exists. Every claim is confidence-labeled and a module owner spot-checks a sample — extraction, not decree (B4 Agent archaeology, B1). This note is what Phase B's generation prompt scopes itself to.

DONE = architecture note exists, claims confidence-labeled, owner spot-check recorded; before-numbers copied into report §2.
The four wrong orderings (B1) — check you are not in one

WO1 write-first pilot: skipping the chain entirely; detected by change-failure rate and rework climbing on agent-touched modules. WO2 coverage-not-quality tests: "run the generator until coverage is green" — corrupts the net; detected by mutation score (Phase C), never by coverage. WO3 AGENTS.md by decree: writing the context file from aspiration before the evidence exists — agent-written context files measurably hurt (−2% success, +23% cost in the ETH Zurich study); extraction comes in PB3, after this runbook. WO4 Phase 0 forever: read-only comprehension that never exits; detected by the calendar — if A2's criteria have been met for weeks and no characterization target is picked, you are stalling, not derisking.

3Phase B — The characterization safety net

Phase B is B2's six-step workflow compressed to four steps: generate under a constraint, filter deterministically, review by sampling, commit read-only (the fifth step, the mutation gate, is Phase C). Two techniques cover almost every module: characterization tests (Feathers 2004 — call the code, observe the actual result, assert that result; per-behavior granularity, reads as documentation) and approval / golden-master tests (Falco — capture entire outputs to files, approve once, diff forever; needs scrubbers for timestamps and GUIDs, and works with zero understanding of internals, which is why it is the natural first agent-write task on brownfield). Approval, golden master, snapshot, characterization: one mechanism, four emphases — if the client's frontend already uses Jest or Vitest snapshots, reuse that vocabulary (B2 Characterization & golden master).

Run the agent's generation pass with the prompt constraint below: pin what the code does today, never what it should do; mark anything suspicious SUSPECTED-BUG with one sentence why; never silently fix. Why: the generator is never asked to adjudicate correctness — pinning warts is deliberate (Hyrum's Law: with enough users, every observable behavior of the system will be depended on by somebody), and letting the generator "correct" what it reads closes the bug-as-spec loop §0 warned about (B2).

DONE = candidate test set generated; zero silent fixes on inspection; every SUSPECTED-BUG marker carries a one-sentence rationale.
# Characterization generation prompt — skeleton (adapt per module)
Role: you are generating CHARACTERIZATION tests for <module>.
Rules — document, don't fix:
1. Pin what the code DOES today. Never assert what it should do.
2. Never modify production code. Never "correct" an odd result.
3. If a pinned behavior looks wrong, keep the pin AND mark it:
   // SUSPECTED-BUG: <one sentence — why this looks wrong>
   One sentence, always; no fix, no workaround, no TODO.
4. Nondeterminism (time, GUIDs, ordering): scrub before pinning
   ([date], [guid-1], sort before compare) — never pin raw values.
5. One behavior per test; name each test after the behavior it pins.
Scope: <entry points / public surface from the Phase A note>
Out of scope: <anything outside the module boundary>

Wire the three TestGen-LLM gates before any human reads a candidate: it must build, it must pass N repeated runs (N=5 screens flakiness), and it must raise coverage over the baseline; failures are discarded unread. Why: review capacity is the scarcest resource in the engagement — the filter spends compute to protect it. Meta's industrial deployment (TestGen-LLM, FSE 2024) reports 75% of candidates built, 57% passed reliably and 25% raised coverage on its Reels/Stories evaluation, and 73% of surfaced recommendations accepted for production across the broader test-a-thons — the filter is what made the human side of that loop bearable (B2).

DONE = filter runs in CI; only surviving candidates reach review; discard counts logged in report §3.
# Candidate-test filter — CI skeleton (GitHub Actions flavour; adapt to your CI)
# The three TestGen-LLM gates: builds, passes N repeats, adds coverage.
name: characterization-filter
on: pull_request            # candidate tests arrive as agent PRs
jobs:
  gate1-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: <build command>                 # gate 1: must build
  gate2-stability:
    needs: gate1-build
    runs-on: ubuntu-latest
    strategy: { matrix: { run: [1, 2, 3, 4, 5] } }   # gate 2: N=5 repeats
    steps:
      - uses: actions/checkout@v4
      - run: <test command, candidate tests only>    # any red run = discard
  gate3-coverage-delta:
    needs: gate2-stability
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: <coverage command>
      - run: <compare against baseline; fail unless delta > 0>   # gate 3
# Discarded candidates are logged, never reviewed — the filter protects
# human review capacity; judging test QUALITY is Phase C's job.

Humans review 100% of SUSPECTED-BUG items and a ~20% sample of the rest — for sense (are these the behaviors worth pinning), never for "is the pinned value correct": the code is the oracle, so that question is circular. Every SUSPECTED-BUG exits through one of two doors: load-bearing behavior (Hyrum's Law) → keep the pin, write an ADR (skeleton in §6); real bug → separate ticket, fixed after the net exists, the fix then flips the pin on purpose. The third door — fixing quietly during characterization — is forbidden. Scope pins narrowly ("pin the chart, not the dashboard") and record keep-or-discard per pin: approve-all fatigue turns pins into inventory, not insurance (B2).

DONE = zero untriaged SUSPECTED-BUG markers; each one → ADR or ticket, listed in report §3.

Commit the approved tests, then enforce read-only in the harness — deny-rules, CODEOWNERS, protected paths (snippet below) — never just a request in the prompt. Why: agents demonstrably edit failing tests to reach green — in ImpossibleBench's adversarial setting GPT-5 cheats 54.0% of the time when tests conflict with the spec, and hiding the tests drops cheating to near zero. Visibility of the oracle is the attack surface, so the defence must be structural (B2, I2 Lifecycle & write permissions).

DONE = a deliberate agent attempt to edit a pinned test is blocked by the harness; verified once, logged in report §6.
# Read-only enforcement for approved characterization tests — two layers
# Layer 1: repository — CODEOWNERS (any change requires named review)
/tests/characterization/  @module-owner @qa-lead

# Layer 2: agent harness — deny rule (enforcement, not a prompt request)
# Claude Code settings.json shown; most harnesses have an equivalent:
{
  "permissions": {
    "deny": [
      "Edit(tests/characterization/**)",
      "Write(tests/characterization/**)"
    ]
  }
}

# blocked by the harness and visible in the session log.

4Phase C — Mutation-testing the net

Coverage cannot grade the net you just built. Suites with 100% line coverage and a 4% mutation score exist (MUTGEN, arXiv:2506.02954); the best frontier model measured on real-world test generation hit 35.2% coverage but only 18.8% mutation score (TestGenEval); and the ICSE 2014 study of 31,000 suites concluded that coverage "should not be used as a quality target because it is not a good indicator of test suite effectiveness" (Inozemtseva & Holmes) — the sentence to quote when a coverage KPI is proposed. Mutation testing is the exam: inject small faults, count how many the net kills. Its validity is empirical, not folklore — mutant detection predicts real-fault detection for 73% of 357 real faults, independently of coverage (Just et al., FSE 2014) (B3 The mutation-testing gate).

Before any full run, do the cheapest audit: extreme mutation (Descartes-style) — one mutant per method, the whole method body removed. It surfaces pseudo-tested methods (covered, verified by nothing) immediately. Why: pseudo-tested code was found in every one of the 21 mature Java projects studied — this is the fastest possible demonstration that "covered" lies, at a cost of one mutant per method, cheap enough for the whole module on day one (B3).

DONE = pseudo-tested inventory produced, reviewed with the module owner, recorded in report §4.

Quarantine or fix flaky tests at module scope before the gate run. Why: mutation scores vary about ±4 percentage points run-to-run on flaky suites — a ~70% verdict can flip on noise alone. Generated tests flake at 0.29–0.71% versus a 0.07% human baseline, and 63% of those flakes share one root cause: asserting on unordered collections (ICSE SEIP 2026, whose advice is verbatim to fix flaky tests before applying LLM generation). Then demand N consecutive fully green runs (B3, B2 — the flaky-pins trap).

DONE = quarantine list recorded in report §5; N consecutive green runs on the module, zero flakes.

One full mutation run on the pilot module only — never the codebase — using fresh, held-out mutants the generator has never seen: the practice sheet is not the exam, and the visible-versus-held-out gap is measured, growing with codebase size. One tool, one config, equivalent-mutant filtering on: TCE mechanically removes 7.4% / 5.7% of C/Java mutants, and it matters because ~45% of undetected mutants were equivalent in one study, at ~15 minutes each to adjudicate by hand. Score = killed / non-equivalent mutants. The bar: ~70% is our working heuristic (Report 4), not an industry standard — record bar, tool, config and mutant source in the ADR (§6) (B3).

DONE = score recorded in report §4 with the gate ADR; at or above the bar → input to the D2 gate; below → survivors exported as the agent's next task list.

After the gate, keep Mode 1 running forever: diff-scoped mutation on changed, covered lines only, at most one mutant per line — Google's design, validated on ~70,000 diffs and 1.1M mutants, with arid-node suppression lifting the usefulness of surfaced mutants from ~20% to 80%. Survivors become review comments and agent tasks; the job never blocks a merge. Why: feeding live survivors back to the generator is the strongest measured lever in this runbook — a mutation score stalled at 53% for four iterations jumped to 89.5% once survivors entered the prompt (MUTGEN) (B3).

DONE = diff-scoped job running in CI, non-blocking; survivors visible in review as tasks.
# Diff-scoped mutation — standing advisory job (Mode 1; generic YAML, adapt)
name: mutation-diff
on: pull_request
jobs:
  mutate-diff:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }         # need the merge base for the diff
      - run: <diff-scoped mutation command>
      # Per-ecosystem one-liners (versions = mid-2026, re-verify):
      # JVM:   mvn org.pitest:pitest-maven:scmMutationCoverage
      # JS/TS: npx stryker run --incremental
      # Rust:  git diff origin/main.. > git.diff && cargo mutants --in-diff git.diff
      - run: <post survivors as PR review comments / agent tasks>
# Advisory by design: this job NEVER blocks the merge. Filter the survivor
# stream (TCE / arid-node-style suppression) so reviewers see a credible
# list, not equivalent-mutant noise.
What never to do with the gate (B3)

No org-wide big-bang mutation CI — Google itself calls a codebase-wide score "infeasably expensive"; start at module scope, expand by the heatmap. No mutation in the agent's inner loop — it belongs at review and phase boundaries. No gating before the flake quarantine — ±4pp noise flips verdicts. No comparing scores across tools — operators differ; compare only within one tool and config. No score-as-KPI dashboard — Goodhart applies; it is a threshold at a boundary, not a graph to optimize. No chasing 100% — the last decile is mostly equivalent-mutant adjudication. And keep the survivor stream credible: unfiltered equivalent-mutant noise trains reviewers to bulk-dismiss (C3/C4's suppression and TCE).

5Phase D — Minimal CI & the F0→F2 gate

Confirm (or add) the minimal CI loop on the pilot module — build, lint, characterization suite — then prove it the way PB1's step A2 did at codebase level: insert a deliberately broken commit and watch it go red end-to-end, with a machine-readable failure the agent can consume. Why: CI is the precondition both AGENTS.md and scoped writes stand on in B1's chain, and the red/green signal is what the agent converges against (B1, E4 Verification-first). Keep it fast — as practitioner reasoning, not a cited statistic: an agent iterates many times per session, so CI wall-clock multiplies into every loop; no independent study pins a number on this, and the "<10-minute feedback" figures circulating in vendor blogs are not attributable to DORA.

DONE = broken commit demonstrably red end-to-end; failure artifact machine-readable; linked in report §7.

Convene the gate: walk the exit-criteria table in §7, fill the module bootstrap report (§6), and record go / no-go with date, criteria and signatories, with STOP criteria armed for the coming write phase. Why: you advance on evidence, never on calendar pressure or felt progress — the METR RCT found experienced developers 19% slower with AI while forecasting 24% faster and still believing 20% faster afterwards, which is exactly why every criterion in this gate is deterministic and none is self-report (B1, B6 Gates & STOP criteria).

DONE = report delivered; go / no-go recorded with date, criteria and signatories.

7Exit gate & takeaways

The module counts as bootstrapped when every row below passes its deterministic check — the criteria come straight from B1 §6 and B2 §6, and none of them is a self-report.

Exit criterionDeterministic check
Net catches changeinject a behavior change by hand; at least one test goes red
Net catches faultsmutation score at or above the ADR-recorded bar (~70% working heuristic) on held-out, non-equivalent mutants
Warts adjudicatedzero untriaged SUSPECTED-BUG markers; each one → ADR or ticket

Exit routing

If GO → proceed to PB3 Harness for this module's write scope: the F2 boundary opens exactly as wide as the net that guards it. If NO-GO → the survivors are the agent's next task list and the module stays at F1 — a working state with a scoped backlog, not a failure; rerun the gate when the list is burned down.

Three sentences to remember PB2 by

Legacy code cannot verify the agent, and the agent is the cheapest way to build the verification — the forced ordering (read, pin, audit, prove CI) is what keeps that circle from closing into bug-as-spec. The net is graded by mutation score on held-out mutants, never by coverage — and ~70% is our recorded decision in an ADR, not an industry standard. The agent never adjudicates correctness and never touches an approved test: SUSPECTED-BUG routes judgment to humans, and read-only lives in the harness, not in the prompt.

Sources and caveats for the empirical claims. Strong independent evidence: Just et al. FSE 2014 (73% coupling on 357 real faults), Inozemtseva & Holmes ICSE 2014 (31,000 suites), the Google mutation trilogy (~70,000 diffs, 1.1M mutants; diff-scoped, no absolute codebase score), MUTGEN arXiv:2506.02954 (the 100%-coverage/4%-score suite; the 53%→89.5% survivor-feedback gain), TestGenEval (35.2% coverage / 18.8% mutation), the bug-as-spec measurement (Konstantinou, Tambon & Papadakis 2026), the flakiness study (ICSE SEIP 2026), ImpossibleBench, the METR RCT (arXiv:2507.09089), DORA 2024's −7.2% delivery stability (the 2025 report flips the throughput sign positive while instability persists), and the replication finding that test code injects future defects at a rate comparable to production code (arXiv:2603.15935). Industrial self-report: Meta TestGen-LLM (FSE 2024).
