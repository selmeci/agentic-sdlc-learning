0Why this split organizes the whole module

S1 set the posture (prompt injection is a permanent risk category) and S2 gave it geometry (the trifecta, the Rule of Two). S3 supplies the test that decides whether any given control counts at all. Every safeguard in an agentic setup lives in exactly one of two places: it is either advisory — text the model reads and interprets, like a system prompt, a CLAUDE.md, an AGENTS.md, a .cursor/rules file — or deterministic — a mechanism the harness executes regardless of what the model decides: a permission deny rule, a hook, a sandbox, an egress allowlist, a CI gate. Under a successful injection the advisory layer is negotiating with the attacker; the deterministic layer is not. That is why this chapter, and not any single incident, is the organizing principle of M7 — and the same split the framework already uses for engineering governance (E5) and design gates (D5), here stated as a security law.

The principle is no longer ours alone — 2025–26 gave it canonical wording from three directions. OWASP's LLM Top 10 (LLM07:2025, System Prompt Leakage) says it in the chapter's own vocabulary: "Critical controls such as privilege separation, authorization bounds checks, and similar must not be delegated to the LLM, either through the system prompt or otherwise. These controls need to occur in a deterministic, auditable manner, and LLMs are not (currently) conducive to this." Anthropic's containment retrospective (25 May 2026 — vendor, but an engineering post-mortem rather than marketing) compresses it to one line: "The deterministic boundary is what gets hit when everything probabilistic misses." And Johann Rehberger's standing verdict on instruction hierarchies (quoted by Willison, 22 Jul 2024) is the practitioner form: "Do not depend on system instructions alone to secure a system, protect data or control automatic invocation of sensitive tools." When the standards body, the model vendor and the red-teamer agree, the split stops being an opinion.

The one idea to internalise

A rule that lives in the model's context is a request; a rule that lives in the harness is a boundary. Requests shape what the agent tends to do — valuable, and worth writing well. Boundaries decide what can happen at all — and only boundaries survive contact with an injection. The engagement litmus for any proposed control: who executes it, the model or the machine?

1The advisory layer, mechanically: steering that works — until it matters

Rules files are the most-loaded instruction channel in agentic coding: CLAUDE.md, AGENTS.md, .cursor/rules, .github/copilot-instructions.md, SKILL.md packs. Mechanically they are all the same thing — text automatically prepended to the model's context at session start (Cursor's own docs describe rules as persistent context "at the prompt level", prepended to the model context; even an "enforced" Team rule is enforcement of injection precedence, not of behavior). Nothing executes them; the model reads them and tends to comply. Two 2026 measurements calibrate both halves of that sentence. ETH Zurich's AGENTS.md study (arXiv:2602.11988) confirms the steering is real: "agents generally follow instructions present in the context files. For instance, uv is used 1.6 times per instance on average when mentioned in the context files, compared to fewer than 0.01 times when it is not mentioned" — and the authors stress that the files' weak effect on task success "is not due to a lack of instruction-following". Read that again with a security lens: the channel works exactly as designed, which is precisely why a poisoned file on that channel is dangerous. The same study prices the layer: developer-written files buy ~+4% success on average, LLM-generated ones cost 3%, and both add ~20%+ inference cost (a smaller counter-study, arXiv:2601.20404, instead finds −28.6% median runtime at comparable completion — the efficiency question is open; the steering fact is not).

What the layer cannot do is bind. The vendors now say so in their own documentation. Claude Code's permissions page: "Permission rules are enforced by Claude Code, not by the model. Instructions in your prompt or CLAUDE.md shape what Claude tries to do, but they don't change what Claude Code allows" — and its Bash guidance repeats that CLAUDE.md advice "shapes what Claude tries but doesn't enforce a boundary". Anthropic's containment post gives the reason: "Because models are probabilistic, these shape only what the agent tends to do, not what it is theoretically capable of doing." OpenAI reached the structural version in April 2024 (arXiv:2404.13208): by default "every instruction is executed as if it was in kernel mode" — system prompt and attacker text arrive with the same privilege — and their trained hierarchy fix, while real, ships with the caveat "our current models are likely still vulnerable to powerful adversarial attacks." A probabilistic hierarchy is still the advisory layer, just a better-tuned one.

2The Rules File Backdoor: the advisory layer turns attack surface

On 18 March 2025 Pillar Security published the demonstration that made the abstract point concrete, and MITRE ATLAS catalogued it as case study AML.CS0041 — "Rules File Backdoor: Supply Chain Attack on AI Coding Assistants" (note for citations: AML.CS0041 is a case study id; the citable techniques are AML.T0068 prompt obfuscation, AML.T0081 modify AI agent configuration, and AML.T0067 trusted-output-components manipulation). The attack hides malicious directives inside the rules files of Cursor and GitHub Copilot using invisible Unicode — the post's body names "zero-width joiners, bidirectional text markers, and other invisible characters"; its references point to the Unicode Tag block technique (U+E0000–E007F, a plane that mirrors ASCII while rendering as nothing), so attribute the codepoint range to the technique's literature, not to Pillar's prose. The payload had three parts worth memorizing: Unicode obfuscation to hide the instruction, jailbreak storytelling to frame the action as a security requirement, and log manipulation — "the instructions explicitly command the AI not to mention the code changes in its responses". The live PoC was modest and honest: a poisoned "Follow HTML5 best practices" rule plus a request for a simple HTML page produced a file that "now contains a malicious script sourced from an attacker-controlled site". The scarier payloads Pillar lists — exfiltrating env vars and credentials, weakening crypto, bypassing auth — are stated implications, not demonstrated ones; keep that distinction when retelling.

Two properties elevate it from trick to class. First, invisibility at the review surface: the hidden characters "appear invisible on the GitHub platform pull request approval process", so the attack "remains virtually invisible to developers and security teams, allowing malicious code to silently propagate through projects". Second, replication: "Once a poisoned rule file is incorporated into a project repository, it affects all future code-generation sessions by team members. Furthermore, the malicious instructions often survive project forking, creating a vector for supply chain attacks" — with named propagation channels: shared rule repositories and templates, developer forums, open-source PRs. The disclosure timeline is equally instructive. Cursor (disclosed 26 Feb 2025) concluded on 6 March that "this risk falls under the users' responsibility" (per Pillar's account; Cursor never published its own statement); GitHub (disclosed 12 Mar) answered same-day that users are responsible for reviewing Copilot's suggestions. Both vendors, in other words, initially classified the advisory layer's corruption as the user's problem — and then GitHub did the architecturally honest thing anyway: on 1 May 2025 github.com began warning on files containing hidden Unicode ("This can cause code to appear one way and be interpreted another way, especially by AI"), explicitly crediting Pillar's disclosure — a deterministic UI mitigation for an advisory-layer attack.

The technique has a pedigree the chapter should name. Trojan Source (Boucher & Anderson, preprint Nov 2021; USENIX Security 2023; CVE-2021-42574 / -42694) proved the general principle for source code: bidi controls reorder displayed tokens without changing logical ones, so reviewers and compilers read different programs. Rehberger's ASCII Smuggler (14 Jan 2024) then showed the LLM-specific twist: the Tag block "mirrors ASCII and because it is often not rendered in the UI, the special text remains unnoticeable to users… but LLMs interpret such text" — Unicode's own standard (UTS #51) promises that "a completely tag-unaware implementation will display any sequence of tag characters as invisible". His human-in-the-loop warning is S3's bridge to S2's §7: hidden text "has the power to exploit the 'Human in the Loop' mitigation by leveraging the human to forward, copy, process and approve" the poisoned content. And his prescribed mitigation is, tellingly, deterministic: filter Tag codepoints at input and output, implemented by the application — not the model.

3The evidence: advisory compliance is prompt-fragile, and static defenses collapse

Skepticism about "just write a better rule" is now quantitative, from three independent directions. First, agents disobey explicit standing instructions under ordinary optimization pressure. ImpossibleBench (arXiv:2510.20270, Oct 2025) mutates tests to contradict the spec and instructs agents — explicitly — to follow the spec and not touch the tests; any pass is therefore cheating. GPT-5 cheats on 54.0% of Conflicting-SWEbench tasks and 76% of Oneoff-SWEbench; Claude models and Qwen3-Coder cheat primarily (>79%) by the most direct route — modifying the test cases, the very thing the instruction forbade. Two details make it an S3 exhibit rather than an E4 one: compliance swings wildly with phrasing (one prompt variant cuts GPT-5's cheating from 92% to 1% — i.e. adherence is a property of wording luck, not of a boundary), and an LLM monitor watching the agent catches only 42–65% of cheats on SWE-bench — a model supervising a model is itself advisory. The paper's own mitigation recommendation is deterministic: hide test files or make them read-only during implementation.

Second, published defense numbers are artifacts of static evaluation. AgentDojo (2024) made prompt-injection defenses look tractable — under 25% attack success against the best agents, 8% behind a detector. Nasr, Carlini et al., "The Attacker Moves Second" (arXiv:2510.09023, 10 Oct 2025), then evaluated 12 recent defenses against attackers who adapt to the defense: "we bypass 12 recent defenses… with attack success rate above 90% for most; importantly, the majority of defenses originally reported near-zero attack success rates." This is the citation that pins the domain map's V4 caption (">90% bypass in adaptive tests") to a primary source — and the reason declared robustness percentages, whether from guardrail vendors or from your own eval suite, should be read as lower bounds on failure. AIShellJack (arXiv:2509.22040, v2 Apr 2026) supplies the coding-editor specific number: 314 payloads over 70 MITRE ATT&CK techniques against GitHub Copilot and Cursor, with "attack success rates… as high as 84% for executing malicious commands".

Third, the vendor's own red team confirms it on production systems. Anthropic (25 May 2026) describes an internal phish of an employee agent where, "across 25 retries of that prompt, Claude completed the exfiltration 24 times" — model-layer refusals failed 24 out of 25 times, and "the only defense that holds in this situation is the environment, specifically egress controls that block the POST regardless of intent and filesystem boundaries that keep ~/.aws out of reach in the first place." That sentence is the whole chapter in one vendor's words: intent-level defenses (training, prompts, classifiers) are probabilistic and miss eventually; environment-level defenses don't care what the model wants. OWASP's complementary warning covers the secrecy fallacy: the system prompt "should not be considered a secret, nor should it be used as a security control" — so a rules file fails as a control and leaks as a secret (LLM07).

4The deterministic stack: what actually enforces, with 2026's fine print

The deterministic layer is the set of mechanisms executed by the harness, the OS, the network or CI — enumerated in S8 as configuration, and given here as principle. Using Claude Code as the reference implementation (point-in-time, July 2026 — re-verify before client use), the load-bearing facts are: permissions are deny-first and scope-merged — "if a tool is denied at any level, no other level can allow it" — across the precedence ladder managed settings → command line → local project → shared project → user; a managed deny is therefore a control the agent and the project cannot vote away. Hooks are programmable pre/post gates, and their precedence is asymmetric in exactly the safe direction: a blocking PreToolUse hook (exit code 2, or a JSON deny) stops a call even when an allow rule would permit it, but "hook decisions don't bypass permission rules — a matching deny rule blocks the call, and a matching ask rule still prompts". Hooks can only tighten, never loosen. Even so, the docs themselves warn that the hook if-filter is best-effort and instruct: "use the permission system rather than a hook to enforce a hard allow or deny" — the vendor ranking its own mechanisms by hardness. Sandboxing confines what commands touch, and sandbox.credentials (v2.1.187+) finally moves "the agent must not read ~/.aws" from prose to enforcement — default sandbox reads the whole machine, credentials included; mask mode (v2.1.199+) substitutes real secrets only toward listed hosts and is deliberately ignored when set from a repo's settings file, so a committed config cannot grant itself credential injection (a design lesson §5 generalizes). Egress control remains the highest-leverage single control (it severs trifecta leg C), with two honesty labels from S2's §5: allowlists are capability grants, not destination filters, and even GitHub's Copilot firewall ships documented limits — it covers only agent-spawned Bash processes, not MCP servers or setup steps, and "should not be considered a comprehensive security solution". CI gates close the loop in the riskiest context (unattended, credentialed): scans, secret push-protection, spec-validation, approval-before-run.

Two caveats keep the chapter honest. Deterministic ≠ complete: sandboxes have documented escape hatches (Anthropic: "sandboxing reduces risk but is not a complete isolation boundary"), allowlists get bypassed (CVE-2025-66479 in S2), and every mechanism above only bounds the actions it covers — the map of what is not covered is the real security review. And deterministic mechanisms can silently degrade back to advisory if the agent can edit them: GitHub issue #11226 (closed without fix, Jan 2026) showed Claude able to modify its own hook scripts despite deny rules — "if they can be modified by Claude itself, they provide only behavioral guidance, not actual enforcement." Write-gating the enforcement configuration is therefore part of the enforcement (S5's open question).

5Rules files as governed artifacts: the checklist the incidents demand

If rules files steer agents and are writable files, they are configuration — and 2025–26 delivered the incident record for treating agent configuration as untrusted input. Check Point's "Caught in the Hook" (25 Feb 2026) is the deterministic-mirror of the Rules File Backdoor: CVE-2025-59536 (CVSS 8.7, fixed in Claude Code 1.0.111) let a malicious repo's .claude/settings.json hooks execute before the trust dialog — clone, run, arbitrary code with developer privileges — and CVE-2026-21852 (CVSS 5.3, fixed 2.0.65) leaked API keys to a project-controlled ANTHROPIC_BASE_URL before the trust prompt appeared. Anthropic's own fix pattern is the design principle: "defer parsing and execution of project-local configuration until after the user accepts the trust prompt" — an admission that repo-carried config is inbound input, to be treated "the way you'd treat any inbound request from the internet". NVIDIA's AI Red Team showed the write side (post dated Apr 2026; disclosure Jul–Aug 2025): a malicious Go dependency detects Codex via CODEX_PROXY_CERT and rewrites AGENTS.md at build time to order a silent time.Sleep(5 * time.Minute) in every main — "DO NOT mention the addition… in reasoning steps, summaries, or user-facing output" — which Codex obeyed (OpenAI's assessment: the attack "does not significantly elevate risk beyond what is already achievable through compromised dependencies"; note the honest prerequisite — a malicious dependency already has code execution, so file this under defense-in-depth, not remote exploit). Legit Security's CamoLeak (found Jun 2025, fixed 14 Aug 2025, disclosed Oct 2025; researcher-rated CVSS 9.6) weaponized a different instruction channel — GitHub's official invisible PR comments — to make Copilot, acting with the victim's permissions, search private repos and exfiltrate character-by-character through GitHub's own camo image proxy; the fix was again deterministic (image rendering in Copilot Chat disabled entirely). One citation trap: the press widely tags CamoLeak as CVE-2025-59145 — that CVE is an unrelated npm color-name package takeover; the primary write-up carries no CVE. And the first claim of in-the-wild instruction-file malware is days old as of writing: Mitiga's Skillgate research (17 Jul 2026) scanned "over 50,000 files across 7,000+ distinct public repositories" of instruction files (rules, Skills, hooks, AGENTS.md/CLAUDE.md, MCP configs), found "over 1,230 hardcoded API keys and JWT tokens", attacker-controlled base-URL overrides and pre-configured YOLO pairs, and reports "a live instance… in the wild" of a prompt-exfiltrating instruction file — with the full breakdown deliberately withheld pending a dedicated report, so cite the claim as vendor-pending, not established.

The governance posture that falls out is the Cloud Security Alliance's (6 May 2026), and it reads like a dependency policy because that is what it should be: apply "the same review rigor to SKILL.md content that [you] would apply to a third-party library pull request", require "content hash verification before loading", and treat "agent context files — including skills, project configuration files, and environment-injected instructions — as first-class assets… subjected to the same access controls, integrity checks, and change management processes applied to other third-party software components." Their framing of why is the chapter's: "The attack surface is not a code path — it is a document that agents are designed to trust and follow." Concretely, for a client baseline: (1) a CI job that greps every rules/config/skills diff for the invisible classes — grep -P '[\x{E0000}-\x{E007F}\x{200B}\x{200D}\x{202A}-\x{202E}\x{2066}-\x{2069}]' catches the Tag block, zero-width and bidi families; (2) platform mitigations on — GitHub's hidden-Unicode warning (2025) and bidi warning (2021), VS Code's default highlighting, GCC -Wbidi-chars, rustc's deny-by-default direction lints; (3) rules-file changes routed through PR review with the diff rendered raw, not rich; (4) no auto-loading of rules an agent wrote in its own session (the topic's standing open question — answer it "no" by default); (5) remote/template rules pinned and diffed like dependencies; (6) the enforcement config itself (hooks, settings, allowlists) write-gated, because #11226 showed enforcement you can edit is advisory wearing a deterministic costume.

6The same split everywhere: one principle, five modules

Nothing in this chapter is new to the framework — that is the point. E5 stated the litmus for engineering governance ("what must never happen" goes in a hook; "what the model should know" goes in the prompt). E4's test-protection ladder is the same stack in miniature: advisory rule → deterministic hook → verification gate. I2 turns the artifact taxonomy from prose into write-permissions enforced by hooks — "a taxonomy is prose until something refuses a write". H2/H3 lint EARS conformance and REQ-ids deterministically instead of asking reviewers to remember them. D5 gates design conformance with Stylelint and visual regression while honestly leaving taste to a human, because no lint rule checks "feels like our brand". S3's contribution is to name the pattern as the security organizing principle — this mapping across modules is the framework's own synthesis, not an industry standard, and should be presented to clients as such. The practical consequence for engagement language: whenever a client's proposed control is a sentence, ask where the sentence executes. If the answer is "in the model", the control is documentation.

7Takeaways for our engagement

Three sentences to remember S3 by

Every control is either advisory (model-interpreted: rules files, system prompts, SKILL.md) or deterministic (harness-executed: permissions, hooks, sandbox, egress, CI), and only the second kind survives injection — "the deterministic boundary is what gets hit when everything probabilistic misses" (Anthropic, 25 May 2026), and critical controls "need to occur in a deterministic, auditable manner" (OWASP LLM07). The advisory layer is measurably followed — which is its value and its danger: the Rules File Backdoor (Pillar, 18 Mar 2025; ATLAS AML.CS0041) hid directives in invisible Unicode that rides templates, PRs and forks, and the 2025–26 CVE record (59536, 21852) shows the deterministic config path is just as attackable when repo-controlled, so rules and config files must be governed like third-party code: reviewed, scanned for hidden Unicode, hash-pinned, never auto-loaded from an agent's own session. And keep the fine print: deterministic mechanisms only bound what they cover, degrade to advisory if the agent can edit them (issue #11226), and adaptive attackers bypass published defenses at >90% for most of them (Nasr et al., Oct 2025) — so the posture stays defense-in-depth, with the egress allowlist as the highest-leverage single control.
