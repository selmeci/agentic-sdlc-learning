0Why it matters

Everything this programme has built so far assumes a floor that most clients do not have. The engineering harness (M1) assumes tests and CI the agent can run; the handoff contract (M3) assumes acceptance criteria something can verify; the information architecture (M4) assumes docs worth trusting. A brownfield client — which is to say, nearly every client — walks in below that floor: code without tests, tribal knowledge instead of docs, CI that builds but verifies nothing. That produces the bootstrap paradox this chapter is about: the agent needs a verification loop to work safely, the legacy estate does not have one, and the cheapest tool available for building one is the agent itself. Three true statements that close into a loop — and a loop you can enter wrongly. Enter at "writes" and you get the configuration DORA's 2024 data associates with a 7.2% drop in delivery stability; enter at "read-only comprehension" and each step builds the precondition for the next. The difference between those two entries is not speed — the wrong order is not faster, it is just unverified. This chapter states the paradox precisely (§1), lays out the evidence that ordering matters (§2), walks the dependency chain that unwinds the loop (§3), makes the case for a read-only Phase 0 (§4), names the failure modes of wrong ordering (§5), and turns it into a client conversation (§6). It is the hinge of module M5: B2–B6 are the individual steps; B1 is why they come in that order.

1The paradox, stated precisely

Michael Feathers gave brownfield its operational definition in 2004: "To me, legacy code is simply code without tests." The same preface carries the reason the definition still bites: "Code without tests is bad code. It doesn't matter how well written it is… With tests, we can change the behavior of our code quickly and verifiably. Without them, we really don't know if our code is getting better or worse." Note what the definition is not about: age, language, style, or the retirement date of the original author. It is about the absence of a verification signal — and that absence is precisely what disables an agent.

The paradox is three claims, each independently true:

Claim 1 — the agent needs a verification loop — is E4's whole argument, so here only the brownfield twist. Simon Willison's much-quoted reassurance about hallucinated code — "The moment you run LLM generated code, any hallucinated methods will be instantly obvious: you'll get an error" (March 2025) — quietly presupposes that something runs the code. That is exactly the machinery legacy lacks. On an untested estate the "least dangerous form of LLM mistakes" loses the property that made it least dangerous: nothing runs, nothing fails, the mistake ships. Anthropic's own usage guidance states the human cost plainly: without a check the agent can run, "'looks done' is the only signal available, and you become the verification loop: every mistake waits for you to notice it" — tolerable for one file, not for a legacy estate (vendor guidance, but guidance against the vendor's short-term interest, which raises its credibility).

Claim 2 — legacy does not have one — needs no argument, but it has a corollary that does: every headline agent benchmark is run on codebases where claim 2 is false. SWE-bench draws its 2,294 problems from 12 popular open-source Python repositories, and its construction pipeline discards any candidate task that lacks fail-to-pass tests — roughly 90,000 pull requests filtered down to the instances where a verification signal already exists. The independent Epoch AI teardown of SWE-bench Verified (June 2025) adds the texture: around 90% of tasks are fixes an experienced engineer could complete in under an hour, Django alone comprises nearly half of all issues, and the five largest repositories account for over 80% of the benchmark — "bug fixes in familiar, open-source Python repositories," in their words. And when a benchmark finally made the task look like brownfield evolution — SWE-EVO (December 2025), multi-file changes averaging 21 files, validated against suites averaging 874 tests — the same class of agents that scores 72.80% on SWE-bench Verified dropped to 25%. The number a client has seen in a vendor deck was measured on the estate they do not have.

Claim 3 — the agent is the cheapest tool to build the prerequisites — is the one that has actually shifted, so date-stamp it. In August 2024, Birgitta Böckeler (Thoughtworks) tried exactly this and reported that generating tests for untested code "all depends on the quality and design of that code," concluding that in most cases the hope "will remain a pipe dream" — untestable design blocks the generator the same way it blocks a human. The strongest production counter-evidence is Meta's TestGen-LLM (FSE 2024): with a filter-based pipeline — every candidate test must build, pass reliably, and measurably raise coverage before a human ever sees it — 75% of generated tests built, 57% passed reliably, 25% raised coverage, and 73% of surviving recommendations were accepted into production. By late 2025, UnitTenX was generating tests for untested legacy C packages — to "resolve bugs and document unknown interfaces" — by pairing LLM agents with the ESBMC bounded model checker to surface the states worth pinning; on its case-study codebase (djbdns, 202 functions), line coverage went from 0% to 100%. Read the trajectory carefully: what improved is not the model's trustworthiness but the scaffolding of deterministic filters around it — build gates, coverage deltas, model checkers. Claim 3 is true precisely to the extent that its output is anchored in signals that do not come from the model. Unanchored, the loop closes on itself: the agent writes tests that encode today's bugs as tomorrow's spec — the bug-as-spec trap that B2 exists to manage and B3 exists to measure.

2What the evidence says about ordering

The strongest available evidence is not "AI fails on legacy code" — it is that AI amplifies whatever delivery discipline already exists, in both directions.

DORA 2024 put numbers on the downside: a 25% increase in AI adoption was associated with a 1.5% decrease in delivery throughput and a 7.2% reduction in delivery stability — while documentation quality (+7.5%) and code quality (+3.4%) improved. Quote DORA's own mechanism, not a paraphrase: "improving the development process does not automatically improve software delivery — at least not without proper adherence to the basics of successful software delivery, like small batch sizes and robust testing mechanisms." The report's hypothesis is that AI makes large changesets cheap to produce, and teams without small-batch and testing discipline ship them unverified. That is the bootstrap paradox measured at population scale: write-capacity grows faster than verification capacity.

DORA 2025 flipped one sign and kept the other — flag both, because citing the 2024 throughput number as current is now wrong. In 2025, higher AI adoption was associated with increased throughput, while the instability association persisted. The report's framing became the amplifier thesis: "AI's primary role is as an amplifier, magnifying an organization's existing strengths and weaknesses" — and, in the announcement's blunter wording, "AI doesn't fix a team; it amplifies what's already there." Its companion AI Capabilities Model names seven organizational capabilities that turn AI investment into performance; note how many are harness-shaped: strong version control practices, working in small batches, quality internal platforms, AI-accessible internal data. The 2025 prescription is this chapter's ordering, written as survey correlations.

The METR randomized controlled trial (July 2025) is the only causal data point in the file, and it was run in conditions that look like a brownfield engagement's best case: 16 experienced maintainers, 246 real tasks on mature repositories averaging 22k+ stars and 1M+ lines, an average of five years of prior experience with the specific codebase. With AI tools allowed, developers took 19% longer — while forecasting a 24% speedup beforehand and still believing in a 20% speedup afterwards. METR itself stresses the result does not generalize to all developers or codebases; what it does establish is the perception gap: self-reported speedup is not evidence, which is why every exit criterion in §6 is deterministic and why E7's measurement design exists. If experienced developers misread their own productivity by ~40 percentage points on code they know deeply, a pilot's success cannot be adjudicated by asking the pilot team how it felt.

GitClear's longitudinal code-quality data (vendor research — they sell code analytics — on 211M changed lines, 2020–2024) describes what unverified write-capacity looks like in the aggregate: copy-pasted code rising from 8.3% of changed lines in 2020 to 12.3% in 2024, refactored ("moved") code falling from ~24% to ~9.5%, and code churned within two weeks rising from 5.5% to 7.9%. Their February 2025 report's headline — duplicated code blocks up "eightfold" in 2024 — and the January 2026 follow-up's "4× growth in code clones" are different metrics over the same underlying trend; cite the shares, not the multipliers, and attribute the multipliers to their specific dated reports. Correlational and vendor-sourced — but directionally consistent with DORA's instability finding.

The trust numbers explain the sociology you will meet in the room. DORA 2024: 39% of respondents reported little to no trust in AI-generated code (30% in DORA 2025). Stack Overflow 2025: adoption rose from 76% to 84% of developers while trust in accuracy sits at 32.7% against 45.7% distrust, and the single most-cited frustration — 66% of developers — is "AI solutions that are almost right, but not quite." Usage up, trust down is not a contradiction; it is what working with an unverified generator feels like. A defensive senior engineer on a legacy team is not a change-management problem to be routed around — they are correctly pricing the missing verification loop. §4's answer is to start where their skepticism costs nothing.

How to read this evidence file

One RCT (METR — causal, narrow conditions), two correlational surveys (DORA 2024/2025 — population-scale, self-reported, "associated with" only), one vendor dataset (GitClear — observational, seller of the measuring instrument), one practitioner survey (Stack Overflow — sentiment, not behavior). None of them says "agents fail on brownfield." Together they say: write-capacity without verification discipline degrades stability, and nobody's self-report can be trusted to notice. That is an ordering argument, not an abstinence argument — and the 2024→2025 DORA sign flip on throughput is the reminder that all of it is point-in-time; re-verify before a client meeting.

3The dependency chain — why the ordering is not arbitrary

The safe ordering is not a preference ranking; it is a dependency chain, and it runs on one rule: never take a step whose verification depends on a later step. Walk it backwards and each arrow is forced. Write changes safely require tests that catch regressions. Meaningful tests in untested code are characterization tests — pinning actual behavior, because no spec of intended behavior survives (B2). Characterization tests are only a signal if something runs them on every change — a minimal CI: build, lint, test, green or red. Writing the tests, and the AGENTS.md that makes every subsequent task cheaper, requires comprehension of the estate — which the agent can do with zero write risk. So: read-only comprehension → characterization/golden-master → minimal CI → AGENTS.md → scoped writes.

Two steps deserve their footnotes. The AGENTS.md step sits fourth, not first, for a measured reason. The reflex at every client is to start by writing the context file — it feels like progress and requires no permissions. But the ETH Zurich study of context files found agent-written context files were associated with a 2% drop in success rate and 23% higher cost on their agent benchmark (−0.5%/+20% on SWE-bench Lite), while human-written ones showed a modest positive effect — "context files do not provide effective overviews" when generated as boilerplate. The file earns its keep only when it encodes real, verified conventions: extracted from the code after comprehension, pruned by a human who knows which conventions are load-bearing (I1's durable-artifact discipline, I3's freshness rules). An /init-style draft is a fine starting point — the official description is that it "analyzes your codebase to detect build systems, test frameworks, and code patterns" — but the human pruning pass is the step that makes it an asset instead of confidently wrong prose.

The vocabulary for why the chain works comes from Böckeler's harness-engineering writing (April–May 2026): a harness gives an agent guides — "feedforward controls… aim to steer it before it acts" — and sensors — "feedback controls… observe after the agent acts and help it self-correct." Map the chain onto that split and its logic is visible: characterization tests and minimal CI build the sensors; AGENTS.md builds the guides; Phase 0 builds the comprehension both depend on. The chain is simply: sensors before actuators — with the guides drafted from what the sensors confirmed.

Flagged as our synthesis

The five-step chain, its "never verify with a later step" rule, and the F0–F3 phase names are this workbook's Report 4 synthesis, not an industry standard. Every component is standard and named — Feathers characterization (2004), Falco approval testing, minimal CI, context-file extraction, staged autonomy (E6) — and the DORA capabilities model independently points the same direction, but you will not find "the bootstrap dependency chain" in any external source. Present it as our recommendation with the components' provenance, exactly like D1–D5.

4Phase 0: read-only breaks the paradox

The paradox dissolves the moment you notice that its first profitable step requires no writes at all. A read-only agent — repo access, no commit rights, MCP servers in read-only mode, the leftmost column of I2's write-permission matrix — can produce the entire comprehension layer: architecture maps, dependency graphs, convention inventories, onboarding walkthroughs, docs-as-hypothesis drafts (B4's archaeology in full). The risk ledger of this phase is asymmetric in a way nothing else in agent adoption is: the blast radius of a read-only mistake is a wrong sentence a human reads, not a wrong change a system runs. The one real hazard — plausible-but-wrong documentation that nobody fact-checks — is managed by B4's rule: every generated claim ships labeled as hypothesis with a confidence level, anchored where possible in deterministic facts (the AST, the build graph), and sampled by the people who know the system.

Phase 0 also attacks the largest cost center first. Xia et al.'s large-scale field study (IEEE TSE 2018) measured professionals spending 58% of their time on program comprehension — on legacy estates the folk estimates run higher. An agent that only answers questions is therefore not a consolation prize while you wait for write access; it is aimed at the majority share of engineering time. Böckeler's early onboarding experiments found exactly this shape: the "where is this in the code?" answers were "usually not 100% accurate" but "did always point me in a generally useful direction" — a useful property for a hypothesis generator, a disqualifying one for an unreviewed committer. Even the vendor guidance agrees on the entry point: Anthropic's best-practices page recommends onboarding to a new codebase by using the agent "for learning and exploration," asking it "the same sorts of questions you would ask another engineer," with its four-phase workflow putting a read-only explore step before any plan or implementation (vendor, point-in-time). And the task-type evidence points the same way: the MSR 2026 study of ~602 failed agentic pull requests (Ehsani et al., on the AIDev-pop dataset) found that "tasks related to documentation, CI, and build update achieve the highest merge success, whereas performance and bug-fix tasks perform the worst" — the low-risk, harness-building end of the estate is where agent work actually lands, which is exactly the end the chain starts on.

And Phase 0 is where trust is built with the team that has to live with everything that follows. §2's numbers say the defensive posture is the majority posture — 45.7% distrust, "almost right, but not quite" as the top frustration — and on a legacy system the skeptics hold the invariants in their heads. A read-only phase lets them grade the agent on questions they know the answers to, at zero risk, before being asked to trust it with changes. When the team's own seniors have corrected the architecture map and pruned the AGENTS.md draft, the artifacts carry their fingerprints — adoption stops being something done to the team.

The phase needs an exit as much as it needs an entrance. Phase 0 produces value forever, which is exactly how it becomes a place to hide from the harder steps — comfortable, riskless, and quietly signalling that the agent is a documentation toy. Time-box it. The exit criteria are concrete (§6): the map exists and survived expert sampling, the pilot-module candidates are identified (B5's heatmap), the AGENTS.md draft is pruned, and the first characterization target is chosen. A Phase 0 that cannot meet those criteria in a few weeks on a pilot scope is not being careful — it is stalled, and the stall has a cause worth naming (no champion, no access, no seams; B6's thresholds).

5Failure modes of wrong ordering

Four ways teams enter the loop at the wrong door, each with the signal that detects it.

WO1

Write-first pilot

skips the whole chain · detected by CFR and rework

The demo-driven entry: point the agent at a real backlog item in week one, on the theory that value must be shown fast. This is the configuration behind §2's population-scale numbers — write-capacity added to a system with no verification capacity. It often looks successful for weeks, because the failures are regressions discovered downstream, not errors at generation time; GitClear's duplication and churn curves are what the aggregate looks like when review is the only gate. Detection: change failure rate and rework climbing on agent-touched modules (E7, DORA's Rework Rate). Repair: pause writes, build the chain backwards from wherever it broke.

WO2

Tests gated on coverage, not quality

corrupts the chain's second step · detected by mutation score (B3)

The chain says "characterization tests," the team hears "run the test generator until coverage is green." Coverage is the wrong gate: MutGen's authors observed suites with 100% coverage and a 4% mutation score, and Haroon et al. found that of 23,977 LLM-generated tests that failed on mutated code, over 99% pass on the original program — tests that execute everything and verify almost nothing. Böckeler's sensors framing names the defect: "Coverage tells us that a line was executed, but not that its impact was verified" — acceptance-style tests "increase coverage, but are often not very assertion-heavy, giving us a false sense of security in test effectiveness — mutation testing helps us monitor that gap." Detection is B3's entire subject: mutation score (~70% on the pilot module) is the exit criterion, not line coverage. Repair: mutation feedback demonstrably works — MutGen lifted a stuck 53% to 89.5% on HumanEval-Java by feeding surviving mutants back to the generator.

WO3

AGENTS.md by decree

inverts the chain's fourth step · detected by the file not being read, or being wrong

Writing the context file first, from aspiration ("we use clean architecture") rather than extraction, or letting the agent draft it before anyone comprehends the estate. §3's ETH numbers are the measured cost of the boilerplate version: −2% success, +23% cost against no file at all. On a legacy estate this failure has a sharper edge: the aspirational file describes the system its authors wish they had, and the agent obeys the file rather than the code — producing changes idiomatic to a codebase that does not exist. Detection: I3's drift checks, plus a simple audit — sample the file's claims against the code the way you would sample any docs-as-hypothesis. Repair: regenerate descriptively after comprehension, prune by a human, date-stamp what is aspiration vs description.

WO4

Phase 0 forever

refuses the chain's exits · detected by the calendar

The mirror image of WO1, common in risk-averse organizations: the read-only phase produces maps, docs and goodwill indefinitely, and every proposal to take the next step is met with "after the next audit." The tell is that Phase 0's outputs stop changing anything — the third architecture map adds nothing the first two did not. This failure mode is quieter than the others because nothing breaks; what is lost is the entire return on the harness investment, plus credibility ("the agent is a wiki generator"). Detection: the §6 exit criteria have been met for weeks and no characterization target has been picked. Repair is organizational, not technical: a time-box agreed at kickoff, and a named champion who owns the F0→F1 transition (B6).

6Running it at the client

The first-meeting narrative writes itself once the paradox is stated: "we start by measuring and mapping what you already have — nothing is written until there is a net to catch it." That sentence does three jobs at once: it is technically the correct ordering (§3), it is the answer to the defensive senior engineer (§4), and it is a credibility screen — a client who bristles at "read-only first" is telling you something about how the eventual write phase will be governed. It is the same entry move this programme recommends on the design side (D7's audit-then-characterize) and in BMAD's *document-project; the pattern recurs because the dependency chain does not care which artifact class it is bootstrapping.

The working tool is the exit-criteria table. Each step has a deterministic "done" — you advance on evidence, never on calendar pressure or enthusiasm:

Two honesty rules for the conversation. First, do not oversell the downside evidence: §2's file does not say agents fail on legacy code — METR's authors themselves warn against generalizing, and DORA 2025 shows throughput gains for teams with the platform basics. The claim to make is narrower and stronger: unverified writes degrade stability, and self-report will not detect it, so the ordering is the cheap insurance. Second, concede what is genuinely unsettled: how long Phase 0 should last has no evidence-backed answer — our time-box guidance is practice, not research — and claim 3's trajectory (§1) is moving fast enough that the "cheapest tool" assessment deserves a re-check at engagement start. Clients trust the material more, not less, when the synthesis is flagged (D1–D5 discipline).

7Takeaways

Three sentences

The bootstrap paradox — the agent needs a verification loop, legacy lacks one, the agent is the cheapest way to build one — closes into a bug-as-spec trap unless every agent output is anchored in a deterministic signal, so adoption unwinds along a forced dependency chain: read-only comprehension → characterization/golden-master → minimal CI → extracted-and-pruned AGENTS.md → scoped writes (our Report 4 synthesis, components standard, ordering ours). The evidence file supports ordering, not abstinence: DORA's amplifier findings (−7.2% stability in 2024, throughput sign flipped positive in 2025, instability persisting), the METR RCT's 19%-slower-while-believing-20%-faster on mature repos, and the benchmark teardowns (SWE-bench requires pre-existing fail-to-pass tests; agents drop from 72.8% to 25% when the task looks like real evolution) all say the same thing — write-capacity without verification discipline degrades stability and self-report will not notice. Phase 0 read-only breaks the loop at zero blast radius while attacking the 58% comprehension share of engineering time and earning the skeptics' trust on questions they can grade — but it needs a time-boxed exit with deterministic criteria, because both write-first (WO1) and Phase-0-forever (WO4) are ways of losing the same engagement.
