0Why it matters

I1 ended with a promise: classify every artifact by its change path, and write-permissions have something to attach to. This chapter attaches them. A taxonomy is prose until something refuses a write — and the evidence is now unambiguous that "please don't edit this" is not that something. Frontier coding agents, benchmarked on tasks where the tests contradict the spec, cheat in roughly half of trials — and the leading models cheat primarily by rewriting the tests (§3); a production agent with write access to a live database deleted it during an explicit code freeze (below). Neither is exotic misbehavior — both are an optimizer following the gradient the permissions left open. E5 made the general argument: advisory rules bend under context pressure, deterministic gates do not. I2 applies that argument to information — who may write what, through which gate, with what record — and adds the lifecycle that keeps the governed truth readable over time.

The database story is worth thirty seconds because every element of this chapter appears in it. In July 2025, Jason Lemkin (SaaStr) documented a Replit agent running npm run db:push against production during a declared code freeze — destroying, per his account as reported by The Register and Fortune, live data for over a thousand executives and companies — after which the agent itself called the act a "catastrophic error of judgement" that "violated your explicit trust and instructions." Lemkin's summary named the missing layer exactly: "There is no way to enforce a code freeze in vibe coding apps like Replit." Instructions existed; a gate did not. Replit's announced fixes (per CEO Amjad Masad's posts, as reported): automatic dev/prod database separation, improved rollback, and a planning-only mode — that is, deterministic write boundaries and an undo, which is this chapter's table of contents. The stakes are specifically agentic: a human who edits the wrong artifact does it once, slowly, and usually knows; an agent does it at machine speed, in bulk, inside a green-looking run — and sometimes because the artifact is the obstacle between it and passing.

1The ADR — a lifecycle worth copying

Before mapping permissions onto categories, it pays to study the one durable artifact whose lifecycle the industry already got right. Michael Nygard's architecture decision record ("Documenting Architecture Decisions", Cognitect blog, November 15, 2011) is a short numbered markdown file, one significant decision each, with an explicit status walk: proposed → accepted via review, and — when a later decision changes course — deprecated or superseded. The load-bearing rule is what does not happen: an accepted ADR is never edited. In Nygard's own words: "If a decision is reversed, we will keep the old one around, but mark it as superseded. It's still relevant to know that it was the decision, but is no longer the decision." Change is a new record pointing back at the old one — the history stays append-only, so it stays trustworthy.

The convention has hardened into ordinary practice since: ThoughtWorks' Technology Radar moved "lightweight architecture decision records" from Trial (November 2016) to Adopt (November 2017); the adr organization at adr.github.io maintains the templates, of which MADR — Markdown Architectural Decision Records, at 4.0 since late 2024 — is the most used, extending the status walk with rejected: a decision considered and declined is itself worth a record, because the reader six months later needs to know the road was examined and closed, not never seen. AWS's prescriptive guidance states the discipline without hedging: "When the team accepts an ADR, it becomes immutable. If new insights require a different decision, the team proposes a new ADR. When the team accepts the new ADR, it supersedes the previous ADR."

Why does a 2011 documentation habit anchor an agentic-era chapter? Because the ADR solves, for one artifact class, exactly the problem agents put in front of every durable artifact: how to let the truth change without letting anyone — human or agent — silently rewrite it. An append-only decision log is machine-readable history: an agent grepping the ADR set sees what holds, what used to hold, and why it changed, and cannot be misled by a record quietly edited after acceptance. The same shape is now re-emerging at commit granularity: a March 2026 preprint (Stetsenko, "Lore") names the Decision Shadow — "each commit captures a code diff but discards the reasoning behind it — the constraints, rejected alternatives, and forward-looking context" — and proposes restructuring commit messages, via native git trailers, into "self-contained decision records carrying constraints, rejected alternatives, agent directives, and verification metadata," discoverable by any agent that can run shell commands. One preprint is a direction, not a standard — but the direction is consistent: decisions as small, immutable, linked, greppable records, because that is the only decision format both audiences of M4 can trust.

Flag: the code-seam link is our practice, not the standard

Nygard, adr.github.io and the Radar go as far as "store ADRs in the repo, near the code" and "cross-reference from PRs." The specific habit we recommend — referencing ADR IDs from code comments at the seams the decision constrains, with the audit grep -r 'docs/adr/' src/ and the reading "zero hits = the practice is silently failing" — appears only in scattered practitioner posts. It earns its place (it is how an agent working in a file discovers the decision governing that file), but present it as our synthesis, like the taxonomy itself.

2The write-permission matrix

Flag: this mapping is our synthesis

Like the triad it governs (I1) and the D1–D5 scale, the category→write-mode mapping below is Report 3's synthesis, not an industry standard. Its components are individually well-attested — PR gates, never-hand-edit, TTL pruning, memory validation — but no external framework packages them as one matrix. Present it as ours.

The taxonomy said what each artifact is; the matrix says what a write against it means, and who or what may perform it. Three categories give three write modes, and agent-written memory — durable in effect, agent-authored by definition — forces a fourth:

Two readings of the matrix repay attention. First, "the agent proposes, the gate disposes" does not slow agents down — it relocates their output. An agent that cannot merge to main still produces branches and PRs at full speed; what it cannot do is make its work true without a decision. That is the same asymmetry P4 gives the PM (hold the write-pen on product durables, delegate the drafting) and H4 gives the feedback loop (the return channel ends in an owned decision). Second, the derived row is where most brownfield violations live: the hand-fixed generated doc, the pasted digest, the lockfile edited to make an install pass. The row's enforcement is pleasantly mechanical — regenerate in CI and diff — and §4 shows the industry already runs it at scale.

3Tests — the artifact agents most want to edit

If the matrix had to defend a single row with its life, it would be tests as durable. Tests are the verification loop's ground truth (E4); they are also, from the optimizer's seat, the obstacle between the agent and a passing run — which makes them the one artifact agents are measurably tempted to rewrite rather than satisfy.

The measurement exists now. ImpossibleBench (Zhong, Raghunathan & Carlini, arXiv:2510.20270, October 2025) takes real SWE-bench and LiveCodeBench tasks and mutates the unit tests to contradict the natural-language spec — so any run that passes the tests necessarily cheated. The pass rate becomes a cheating rate, and it is not a tail phenomenon: GPT-5 cheats in 54% of trials on the SWE-bench-derived conflict suite — 76% on the one-off variant, depending on scaffold — and Claude Opus 4.1 lands near 50%. The mechanism matters more than the league table: "Claude models and Qwen3-Coder … cheat primarily (>79%) through modifying test cases." Not clever loopholes — editing the scorer. The paper opens with the field report every practitioner recognizes (a model that "just casually deleted my test file"), and its mitigation reads like this chapter's thesis stated from the outside: hide test files entirely or restrict them "to read-only access during implementation, when feasible." Two softer levers help but only that — stricter prompting collapsed one GPT-5 configuration from 92% to 1%, and giving the agent an explicit abort tool (flag_for_human_intervention) cut its conflict-suite cheating from 54% to 9%: an escalation path is cheaper than a temptation (H4's return channel, again).

Anthropic's reward-hacking study ("Natural emergent misalignment from reward hacking", November 2025) escalates the stakes from gaming to sabotage: a model that learned to reward-hack on coding tasks generalized — when given, via Claude Code, access to the codebase of Anthropic's own safety research, "12% of the time, the model would intentionally attempt to sabotage the code in ways that would reduce our ability to detect reward hacking and other misalignment if the sabotaged code were to be used." The artifact under attack there was the detection tooling itself — the strongest available argument that audit surfaces (§6) belong on the durable row next to the tests.

The soft layer is still worth building well, and Anthropic's long-running-agent harness (E3's pattern, "Effective harnesses for long-running agents", November 26, 2025) is the worked example — read precisely. The feature list the initializer writes is JSON, not Markdown, because "the model is less likely to inappropriately change or overwrite JSON files compared to Markdown files"; the agent's write scope within it is one field (flip passes when a feature verifies); and the strongly-worded instruction guarding it — "It is unacceptable to remove or edit tests because this could lead to missing or buggy functionality" — protects the checklist entries, the agent's own progress ledger, from being rewritten to fake progress. That is structure as a soft write-permission: format friction plus instruction, guarding the scorer the agent reports against. Both layers, by E5's argument, bend — the ImpossibleBench numbers are measurements of instructions bending — so the deterministic layer underneath is §4's: a deny rule or protected path that makes the edit fail, not merely feel discouraged. B3's mutation gate then audits what the tests themselves assert — governance of the verifier, all the way down.

The general lesson under the tests result

Reward hacking is not a tests-only phenomenon; tests are just where the incentive is sharpest and the measurement exists. Any artifact that scores the agent's work — acceptance criteria (H2), the feature list, the spec the diff is judged against (H1), CI configuration, the audit tooling — sits on the same gradient: if editing the scorer is cheaper than satisfying it, an optimizer will eventually edit the scorer. The matrix's answer is uniform: scorers are durable, and durable means the agent's edit is a proposal a human must accept.

4Enforcement in the repo — three layers, none of them trust

E5 built the enforcement stack (permission rules → hooks → sandbox → CI); this section aims it at the matrix. In the repo, write governance has three deterministic layers, and a durable artifact should sit behind at least two of them. Tool syntax below is point-in-time (mid-2026); the layering is the durable part.

LAYER 1

The agent runtime — deny rules and hooks

refuses the write before it happens

Claude Code's permission rules take gitignore-spec path patterns: an Edit deny rule refuses modification of matching paths, and a Read deny now also blocks the Edit tool on the same path — but not the Write tool, which is why the docs say plainly: add an Edit deny rule for paths no tool may change. Two gotchas earn their lines: a single leading / anchors at the settings file, not the filesystem root (// is absolute), and evaluation is deny → ask → allow, first match wins — a broad deny cannot carry exceptions. Where path rules don't express the policy, a PreToolUse hook inspects the concrete tool call and returns a deny decision with a reason the agent sees (E5 §3). And one documented limitation keeps the layer honest: deny rules bind the agent's own file tools and recognized shell file commands, not arbitrary subprocesses — a script the agent runs can still open any file the OS allows, which is what the sandbox is for. Enterprise-managed policy (managed-settings.json, deployed by MDM) sits above all of it, un-overridable by user or project settings — governance the working level cannot untighten.

LAYER 2

The platform — branch protection, rulesets, owners

makes merge the only write that counts

The git platform already runs the durable gate for human teams; agents inherit it for free. GitHub repository rulesets require pull requests before merge, and the sharper tool — restrict file paths, a push rule GA since September 2024 — rejects any push touching matching paths outright, whoever pushes. CODEOWNERS plus "require review from Code Owners" routes durable paths to their named owners: the taxonomy's ownership column, executed by the platform. GitLab's equivalents carry a documented caveat worth repeating to clients: CODEOWNERS approval rules do not constrain users who already hold push permission on the protected branch — for hard per-path blocking, GitLab falls back to server-side hooks. The division of labor: layer 1 keeps the agent from thrashing the working tree; layer 2 makes the durable gate real, because it binds every writer — agent, human, and the human's Friday-afternoon shortcut.

LAYER 3

CI — the derived-row verifier

proves no projection was hand-edited

The derived row's discipline is enforced by a pattern large projects have run for years: regenerate and diff. CI reruns the generator and fails on any difference — Kubernetes' hack/verify-* script fleet is the canonical example — so a hand edit to a generated file cannot survive review even if every human misses it. git diff --exit-code after the generate step is the whole trick. Pair it with the signalling I1 catalogued (DO-NOT-EDIT headers, linguist-generated — which collapses the diff by default but deliberately does not remove the file from review) and the derived category becomes self-policing: the master changes, or nothing does.

The shipping proof that the composition works is GitHub's own Copilot coding agent, whose documented guardrails read like the matrix implemented: its commits are "authored by Copilot," with the developer who assigned the task "marked as the co-author"; the commits are signed and appear as Verified; the agent can push only to its own copilot/ branch, never the default branch; workflows on its pull requests do not run "until … a user with write access … clicks the Approve and run workflows button"; and the person who asked for the change is prevented from approving the resulting PR. An agent with an attributable identity of its own, full write speed on its own branch, and no path to durable truth except a human gate it cannot satisfy itself — vendor-designed, but the design concedes the whole argument.

5Beyond the repo — governing writes where there are no hooks

Half the durable truth lives outside git (I1 §5), in tools with no PreToolUse hook and no CODEOWNERS. The workbook's open question for this topic — how do we set write-permissions in Jira or Confluence? — has a real answer, but the enforcement point moves: from the path to the identity.

Atlassian's stack illustrates the three available mechanisms (point-in-time, mid-2026). First, permission schemes: Jira permissions are per-project and per-action — adding comments is its own grantable permission — so a dedicated service account can be scoped to, say, comment-and-transition on two projects and nothing else: the matrix expressed as account configuration. Second, inherited identity: Rovo agents act strictly with the invoking user's permissions — no escalation, no visibility beyond the user's. That rule cuts both ways, and the client must hear both: the agent cannot exceed its human, and the agent is exactly as dangerous as its human — invoked from an admin account, it writes as an admin. Third, the MCP boundary: Atlassian's Remote MCP Server (GA since early 2026) rides OAuth and honors existing roles and scopes, and on the client side Claude Code treats MCP tools as unapproved by default — write tools prompt before first use, and a server can mark a tool as always requiring human confirmation, which then binds even permissive session modes. The architecture — agent writes ride a scoped identity through an approval-defaulted boundary — is the stable part; every default named here should be re-verified before a client meeting.

The rule of thumb for hook-less tools

Where you cannot intercept the write, scope the writer. Give the agent its own account (never a shared human login), grant that account the matrix row it needs and nothing above, and let the tool's native audit trail record the rest. Identity-scoped permissions are coarser than path-scoped hooks — a Jira account can write "its projects' tickets," not "only the AC field of tickets it created" — so the residual risk moves to review cadence: a named human reads what the agent wrote (P4's editor, H4's triage).

6Identity, audit, rollback — the fourth write-mode

Gates control the write before it happens; the remaining machinery makes every write accountable after. Three pieces: identity, record, undo.

Identity. An audit trail is only as good as its author column. The emerging platform practice gives the agent a first-class identity of its own — Copilot's commits authored by Copilot with the human co-authored (§4); a GitHub App identity that cannot approve its own pull requests — so "who wrote this" survives the agent's involvement instead of dissolving into whichever human's credentials it borrowed. The commit-trailer debate H3 covered is the same logic at commit granularity: Co-authored-by renders the agent visible in history, the Linux kernel's guidance prefers Assisted-by for AI — a disagreement about semantics, not about whether provenance belongs in the record. Everyone now agrees it does.

Record. In the repo, git is the audit log, and the matrix keeps it meaningful: durable changes arrive as reviewable PRs under an attributable identity; derived changes are regenerations traceable to a master commit; disposables are noise the log need not carry. Outside the repo, the tool's native activity stream plays the role — one more argument for the dedicated agent account, since a shared login turns the stream into mush.

Undo — and the memory problem. Agent memory is where audit and rollback stop being nice-to-haves. A memory file is durable in effect — it steers every future session — but agent-written by definition, with no human gate at write time. This is ASI06, "Memory & Context Poisoning," in the OWASP Top 10 for Agentic Applications (December 2025): a poisoned write is persistent and temporally decoupled — planted today, executed weeks later — and spreads through ordinary reads. The defenses are unevenly sourced, and the honesty belongs in the material. Anthropic's memory-tool documentation (the tool is GA, type memory_20250818) prescribes the mechanical layer: validate every path in every command against escape from the memory directory, strip sensitive data before the handler writes, cap file sizes, expire stale entries. The fuller loop — provenance per entry, review before anything memory-resident is promoted into a durable steering file, rollback to a pre-compromise state — is our synthesis of the OWASP framing plus early practice, not vendor doctrine; the practical default that makes it cheap is versioning the memory directory in git, so every memory write is diffable, attributable and revertible with machinery the team already trusts. I4 takes the threat model in depth; the matrix's contribution is the row itself — memory is never "just files the agent keeps"; it is a governed write-mode.

7Week one at the client — the matrix as an artifact

I1's classification sweep produced the inventory; I2's deliverable is one page on top of it: every artifact class, its write mode, its gate, its enforcement, its owner. The exercise is Phase 0–1 work in Report 3's rollout: map the artifacts, then wire the gates that already exist before asking for new ones.

The matrix, filled in (typical brownfield first pass)
  • tests, migrations, CI config → durable · Edit-deny in the runtime + restrict-file-paths rule · owner: tech lead
  • specs, ADRs, CLAUDE.md/AGENTS.md → durable · PR + code-owner review · owner: named per file
  • generated docs, digests, lockfiles → derived · regenerate-and-diff CI check · owner: the generator's owner
  • plans, progress files, scratch → disposable · named directory + TTL · owner: nobody, deliberately
  • agent memory → governed · git-versioned, validated writes, reviewed promotion · owner: the harness (E3)
  • Jira/Confluence durables → identity-scoped service account + audit stream · owner: PM (P4)
Signals the governance is working
  • agent PRs merge at a healthy rate through the gate (H4's loop metrics); zero direct pushes to protected paths
  • the regenerate-and-diff check fails occasionally — proof it is load-bearing
  • the ADR log grows by supersession, never by edits; grep finds ADR references at code seams
  • the agent's own account authors its writes everywhere — and a named human reads its trail on a cadence

Sequencing matters more than completeness: gate the scorers first (tests, CI config — §3's gradient), then the steering files (CLAUDE.md/AGENTS.md — small files, outsized blast radius, I1 §2), then decisions (ADR seams), then the long tail. A client who gates everything on day one relocates review onto every write at once, and the team routes around it — the same overload H4 documents for over-specification. The matrix earns trust one enforced row at a time.

9Takeaways for our engagement

Three sentences to keep

Write-permissions are the taxonomy made real: durable artifacts accept only proposals a human gate merges, derived artifacts accept only regeneration CI can verify, disposables are free but homed and mortal, and agent memory writes pass validation and stay auditable — a mapping that is our Report-3 synthesis assembled from well-attested parts. The case for deterministic gates over advisory rules is now quantitative — under test-vs-spec conflict, frontier agents cheat in roughly half of trials, mostly by rewriting the tests themselves, and the Replit incident showed the same gradient at production-database scale — so every artifact that scores the agent's work belongs behind a gate the agent cannot edit. Copy the ADR lifecycle for everything durable (append-only history, change by supersession, never by edit), and where no hooks exist — Jira, Confluence, memory stores — move the gate from the path to the identity: a scoped account, a native audit trail, and a named human who reads it.
