0Why it matters

Every chapter of M4 so far has assumed something this one finally confronts: that the artifacts agents read are true. I1 classified them, I2 decided who may write them — I3 is about what happens between writes, when the codebase moves and the prose does not. Documentation drift is an old disease, measured long before AI sped anything up. What agentic development changes is not the drift itself but its consequence: documentation used to be output — read by humans who could squint at a stale README and discount it. Now it is input — loaded into an agent's context at the start of every session and acted on without skepticism. A practitioner put the asymmetry in one line in July 2026: "A human reads a stale README and thinks 'hmm, that looks old.' An agent reads your AGENTS.md and acts on it." The academic version arrived a month earlier, with data: stale descriptions "can degrade AI assistance without any visible error" (Treude & Baltes, June 2026). No exception is thrown; the agent simply builds on a fact that stopped being true — and the code it generates makes the prose one change further behind. The countermeasure is E5's move applied to documentation: stop asking people (or agents) to remember to update the prose, and make consistency a failing build — docs-as-tests, the deterministic gate on the one artifact class everyone still governs by good intentions.

1Drift predates AI — the measured baseline

It would be convenient to blame agents for documentation rot, but the baseline was measured before they arrived and it is bad. Tan, Wagner and Treude built DOCER, a detector for outdated code-element references — a method or file named in the documentation that has since been deleted from every source file — and ran it across the most popular GitHub projects ("Detecting Outdated Code Element References in Software Repository Documentation", arXiv:2212.01479, December 2022; journal version in Empirical Software Engineering, 2023). The result: 28.9% of projects (265 of 918 analyzable) contained at least one outdated document, and the stale references had been wrong for 4.7 years on average. These are the industry's flagship repositories, and the definition is conservative — it counts only references that provably point at nothing, not descriptions that are merely misleading.

The mechanism behind that number is older still. Fluri, Würsch and Gall asked "Do Code and Comments Co-Evolve?" (WCRE 2007) across ArgoUML, Azureus and Eclipse's JDT Core, and found that 97% of comment changes happen in the same revision as the associated code change — but newly added code, despite its growth rate, barely gets commented in the first place. Translation: prose that is going to be maintained gets maintained with the change or never. There is no later. Every process that schedules documentation updates "for after the release" is scheduling them for the 4.7-year bucket.

The phenomenon has a name in the technical-debt literature — documentation debt is a recognized debt type in the standard taxonomies (Alves et al.'s technical-debt ontology; Rios, Mendonça-Neto & Spínola's 2018 tertiary study in Information and Software Technology, which catalogues it among fifteen types and maps it to SDLC phases). Naming matters here for one reason: it moves stale docs from the moral register ("we should be better") into the engineering register ("a liability with a principal, an interest rate, and repayment options" — §6).

And the stakes were quantified before AI raised them. DORA's 2022 State of DevOps report made quality documentation a headline capability and found it does not merely add value — it multiplies the value of everything else: "In the presence of quality documentation, technical practices help elevate an organization's performance… This interaction and impact applies to every single technical capability we studied in 2022." The published per-capability lifts for teams with above-average documentation are startling — trunk-based development's contribution to organizational performance amplified 1525%, continuous integration 750%, continuous delivery 656% (dora.dev, documentation-quality capability page). A widely-circulated "2.4× more likely" figure floats around secondary blogs; we could not pin it to any primary DORA text, so it does not appear in this workbook — the amplification framing and the published lift percentages are the citable versions.

2What agentic development changes

Three things change when agents enter the loop, and together they turn a chronic condition into an acute one.

Docs became input

The context files that steer coding agents — CLAUDE.md, AGENTS.md, copilot-instructions.md, .cursorrules — are documentation that is consumed mechanically. The population is no longer marginal: an exploratory study of agentically-engineered repositories (Galster, Mohsenimofidi, Lulla, Abubakar, Treude & Baltes, AIware 2026) identified 4,768 context files across 2,586 of the 2,853 repositories in its sample — 90.6% — with CLAUDE.md the most common (34.4%), then AGENTS.md (31.6%) and copilot-instructions.md (29.2%), and found the files substantive rather than mere pointers. Freshness in these files is not cosmetic: a companion study (Lulla et al., JAWs 2026 @ ICSE, arXiv:2601.20404) associates the mere presence of AGENTS.md with roughly 29% lower median agent runtime and 17% lower output-token consumption at comparable task completion, across 10 repos and 124 PRs — the file is doing real work every session, which is exactly why it must not lie.

And it does lie, at a measurable rate. Treude & Baltes ("Context Rot in AI-Assisted Software Development", arXiv:2606.09090, June 2026) took DOCER — the same detector from §1's baseline, unmodified — and pointed it at agent config files: 23.0% of repositories in a 356-repo representative sample carried at least one stale code-element reference in their agent configuration (95% CI 18.8–27.2%; 230 stale references across 612 config files; the authors read it as "a feasibility signal rather than a precise prevalence" after manual inspection rated 64% of flagged cases genuine). The examples are mundane and deadly: a CLAUDE.md pointing at a function renamed away in a UI rewrite; an AGENTS.md referencing a type-definition path that no longer exists; a copilot-instructions file mandating a library the build had already dropped. Their definition is the one this chapter runs on: "We call this context rot: the gradual divergence between what a configuration file says about a codebase, its tools, its architecture, or its conventions, and what actually holds."

"Context rot" — two senses, one lesson

The term originates with Chroma's July 2025 report ("Context Rot: How Increasing Input Tokens Impacts LLM Performance", Hong, Troynikov & Huber — the study behind I1's context-hygiene rule): across 18 models, "model performance varies significantly as input length changes, even on simple tasks" — degradation inside the context window. Treude & Baltes explicitly extend the term to repository-versioned artifacts: the file itself rotting relative to the codebase. The two compound: a bloated config file degrades attention (Chroma's sense) and a stale one injects falsehoods (the new sense). Both argue for the same discipline — short, load-bearing, verified context files.

The loop this closes is the reason I3 outranks its pre-AI ancestor in urgency. Stale doc → wrong context → wrong code, merged at agent speed → codebase and prose drift further apart → the next session loads a slightly more wrong context. Nothing errors. The practitioner framing from §0 completes it: the agent "runs the build command you renamed three sprints ago, skips the guardrail you added last week, follows the convention you already abandoned. Confidently." For an agent, a stale cache is worse than an empty one — absence triggers exploration, staleness triggers confident wrongness.

Docs get produced at AI volume — and read polished

The second shift is on the output side. Agents produce documentation effortlessly — I1 measured the sprawl reflex — and the output reads authoritative regardless of whether it is true. A practitioner essay titled "Impressive, But Wrong" names the failure honestly: "The problem isn't just bad information — it's false confidence." Hand-written bad docs usually look bad; generated bad docs look like good docs. Review effort concentrates exactly where the surface gives no signal — which is why §5 insists AI-generated documentation carries a label, a timestamp and an expiry, and why I1 routes it through the derived category (regenerate, never hand-patch) or an explicit promotion gate.

LLMs cannot police their own drift — naively

The tempting fix — "have the agent check whether the docs match the code" — fails in a specific, measured way. DocPrism (Xu, Wahab, Holmes & Lemieux, UBC, arXiv:2511.00215, October 2025) found that naively-prompted LLMs "identify natural gaps between high-level documentation and detailed code implementations as inconsistencies" — abstraction itself gets flagged as drift, drowning the real signal: their baseline flagged 98% of documentation as inconsistent, against 14% with their filtered approach (accuracy 14%→94%, precision 0.62 across four languages). The architecture that works is a division of labor both papers converge on: a deterministic detector finds the stale element (DOCER's check is "a lightweight two-snapshot git grep procedure" that "needs no new tooling and runs in continuous integration"), and the LLM proposes the repair — inspecting git history to suggest the updated reference — with a human gate on the merge (I2). Detection is a grep; judgment is the expensive part; spend the model on judgment.

The organizational stakes were restated by DORA's 2025 report, which reorients the whole research program around AI and lists "AI-accessible internal data" among its seven capabilities that magnify AI's positive impact: "AI doesn't fix a team; it amplifies what's already there. Strong teams use AI to become even better and more efficient. Struggling teams will find that AI only highlights and intensifies their existing problems." Internal documentation is the AI-accessible internal data — and amplification runs both directions. A shop with rotten docs that adopts agents is amplifying rot.

3Docs-as-code — the substrate

None of §4's checks can run against documentation living in someone's head, a wiki nobody exports, or an attachment named final_v2. The precondition is docs-as-code — Write the Docs' canonical definition: "a philosophy that you should be writing documentation with the same tools as code" — issue trackers, version control, plain-text markup, code review, automated tests. For engineering durables this is nearly free (the docs sit next to the code they describe); for product durables mastered in Notion or Confluence it is the SSOT-boundary problem I1 §5 handled — the bridge is a pointer or a labelled derived digest, never a fork.

The canonical industrial statement is Software Engineering at Google, chapter 10: "Documentation is often so tightly coupled to code that it should, as much as possible, be treated as code" — owned by someone, under source control, changed through review, its bugs tracked like code bugs, and — the part most shops skip — measured for freshness. Google's g3doc pages carry last-reviewed metadata, "and metadata in the documentation set will send email reminders when the document hasn't been touched in, for example, three months." Microsoft Learn runs the same discipline as editorial policy: contributors bump an ms.date field on substantial edits or freshness reviews, and the date is surfaced to every reader. Note what these precedents are and aren't: review-date metadata plus a reminder loop — a nudge, not a gate. They keep humans honest about age; they do not stop a stale doc from shipping. That is the right division: freshness metadata is the advisory layer (cheap, universal), §4's checks are the deterministic layer aimed at the failures that matter most.

The same discipline now has an official restatement for agent context files. Anthropic's Claude Code best-practices page: "Treat CLAUDE.md like code: review it when things go wrong, prune it regularly, and test changes by observing whether Claude's behavior actually shifts." The page's exclusion list is drift-avoidance stated as policy — keep out "information that changes frequently" and "detailed API documentation (link to docs instead)": volatile facts don't belong in an always-loaded file, because every volatile fact is a future stale fact. (It also warns that "bloated CLAUDE.md files cause Claude to ignore your actual instructions!" — Chroma's sense of context rot, from the vendor's own docs.) Treude & Baltes close the loop with the review rule: treating configuration files as code, "reviewing them alongside refactors that rename or delete elements, prevents much of this rot at the source" — the same-PR discipline §1's Fluri result predicted in 2007: with the change or never.

4Docs-as-tests — consistency as a failing build

E5 drew the line this workbook keeps returning to — and the vendor docs now state it verbatim for context files: "Unlike CLAUDE.md instructions which are advisory, hooks are deterministic and guarantee the action happens." Applied to documentation: "keep docs updated" is advisory and bends under deadline pressure exactly like every other advisory rule; a CI check that fails the build on a stale reference is deterministic and does not. The term of art is docs-as-tests — Manny Silva's formulation (book: Docs as Tests: A Strategy for Resilient Technical Documentation): documentation "shouldn't just inform — it should also verify," treating each piece of documentation as a test case executed against the product. The checks form a ladder, from trivial to ambitious:

Rung 1 is older than the docs-as-code movement and proves the concept ships in the standard library: Python's doctest (Tim Peters, standard library since the early 2000s) runs the examples embedded in docstrings; Rust made it a default — cargo test compiles and runs every documentation example, so a Rust crate's examples cannot silently rot; Elixir's ExUnit.DocTest generates tests from @doc examples. Rung 2 is the cheapest high-value check for prose: lychee (an actively-maintained Rust link checker with a first-class GitHub Action) fails the build on dead links and file references; DOCER's two-snapshot git grep does the same for code-element references — including, per Treude & Baltes, in CLAUDE.md and AGENTS.md, where §2 says it matters most. Rung 3 is Silva's own tooling: Doc Detective (open source, actively released) executes documented procedures — click this, run that, expect this — against the product. Rung 4 makes the API spec an executable contract: Schemathesis generates property-based test cases from an OpenAPI/GraphQL schema (independent evaluation in an ICSE 2022 paper; adopter lists are vendor-reported); Specmatic runs the spec as tests against the provider and as a stub for consumers, catching both sides of the drift. Rung 5 is the agentic frontier and deserves precision: Kiro's "Sync Files" action updates task lists from changed requirements on request, and its agent hooks can regenerate docs on save — useful, but user-initiated sync, not automatic drift detection; third-party posts that call it "drift detection" overstate what the vendor documents (H3 made the same correction).

Point-in-time warning — this shelf churns faster than most

Half the tools once cited for this chapter died while the chapter was being researched: Dredd (the original OpenAPI contract tester, out of Apiary/Oracle) is archived; Optic was acquired by Atlassian in April 2024 and its repo archived in January 2026; driftctl has been in maintenance mode since mid-2023; DeepDocs — a 2025 "AI agent that keeps your docs in sync" GitHub app — already carries a "this app has been deprecated" notice; Swimm, whose patented Auto-sync (code-coupled snippets flagged and auto-patched when the referenced code changes) remains the best-documented mechanism of its kind, pivoted the company to agentic-modernization services. All tool facts here are mid-2026 and need re-verification before a client meeting. The durable content is the rung, not the logo: a deterministic check in CI that executes something the documentation claims.

The pattern has a decade of prior art one floor down the stack. Infrastructure-as-code treats drift — the live system diverging from the declared configuration — as a first-class, detectable condition: terraform plan computes it on demand, and HCP Terraform's Health Assessments re-check it automatically about every 24 hours with refresh-only plans. Nobody in that world proposes fixing drift with a "please keep the infrastructure matching the config" guideline. Documentation is a declared configuration of understanding; it deserves the same machinery.

5A freshness policy for the AI era

Flag: the policy packaging is our synthesis

Each ingredient below is independently attested — Google's review-date reminders, Microsoft's ms.date, Anthropic's prune-and-test guidance, DOCER's CI check, I1's derived rules. The packaging into one per-category freshness policy with thresholds is Report 3's synthesis, not an industry standard — present it as ours, like the taxonomy and the write-permission matrix it extends.

Detection (§4) needs a policy to enforce. The taxonomy already tells us freshness means something different per category, so the policy writes itself as a third column on I1's table — what "fresh" means, and which check proves it:

Two design choices deserve defense. First, expiry means "suspect", not "auto-delete" — H3's semantics: an expired AI-generated doc stops being citable context until a human revalidates it, which is cheap insurance against "impressive but wrong" surviving on its polish. Second, the agent-config row gets the strictest contract — zero stale references, checked every build — because §2 showed it is the highest-leverage rot in the system (loaded every session, trusted completely, measured at 23% of repos) and because the check is nearly free: the references are code elements, and grep is grep.

The client's Phase-1 question — "what staleness threshold triggers an alarm, and can we measure it from day one?" — now has a concrete answer derived entirely from git: (a) stale-reference count in agent config files (alarm at >0 — cheap to hold at zero); (b) dead links/references across durable docs (lychee's count — alarm on increase, burn down the backlog); (c) review-interval breaches per durable (alarm per owner, not per repo); (d) the age gap between each durable doc's last commit and the last commit to the code it describes (a rough but automatable early-warning signal — a doc directory untouched for a year while its module churned weekly is §1's 4.7-year bucket forming in real time). None of these needs tooling the client doesn't already have.

6Documentation debt — triage, not guilt

Everything §4–§5 catch lands in a pile, and a brownfield client's pile is years deep on day one. The technical-debt frame earns its keep here because it forbids both bad responses: pretending the pile isn't there, and declaring bankruptcy ("the docs are useless, ignore them"). Fowler's quadrant (bliki, October 2009) — deliberate vs. inadvertent, prudent vs. reckless — sorts documentation debt as well as it sorts code debt. Prudent-deliberate: "we're shipping without updating the runbook; ticket filed, owner named" — debt as a financing decision. Reckless-inadvertent: the 23% of §2 — nobody decided anything; the drift just accrued, silently, in the files agents trust most. The entire point of §4's machinery is to convert inadvertent debt into deliberate debt: once a stale reference fails a build, ignoring it requires a decision someone owns, and I2's audit trail records who made it.

Triage per drifted artifact is I1 §6's promotion/demotion logic, applied at volume — three honest exits: delete or demote (the doc describes a world that no longer exists and nobody needs the history — most of the pile); regenerate (it's derived, or should be: fix the master, rerun the generator, and add the regenerate-and-diff check so it can't rot again); rewrite and own (it's genuinely durable: validate it, name an owner, stamp a review date, put future changes behind the PR gate). The state to refuse is the one I1 already refused: half-trusted, unowned, hand-patched — the middle state agents keep reading and compounding.

Agents themselves are the labor force that makes repayment tractable — with the division of labor §2 established. The deterministic detector produces the worklist (stale references, dead links, expired stamps); the agent drafts the repair, using git history to find what the reference became (DocPrism's filtered architecture and Treude & Baltes' proposed hybrid are both exactly this shape); a human gate merges, because every one of these repairs is a write to a durable artifact and I2's matrix does not care that the proposer was fixing documentation. M5 extends the same pattern to generating missing docs for legacy code (agent archaeology) — with the I1 rule that generated legacy docs are derived until a human validates and adopts them. What agents must not do is the naive sweep — "review all docs and fix anything outdated" — which DocPrism measured: 98% of documentation flagged, the real signal drowned, and a mountain of confident rewrites for the human gate to review. Point the agent at a grep hit, not at the corpus.

7Week one at the client — the drift audit

I3's week-one move rides on I1's classification sweep (which listed the artifacts) and I2's matrix (which assigned the gates); it adds the freshness column and the first checks. Concretely, in order of effort-to-payoff:

Do in week one
  • run lychee across the repo's markdown; count, don't fix — the number is the baseline
  • run the two-snapshot grep on CLAUDE.md / AGENTS.md / copilot-instructions; fix these immediately (small, high-leverage, §2)
  • wire both checks into CI as warnings; flip to failing once the baseline burns down
  • stamp every durable doc: owner, last-reviewed, interval (a header template, one PR)
  • label + timestamp + expiry on anything AI-generated the team already trusts
  • write the same-PR rule into CLAUDE.md/AGENTS.md stewardship: renames and deletes update the config files that mention them, in the same PR
Refuse in week one
  • "we'll keep docs updated going forward" as the plan — advisory; it is how the client got here (E5)
  • an LLM sweep over the whole doc corpus — the 98% flag rate is the measured outcome
  • a doc-sync tool purchase as the first move — §4's graveyard; principle first, vendor later, if ever
  • rewriting the wiki — triage (§6), don't refinance everything at once
  • contract tests where no OpenAPI spec exists yet — rung 4 needs H1/H2's groundwork first
SDLC placement
  • drift checks live in CI (the E4/E5 verification loop), freshness review lives in the maintenance cadence, and the debt worklist feeds sprint planning like any other debt — documentation stops being a separate hygiene project and becomes part of the same delivery loop as code

9Applying it — decision guide

Three sentences to keep

Documentation drift was already endemic before AI — 28.9% of flagship repos carried references stale for 4.7 years on average — but agents change its physics: docs became input, loaded every session and acted on without skepticism, so a stale line no longer decays quietly but is re-injected into generation "without any visible error," and the config files agents trust most are measurably rotten in about a quarter of repositories. The countermeasure is E5's move applied to prose: freshness metadata and same-PR review as the advisory layer, and docs-as-tests as the deterministic one — executable examples, reference and link checks (a two-snapshot grep covers CLAUDE.md and AGENTS.md), docs exercised against the product, and contract tests from the spec — each a build-failing check, because "keep docs updated" is how every client got here. Triage the resulting pile like the debt it is — delete or demote, regenerate, or rewrite-and-own; never the unowned middle — with deterministic detectors producing the worklist, agents drafting repairs, and humans gating the merge, since a documentation fix is still a write to a durable artifact.
