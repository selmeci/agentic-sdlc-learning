0Why the design layer needs its own bootstrap order

Every brownfield engagement opens with the same paradox (B1): the system that would benefit most from an agent's speed is precisely the one where an agent is least safe, because nothing exists to verify the agent's work. Module M5 answered the paradox for code — read first (B4), characterize before you change (B2), gate before you generate (B3), roll out in measured phases (B6). This chapter is the same argument, run once more on the visual layer. A brownfield client almost never has the artifact module M6 presumes: there is no token JSON in git, no component schema, no visual baseline — there are years of accumulated CSS, drifted components, and values hardcoded by people who left. D1–D5 describe how to govern a design system with agents in the loop. D7 is about how that system comes to exist on brownfield at all, and in what order you may safely grant an agent access to it.

The ordering is not negotiable, and it is identical to Report 4's: read before write, characterize before change, gates before generation, pilot before scale. What makes the design layer different is only what gets read and what gets characterized. The code estate hides its behaviour in execution paths; the visual estate hides it in fifty stylesheets and a Figma file nobody owns. And the estate is always worse than its documentation: the style guide says eight brand colors, the shipped CSS contains forty-seven. This gap is not cosmetic for an agentic engagement — agents amplify whatever context they are given. Point a generator at a codebase with forty-seven de-facto colors and it will cheerfully invent the forty-eighth, lint-clean and on-brand nowhere. The de-facto system is the system: whatever ships is what users depend on (Hyrum's Law holds on pixels as much as on APIs — B5) and what every agent will learn from. Governing it starts with measuring it.

That measurement-first posture is also the commercial move, as D1's selling chapter already noted: the first-meeting narrative is "we start by measuring what you already have — nothing is written until there is a net to catch it." A client who hears an audit before a pitch trusts the rest of the rollout; a client who hears "we will rebuild your design system with AI" hears a big-bang rewrite and prices the risk accordingly. F0's outputs are deliberately useful even if the engagement stops there — the audit is the deliverable, the agent pilot is the optional sequel.

The one idea to internalise

You cannot govern what you have not measured, and you cannot let an agent change what you have not characterized. Design archaeology — a read-only excavation of the de-facto visual system from shipped CSS, rendered pages and component inventories — is the design layer's characterization test: it documents how things are, so that every later agent-driven change can be diffed against it. Everything in F1–F4 is built on, and only on, the artifacts F0 produces.

1Design archaeology: the characterization parallel

B2's characterization tests pin how legacy code behaves, not how it should behave; the design-layer equivalent is a pair of artifacts: a golden-master visual baseline (visual-regression snapshots of the UI as it renders today) and a de-facto token extraction (the colors, spacings, radii, shadows and type the shipped CSS actually uses, with usage counts). Together they play the exact role Feathers' seam plays in code: a change detector wired around a system you did not write and do not yet understand. Their purpose is protective, not aspirational — once the baseline exists, any agent-driven change that silently alters the visual status quo fails a deterministic check instead of reaching a user. That is the answer to this topic's self-check: the design-layer characterization test documents how things ARE, and it protects against quiet visual drift at agent speed.

The practice is much older than agents, and the lineage matters because it makes our pitch credible rather than novel. Brad Frost named the core move in 2013: the interface inventory — "a comprehensive collection of the bits and pieces that make up your interface" — built by screenshotting every distinct treatment of each component (not every instance) and laying the variants side by side, until the organization sees its seventeen button styles and, in his words, cries realistic tears. Frost's key distinction survives unchanged into the agent era: learn to separate intentional from unintentional deviation — the first is brand, the second is debt, and only an inventory tells them apart. Nathan Curtis turned the same move into consulting method: "Starting a Design System" (2017) opens every engagement with as-is system reviews of design assets, code libraries, standards documentation and governance; his token-taxonomy practice (2022) literally says "excavate tokens" — from Figma styles, Style Dictionary configs, component-repo variables, doc sites — logging every rogue value in a tracking spreadsheet, because "design system teams are usually unaware of these additional layers of transformation… This is why we audit." Alla Kholmatova's functional/perceptual pattern split (2017) supplies the vocabulary that later became tokens. And the retrofitting framing is documented too: a 2019 24 ways article already positioned extracting "the design system you already have" from a live site as the brownfield first step. Design archaeology is an old technique with a new urgency: what agents changed is the cost curve in both directions — the audit got cheap (§2), and skipping it got dangerous (§7).

What F0 produces (read-only)
  • de-facto token report — extracted values with per-value usage counts; the counts are the consolidation map
  • interface inventory — components, duplicates, drift; intentional vs unintentional deviation marked
  • golden-master baseline — visual snapshots of the current state, committed to git
  • agent-readiness heatmap — machine-legibility × brand risk per area (§4); the pilot shortlist
What F0 never does
  • no writes to product code or styles — read-only like B4's archaeology
  • no token "cleanup" before the baseline exists — consolidation is F1
  • no generation pilots — the first agent output waits for F2's gates
  • no promises about the new system — the audit reports what is, not what will be

2The F0 audit toolbox

The audit is mechanized — what Frost did with screenshots and Keynote, tooling now does from the shipped artifact itself. The anchor is Project Wallace's CSS Design Tokens Analyzer: paste a URL or raw CSS and it statically extracts the de-facto token set — colors, font sizes, radii, shadows, easings — emitting DTCG-format JSON ($type/$value/$extensions), and every token carries a com.projectwallace.usage-count extension. The counts are the point: "47 colors, but 80% of usage in 6 of them" is the consolidation map for F1, and the project's own homepage makes the same joke about "47 shades of gray." Two honesty notes: values the spec cannot express are emitted as type-less tokens (valid DTCG, no $type), and the package is pre-1.0 though actively maintained (v0.11.x as of July 2026 — point-in-time). Its natural partner is Dembrandt, an open-source CLI that renders a live site via Playwright, reads computed styles, and exports tokens (--dtcg) — and then closes the loop F1 needs: --compare baseline.json diffs the live site against a committed baseline and exits 1 on drift with a machine-readable report (a 0–100 drift score, per-category summaries, per-token before/after changes). Its exit codes are a small lesson in gate design: 0 stable, 1 drift, 2 extraction failure, 67 navigation timeout — the gate can tell "the design changed" apart from "the tool broke," which is exactly the distinction B3's mutation gate enforces on the test side. --approve blesses a new baseline deliberately. Young tooling (first released November 2025) but very active, MIT-licensed; note the commercial seam — comparing against a cloud baseline routes to the vendor's paid platform, so flag it as freemium.

Around the anchor pair sit the triangulation tools, and they carry the chapter's first caution. CSS Stats (colors, font stacks, specificity graphs from any URL) is live but its repo has been dormant since 2023; Superposition (the extract-tokens-from-a-live-site desktop app from the 2019 retrofitting lineage) is online with no verifiable releases since ~2020. Both still work for a one-off cross-check; neither belongs in a client's CI. On the codebase side, csstokens scans CSS/SCSS/TSX sources for repeated values and emits a token file plus a refactor plan — closer to a brownfield repo audit than the website-side extractors — and TokenLens audits a live URL against an intended token set and scores compliance over time, which is already an F3-shaped metric. All statuses below verified July 2026; this tool class churns fast (§3 has the cautionary obituary).

3The golden-master visual baseline

Tokens are half the characterization; the other half is pixels. The golden-master baseline is a set of visual-regression snapshots of the UI as it renders today — committed to git before any agent touches a stylesheet, so that every later diff is measured against the state users actually depended on. This is B2's golden-master master move with no adaptation needed, and the baseline discipline is D5's: pin the render environment, store baselines in the repo, update them deliberately and review the update, mask dynamic regions and freeze clocks before touching thresholds, track flake without letting it block. What the archaeology phase adds is scope: the baseline starts with the screens the heatmap (§4) marks as pilot candidates plus the highest-brand-risk surfaces, not with the whole estate — coverage grows in F3 with the strangler migration (§6).

Two honesty notes belong with this table. First, the fatigue vendors sell against is real — a golden-master gate that fires on noise trains humans to approve-all, which is why D5 insists on masking and thresholds before volume. Second, the landscape churns: Lost Pixel was archived in April 2026 with no shutdown announcement, which is why the baseline discipline (baselines in git, renderer pinned) matters more than any vendor choice — the snapshots outlive the tool.

4The agent-readiness heatmap

F0's last artifact answers the question every rollout gets wrong by intuition: which area pilots first? The instrument is the design-layer version of B5's strangler heatmap — same logic, different inputs. Score every candidate area on two axes. Machine-legibility: does the area already have what an agent needs to be safe — Storybook stories, tokens in code, Code Connect mappings, CI that would catch a regression? Brand risk: what does it cost if the agent errs here — a settings page nobody loves, or the checkout? Pilot where legibility is high and brand risk is low; the checkout with no stories and no tests is where pilots go to die, and it is also the exact profile of the F0 STOP condition (no harness possible AND brand risk high). Like B5's criticality gate, the axes are exclusionary, not averaged: a high brand-risk area does not become a pilot by scoring well on everything else.

The dimensions are not invented — each has documented practice behind it, and assembling them into a 2×2 is our synthesis, flagged as such. For the legibility axis, Ben Callahan's design-system maturity model (Sparkbox) supplies per-stage "what to measure" lists — adoption, variance from documented patterns, accessibility regressions, release-adoption lag — i.e. coverage and conformance metrics by area; Nathan Curtis ran a per-team 0–4 adoption scorecard with named stakeholder leads (UXLx 2020), which is a pilot-selection instrument; Shopify's stylelint-polaris ships 40+ rules in ten coverage categories explicitly "to measure the Polaris design system's coverage" — lint output doubling as the adoption metric; and component analytics (Omlet) operationalize Curtis's dictum to measure who is using which components how, not just how much. For agent-legibility specifically, the 2026 checklists have arrived: Into Design Systems' "Your Design System Is Not Ready for AI Agents" (April 2026, conference-sourced — flag as practitioner claims) names five failure modes that read as a readiness audit — documentation drift across docs/tokens/components, Markdown dumped into MCP without benchmarking (Indeed's 77 components: JSON metadata beat Markdown at ~80% fewer tokens, per D3), no trust levels per agent action, no always-on foundation rules, monolithic component definitions; and zeroheight's 2026 report lists "AI & LLM readiness" among the industry's biggest tooling gaps. The honest framing: a design-system heatmap scores human adoption; the agent-readiness overlay scores whether a machine can navigate the area — both are needed, and F0 measures both.

5The F0–F4 rollout

The roadmap assembles everything above into phases, and each phase grants a D-level rather than declaring one — D6's scale applied as a permission ladder, the exact analogue of B6's F0–F3 engineering spine. In fact the two plans share structure deliberately: B6's framing is that F0–F3 is the spine and every other report's staged rollout is rails on the same four permission tiers — this chapter is the design rail, with F4 appended because the design layer's top rung (D4, agent-proposed system changes) has no engineering counterpart in B6's scope. The gate discipline is B6's too: movement between phases is earned by measured exits, and demotion on a failed gate is automatic and unembarrassing.

6Strangler fig on the system itself

F3's migration mechanics deserve their own section, because "never big-bang" is easy to say and the how is usually missing. Start with honesty: "strangler fig applied to a design system" is our term — no published source names the pattern for the visual layer. What is documented, by named systems, is every mechanic the pattern needs. Coexistence with adapter shims: Microsoft's Fluent UI officially supports running v8 and v9 side by side and ships @fluentui/react-migration-v8-v9, a shim that renders v9 internals behind the v8 prop interface — B5's anti-corruption layer in design clothes, letting product teams migrate screen by screen. Dual-running: Morningstar shipped old and new versions of a component side by side for six months until removal at the next major (via Curtis). Versioned grace windows: IBM Carbon's rule that tokens deprecated in one major keep working through the next and are removed in the one after. Lint-enforced adoption: Pinterest's eslint-plugin-gestalt progressively forbids legacy patterns — the envelope is enforced by the compiler and CI, not by announcements, which is exactly E5's deterministic-over-advisory principle. The new tokenized system envelops the old the same way: new work imports from the system, lint denies the legacy paths, and the old surface shrinks area by area until it can be deleted.

The deprecation machinery is equally documented, and it is what makes the envelope finishable instead of permanent. Shopify Polaris publishes a five-stage component lifecycle (Alpha → Beta → Stable → Legacy → Deprecated) with explicit notice rules — deprecation announced at least a month ahead, migration paths documented and available before removal. Nathan Curtis's "Versioning Design Systems" (2018) is the canonical text: SemVer as the only versioning vocabulary that survives contact with consumers, tokens versioned as their own package, and published deprecation windows — Salesforce Lightning's 18 months, FT Origami's 3–6 months, Atlaskit's six-step end-of-life process. Salesforce even automated the warning phase: sass-deprecate makes the Sass compiler emit warnings for selectors scheduled for removal at the next major. And because breaking changes are inevitable, mature systems treat migration as a codemod problem: Shopify ships polaris-migrator with every major, and Style Dictionary upgrades arrive with codemods. An agent in F3–F4 consumes exactly these artifacts — a deprecation is a machine-readable event (SemVer bump + codemod + lint rule), which is why a governed system can let an agent propose one at D4 while an ungoverned one cannot.

Why must archaeology come first? Spotify's Encore is the cautionary tale: consolidating twenty-two design systems into one, the team discovered they could not safely change Spotify Green — the most load-bearing token in the company — because nobody knew where and how it was actually used; they had to build usage telemetry and a semantic token layer before the token could move. That is F0's de-facto token report with usage counts, learned the expensive way. The scale cases run the same order: Pinterest migrated roughly two thousand hardcoded colors onto Gestalt tokens (first-person account by the initiative's co-lead) and GitHub moved GitHub.com onto Primer's functional color tokens behind published migration tables — extraction and remap first, enforcement second. Our "3-phase deprecation" shorthand (announce/warn → dual-run → remove) is synthesis assembled from the sources above; say so when a client asks.

The envelope is enforced by lint, not by memos

Every documented success enforces the migration in tooling: Gestalt's ESLint plugin, Primer's no-override stylelint rules, Polaris's migrator codemods, Carbon's compiler warnings. The announcement is the advisory layer; the deny-rule is the deterministic one (E5). When you install the strangler at a client, the first import restriction pointing at the new system is the migration start date — everything before it is preparation.

7The risk ledger

Three risk classes stand between the pilot and the rollout, and each is measured rather than hypothetical. Accessibility first, because it breaks silently and now carries legal weight. The peer-reviewed anchor is CodeA11y (Mowar et al., CHI 2025): in the formative study, 0 of 16 developers prompted with accessibility in mind — including the two who knew accessibility — and without explicit prompting the assistant rarely produced accessible markup; participants pasted empty alt placeholders straight through and deleted <label> elements. Frame it precisely: the paper does not claim AI always degrades accessibility — it shows assistants omit accessibility unless asked, and that novices never ask; the same team's follow-up showed a guardrail extension significantly improving output, which is the harness argument in miniature. The backdrop moved the wrong way meanwhile: WebAIM's 2026 Million found 95.9% of home pages with automatically detectable WCAG failures, up from 94.8% and reversing six years of slow improvement, with WebAIM itself attributing part of the regression to AI-assisted "vibe coding." And since 28 June 2025 the European Accessibility Act applies to e-commerce, banking, transport and e-books sold in the EU — with the nuance consultants get wrong: the directive does not name WCAG; the presumption of conformity runs through EN 301 549, whose current version maps to WCAG 2.1 AA. An agent shipping unverified UI in F2 accumulates legal exposure at machine speed; the F1 axe gate is not optional.

Second, the quality and security profile of generated code. Two independent vendor studies — label both as such. CodeRabbit (December 2025; 470 open-source PRs, authorship inferred from signals, not ground truth): AI-co-authored PRs carry 10.83 issues per PR vs 6.45 human (~1.7×), and 2.74× more XSS vulnerabilities specifically — note that the sharpest finding is a front-end vulnerability class, i.e. this module's home turf. Veracode's 2025 GenAI Code Security Report (July 2025; 100+ LLMs, 80 tasks): 45% of AI-generated samples introduced OWASP Top 10 vulnerabilities — worst in Java at 72%, JavaScript at 43% — and, the load-bearing detail, security performance was flat across model generations even as syntactic correctness improved, with the Spring 2026 follow-up reporting no improvement. The strategic conclusion: gates are permanent architecture, not scaffolding you remove when models get smarter — capability growth is not judgment growth (D6).

Third, the injection surface — where D7 meets M7. Design tools are MCP clients and design files are content, and 2025 supplied the canon. The checklist backbone now exists: OWASP's Top 10 for MCP (beta, mid-2026 — point-in-time). The incidents: Invariant Labs' GitHub MCP "toxic agent flow" (May 2025) — a malicious public issue coerced an agent into leaking private-repo data, with no CVE because the server worked as specified; the runtime is the vulnerability. The first malicious MCP package in the wild (postmark-mcp, September 2025) hid a BCC-to-attacker in a routine update. CVE-2025-49596 (CVSS 9.4) was an unauthenticated RCE in the MCP Inspector — even the ecosystem's reference developer tool shipped insecure by default. And the design-specific entries: CVE-2025-53967, a command-injection RCE in the popular community figma-developer-mcp server (fixed in v0.6.3), and Figma-Context-MCP issue #303 (March 2026) documenting that text layers, comments and hidden layers — 0% opacity, off-canvas, 1px fonts — flow into agent context unsanitized: any collaborator in a shared Figma file can plant an instruction the agent will read. The F0 consequence is concrete: the audit inventories not just tokens and components but trust — who can edit the libraries, community files and canvases the agent will consume. The mitigations are M7's, applied here: treat every MCP server as untrusted, least-privilege scopes, read-only channels by default, a human gate on any write-to-canvas path, and no credentials in MCP context.

8The mature end-state

What does F4 look like when it works? The system stops being documentation humans read and becomes infrastructure machines navigate — D1's third artifact in full production form. A JSON schema per component (identity, props, variants, states, usage, explicit anti-patterns) replaces prose as the primary encoding — codify knowledge as structured metadata, not prose; Indeed's published experience parsing 77 components from MDX into JSON metadata is the reference case, with JSON beating Markdown on both accuracy and token cost. MCP exposure (Storybook MCP, Code Connect) makes the system queryable in the agent's native idiom; AGENTS.md is the orchestration file that tells the agent which channel to use for what; Storybook doubles as coverage report and validation harness. The vendors are converging on the same shape from the other side: Figma's Code Connect launch blog already floated "automatic audits of design system usage in both code and design," and Schema 2025 shipped AI-suggested component-to-code mapping — the audit, in other words, stops being an F0 event and becomes standing telemetry. You can see the convergence in the small tools too, all point-in-time: Dembrandt generates a DESIGN.md for agents and exposes its own MCP server; Argos lets an agent approve a visual build from the CLI; Argos and Percy route diff triage to agents and AI respectively.

Hold the end-state against the honesty discipline and it shrinks to a defensible claim: none of this removes the designer gate — D6's split survives every tool above, because every one of them moves information and conformance, none moves judgment. The mature system is one where the machine-verifiable half is total (every token, component, story and diff flows through typed, queryable, gated channels) so that the human half — taste, brand, the last 40% — gets the designer's full hour. F4's "agent proposes system changes" is safe precisely because the proposals arrive as PRs against a fully characterized, fully gated system: the same artifacts F0 excavated by hand are by then maintained by the harness itself.

The deliverable of D7 in one line

You arrive with an audit, not a pitch. F0's four artifacts — the de-facto token report, the interface inventory, the golden-master baseline, the heatmap — are valuable to the client even if the engagement ends there, which is exactly PB2's posture when the assessment says NO-GO: the bootstrap is the deliverable. If the engagement continues, every later phase is built from those same artifacts and nothing else.

9Applying it — decision guide

Assess (F0 questions)
  • how many de-facto colors/spacings/radii does the shipped CSS contain — and how many were intended? (the usage counts tell you which six matter)
  • is there any visual baseline at all — and would anyone notice a silent 2px shift on the checkout?
  • which areas have stories, tokens, Code Connect and CI — the heatmap's legibility inputs?
  • what is the brand-risk map — which screens must never visibly regress?
  • who can edit the shared Figma libraries, community files and MCP servers an agent would read? (the trust inventory)
Commit (the defaults we bring)
  • audit before writes, always: the four F0 artifacts are fixed scope, and they are useful even on a NO-GO
  • gates in CI before any generation pilot: lint + axe + visual baseline + drift compare — permanent, not scaffolding (Veracode's flat-across-generations)
  • envelope, not big-bang: new system beside the old, adoption enforced by lint and codemods, deprecation on published windows
  • designer gate at L3 on every agent PR; D4 only with measured review capacity; never D5
  • MCP least-privilege, read-only by default, human gate on any write-to-canvas; design files treated as untrusted content
  • evidence honesty: CodeRabbit/Veracode labeled vendor, conference numbers labeled claims, every tool fact dated
Three sentences to remember D7 by

Design archaeology is the design layer's characterization test: before any agent touches a brownfield UI, excavate the de-facto system — tokens with usage counts (Project Wallace, Dembrandt), an interface inventory (Frost's 2013 move, Curtis's audits), a golden-master snapshot baseline — because agents amplify whatever context they find, and an undocumented system hands them forty-seven colors and asks for a forty-eighth. The rollout that follows is a permission ladder, B6's spine with design rails: F0 read-only audit (STOP if no harness is possible and brand risk is high) → F1 tokens-in-git plus deterministic gates → F2 a pilot where the heatmap scores legibility high and brand risk low → F3 scaling under the harness with strangler-fig mechanics (Fluent's shims, Carbon's versioned windows, Gestalt's lint enforcement — "strangler on a design system" is our term, the parts are documented) → F4 gated agent proposals, never D5. And the risk ledger is measured, not hypothetical: 0-of-16 developers prompt for accessibility (CodeA11y, CHI 2025) against a WebAIM baseline already at 95.9% and an EAA deadline now in force; generated code carries ~1.7× more issues and 2.74× more XSS (CodeRabbit) with 45% of samples introducing known vulnerabilities, flat across model generations (Veracode); and the design tools themselves are an injection surface (OWASP MCP Top 10, the GitHub MCP toxic flow, hidden-layer payloads in shared Figma files) — so the gates are permanent, the audit inventories trust as well as tokens, and every number above carries its source label.
