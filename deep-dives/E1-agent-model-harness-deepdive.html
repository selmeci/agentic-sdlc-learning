0Why this matters for us

E1 is the conceptual foundation for the entire engagement. If we get this framing right, every later decision — how much autonomy to grant, which guardrails are deterministic, where the human gates sit — follows from it. If we get it wrong, we sell the client a "smarter autocomplete" and it fails in production the way DORA's data predicts.

Our goal is to introduce an agentic engineering harness at a brownfield client. This document unpacks the seven-component framework so that we can (a) explain it credibly to a technical audience, (b) assess what a client already has versus what we must build, and (c) know the input and output of each component so the pieces actually connect into a working loop rather than a demo.

The one idea to internalise

A model is a capability; an agent is a system. The difference — and almost all of our engineering value — lives in the harness. When people say "the agent isn't good enough," they usually mean the harness around a perfectly capable model is missing or misconfigured.

1The core equation: Agent = Model + Harness

The framing comes from Birgitta Böckeler (ThoughtWorks) and is echoed in Anthropic's engineering writing: an agent is a language model wrapped in a harness that gives it state, tool execution, feedback loops, and enforceable constraints. Simon Willison's compact definition of the runtime pattern completes it: an agent "runs tools in a loop to achieve a goal."

The practical consequence is where the discipline's centre of gravity has moved. Two years ago the leverage was in prompt engineering — phrasing the request well. Today the leverage is in context engineering (curating what the model sees at each step) and harness engineering (designing the scaffolding around it). The model is increasingly a commodity you swap out; the harness is the durable asset you build and own.

A principle that will keep recurring

Every harness component encodes an assumption about what the model cannot do on its own. When the model improves, some components become unnecessary — and the right move is to remove them. A harness is therefore never "finished"; it is continuously pruned as models get stronger. Anthropic saw this directly: moving from one model generation to the next let them delete scaffolding they had previously needed.

This is also why we resist the temptation to over-build. The framework below is not a checklist of things every client must have on day one — it is a map of the surfaces where engineering effort pays off, to be applied in proportion to the client's maturity (assessed in E6) and rolled out in phases (E7).

2The seven components — with inputs & outputs

Below is the whole framework as one picture, then each component in turn. For each, we give a plain explanation, what it requires on input, what it produces on output, where it sits in the SDLC, and a short note for our rollout. Read the diagram first: notice that the components form a loop, not a pipeline — output feeds back in.

01

Prompt

the instruction layer

The standing instructions plus the specific task: who the agent is, how it should behave, and what it is being asked to do now. In coding agents this splits into a durable system/role layer (the CLAUDE.md/AGENTS.md conventions) and the per-task request.

Requires on input
  • a clearly scoped goal (from the handoff contract, M3)
  • role/behaviour conventions for the repo
  • explicit non-goals and constraints
Produces on output
  • a bounded instruction the model can act on
  • the frame that determines whether context is even interpreted correctly
Where it sits in the SDLC
  • Refinement / task definition — the prompt is the downstream face of the handoff contract; a vague prompt is an under-specified ticket wearing a costume.

For our rollout: the prompt layer is advisory — the model may drift from it under pressure or injection. Treat it as documentation of intent, never as a security or correctness boundary. Keep the durable part short and load-bearing (every line earns its place).

02

Context

what the model sees

The working set of information assembled for each step: relevant code, documentation, retrieved snippets, prior messages, tool results. Context engineering is the iterative discipline of curating "the smallest set of high-signal tokens" at every decision — not stuffing everything in.

Requires on input
  • a retrieval mechanism (agentic search / grep, or RAG)
  • source-of-truth artifacts to draw from (M4)
  • a budget discipline — what to include, what to link
Produces on output
  • a curated, task-relevant window handed to the model
  • the single biggest lever on output quality
Where it sits in the SDLC
  • Cross-cutting, but heaviest at implementation time — the agent assembles context each turn as it navigates the codebase.

For our rollout: beware context rot — recall degrades as the window grows. Prefer "reference, don't copy": link to masters, inject summarised excerpts. Every piece of retrieved content is untrusted until proven otherwise (this is the entry point for injection — see M7).

03

Harness

the orchestrating scaffold

The component the whole equation is named for: the software around the model that manages sessions, assembles context, invokes tools, enforces constraints, and carries state between turns. It is where our engineering lives. Patterns include the initializer agent (first session sets up init.sh, a progress file, an initial commit) and the clean-state principle (every session ends merge-ready).

Requires on input
  • a runtime (CLI / SDK / IDE integration)
  • configuration: which tools, which permissions, which hooks
  • a state convention (progress file, git history)
Produces on output
  • a repeatable, constrained execution environment
  • continuity across sessions (the "engineers in shifts" model)
Where it sits in the SDLC
  • Infrastructure/platform layer — set up once (Phase 1 of rollout), then maintained and pruned as models improve.

For our rollout: this is the "internal product" we build and own (see trajectory T5). Its quality determines everything downstream; treat it with versioning, ownership, and — eventually — its own evals.

04

Loop

the runtime cycle

The cycle that makes it agentic: the model proposes an action, the harness executes it via a tool, the result returns to context, the model reasons again — repeating until the goal is met or a stop condition fires. Willison's "tools in a loop" is exactly this.

Requires on input
  • tools it can call and observe results from
  • stop conditions (done, budget exhausted, needs human)
  • a way to detect and recover from being stuck
Produces on output
  • a sequence of actions converging on the goal
  • a trace/audit of what was attempted
Where it sits in the SDLC
  • Implementation runtime — this is the agent "doing the work" between task hand-off and PR.

For our rollout: loops need blocker-recognition. Devin completed only 3 of 20 tasks in Answer.AI's independent test; the failures came from a loop that pushed ahead on impossible tasks instead of stopping. A good loop knows when to ask a human — which is where autonomy level (E6) is really decided.

05

Tools

the hands

The concrete actions the agent can take: run shell commands, read/edit files, search, call external systems via MCP. Tools are what turn reasoning into effects on the world — and therefore where capability and risk both concentrate.

Requires on input
  • well-described tool interfaces the model can use
  • scoped credentials (least privilege)
  • an allowlist of what may be called
Produces on output
  • real effects: file changes, command results, API calls
  • observations that re-enter the loop
Where it sits in the SDLC
  • Implementation & integration — and, via MCP, the seam to ticketing, design, and deployment systems.

For our rollout: tools are the richest attack surface (M7). Every tool is a privilege you hand an agent that may be injected. One-repo-per-session, scoped tokens, no service_role, human gate on writes.

06

Verification / feedback

the external truth

The deterministic signal that tells the agent — and us — whether the work is actually correct: tests, type-checks, linters, CI results. This is the component that makes delegation safe, because it is truth the model cannot argue away. TDD fits agents naturally: run tests, read failures, self-correct.

Requires on input
  • a runnable, reasonably fast test suite
  • CI able to return results the agent can read
  • machine-verifiable acceptance criteria (EARS/Gherkin, M3)
Produces on output
  • a pass/fail signal that drives the loop
  • the merge gate's evidence base
Where it sits in the SDLC
  • Testing & CI — but shifted left: verification is written before or alongside implementation, becoming the specification.

For our rollout: this is the make-or-break prerequisite. A client without a fast, trustworthy test loop cannot safely go past L2. Agents also "cheat" tests — protect them (three-layer defence: rule + hook + verification gate).

07

Governance

the enforced boundary

The deterministic constraints on what the agent may do, enforced by the harness rather than requested of the model: permission rules, pre/post-tool hooks, sandboxing, network policy. The defining distinction of the whole framework lives here — advisory vs deterministic.

Requires on input
  • a policy: what is always allowed / denied / gated
  • enforcement points (hooks, OS sandbox, egress proxy)
  • an audit mechanism
Produces on output
  • hard guarantees independent of model behaviour
  • bounded, observable, reversible blast radius
Where it sits in the SDLC
  • Cross-cutting platform concern — spanning dev, CI, and deploy; the organisational layer sits above it (T7).

For our rollout: the litmus test for any rule — "what must never happen" goes in a hook/permission (deterministic); "what the model should know" goes in the prompt (advisory). Under injection, only the deterministic layer holds.

+

Memory & Orchestration

cross-cutting & conditional

Memory runs across all seven: session/working memory (the context window), persistent files (progress logs, /memories), cross-session state (git history), and organisational stores. It is what lets a memoryless model behave like "engineers working in shifts." Input: a state convention and a place to write. Output: continuity — and, if unguarded, a poisoning surface (M7, ASI06).

Orchestration (multi-agent) is a conditional choice, not a default eighth component. Parallel writers are fragile; the pattern that works keeps writes single-threaded while other agents contribute intelligence, not actions. Reach for it only when a task genuinely parallelises — and treat the whole cluster as one trust boundary.

3Mapping the components to the SDLC

The framework is not a phase model, but each component has a natural home in the classical software lifecycle. This table is how we explain "where does the agent actually plug in?" to a client who thinks in SDLC stages.

SDLC stageDominant component(s)What good looks like
Requirements / refinementPrompt · (handoff contract)scoped tasks with non-goals and machine-verifiable acceptance criteria
Design / planningContext · Harnessthe agent can retrieve architecture and decisions; a plan mode before writes
ImplementationLoop · Tools · Contextthe agent works in a constrained loop, assembling context per turn
Testing / CIVerificationfast trustworthy suite as the external truth; tests protected from the agent
Review / mergeGovernance · Verificationhuman gate on intent; deterministic gates on correctness & safety
Operate / evolveMemory · Governancecontinuity across sessions; audit trail; blast radius bounded in prod
The shift to notice

Classical SDLC runs left-to-right once. The agentic version loops: verification feeds implementation, and memory carries state between whole cycles. The components don't replace the SDLC — they instrument it with a fast inner feedback loop, which is exactly why a weak test/CI base (a weak verification component) makes the whole thing regress, as DORA's stability data shows.

4Takeaways for introducing a harness

Distilled to what we act on:

Assess before you build (feeds E6)
  • Is there a fast, trustworthy test suite? (verification)
  • Can CI return signal an agent can read?
  • Are source-of-truth artifacts retrievable? (context)
  • Do dev/prod separation, RBAC and secrets hygiene exist? (governance)
Build in this order (feeds E7)
  • make the repo agent-ready (harness + governance skeleton)
  • establish the verification loop before granting write autonomy
  • start at L1–L2, prove the base, then move to L3
  • prune harness components as the model improves
Three sentences to remember E1 by

An agent is a model plus a harness; the harness is where our engineering value and the client's risk both live. Some components are advisory (prompt, context) and some must be deterministic (governance, verification) — never confuse the two. Build the loop, verify with external truth, bound the blast radius, and remove scaffolding as the model grows.
