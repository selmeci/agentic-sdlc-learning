0Why it matters

I5 closed on a promise: disclosure stages what is known to matter; retrieval finds what no map anticipated. This chapter is about that finding — where the agent's knowledge of the codebase comes from at query time. Three architectures compete for the job: a live search loop the model drives itself (grep, glob, read — nothing precomputed), a precomputed embedding index queried by similarity (the RAG pattern), and a graph of code entities and relations. The stakes are easy to underestimate because 2023-era folklore settled the question early: "AI needs your code in a vector database" was the default enterprise assumption, and vector-store line items are still in many clients' AI budgets. The production reality of 2025–26 is stranger and more instructive: the most-used coding agent ships no index at all — Boris Cherny, who created Claude Code, put it flatly: "Claude Code doesn't use RAG currently. In our testing we found that agentic search out-performed RAG for the kinds of things people use Code for" — while Cursor runs a hybrid it defends with measurements, Sourcegraph publicly walked its embeddings back, and Augment doubled down on vectors for 100-million-line codebases. Every position on the board is occupied by a serious team with a serious argument, which is precisely why this is a harness architecture decision (E1) and not a tool detail: the retrieval choice determines what your client must build, sync, secure and pay for — an index is a second copy of the codebase with its own freshness problem (I3), its own attack surface (M7), and its own bill. The chapter maps the lanes, reads the evidence, prices the index tax, deflates the knowledge-graph hype, and ends in an escalation ladder you can defend in a client meeting.

1The four retrieval lanes

Strip the marketing and there are four ways an agent can find things in a codebase. They differ on one axis that matters more than any other: how much is precomputed — because everything precomputed must be stored, synced, secured and paid for, and everything computed live must be paid for in tokens and turns instead.

LANE A

Agentic search

nothing precomputed · the model drives

The model composes searches itself — find files by name pattern, grep file contents, read what looks promising — inspects the results, and refines. Claude Code's tool tier is cost-ordered exactly this way: Glob returns paths for near-zero tokens, Grep (ripgrep underneath) returns matches, Read pays hundreds to thousands of tokens per file — so a competent loop spends cheaply before it spends dearly. Search-relevance veteran Doug Turnbull's diagnosis of why this suddenly works is worth keeping: the win is not grep, it is the loop around grep — an agent that reacts to what it finds, backtracks, and is checked by an outer validation loop. His caveat travels too: "feedback that's not actionable doesn't help an agent" — a tool that returns 4,000 unranked matches has not helped. Zero build cost, zero sync, structurally fresh (it searches the working tree as it is right now); the price is per-query tokens and latency measured in seconds.

LANE B

Embedding index — the RAG lane

chunks precomputed · similarity lookup

The pattern the term RAG named (Lewis et al., NeurIPS 2020, arXiv:2005.11401): split the corpus into chunks, embed each into a vector, store the vectors; at query time embed the question and return the nearest chunks. Its unique power is matching meaning without shared words — "where do we throttle API clients?" can land on RateLimiter even if no file contains the word "throttle". Its unique costs are the pipeline: an index to build, re-embed on every change, host somewhere, and secure — §4 prices all three. Latency, once built, is milliseconds — which §2 shows is exactly why the products that kept embeddings are the ones with millisecond features.

LANE C

Symbol & structure index

derived from syntax · precise relations

Not semantic, not lexical: structural. Tree-sitter or a language server parses the code and answers questions grep answers badly — definitions, references, call sites, "who uses this symbol". Two shipped forms: aider's repo map (tree-sitter symbols, graph-ranked, packed to a 1k-token default budget — I5's computed map), and LSP-backed tool servers such as Serena, which expose find_symbol / find_referencing_symbols to any MCP-capable agent. In I1's taxonomy these are derived artifacts: regenerated from code, never hand-edited, cheap to keep fresh because the compiler toolchain already does the parsing. This lane is the quiet workhorse — most of what a "code knowledge graph" promises, it already delivers without a database.

LANE D

Code knowledge graph

entities + relations · a database of the repo

The maximal precompute: extract entities (modules, classes, functions) and typed relations (calls, imports, inherits) into a graph database the agent queries — CodexGraph (arXiv:2408.03910) is the cleanest research statement of the idea. The pitch is multi-hop structural questions in one cheap query instead of a grep chain. The status — measured results, self-reported gains, and what has actually shipped — gets §5 to itself, because this is where the hype concentration is highest.

Lanes are tools, not religions

Nothing forces a harness to pick one lane. The visible trend of 2026 is that retrieval choices migrate to the tool boundary: Serena ships lane C as an MCP server, Augment ships its context engine as one, research prototypes ship lane D the same way — so the agent picks grep for an exact symbol, the symbol graph for "who calls this", and semantic search for a fuzzy concept, per query. The architecture question then becomes not "which retrieval?" but "which retrieval tools earn a place on the agent's belt?" — which is E9's harness-tuning question wearing different clothes.

The no-index camp

Claude Code is the flagship datum because the team tried the alternative first. Cherny on Hacker News (February 2025): "Claude Code doesn't use RAG currently. In our testing we found that agentic search out-performed RAG for the kinds of things people use Code for." He later restated the history with the reasons attached: "Early versions of Claude Code used RAG + a local vector db, but we found pretty quickly that agentic search generally works better. It is also simpler and doesn't have the same issues around security, privacy, staleness, and reliability." The Pragmatic Engineer's write-up of the story (March 2026) adds texture: the team tried "local vector databases, recursive model-based indexing, and other fancy approaches," all with downsides ("stale indexes, permission complexity") — "plain glob and grep, driven by the model, beat everything," an approach reportedly inspired by how engineers searched code at Meta when fancier tooling broke. Note what the stated reasons are not: nobody claims grep finds more relevant code than a good embedder — the case is systems-shaped (simplicity, freshness, privacy, reliability), and §3 shows the benchmark literature is consistent with that reading. OpenAI's Codex sits in the same lane with less ceremony: its core prompt bakes in a ripgrep preference ("prefer rg… since rg is faster than grep"), and semantic indexing exists mainly as an open feature request — a default, not a manifesto. Don't cite Codex as an anti-RAG position; cite it as the same architecture arrived at quietly.

The hybrid, defended with a number

Cursor is the measured counterpoint. Its semantic-search post (November 6, 2025) reports that adding embedding search to a grep-capable agent yields "on average 12.5% higher accuracy in answering questions (6.5%–23.5% depending on the model)" — measured on Cursor Context Bench, an internal, unpublished eval — and is explicit that the agent "makes heavy use of grep as well as semantic search, and the combination of these two leads to the best outcomes." Two details are worth repeating precisely: the 12.5% attaches to semantic search's lift on their QA eval, not to some grep-vs-index showdown; and the embedding model is custom, trained on ranked traces from their own agent sessions — a reminder that a serious lane-B commitment includes training and serving infrastructure, not an off-the-shelf vector DB.

The retreat, and the counterexamples

Sourcegraph ran the experiment in the other direction and published the result (February 15, 2024): "Embeddings have been at the backbone of Cody's retrieval stack since we launched… and now that Cody Enterprise is generally available, we're leaving them behind (for now)." Three stated reasons, all of them §4's index taxes in vendor prose: code "has to be sent to a 3rd party (OpenAI) for processing"; "creating embeddings and keeping them up-to-date introduces complexity"; and vector search over ">100,000 repositories is complex and resource-intensive." (Precision on the product's fate: Cody's Free and Pro tiers were discontinued July 23, 2025 as Sourcegraph pivoted to its agentic tool Amp; Cody Enterprise continues — the embeddings retreat was 2024 and architectural, not part of the sunset.)

And the honest board has counterexamples. Devin indexes — its DeepWiki "automatically indexes your repos and produces wikis with architecture diagrams, links to sources, and summaries," and Devin's answers are wiki-grounded with code search layered on top; an earlier draft of this workbook claimed Devin followed the no-index pattern, and that was simply wrong. Windsurf's Riptide attacks the retrieval-quality problem with more compute rather than less: an LLM evaluates candidate snippets for relevance at search time, claiming "3x the recall of state-of-the-art embedding systems" (a Codeium-era post now hosted on Cognition's domain after the acquisitions — treat the details as historical). Augment made the strongest embeddings-at-scale case: quantized vector search serving 100-million-line codebases, with completions latency "improved by over 40%" and index memory cut "8x (from 2GB to 250MB for a 100M LOC codebase)" (June 2025, updated June 2026). Read the counterexamples carefully and a pattern falls out — flag it as this workbook's synthesis, not anyone's official doctrine: the split tracks product shape more than conviction. Millisecond features (completions, real-time navigation, answering over repos you haven't cloned) cannot wait for a grep loop, so they need an index; task agents that think in seconds can afford to search live. Where on that spectrum your client's use case sits is the first question of §6.

3The evidence — what independent sources actually show

Separate the lanes of evidence the way E7 taught: shared benchmarks first, practitioner testimony second, vendor numbers last.

Retrieval caps single-shot performance — and the cap moved the paradigm

The original SWE-bench paper (Jimenez et al., ICLR 2024, arXiv:2310.06770) ran the experiment everyone now forgets it ran: give the model issue text plus retrieved files (BM25), versus issue text plus the oracle files — the ones the real fix actually edited. Claude 2 resolved 1.96% with BM25 retrieval and 4.80% with the oracle; GPT-4, 1.74% even with the oracle (on a 25% subset). Two readings, both load-bearing. First, retrieval quality strictly caps a retrieve-then-edit pipeline — BM25 roughly halved the oracle score. Second, and more important for this chapter: even perfect retrieval left single digits on the table, so the field's answer was not better retrieval, it was abandoning single-shot — letting the model search, read, run and iterate. Retrieval stopped being a preprocessing stage and became part of the loop. That is the deepest sense in which "agentic search won": not grep over embeddings, but iteration over one-shot context stuffing.

Agentless (Xia et al., arXiv:2407.01489) is the cleanest demonstration that the middle ground works too: no autonomous agent, no vector index — a fixed three-phase pipeline whose localization is hierarchical and LLM-driven (repo structure → files → classes/functions → edit locations), scoring 32.00% on SWE-bench Lite at $0.70 average per issue (its 2024 numbers; the leaderboard has moved). Locating code by structure, letting the model read the tree the way a new hire would, needed no embeddings to beat contemporaneous agent frameworks.

The folklore correction: grep does not win on retrieval quality

CodeRAG-Bench (Wang et al., NAACL 2025 Findings, arXiv:2406.14497) benchmarked 10 retrievers × 10 generators on code tasks and reads as a caution in both directions: "notable gains are made in final code generation by retrieving high-quality contexts," but "current retrievers still struggle to fetch useful contexts especially with limited lexical overlap, and generators fail to improve with limited context lengths or abilities to integrate additional contexts." And here the folklore needs correcting: on these code benchmarks, dense code-trained embedders frequently surpass BM25 — notable precisely because in general-domain IR, BM25 usually wins out-of-domain (CoIR, arXiv:2407.02883, is the dedicated code-retrieval benchmark to pin this). So do not repeat "lexical search is just better for code." The defensible claim is different: embeddings retrieve somewhat better on average, and the no-index camp wins anyway — on freshness, simplicity, privacy, and above all on the agent's ability to compensate for a dumb tool by iterating. Colin Flaherty, who built Augment's SWE-bench-topping agent before the industry's agentic turn, says exactly this: "We explored adding various embedding-based retrieval tools, but found that for SweeBench tasks this was not the bottleneck — grep and find were sufficient" — his stated reasons: benchmark repos are modest in size, code is keyword-dense, and a persistent agent recovers from bad first searches.

Benchmark shape is doing work in every direction

SWE-bench repos fit comfortably in the regime where grep thrives (Flaherty's own caveat), so "grep sufficed on SWE-bench" does not settle the 100M-LOC monorepo question — that is Augment's regime, and no shared benchmark covers it. Symmetrically, none of §2's vendor numbers (12.5%, 3× recall, 40% fewer tokens, 99.9% accuracy) were produced on a benchmark anyone else can run. Mid-2026 there is no public, adversarial grep-vs-index benchmark at enterprise scale. When the client asks "which is better?", the honest answer is: for task agents on normal repos, the production consensus and the SWE-bench-era evidence favor search loops; past that, you are choosing between vendors' self-measurements — so run your own (§6).

The one measured head-to-head with structure

The nearest thing to a controlled lane-A-vs-lane-D comparison is Codebase-Memory (arXiv:2603.27277, March 2026 — a young preprint, weight it accordingly): a tree-sitter-derived knowledge graph served over MCP reached "83% answer quality versus 92% for a file-exploration agent, at ten times fewer tokens and 2.1 times fewer tool calls" across 31 real repositories, matching or exceeding the explorer on graph-native queries (hubs, callers) in 19 of 31. That is the honest shape of the whole trade: the grep loop keeps a quality edge; the structure pays a token tax an order of magnitude smaller. Chroma's context-rot report (July 2025, vendor technical report, 18 models) and RepoQA (arXiv:2406.06025) close the loop from the long-context side: performance degrades as input grows and needle-finding in repos stays imperfect — so "just give the model a huge window and skip retrieval entirely" is not the exit; something still has to choose what enters the window (E2).

4The index tax

Whatever an index buys, it costs along three axes. This section is the part of the chapter a client's platform team should read twice, because every line item lands on them.

Tax one: staleness — the index is a derived artifact that drifts

An embedding index is a second copy of the codebase in another representation. In I1's taxonomy it is a derived artifact; I3's law applies in full: every derived copy is drifting toward wrong the moment it is written. Cherny's list put "staleness" in the middle of the reasons to skip the whole problem; Sourcegraph's retreat named "keeping them up-to-date" as a core complexity. The engineering answer, where the index is kept, is real but nontrivial: Cursor's indexing pipeline (January 27, 2026 post) hashes every file into a Merkle tree ("a cryptographic hash of every file, along with hashes of each folder that are based on the hashes of its children"), re-embeds only changed chunks, and caches embeddings by chunk content — and even the bookkeeping is material: "In a workspace with fifty thousand files, just the filenames and SHA-256 hashes add up to roughly 3.2 MB." Grep's freshness, by contrast, is structural — it searches the bytes on disk at query time, so there is nothing to sync and no window in which the agent can be confidently wrong about code that changed a minute ago. For a brownfield client, put it in governance terms: an adopted index needs an owner, an automated sync, and a freshness check (I3's per-category policy) exactly like any other derived artifact — if nobody will own that, that alone decides §6.

Tax two: security — embeddings are recoverable code

The comforting intuition that a vector index is an abstract mathematical residue of the code is false. Vec2Text (Morris et al., EMNLP 2023, arXiv:2310.06816) showed an iterative inversion attack that "is able to recover 92% of 32-token text inputs exactly"; follow-on work relaxed the assumptions — transferable attacks that need no query access to the victim embedder (ACL 2024, arXiv:2406.10280) and few-shot variants (arXiv:2502.11308). The demonstrations are on text, but the mechanism is content-agnostic; treat an embedding store as a recoverable copy of the source and classify it accordingly (M7): where the vectors live, who can query them, and what leaves the building are now questions about the code itself. This was Sourcegraph's first stated reason (embedding meant code "has to be sent to a 3rd party… for processing"), and it is what Cursor's content-proof design is engineered against — the server stores the Merkle tree "as a set of content proofs" and "the client can never see results for code it doesn't already have." Note the limit of the public record: Cursor's fetchable pages do not state that plaintext code is never persisted server-side — verify current guarantees directly before an enterprise adoption, point-in-time. And agentic search is not tax-free here either: it inherits whatever file-access the agent has (I2's write-permission discipline is the control surface), it just adds no second store to defend.

Tax three: cost — the crossover arithmetic

Practitioner Harrison Guo frames the choice as one formula worth stealing for client conversations: total cost = build cost + maintenance × time + per-query cost × queries. Grep sets the first two terms to zero and pays per query in tokens and round-trips; an index pays heavily up front to make queries nearly free. The crossover is therefore about scale and query volume, and the extremes are documented. At the top: Augment's stated arithmetic — good embedding models "require 20+ bytes per LOC to store and 20+ nanoseconds per LOC to search," so a 100M-LOC codebase means ~2 GB of vectors and 2+ seconds per brute search, which is why they quantize (their reported 8× memory reduction and sub-200ms search are the vendor's own numbers). At planetary scale the argument inverts entirely: GitHub's code search (the Blackbird engine, February 2023) indexes 15.5B documents and 115 TB across ~45M repositories into a 25 TB lexical sparse-ngram index — because brute grep at that scale would need "2,048 CPU cores for 96 seconds to serve a single query." Two lessons hide in that number: corpus-scale search needs an index, full stop — and the biggest code-search operator on earth chose an inverted lexical index, not embeddings, to build it. Meanwhile the vector-DB industry runs the token-cost argument against the grep loop — Milvus/Zilliz: "Token usage dropped by over 40%, without any loss in recall" with their semantic plugin (August 2025) — a claim from the vendor selling the fix, quoted here as the named opposing voice, not as evidence; the independent Codebase-Memory measurement (§3) at least confirms the direction that structure saves tokens. The honest summary: on a normal repo with a task agent, the grep loop's token tax is real but the index's three taxes are realer; the regimes where that flips are §6's triggers.

5Knowledge graphs over code — the hype check

Lane D attracts the most consultant enthusiasm and has shipped the least. Hold three facts together and the picture is sober.

The most common miscitation in this topic: GraphRAG is not a code technique

Microsoft's GraphRAG (arXiv:2404.16130) — the paper that made "knowledge graph + RAG" a boardroom phrase — is about query-focused summarization over document corpora: podcast transcripts and news articles at ~1M tokens, where an LLM-extracted entity graph plus community summaries answers global "what are the main themes" questions. Its reported wins over vector RAG (comprehensiveness 72–83%, diversity 62–82%) are LLM-as-judge pairwise win rates — self-evaluated by a method with known biases, and on the wrong corpus type entirely. Citing GraphRAG to justify a code knowledge graph is a category error; do it in front of the wrong audience and it costs you the room.

The actual code-KG literature is real but self-reported. CodexGraph (arXiv:2408.03910) puts an LLM in front of a graph database of repo structure and labels itself "work in progress"; RepoGraph (arXiv:2410.14684) is a plug-in repo-level graph reporting SWE-bench-Lite improvements for the systems it augments; CGM (arXiv:2505.16901) and KG-guided repair (arXiv:2503.21710) continue the line. Every headline gain in that list is the authors' own evaluation — no independent replication, no adversarial benchmark (E7 files these where vendor claims go). The one measured head-to-head we have is §3's Codebase-Memory result: ~90% of the explorer's answer quality at a tenth of the tokens, better on graph-native questions, worse when the answer needs full source context — a genuine, quantified niche, not a revolution.

And the shipping test: as of mid-2026 there is no public evidence that any major production coding agent uses a code knowledge graph as its retrieval backbone (a hard-to-prove negative — state it as "no public evidence," not impossibility). The nearest shipped relatives are telling about what actually survives contact with production: Devin's DeepWiki (a generated wiki plus code graph — an onboarding and Q&A layer more than a retrieval backbone), and Greptile's AST-derived docstring-embedding graph — built for code review, a batch workload whose economics differ from an interactive agent's. The practical verdict for a client: the structural benefits a KG promises are mostly available today from lane C — LSP and tree-sitter tools, derived artifacts with toolchain-grade freshness and no graph database to feed. Put lane D on the watchlist, revisit when someone publishes an independent benchmark win plus a staleness story; do not put it in the harness.

6The escalation ladder

This rubric is our synthesis — assembled from the evidence above, not an industry standard; flag it the way D1–D5 are flagged. The default is set by the production consensus and the price list; each rung above it is bought only by a measured trigger, never by architecture enthusiasm.

L0

Agentic search + the map

the default — start every engagement here

Grep/glob/read driven by the model, over a repo whose always-loaded core is a map (I5): ARCHITECTURE.md for the hand-written topology (matklad's "10x more time to figure out where" is the cost being avoided), context files naming the load-bearing directories. The map is what makes search cheap — it turns the first three exploratory turns into zero turns. Cost: tokens per query. Prerequisites: none. This is what Claude Code and Codex ship, and it needs no procurement meeting.

L1

Derived structure tools

cheap, fresh, undersold

Add lane C: a repo map computed from tree-sitter (aider-style), LSP-backed symbol tools (Serena-class MCP servers) for definitions/references/call sites. These are derived artifacts the toolchain keeps fresh nearly for free, they answer exactly the questions grep answers worst, and they capture most of the KG promise without a database. For a brownfield client with a real language server already in CI, this rung is close to free — take it early.

L2

Hybrid: add an embedding index

bought by a trigger, run under governance

Adopt lane B only when a §-below trigger is measured, and adopt it with the full I-module governance: a named owner, automated Merkle-style sync, a security review of where vectors live and who queries them (§4, M7), and a before/after eval on the client's own query set — not the vendor's. Cursor's experience says the ceiling is real (+12.5% on their eval, with a custom-trained embedder); Sourcegraph's says the floor is real too.

L3

Code knowledge graph

watchlist — not a client recommendation

Experimental (§5). Revisit on independent benchmark evidence plus a credible freshness story. If a client is already being sold one, the questions that cut fastest: who re-extracts the graph on every merge, what did it score on a benchmark the vendor doesn't own, and what does lane C not give you?

Measure before you buy — the twenty-query bench

Before any L2 conversation, collect ~20 real retrieval queries from the client's actual agent sessions — the searches that preceded wrong answers are the valuable ones — and run them against L0/L1 and a candidate index. Score answer quality, tokens, and wall-clock. This is E7's discipline applied locally: every number in §2 is a vendor's self-measurement on a private eval; the only benchmark that binds your client's decision is the one built from their queries on their code. It also answers this topic's standing open question — "at what repo size does plain grep start losing?" — empirically, for the one repo that matters.

7Week one at the client

The retrieval audit runs in parallel with I5's disclosure audit — same week, same interviews.

Do — in this order
  • 1 · Inventory the retrieval surface. For each agent in use: which lanes does it actually have — grep tools, an embedding index (whose? synced how?), symbol tools, MCP search servers? Most clients cannot answer this; the answer is the architecture.
  • 2 · Build the map before any index. ARCHITECTURE.md + context files naming load-bearing directories (I5). Cheapest retrieval upgrade that exists.
  • 3 · Take the free L1 rung. If the stack has a language server, wire symbol tools (repo map, LSP/MCP) into the harness.
  • 4 · Collect the twenty-query bench from real session transcripts (§6) and classify failures by trigger: scale, query shape, latency, structure.
  • 5 · If an index already exists, govern it: name an owner, verify sync automation, run the M7 questions (where do vectors live, who queries, what leaves) — and check it against I3 drift today.
Don't
  • let "we need a vector database" enter the SOW before the twenty-query bench exists
  • cite Cursor's 12.5%, Riptide's 3×, or Milvus's 40% as if they were independent — all vendor, all private evals (E7)
  • cite GraphRAG for code retrieval — wrong corpus, LLM-judged wins (§5)
  • treat an existing embedding index as harmless metadata — it is recoverable code (Vec2Text, M7)
  • promise the client grep "beats" embeddings on retrieval quality — the honest case is freshness, simplicity, privacy and iteration (§3)

9Takeaways for our engagement

Three sentences

Production coding agents mostly answered the retrieval question against the 2023 folklore — no index, model-driven grep/glob/read — and the stated reasons are systems reasons (simplicity, freshness, privacy, reliability), not retrieval-quality reasons: on quality alone, dense code-trained embedders often beat lexical search, and the agent's iteration is what closes the gap. An index is a second copy of the code with three taxes — it drifts (I3), it is recoverable source (Vec2Text, M7), and it bills for build, sync and serving — so it must be bought by a measured trigger (scale, low-overlap queries, millisecond latency, corpus breadth), never by default. Our ladder — L0 search+map, L1 derived structure, L2 governed hybrid, L3 KG-watchlist — is this workbook's synthesis; the twenty-query bench on the client's own code is what moves anyone up a rung.
