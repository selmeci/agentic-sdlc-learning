0Why it matters

Every chapter of M4 so far has decided something about the artifacts an agent works from: what they are (I1), who may write them (I2), how they stay true (I3), what the agent itself remembers (I4). This chapter decides the last variable — how much of it the agent sees at once. The naive answer is "all of it": paste the PRD, the architecture notes, the coding standards and the tribal knowledge into one always-loaded file and the agent can never claim it wasn't told. The measured reality is that this fails twice. It fails on capacity — the context window is the scarcest resource in the system (E2), and Chroma's context-rot report showed accuracy degrading as input grows even on trivially simple tasks, across all 18 models tested (July 2025). And it fails on attention — an instruction buried in a 900-line context file competes with everything around it, which is why the official Claude Code guidance is blunt: files load in full regardless of length, "though shorter files produce better adherence." The engineering answer that emerged across the industry in 2025–26 is progressive disclosure: keep a small always-loaded core, make that core a map of where everything else lives, and let the agent pull the rest just-in-time. Anthropic's context-engineering team states the inversion directly: agents should "maintain lightweight identifiers (file paths, stored queries, web links, etc.) and use these references to dynamically load data into context at runtime" — which "enables progressive disclosure… allows agents to incrementally discover relevant context through exploration." Skills, nested context files, scoped rules and index files are four implementations of that one idea, and this chapter takes them apart mechanism by mechanism — including the evidence that content quality, not raw length, is what the budget is actually for.

1From UX pattern to context discipline

The name is borrowed honestly. Progressive disclosure is a user-interface pattern — show only what the user needs now, defer the rest behind an explicit next step — popularized by Jakob Nielsen's 2006 Nielsen Norman Group article of that name, with research roots in John M. Carroll and Caroline Carrithers' "Training Wheels in a User Interface" (Communications of the ACM, 1984), which found novices learned faster when advanced functions were blocked off. Both findings transfer to agents almost without translation: attention is finite, everything visible competes for it, and the cure is not more information but staged information.

Applied to an agent's context, the pattern gives a three-tier architecture — the vocabulary core / map / leaves is this workbook's framing, but every mechanism in this chapter implements it:

TIER 1

The always-loaded core

every session · paid every time

What enters the context before any work happens: the system prompt, root context files (CLAUDE.md/AGENTS.md), skill metadata. Every line here taxes every session, relevant or not — which is why §5's gardening budget applies here and only here.

TIER 2

Relevance-triggered bodies

when the task matches

Content that loads itself when a signal fires: a skill body when the model judges its description relevant, a subdirectory context file when the agent reads files there, a glob-scoped rule when a matching path enters the conversation. The trigger — a description, a path pattern — is the whole interface.

TIER 3

On-demand leaves

pulled by tools · unbounded

Everything the agent fetches deliberately: bundled skill resources, the document an index line points to, any file grep can reach. Costs nothing until touched; effectively unlimited, because the filesystem is the overflow — Manus calls it "the ultimate context… unlimited in size, persistent by nature, and directly operable by the agent itself" (July 2025).

The clearest proof that staging pays comes from the tool layer, where the numbers are largest. MCP tool definitions historically loaded up-front, like a kitchen-sink context file: Anthropic's advanced-tool-use post (November 24, 2025) measures a five-server setup at "58 tools consuming approximately 55K tokens before the conversation even starts." The fixes are progressive disclosure verbatim — a Tool Search Tool that loads definitions on demand (defer_loading: true), reported at an 85% reduction in token usage with accuracy improving, from 49% to 74% on MCP evaluations for Opus 4 and 79.5% to 88.1% for Opus 4.5; and the code-execution-with-MCP pattern (November 4, 2025), which presents tools as files precisely because "Models are great at navigating filesystems. Presenting tools as code on a filesystem allows models to read tool definitions on-demand, rather than reading them all up-front" — the post's worked example drops 150,000 tokens to 2,000, "a time and cost saving of 98.7%."

Vendor numbers, one direction of evidence

Those token and accuracy figures are Anthropic internal testing on Anthropic products — vendor claims by E7's standard, not independent replication. They are quoted here because the direction is corroborated from the independent side: Chroma's context-rot measurements (more input, worse recall — 18 models) and §5's ETH study (redundant context: higher cost, no success gain) point the same way from outside. Loading less is not merely cheaper; past a threshold it is more accurate. Treat the exact percentages as point-in-time marketing precision.

2Skills — the canonical three-level mechanism

Anthropic's Agent Skills (launched with "Equipping agents for the real world with Agent Skills," October 16, 2025) are the cleanest productized form of the pattern, worth studying even if a client never runs Claude: a skill is just a folder with a SKILL.md file — YAML frontmatter carrying a name (max 64 characters; lowercase letters, numbers, hyphens; the words "anthropic" and "claude" reserved) and a description (non-empty, max 1024 characters) — plus optional bundled scripts, references and assets. The three documented disclosure levels map exactly onto §1's tiers, with official context costs attached:

LevelWhat loadsWhenContext cost (official)
L1 metadataname + description of every installed skillalways — "at startup," into the system prompt"~100 tokens per Skill"
L2 instructionsthe SKILL.md body — "Only then does this content enter the context window"when the model judges the description relevant"Under 5k tokens"; best practice: body under 500 lines
L3 resourcesbundled files — scripts executed, references read"only when referenced" during the work"None until accessed"

Two design consequences deserve more attention than they usually get. First, the description is the entire retrieval interface: at the moment of deciding whether to load a skill, the model sees nothing but those ≤1024 characters — which is why the authoring guidance insists a description state both what the skill does and when to use it, in third person. A skill with a vague description is not a weaker skill; it is an unreachable one (§6). Second, the body budget is not a style preference: a triggered skill spends from the same window as the task, so "under 500 lines," with anything longer split into L3 references, is the same gardening rule as §5's — applied one tier down.

What began proprietary is now — point-in-time, mid-2026 — a cross-vendor surface. Anthropic published the format as an open standard on December 18, 2025 (agentskills.io; spec repo Apache-2.0), and the format's frontmatter-plus-lazy-body shape has been adopted natively: OpenAI's Codex reads SKILL.md folders ("Skills build on the open agent skills standard," scanning .agents/skills upward from the working directory), GitHub Copilot added support on December 18, 2025 (reading .github/skills, .claude/skills or .agents/skills, across cloud agent, code review, CLI and IDE modes), Cursor 2.4 loads project skills from .cursor/skills/, and Gemini CLI reads user and workspace skill directories including .agents/skills — with the spec site's showcase adding VS Code, JetBrains Junie, Goose, OpenCode, Amp, Factory and Kiro. The practical upshot for a client: process knowledge packaged as a skill is not tool lock-in the way a proprietary rules file is, and .agents/skills is emerging as the shared path. All of this list is fast-moving; re-verify before relying on any one tool's support.

Skills execute code — the trust boundary

A skill is not inert prose. The official security guidance is unambiguous: "Use Skills only from trusted sources: those you created yourself or obtained from Anthropic. Skills give Claude new capabilities through instructions and code, which also means a malicious Skill can direct Claude to invoke tools or execute code in ways that don't match the Skill's stated purpose" — with a documented worst case of data exfiltration or unauthorized system access for unaudited third-party skills. Treat skill installation like dependency adoption: review before install, pin what you reviewed, and gate additions through the same PR discipline as any durable artifact (I2). M7 covers the supply-chain framing; I4 already met the ancestor — Voyager's "ever-growing skill library" of executable, reviewable procedures.

3Nested context files and scoped rules

The second mechanism family scopes instructions by location. Claude Code's semantics are the fully documented reference: memory files load in a fixed order — managed policy (e.g. /etc/claude-code/CLAUDE.md), then user (~/.claude/CLAUDE.md), then project (./CLAUDE.md), then ./CLAUDE.local.md for private, uncommitted preferences — and the disclosure rule is verbatim: "CLAUDE.md and CLAUDE.local.md files in the directory hierarchy above the working directory are loaded in full at launch. Files in subdirectories load on demand when Claude reads files in those directories." That one sentence is the monorepo pattern: a small root file with the always-true facts, and per-package files that cost nothing until the agent actually works there — packages/billing/CLAUDE.md taxes only billing sessions.

The import illusion

Claude Code's @path imports (recursive, "a maximum depth of four hops") look like lazy loading and are not: "imported files still load and enter the context window at launch." Imports are an organization feature — splitting one logical core across files — not a disclosure feature. Teams routinely "trim" a bloated CLAUDE.md by moving half of it into imported files and celebrate a diet that changed nothing. Deferral comes only from the mechanisms that actually gate loading: subdirectory files, path-scoped rules, skills.

AGENTS.md, the cross-vendor sibling (stewarded by the Agentic AI Foundation under the Linux Foundation; "used by over 60k open-source projects," self-reported), specifies nesting with one sentence: "Agents automatically read the nearest file in the directory tree, so the closest one takes precedence and every subproject can ship tailored instructions." The fine print matters at a client, because implementations diverge: Codex documents walking the tree and concatenating from root down (with AGENTS.override.md for hard replacement), while other tools resolve nearest-first. The defensive authoring rule that works under both: nested files refine their ancestors — they never contradict them, so precedence order cannot change behavior. (Empirically, contradictions may cost less than feared — §5's factorial study found no detectable adherence effect from contradictory adjacent files — but "the agent tolerates our inconsistency" is not an architecture.)

Every major tool now ships the same triggers under its own names — the strongest evidence that this is a convergent pattern, not one vendor's quirk:

Windsurf's hard character caps are worth quoting to clients precisely because they are deterministic where everyone else's budget is advisory (E5's distinction, applied to context itself). Tool details are point-in-time mid-2026.

4The map layer — indexes as tables of contents

Tiers 2 and 3 share a dependency that is easy to miss: the agent cannot ask for what it does not know exists. Lazy loading without a map degrades into blind search; the always-loaded core earns its cost precisely by telling the agent where things are. That is the index pattern — and it predates agents. Alex Kladov's ARCHITECTURE.md convention (February 2021) built the human version on an observation that transfers to agents unchanged: "it takes 2x more time to write a patch if you are unfamiliar with the project, but it takes 10x more time to figure out where you should change the code." His authoring rules are the anti-staleness discipline this workbook keeps arriving at from other directions: name the bird's-eye modules and entry points, keep it short, and "only specify things that are unlikely to frequently change. Don't try to keep it synchronized with code."

The machine-generated end of the same idea is aider's repository map (Paul Gauthier, October 2023 — before any context-file convention existed): tree-sitter parses every file into definitions and references, a graph ranking algorithm over the file graph picks the most relevant identifiers (personalized PageRank in the implementation), and the result — signatures, not bodies — is packed into a fixed budget (--map-tokens, default 1k). It is a table of contents computed from code, derived and regenerated per I1's rules, and its economics are the whole argument in miniature: a 1k-token map that routes the agent to the right 3 files beats 100k tokens of pasted source. Claude Code's MEMORY.md index (I4) is the same shape for the agent's own notes — a one-line-per-memory index always loaded, topic files on demand.

Two governance corollaries. First, indexes are durable, owned artifacts (I1) that are doubly freshness-critical (I3): ordinary stale prose misleads the session that reads it, but a stale map misroutes every session, silently, at the top of the funnel — so the two-snapshot CI grep from I3 (does every path/name the index mentions still exist?) belongs on index files first. Second, an index needs a consumer. The cautionary tale is llms.txt: of 137,210 domains Ahrefs analyzed (May 2026 data), 28% had published a valid llms.txt — and 97% of those files received zero requests of any kind; AI-retrieval bots accounted for ~1.1% of the few fetches that did happen. An index written for readers who never arrive is decoration. Before writing ARCHITECTURE.md, docs/INDEX.md or a repo map, confirm the loop: the core points to the index, the agent's tools can follow the pointer, and something in CI notices when the pointer breaks.

5Gardening the always-loaded core

Everything above moves content out of the core; this section is about what earns a place in it. The official budget is now explicit — Claude Code's memory documentation: "Size: target under 200 lines per CLAUDE.md file. Longer files consume more context and reduce adherence" — and the failure mode above the budget is not truncation but dilution: "CLAUDE.md files are loaded in full regardless of length, though shorter files produce better adherence." (Earlier editions of this workbook said "under ~300 lines"; that matched the practitioner range — HumanLayer's widely-cited guide says "< 300 lines is best, and shorter is even better," and keeps its own root file under sixty — but the official target is 200, and that is the number to hold a client to.) The same guide supplies the capacity argument: frontier models follow roughly 150–200 instructions with reasonable consistency (citing arXiv:2507.11538), and Claude Code's system prompt already spends ~50 of them before your file adds one.

What earns a line, per the official guidance: facts the agent needs every session — build commands, conventions, project layout, "always do X" rules — plus pitfalls and rationale the codebase cannot reveal. What does not: anything derivable — directory listings, dependency lists, architecture overviews the agent can read from the repo. Claude Code's /doctor now automates exactly that triage, proposing trims of derivable content while keeping "pitfalls, rationale, and conventions that differ from tool defaults." Procedures and per-area material go down a tier: multi-step workflows become skills; area-specific rules become subdirectory files or path-scoped rules. HumanLayer's one-liner for why the core deserves this care is the leverage argument: the file "goes into every single session" — "a bad line of code is a bad line of code. A bad line of an implementation plan has the potential to create a lot of bad lines of code."

The evidence on what context files buy is younger than the practice, and it cuts precisely — three studies, three complementary answers. Presence pays: Lulla et al. (ICSE JAWs 2026, arXiv:2601.20404), across 10 repositories and 124 pull requests, found agent runs with an AGENTS.md present completed comparably while showing "lower median runtime (Δ 28.64%) and reduced output token consumption (Δ 16.58%)" — the file works as an efficiency device. Content decides the sign: the ETH Zurich evaluation (Gloaguen, Mündler, Müller, Raychev & Vechev, arXiv:2602.11988) is the study this chapter leans on hardest — "providing context files does not generally improve task success rates, while increasing inference cost by over 20% on average." Underneath the average: LLM-generated context files reduced resolution rates (−0.5% on SWE-bench Lite, −2% on the authors' CTXBENCH) while raising cost (+20% and +23% respectively), whereas developer-written files managed "an increase of 4% on average" — and the paper's diagnosis is the duplication mechanism this module keeps meeting: generated files reproduce what the agent can discover itself (READMEs, layouts, build tooling), so "context files do not provide effective overviews." Auto-running /init and committing the output is, measurably, paying more for slightly worse results. And structure is not the lever: McMillan's factorial study (arXiv:2605.10039; 1,650 Claude Code sessions, 16,050 function-level observations) varied file size, instruction position, file architecture and contradictions between adjacent files — and found no detectable adherence effect from any of them after multiple-testing correction (Bayes factors 0.05–0.10, evidence for the null); what did move compliance was session depth, roughly "5.6% lower odds of compliance per step." Read together: keep the file (Lulla), fill it only with non-derivable knowledge (ETH), and don't expect formatting tricks to substitute for short sessions and deterministic gates (McMillan, E5).

For calibration, the population: 90.6% of 2,853 agentic-tool repositories in the AIware 2026 census carry context files at all — 4,768 of them, led by CLAUDE.md at 34.4% over AGENTS.md at 31.6% (Galster et al., arXiv:2602.14690). The instrument is ubiquitous; the discipline above is what separates the ones that pay from the ones that cost.

6Failure modes

Each mechanism fails in a characteristic way, and all five failures are quiet — nothing errors; the harness just underperforms.

The kitchen-sink core. Everything "important" goes in CLAUDE.md; at 900 lines nothing is important. The signature is instructions the agent visibly ignores while the file's authors add more emphasis — capitals, "CRITICAL", repetition — instead of removing competitors for attention. The fix is §5's triage, and the deterministic backstop is E5's: a rule the team cannot afford to see ignored does not belong in prose at all; it belongs in a hook or a validator.

The auto-generated core. The measured version of the same disease (§5): generated-and-committed context files duplicate the discoverable, cost more, and resolve slightly less. If a generator drafts the file, a human must delete its redundant majority before commit — the ETH numbers are the argument to show whoever asks why the "free" file needs editing.

The dead skill. A skill whose description fails to fire never loads — invisible in any log, discovered only when someone asks why the agent keeps reinventing the workflow the skill encodes. Symmetrically, an over-broad description fires everywhere, taxing unrelated sessions with a 5k-token body. The description is a retrieval query written in advance; it deserves review as such (test: does the skill trigger on the three tasks it exists for, and stay silent on three neighbours?).

The stale map. §4's edge: a renamed package, a moved doc, and the always-loaded pointer now routes every session to a void — or worse, to the old location a lazy git history still contains. Index files get I3's reference checks first, and matklad's rule pre-empts most of it: maps name only what changes slowly.

The import illusion & the nesting mirage. §3's pair of false economies: @imports that reorganize the bill without shrinking it, and nested files that assume nearest-wins semantics one tool honors and another concatenates. Both are verified in minutes — load a session, check what actually entered context (/context in Claude Code) — and both persist for months in teams that never look.

One boundary is not like the others

Dead skills and bloated cores waste tokens; an untrusted skill executes. The failure modes above degrade performance — a malicious or sloppy third-party skill compromises the machine it runs on (§2's warning, M7's supply-chain frame). Disclosure mechanisms move content into the agent's trust perimeter automatically, which is exactly why I2's write-gates guard the directories they load from: whoever can write to .claude/skills/ or the root context file programs every future session.

7Week one at the client

The disclosure audit is a natural early-engagement move: cheap, diagnostic, and it produces its own before/after numbers.

Audit — day one
  • inventory every context artifact: root and nested context files, rules directories, skills, indexes — the census says there will be several
  • measure the always-loaded core: lines and tokens per session before any work happens (Claude Code: /context)
  • classify each core line: needed-every-session · derivable-from-repo · per-area · procedure · dead
  • check the map: does the core say where specs, docs, ADRs and skills live — and do those pointers resolve?
Restructure — the rest of the week
  • trim the core toward the 200-line target: delete derivables (/doctor-style), demote per-area lines to subdirectory files, procedures to skills
  • add the map lines the audit found missing — one pointer per branch, nothing more
  • package the first two skills from real recurring procedures — our own candidates: the task-spec template (H1) and the characterization-test workflow (M5)
  • wire I3's reference check over core + index files in CI, so the map cannot silently rot
  • re-measure: same session type, tokens before work, and instructions-followed spot checks — the before/after pair is week-one evidence for T2's pilot
Rules that keep it healthy
  • every core line answers "which sessions need this?" — "all of them" or it moves down a tier
  • nested files refine, never contradict — safe under both precedence semantics
  • skills from outside the team are reviewed like dependencies, gated like durables (I2)
  • the budget is enforced where possible (CI line count on the core), advised where not (E5)

9Takeaways for our engagement

Three sentences

Progressive disclosure is the discipline of keeping the always-loaded core small and making it a map — Skills, nested context files, scoped rules and indexes are four mechanisms for the same three tiers: always, on relevance, on demand. The budget is official and concrete — target under 200 lines of core, ~100 tokens per skill's metadata, bodies under 500 lines — but the evidence says the real lever is content: files of non-derivable knowledge pay (+2.4%, −28.64% median runtime), generated duplication measurably costs (+23% for −2%), and formatting tricks move nothing. Guard the mechanism like the infrastructure it is: review skills like dependencies, reference-check the map in CI, and put must-hold rules in hooks, not prose.
