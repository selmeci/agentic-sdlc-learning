0Why an autonomy taxonomy matters

E1–E5 built the components of a capable, verifying, governed agent. E6 answers the question a client actually cares about: how much do we let it do on its own? The L1–L5 taxonomy gives us a shared vocabulary for that decision — and, more importantly, reframes it. Autonomy level is not a measure of how good the model is; it is a staffing decision about how much judgment we delegate and how much we keep.

For our engagement this is the spine that ties the other components together. Each level demands specific components be in place (verification for L3, governance for L4), so "what level is this client ready for?" is really "which components do they actually have?" — which makes E6 the bridge from the framework (E1–E5) to the assessment (checklist) and the phased rollout (E7).

The one idea to internalise

Higher is not better. L5 is not the goal and L1 is not failure. The right level is the one the client's ecosystem can support safely — usually L2 today, with robust L3 as the ambitious target. Selling "full autonomy" is how engagements fail; selling "the highest level your verification and governance can back" is how they succeed.

1The five levels

Each level is defined by how work is initiated, how far the agent runs unattended, and — the real differentiator — which harness components must be in place to make it safe. Read the ladder as increasing delegation of judgment, not increasing model capability.

L1

Assistant

inline, single-file

Autocomplete-style suggestions the human accepts or rejects keystroke by keystroke. The human does all the work; the agent shortens it. Needs only prompt + context (E1/E2).

L2

Conversational

multi-turn, navigates the codebase

Back-and-forth in an IDE chat: the agent reads the codebase, proposes changes, explains. The human still drives and integrates. Adds tools (E1). Where most teams realistically are.

L3

Delegated task → PR

today's production ceiling

A scoped ticket is handed off; the agent works asynchronously and returns a pull request a human reviews and merges. This is where real delegation begins — and where the full harness (E3), verification loop (E4) and governance (E5) become mandatory, because the human is no longer watching every step.

L4

High autonomy

narrow tasks only

The agent selects its own work from a stream and acts with rare human intervention — but only for narrowly defined, repeated, deterministically-verifiable tasks. Real examples: Dependabot (dependency bumps), Uber's FlakyGuard (Uber engineering blog: of 1,115 flaky tests, 380 fixed, 197 landed — re-verify before quoting). Adds memory and blast-radius control on top of L3.

L5

Full autonomy

not achieved

No human in the loop for general software work. Not a real option today; treated as a horizon, not a target. Listing it matters mainly to make explicit that we are not aiming there.

2The component prerequisites — what each level requires

This is the operational heart of E6: each rung is unlocked by specific components being genuinely in place. This table is how we translate "what level?" into "what must exist first?" — and therefore how E6 drives both assessment and rollout.

LevelMust have in placeWhat gates the step up
L1–L2prompt + context + tools (E1, E2)basic retrieval; a human watching every change
L3+ harness, verification, governance (E3, E4, E5)a fast, trustworthy test loop; deterministic guardrails; PR review capacity
L4+ memory, blast-radius controla narrow, repeated task with a deterministic oracle; bounded, reversible effects
L5— (not achieved)an unsolved problem, not a configuration
The rule that caps everything

Never scale the number of agents beyond code-review capacity. Autonomy and volume are the same budget seen twice: more agents or higher levels both produce more output needing review. When output outruns review, the effective level must come down, regardless of how capable the model is. This is why E7's review-bottleneck data (+98% PRs, +91% review time — Faros AI vendor telemetry) is really an E6 constraint.

To see what the "staffing decision" headline cashes out to, run the arithmetic (numbers illustrative — measure your own): suppose one senior reviewer can genuinely review — read, question, sometimes reject — about five agent PRs a day. Two such reviewers cap an L3 workflow at roughly ten merged agent PRs a day, regardless of how many agents are provisioned. Adding a third, fifth or tenth agent does not raise throughput; it only deepens the review queue and stretches time-in-review, which is the same failure the Faros AI telemetry above describes. The only levers that raise the cap are more review capacity, cheaper review per PR (smaller diffs, verification evidence attached), or moving a narrow slice to L4 where a deterministic oracle replaces the reviewer.

3How to choose the level — the assessment lens

The level is set by the weakest enabling component, not the strongest. A brilliant model on a repo with no tests is capped at L2, because L3 has no verification loop to make delegation safe. So the assessment is component-by-component:

Signals that support L3
  • fast, trustworthy tests (E4) with real teeth (mutation score)
  • CI that returns a signal the agent can read
  • deterministic governance: hooks, sandbox, egress (E5)
  • enough human review capacity for the PR volume
Signals that cap at L2
  • no/slow/flaky test suite
  • advisory-only config; no enforcement points
  • no dev/prod separation, RBAC or secrets hygiene
  • review already saturated

Phrases like "fast, trustworthy tests" and "enough review capacity" only bite if they are checkable, so turn them into an assessment instrument. The rubric below gives each prerequisite a concrete heuristic and a way to measure it. Every threshold here is a judgment-call practitioner heuristic, not a standard — tune the numbers to the client — but the discipline is the point: each rung's prerequisites must be checkable, or the level decision is vibes.

PrerequisiteConcrete heuristicMeasured how
per-turn signala focused test subset returns in seconds to low minutes; the full suite is bounded and predictabletime the agent's inner loop: relevant-subset wall-clock, full-suite p95 in CI
test trusttests actually fail when the code is wrong — mutation score on the target module around ~70% is a pragmatic bar (E4; practitioner heuristic, not a standard)run mutation testing on the specific modules the agent will touch, not the repo average
review capacityreview queue depth trend is flat; median time-in-review is not growingPR analytics over the last few sprints: queue depth over time, median time-in-review
intervention ratea human rarely has to correct the agent mid-task; a rising rate means the level is set too highcount mid-task human corrections per delegated task; watch the trend, not the absolute
blast radiusthe agent's writes are confined to allowlisted paths; effects bounded and reversibleaudit the hook/sandbox logs (E5): zero writes outside the allowlist over the trial window
The level is a dial, not a ratchet

Autonomy governance is bidirectional: alongside the promotion rubric, define demotion triggers up front — review queue saturating, intervention rate rising, escaped-defect rate climbing, the test suite slowing past its budget. Who decides matters: the accountable reviewer or lead pulls the trigger, never the agent operator, whose incentives all point at keeping the dial up. And the speeds are deliberately asymmetric: promotion is slow — one level at a time, after the rubric above holds for a sustained period — while demotion is instant: the moment a trigger fires, drop one level immediately and investigate afterwards. A team that can only turn the dial up has built a ratchet, and ratchets fail loudly.

Note L4 is not "L3 done well" — it is a different shape of task. You do not graduate a general delegated-task workflow to L4; you identify a narrow, repeated, deterministically-checkable slice (dependency updates, flaky-test repair) and run that slice at L4 while everything else stays at L3. Level is per-workflow, not per-team.

4The parallel scales — L, P and D

E6's engineering ladder has two siblings elsewhere in the workbook, and seeing them together clarifies all three. They share a shape (5 rungs, top rung an anti-pattern or unreached) but differ in what does the verifying.

ScaleLayerWhat verifies each stepTop rung
L1–L5engineering (E6)deterministic tests / CIL5 not achieved
P1–P5product (M2)human judgment + consistency checksP5 anti-pattern
D1–D5design (M6)conformance checks + designer tasteD5 anti-pattern
Provenance — flag this honestly

The three scales do not share a pedigree. L1–L5 is our adaptation of published industry autonomy ladders (Swarmia, Tessl and similar — see further reading); P1–P5 and D1–D5 are original syntheses of this study programme, useful lenses but not industry standards. Say so whenever presenting them — a client who searches for "P1–P5 product autonomy scale" should find nothing and not be surprised.

The instructive contrast: engineering can push furthest (L4 in narrow cases) precisely because it has a deterministic oracle — tests either pass or fail. Product and design cap lower (P5/D5 are anti-patterns) because their verification is human judgment, which cannot be automated away. Autonomy rises exactly as far as verification can be made deterministic — the single principle behind all three scales.

6Takeaways for our engagement

Assess (the level question)
  • which enabling components are genuinely in place (E3/E4/E5)?
  • the level is set by the weakest one, not the model
  • is there review capacity for the volume the level produces?
  • which narrow slices could run at L4 while the rest stays L3?
Build (feeds E7)
  • start L1–L2, prove the verification + governance base
  • move to L3 only once E4 and E5 are real
  • pick L4 candidates by task shape, not team seniority
  • set exit criteria per level; bring the level down if review saturates
Three sentences to remember E6 by

L1–L5 is a staffing decision about delegated judgment, not a measure of model capability — and higher is not better, with robust L3 the realistic target and L5 unreached. Each rung is unlocked by specific components (L3 needs verification + governance; L4 needs a narrow deterministic oracle), so the safe level is set by the weakest enabling component, never the strongest. Level is per-workflow, capped always by the rule "never scale agents beyond review capacity."
