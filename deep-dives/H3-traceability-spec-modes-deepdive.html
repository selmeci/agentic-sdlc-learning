0Why it matters

Every artifact in this module exists so that work can be delegated and still be accountable. H1 built the contract, H2 made its criteria machine-verifiable — H3 is about what happens after the handoff: keeping the chain requirement → spec → task → code → test → PR connected while agents generate most of its links. At human speed, weak traceability is an annoyance that surfaces in a quarterly audit. At agent speed it is structural: an agent produces artifacts faster than anyone can reconstruct their provenance afterwards, so the question "which requirement does this code serve?" stops being an audit question and becomes the operating question of the whole delegation loop. Without the link you cannot review by intent (H1), you cannot re-verify after a change (E4), and you cannot triage drift when spec and code disagree (H4).

The failure mode is already documented — though so far as vendor narrative, not measurement. Jama Software describes a systems-engineering team that ran a traceability audit about six months into an AI-assisted initiative and found that "a significant share" of agent-generated code had no traceable link back to any documented requirement, alongside orphaned test cases and specified features that were never built (Mario Maldari, Jama blog, May 2026). No percentage is given, and Jama sells traceability tooling — treat it as an illustration with an incentive, not a statistic. But it names the mechanism precisely: the agent built something, tests passed, PRs merged, and only the audit revealed that nobody could say which requirements the code answered to.

The link is the memory

A human author can be interrogated months later — "why is this here?" gets an answer from memory, however fuzzy. An agent's session is gone the moment it ends (the amnesia argument P3 made for decomposition). For agent-written artifacts the recorded link is not documentation about the knowledge — it is the only knowledge there is. That inversion is why traceability moves from compliance chore to load-bearing infrastructure in an agentic SDLC.

1Thirty years of prior art

Traceability is the least novel topic in this workbook, and that is its strength: the concepts are proven, only the economics changed. The field's founding problem statement is Gotel & Finkelstein's "An Analysis of the Requirements Traceability Problem" (ICRE '94), built on a study of 100+ practitioners, which also drew the still-useful line between pre-requirements traceability (where a requirement came from — stakeholder, regulation, decision) and post-requirements traceability (where it went — design, code, test). The requirements traceability matrix is a standard artifact formalized in ISO/IEC/IEEE 29148; BABOK calls the same thing a coverage matrix. And requirements-management tools solved change propagation decades ago: in IBM DOORS, Jama Connect or Polarion, editing an upstream requirement automatically flags every downstream link as "suspect" until an engineer reviews and clears it — a mechanism we will steal wholesale in §4.

In regulated industries none of this is optional. DO-178C (avionics) requires demonstrable bidirectional trace from system requirements through software requirements to source code, test cases and results; ISO 26262 (automotive) and IEC 62304 (medical device software) mandate the same discipline with different evidence depth. The audit question "show me the requirement behind this code" has had a mandatory answer in aerospace since before most web developers were born.

What the heritage solved
  • the vocabulary: trace links, suspect flags, coverage matrices (29148)
  • the mechanism: upstream change → downstream artifacts flagged for review
  • the governance: bidirectional trace as an audit obligation (DO-178C, ISO 26262, IEC 62304)
What agents change
  • volume — hundreds of artifacts a week, not per release
  • speed — drift accumulates in hours, not quarters
  • authorship — no human memory behind the artifact; the link is the memory (§0)
  • and one thing in our favour: an instructed agent applies ID discipline deterministically — it never finds it tedious (§3)

The practical conclusion for a brownfield client: agentic teams are converging on what regulated industries always did — minus the tooling weight. The job is not to buy an ALM suite; it is to get the suspect-flag idea and the coverage idea running on git primitives (§7).

2The chain — IDs travel forward

Traceability at its simplest is one identifier that survives every hop. The requirement gets an ID at birth; the spec quotes it; the task names it; the branch, the commits and the PR carry it; the tests encode it in their names. Each hop is written down at the moment the artifact is created — by the author, human or agent — which is what makes the whole chain greppable, auditable and free of inference.

The two spec-driven toolchains implement the forward chain at different resolutions — a genuine design contrast, not just branding (tool facts point-in-time, mid-2026; re-verify). Kiro traces at the level of individual acceptance criteria: requirements.md numbers every criterion (1.1, 1.2 …), and each task in tasks.md carries a trailing line naming exactly which criteria it implements — the format Kiro emits is _Requirements: 1.2, 2.1_ under the task entry. GitHub Spec Kit (v0.12.x as of July 2026) traces at the level of the feature: /speckit.specify creates a numbered directory specs/001-feature-name/ holding spec.md, plan.md and tasks.md, plus a matching git branch 001-feature-name — folder- and branch-level linkage, with no per-requirement ID system. Kiro answers "which criterion does this task serve?"; Spec Kit answers "which feature does this branch serve?". For a brownfield client the difference matters: criterion-level trace is what lets a failing test point back at a specific requirement line.

Outside the toolchains, the carriers are ordinary git and forge conventions: the ID in the branch name, a commit trailer (Refs: REQ-12 in the Conventional-Commits footer position), the forge's closing keywords in the PR body ("Closes REQ-12" auto-links and closes the tracker item), and the ID embedded in test names so a red test announces which requirement is failing. This is also where H1's anatomy pays off twice: component E (context links) is where the contract records the chain backwards — the requirement, ADRs and code paths the task descends from — and component F (Definition of Done) is where the tech lead demands the chain forwards: "PR links REQ-12; tests named for it." A T0 micro-task (H1's sizing tiers) may carry nothing but an issue ID; a T2 contract carries the full set.

3Structural links vs inferred links

There are exactly two ways to know which requirement a piece of code serves: write the link down when the artifact is born (structural), or reconstruct it afterwards from textual similarity (inferred — the research field of automated trace-link recovery). The research verdict on inference is remarkably stable across two decades: never reliable enough for unattended use. The retrospective by Cleland-Huang, Hayes and colleagues ("Automating Requirements Traceability: Two Decades of Learning from KDD", arXiv:1807.11454) documents that classic information-retrieval methods never reached uncritical-use quality; current LLM-based recovery lands around 60–70% recall on benchmark datasets, and even the strongest recent result — a retrieval-augmented pipeline for requirements-to-code recovery in automotive reporting ~85.5% recovery accuracy (RE:FSQ 2025) — comes wrapped in the same caveat every paper in the genre carries: residual "phantom links", naming bias, and an explicit requirement for human review, especially in regulated contexts.

So the defensible claim is not "IDs beat ML" absolutely — LLM recovery is improving fast and the gap is narrowing. It is sharper and more useful: structural links are deterministic, auditable and 100%-precise by construction; inferred links are probabilistic and need a human in the loop. A regulated audit (§1) accepts the first kind as evidence and the second only as a reviewed proposal. ReqToCode (Schlathölter, arXiv:2603.13999, March 2026) makes exactly this argument for the agentic setting: requirement IDs embedded structurally in the codebase persist when AI generates or modifies code, whereas ML-inferred associations degrade and need re-training as the code moves under them.

And here the economics tilt further than they ever did for humans, in an inversion worth savouring: humans find ID discipline tedious, so structural traceability historically decayed — which is what created the market for recovery research in the first place. An agent finds nothing tedious. Put the stamping rule in CLAUDE.md and the deterministic gate in CI (§7), and the agent applies the convention on every commit without complaint or fatigue. The author that caused the traceability problem at scale is also the first author that reliably follows the discipline that prevents it.

Where inference does earn its keep: the brownfield bootstrap

Your client's existing codebase has no links — and no discipline retroactively creates them. A one-shot, human-reviewed recovery pass (LLM-assisted, ~60–85% by method) is the realistic way to bootstrap coverage over legacy code, prioritized by the change heatmap (M5). Inference to bootstrap once; structure to maintain forever. What does not work is running inference continuously as a substitute for discipline.

4Invalidation — the suspect flag

A chain of links is only useful if it reacts to change. The mechanism is §1's suspect flag, generalized: when an artifact at one end of a link changes, the artifact at the other end becomes suspect — possibly stale, pending a human (or gate) re-affirming the link. Code changed? The linked spec, requirement and tests are suspect. Requirement changed? Every task, code path and test tracing to it is suspect. In DOORS or Jama this is a built-in red flag cleared by review; the concept transfers to a git-native stack almost embarrassingly cheaply, because markdown specs versioned in the repo (M4's durable tier) already ride the same diff and the same review as code.

Be precise about what today's agentic tools actually do here, because the marketing suggests more than the products deliver (point-in-time, mid-2026). Kiro does not do drift detection. Its one sync feature — "Sync Files" on tasks.md — analyzes the codebase and checks off tasks that turn out to be already implemented (useful when a colleague did the work in another session). That is one-directional and task-completion-only: nothing in Kiro flags that your code has wandered away from requirements.md, and nothing regenerates a spec from code. Spec Kit has no invalidation mechanism at all. If you want suspect flags, you build them — which is fortunately a CI rule, not a product:

SYNTHESIS

A git-native suspect gate

ours — flag it as such to the client

Two deterministic rules (E5's advisory→deterministic principle), implementable as a CI script over the PR diff: (1) code-side: if the PR touches source paths that a spec's context links (H1 component E) name, but does not touch the spec file, label the PR suspect-spec — advisory at first, a failing check once the team trusts it. (2) spec-side: if a spec file changes but no linked code or test changes with it (and the PR is not labelled spec-only), flag the reverse. Both rules are greps over the diff plus the spec's own link list — no ML, no new tooling, and the suspect label lands exactly where H4's triage question picks it up: is the code wrong, is the requirement wrong, or did we learn something?

The deliberate modesty of this design is the point. The heritage tools implement suspect tracking with full link databases and audit trails because their industries must prove it; a product team needs the reflex, not the database — a PR label that forces the drift conversation while the diff is still open, instead of an audit that discovers it two quarters later.

5The three spec modes

Everything so far assumed the spec survives the task. Whether it should — and what role it plays afterwards — is the sharpest open question in spec-driven development, and the cleanest framing is Birgitta Böckeler's (Thoughtworks, martinfowler.com, October 2025), from her analysis of Kiro, Spec Kit and Tessl. Her three modes, verbatim: spec-first — "a well thought-out spec is written first, and then used in the AI-assisted development workflow for the task at hand"; spec-anchored — "the spec is kept even after the task is complete, to continue using it for evolution and maintenance of the respective feature"; spec-as-source — "the spec is the main source file over time, and only the spec is edited by the human, the human never touches the code." A terminology warning her own institution supplies: the Thoughtworks Radar notes that SDD terms are "used inconsistently … without shared definitions" across the industry — you will meet renamed variants (one circulating blog rebrands the third level "spec-as-truth"); anchor on Böckeler's definitions and cite them.

H3's angle on the modes is operational: each mode implies a different traceability contract. That is what the table adds beyond E8's strategic recommendation.

ModeThe spec is…Traceability means…Invalidation burden
spec-firstscaffolding — may not outlive the taskprovenance note: "this PR grew from that spec" — links die young by designnone (nothing durable to drift)

Spec-as-source deserves respect as an idea and honesty as a status report. The ambition traces to Sean Grove's The New Code (the manifesto H1 dissects — intent as the durable asset, code as its "lossy projection"). Its boldest commercial bet was Tessl — which called its own approach "spec-centric"; spec-as-source is Böckeler's classification of it — where generated code files carried // GENERATED FROM SPEC - DO NOT EDIT and mapped 1:1 to spec files, backed by $125M of funding. The status report: in January 2026 Tessl repositioned toward an "agent enablement platform" — a versioned registry of agent skills ("Skills on Tessl") — and the pure spec-as-source framing is now its history rather than its headline (point-in-time; re-verify before quoting to a client). The most heavily funded attempt at code-as-disposable-byproduct pivoting away is itself evidence about how far the mode is from production reality.

The principled objection was stated by Kent Beck (LinkedIn, December 2025), on descriptions of SDD that require the whole spec before implementation: this "encodes the (to me bizarre) assumption that you aren't going to learn anything during implementation that would change the specification." That is precisely the assumption spec-as-source hard-codes and spec-anchored relaxes: an anchored spec is allowed to be wrong and get corrected by implementation learning — the suspect gate (§4) plus H4's triage are the machinery of that correction. Hence the recommendation E8 already made and H3 prices out: spec-anchored for a brownfield client — with eyes open that "anchored" is not a free lunch but a commitment to the invalidation discipline of §4. (The tool field keeps moving: the Radar now also lists OpenSpec, a fourth workflow built around change proposals — propose → apply → archive — rather than feature folders. Point-in-time.)

6Provenance — who wrote this code?

Traceability has a second axis the heritage tools never needed: not "which requirement does this serve?" but "who — or what — wrote it?" Review policy (E6's autonomy levels), incident forensics, licensing and emerging regulation all hinge on knowing whether a change was human, assisted, or delegated — and that fact, like the requirement link, must be stamped at creation or it is gone.

The evidence that it is gone by default is now quantified. The AIDev census (Khosravani & Mockus, arXiv:2606.24429, June 2026) mined ~180 million repositories for agent activity and found that no single detection method captures more than a fraction of it: bot-account detection alone recovered just 3.3% (28,154) of Claude Code's actual commits — a 30× undercount against the 886,122 commits identifiable when four trace types are combined (bot accounts, commit-message signatures, author-name suffixes, and config-files-only "silent agents" like Copilot or Cursor, which leave no commit fingerprint at all). The census volumes — Claude Code 886k, Replit 315k, Google Jules 216k, Devin 98k in its late-2025→early-2026 window — matter less than the method finding: if provenance is not stamped deliberately, it cannot be reconstructed later, even by researchers trying hard. Strong independent evidence, and the provenance mirror of §0's Jama story.

What stamping looks like is currently a convention debate, not a standard (practitioner synthesis, point-in-time). One camp uses git's Co-authored-by: trailer — which forges render as co-authorship; the counter-argument runs that co-authorship implies shared accountability and "a model can't hold accountability", so the Linux-kernel-style Assisted-by: <agent>:<model-version> trailer is more honest, keeping the human's Signed-off-by as the accountability line. The debate has a cautionary tale: Microsoft made VS Code append a Copilot co-author trailer by default and reverted it around May 2026 after backlash (it fired even with AI assistance disabled). GitHub's own coding agent models the middle ground: Copilot-authored PRs are searchable by author, while release notes credit the human who asked for the work — the agent is part of the provenance, the human remains responsible for whether it ships. Regulation is converging on the same demand from above: the EU AI Act's technical-documentation duty (Article 11) requires "traceability throughout the system's lifetime", with high-risk obligations enforceable from 2 August 2026 — though note the popular reading that maps this onto which model generated which code under which spec is so far a vendor gloss on the statute, not its text; verify against the Act before citing it to a client.

Policy beats archaeology

Pick one trailer convention, write it into CLAUDE.md, and enforce its presence with a CI grep (E5). Record agent, model version and — where the harness supports it — a session or task ID, which links the commit back to the contract that authorized it (H1) and the autonomy level it ran at (E6). Three lines of convention at write time versus a research project at audit time: the AIDev census is the measurement of what the research project costs.

7A minimal ID scheme for a brownfield client

Everything above compresses into one deployable scheme. It is our synthesis — assembled from the heritage mechanisms (§1), Kiro's criterion-level idea (§2), the structural-links argument (§3) and the provenance policy (§6) — not an industry standard, and we present it to clients as such.

The scheme — five moves
  • 1 · Registry. specs/ in the repo; one markdown file per requirement cluster, named REQ-012-discount-codes.md; changed only by PR (M4 durable tier)
  • 2 · Contract. IDs live in the contract's components E and F (H1): context links trace backwards, DoD demands the forward chain
  • 3 · Stamping. CLAUDE.md instructs the agent: branch req/012-*, trailer Refs: REQ-012, provenance trailer (§6), tests named test_req012_*
  • 4 · Gates. Two CI greps: PR must carry a REQ id; the suspect rule of §4
  • 5 · Bootstrap. one reviewed inference pass over legacy hot spots (§3), then discipline only
Deliberately skipped — until scale demands
  • RTM/ALM tooling — the registry is the matrix, greppable
  • continuous ML trace recovery — probabilistic answers to a question the scheme answers deterministically
  • per-line provenance — commit granularity suffices outside regulated code
  • the threshold: ID discipline + grep carries you to hundreds of specs; buy tooling when the grep stops fitting in a head

That last bullet answers the topic's open question — how does traceability scale to thousands of agent-generated artifacts? — with a claim we hold with medium confidence: discipline scales further than intuition suggests, because the marginal cost of stamping is zero for an agent (§3's inversion), and the reading side stays cheap as long as links are text in the repo rather than rows in a silo. The genuine scaling wall is human review capacity for suspect flags — which is E7's review-bottleneck finding wearing a different hat, and it is managed the same way: tighten what triggers a flag, not what gets recorded.

9Takeaways for our engagement

Assess
  • can anyone answer "which requirement does this file serve?" — for human code? for agent code?
  • does any ID survive from tracker to PR today, or do links die at the first hop?
  • where do specs live — repo, wiki, nowhere — and who notices when code drifts from them?
  • is agent-authored work distinguishable in git history at all?
Build (feeds E7 rollout)
  • the five-move ID scheme (§7), advisory first, deterministic once trusted
  • the suspect gate as a PR label wired to H4's triage ritual
  • one provenance trailer convention in CLAUDE.md + CI
  • a metric for E7: % of merged PRs carrying a valid REQ id — the traceability coverage curve
Three sentences to remember H3 by

At agent speed the link between requirement and code is the only memory of intent, so it must be written at creation — structural IDs that travel requirement → spec → task → code → test → PR — because reconstructing links afterwards is probabilistic (~60–85%) while stamping them is deterministic and costs an agent nothing. Change management is the heritage suspect flag rebuilt on git: touch code without its spec and the PR gets flagged, feeding the drift triage. Of Böckeler's three spec modes, recommend spec-anchored and budget honestly for its invalidation discipline; treat spec-as-source as a horizon (its boldest bet, Tessl, pivoted away in January 2026), and stamp provenance now — the AIDev census shows unstamped agent authorship is unrecoverable even by researchers trying hard.
