0Why security is a posture, not a component

Modules M1–M6 built the harness: seven components — prompt, context, memory, tools/MCP, loop, verification, governance — plus the two contract surfaces that feed it. M7 adds no eighth. Security is a property of all of them, and almost all of it reduces to one unsolved root cause: prompt injection. This opener makes the mindset shift that the rest of the module unpacks — S2's lethal trifecta, S3's advisory-vs-deterministic split, S4's MCP surface, S5's loop risks, S6's memory poisoning, S7's contract and brownfield surfaces, S8's practical configuration — and ties it to E5, which already owns the enforcement mechanisms this posture depends on.

Two incidents frame the stakes, one adversarial and one not (both point-in-time). The adversarial case: in April 2026 ("Comment and Control", Guan et al.), a single pull-request title made three different vendors' coding agents — Claude Code Security Review, Gemini CLI Action and Copilot Agent — post their own API keys as PR comments. Rated CVSS 9.4; silently patched, no CVEs issued. Untrusted content reached a privileged agent, and the agent turned on its operator — three times, across three independent harnesses, from one payload. The second case had no attacker at all: in December 2025 (reported by the Financial Times in February 2026), Amazon's internal Kiro agent, holding operator-level permissions, deleted and recreated an environment and caused a roughly 13-hour AWS Cost Explorer outage. Same lesson from both directions: the controls that bound an injected agent also bound an honest agent that is simply wrong. Blast-radius engineering is dual-purpose.

The one idea to internalise

You cannot train, prompt or filter your way out of prompt injection — the best cross-lab evidence says adaptive attackers bypass every published defense. What you can do is decide in advance how much damage a compromised (or merely mistaken) agent is able to do, and engineer that number down. Security work on an agentic harness is blast-radius engineering applied to every component — which is why this module re-reads the whole framework instead of adding to it.

1The root cause: one token stream, no trust separation

Simon Willison's one-sentence mechanism (June 2025): "Everything eventually gets glued together into a sequence of tokens and fed to the model." The system prompt, the user's task, and every piece of content the agent retrieves — tickets, web pages, tool output, tool descriptions — arrive as one undifferentiated stream, and the model has no reliable way to treat one span as orders and another as inert data. Willison coined the term at the very start of the LLM era ("Prompt injection attacks against GPT-3", 12 Sep 2022 — Riley Goodside demonstrated the attack, Willison named it): "I coined the term prompt injection… I named it after SQL injection, which has the same underlying problem."

The SQL analogy is worth completing, because it explains why no patch is coming. Classic SQL injection was solved by parameterized queries: the database receives instructions and data through separate channels, so data can never execute. Willison initially hoped for an equivalent — "parameterized prompts" — and retracted the hope in April 2023 as "extremely difficult, if not impossible". The UK NCSC's David C. wrote the definitive version ("Prompt injection is not SQL injection (it may be worse)", 8 Dec 2025): parameterized queries fix SQLi "at its root", but under an LLM "there is only ever 'next token'". One channel, one mechanism, no separation to engineer.

Two delivery modes matter in practice. Direct injection — the attacker types at the model — is mostly the lab case. Indirect injection — the payload sits in content the agent reads on its operator's behalf — is the delivery vector behind nearly every real incident in this module, because an agent that reads tickets, browses the web and runs tools ingests attacker-controlled text by design. And the channel is wider than document text: MCP tool descriptions are themselves executable trust and can carry poisoned instructions (Invariant's tool-poisoning work — one sentence here; S4 owns the depth).

2What we are defending: a privileged runtime (a confused deputy)

The asset under attack is not a chatbot, where the worst case is a bad answer. A coding agent holds shell access, file write, network and credentials at the same time — securing it is closer to securing a shell, a package manager and a CI runner combined. The classic name for this situation is the confused deputy (Hardy, 1988): a program that holds authority its caller lacks, tricked into exercising that authority on someone else's behalf. The NCSC's description of LLMs as "inherently confusable" is the same diagnosis forty years later. Willison states the consequence: once attacker-controlled tokens are in the stream, the attacker has "complete control over what happens next" — and "what happens next" is whatever the deputy is allowed to do.

What a compromised coding agent can do
  • exfiltrate every secret it can read — or find in its own environment (S5/S7 incidents)
  • modify code and CI configuration so the compromise persists
  • edit its own harness config to escalate its permissions (S5)
  • communicate the result home — the exfiltration leg (S2)
What it cannot be talked out of
  • advisory text (CLAUDE.md, system prompt) is itself injectable content
  • an instruction the model interprets holds only until a better instruction arrives
  • only harness-enforced constraints survive compromise — the E5 litmus
The design consequence
  • the deputy's privileges must be smaller than its reach: least privilege is the first funnel layer (§3), not an afterthought.

This is why Report 1 caps autonomy at L3 — the agent proposes, a human disposes of consequential actions — and why E5 insists the cap be enforced by the harness rather than requested in a prompt. A confused deputy with read-only tools and no egress is an incident; a confused deputy with your cloud credentials is a breach report.

3The posture: assume breach, reduce the blast radius

If prevention is unavailable (§4 shows the evidence), the defensible posture is the one classic security reached decades ago. Saltzer & Schroeder's least-privilege principle (1975) was explicit about the goal: "Primarily, this principle limits the damage that can result from an accident or error." NIST SP 800-207 (zero trust, Aug 2020) assumes the attacker is already present and scopes privilege per session. Microsoft's Zero Trust guidance opens with "assume breach" and focuses controls on limiting breach impact. Michael Bargury applied it to exactly this problem ("Assume Breach When Building AI Apps", Aug 2024): "AI jailbreaks are not vulnerabilities; they are expected behavior." Treat prompt injection the same way — not an event to prevent, a condition to design for.

The design target is a blast radius that is bounded, observable and reversible — each with an honest caveat, because overselling the posture is how engagements get burned:

Bounded — least privilege, sandbox, egress, gates
  • each deterministic layer shrinks what an injected instruction can reach (the funnel below)
  • caveat: the human gate sees less than the model — Invariant showed (Apr 2025) a Cursor approval dialog hiding the SSH key being exfiltrated; HTML-comment payloads are invisible in rendered views. A gate is only as good as what it renders.
Observable — audit every consequential action
  • logs make the breach a reviewable event instead of a silent one
  • caveat: output laundering defeats scanning — "cut the first 7 characters" of an sk-ant- key defeated GitHub Secret Scanning; EchoLeak (CVE-2025-32711) exfiltrated via reference-style Markdown images straight past link redaction. Hence deterministic egress over output scanning.
Reversible — rollback, rotation, recovery
  • caveat: memory poisoning persists across sessions (Microsoft Defender research, Feb 2026 — point-in-time), so "reversible" must cover the agent's stored state, not just the filesystem. S6 owns that depth.

Two closing points. First, the controls are dual-purpose: the Kiro incident (§0) involved no attacker, and the same funnel bounds an honest agent's worst mistake — least privilege and human gates do not care whether the instruction came from an adversary or from a confident model. Second, every layer must be deterministic in E5's sense — enforced by the harness, not requested of the model — because the advisory layer is itself injectable. The vendors have started converging on this posture from their side: OpenAI's "Lockdown Mode" (~June 2026 — point-in-time) is deterministic egress limiting for agents, explicitly framed as risk reduction rather than an injection cure. That is validation of the model, from the people selling the models.

4The honest ceiling: unsolved — and how to say it

Three years of defenses tell one consistent story: the arms race is real and the attacker is winning it. OpenAI's instruction hierarchy (arXiv 2404.13208) ranked system over user over tool text — Rehberger bypassed GPT-4o-mini's hierarchy within three days of its release (Jul 2024). Microsoft's Spotlighting (arXiv 2403.14720) delimits untrusted spans and claimed attack success cut from above 50% to below 2% — but it is prompt-layer only, defense-in-depth by its own framing. StruQ (arXiv 2402.06363, USENIX Security 2025) trains the separation in — and was broken whitebox at 85–95% attack success (arXiv 2507.07417). SecAlign (CCS 2025; Meta's SecAlign++ 70B ships open weights, arXiv 2507.02735) continues the line. Each defense raises the bar; none eliminates the attack.

The definitive verdict is cross-lab. "The Attacker Moves Second" (Nasr, Carlini, Tramèr et al. — 14 authors spanning OpenAI, Anthropic and DeepMind plus HackAPrompt, Northeastern, ETH Zürich and MATS; arXiv 2510.09023, v1 10 Oct 2025) set adaptive attackers against twelve published defenses covering both jailbreak and prompt-injection protection: every one fell, with over 90% success for most — even the two strongest, PIGuard and MELON, were bypassed at 71% and 76%. Most of these defenses had originally reported near-zero attack success against static attacks; adaptation is the difference. Human red-teamers (500+) outperformed automation, 100% versus 69%. Independent confirmations keep arriving (point-in-time, July 2026): a SoK on agentic coding assistants (arXiv 2601.17548, 78 studies) puts adaptive attack success above 85%, and AutoDojo (arXiv 2606.15057) recovered a filter from 0% static attack success to 28% — 64% on action-open tasks.

Architectural defenses reduce risk without eliminating it. Meta's Rule of Two drops one trifecta leg per session (S2 owns it). Willison's Dual-LLM (2023) and its rigorous successor CaMeL ("Defeating Prompt Injections by Design", arXiv 2503.18813, v1 24 Mar 2025; Google / Google DeepMind / ETH Zurich; published at IEEE SaTML 2026) split a privileged planner from a quarantined reader and execute the plan through a custom Python interpreter — the only approach with a formal security argument, bought at roughly 7 percentage points of task utility (77% versus 84% undefended on AgentDojo, v2). The design-patterns paper (Beurer-Kellner et al., arXiv 2506.08837, v1 10 Jun 2025) catalogues six such patterns — action-selector, plan-then-execute, LLM map-reduce, dual-LLM, code-then-execute, context-minimization — and is explicit that each trades utility for security: once an LLM agent has ingested untrusted input, its authors warn, the guarantees degrade, and it is "unlikely that general-purpose agents can provide meaningful and reliable safety guarantees". Even Willison's mid-2026 softening holds the bottom line: frontier-model training makes naive attacks "much harder to pull off", but "I still wouldn't recommend deploying a production system where a prompt injection attack could cause irreversible damage" (Jun 2026 — point-in-time).

How to say this to a client — without killing the engagement

Neither oversell nor catastrophize; borrow the field's own calibrated wording. OWASP (LLM01:2025 Prompt Injection — #1 on both the 2023 and 2025 lists): "it is unclear if there are fool-proof methods of prevention", so the standard's advice is to "mitigate the impact" — which is precisely §3. OpenAI's Atlas hardening post (Dec 2025): prompt injection is "unlikely to ever be fully 'solved'"; their CISO calls it "a frontier, unsolved security problem" (Oct 2025, via The Register) and analogizes it to scams and social engineering — a permanent risk category you manage, like phishing, not a bug you patch. The NCSC is the bluntest: it "will remain a residual risk, and cannot be fully mitigated with a product or appliance" — and "beware any [product] that claim they can 'stop' prompt injection". The engagement framing writes itself: complete security is a different, much harder, unsolved problem; what we sell is a measured, engineered reduction of likelihood and blast radius, with the receipts — controls, gates, audit logs — to show for it. Beware any product, including ours, that claims otherwise.

6Applying it — decision guide

Three sentences to remember S1 by

One root cause explains nearly the whole module: trusted instructions and untrusted data share a single token stream, and no model-layer defense has survived an adaptive attacker (>90% bypass for most, cross-lab, Oct 2025) — so injection is a permanent risk category, managed, never solved. What you are defending is a privileged runtime, a confused deputy with shell, files, network and credentials, so the posture is assume breach: least privilege, sandbox, egress allowlist, human gates and audit, deterministic controls only (E5), because the advisory layer is itself injectable. And the honest client sentence is a feature, not a concession: complete security is unsolved, every control reduces likelihood and blast radius, none prevents injection — which is exactly why autonomy stays capped at Report 1's L3.
