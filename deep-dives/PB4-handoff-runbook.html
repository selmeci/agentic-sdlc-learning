0Why this runbook exists

PB4 turns the caged module into a delivery system. PB1 measured the ground, PB2 built the net, PB3 built the cage — PB4 installs the discipline that decides what the agent inside the cage is told to do: the handoff contract as the unit of delegation (H1), acceptance criteria machine-verifiable enough to be the agent's reward signal (H2), traceability ids that keep spec, code and test linked at agent speed (H3), and the return channel that keeps the contract honest when reality disagrees with it (H4). The exit test is deliberately concrete: one real feature delivered end-to-end against a written contract, with review running against passing acceptance tests.

Like PB3, this runbook has exactly one entry posture: a recorded PB3 GO — the harness-install report with its granted autonomy level and its probe results. Two earlier artifacts are inherited by reference, not rebuilt: PB1's named editor/curator with a protected review budget (the requirement owner of everything below — P4's editor role made concrete) and PB1's delivery baseline including the rework rate (the loop-health number this runbook's ritual exists to move); the acceptance tests built here ride PB2's characterization net and PB3's CI gate. No PB3 GO → there is no cage to run a contract in — go back to PB3 Harness first.

The failure PB4 prevents is the measured default, not an edge case. Agents do not ask: in HumanEvalComm (ACM TOSEM 2025) models produced code instead of a clarifying question in more than 60% of cases that warranted one, and HiL-Bench (preprint) watched 67–88% of agent runs plough ahead where a human answer was required — falling to 1–9% when asking was made part of the task. Ambiguity is expensive: Orchid (preprint) measured a ~7.2-point average Pass@1 drop on ambiguous requirements, 31 points in the worst case. And the other half of the loop fails silently too: SWE-Bench+ traced 31.08% of benchmark passes to weak tests and 32.67% to leaked solutions — green that means nothing. The repair mechanism is also measured: agents that detect ambiguity and ask before patching resolve under-specified issues better than agents that guess (Ambig-SWE, CMU, arXiv:2502.13069 — we cite the mechanism and deliberately no single percentage), and clarification pays even for plain codegen (ClarifyGPT, FSE 2024: +11.52% relative Pass@1).

The ticket is now the prompt — literally

GitHub Copilot's coding agent ↔ Jira integration went GA on 25 June 2026: assign a Jira issue to the agent and a draft PR appears, status streaming back to the ticket. Devin takes Linear issues by assignment or label; Cursor agents take Linear issues and Slack threads. (All point-in-time, verified 2026-07-20.) This is why acceptance-criteria discipline in the issue body is not bureaucracy: the issue body is the prompt, and H1's contract anatomy is what makes it a good one.

What this is not

Not a documentation bureaucracy. The whole discipline is a handful of committed templates, one appended steering-file section, two CI greps and a fifteen-minute weekly ritual; a full contract is 10–30 lines, and T0 work needs no artifact at all (H1's sizing). The counterweight anti-pattern is real: over-specification is HAP1 in H4's catalogue, and AI-assisted code already draws 29.6 review-tokens per changed line against 4.1 on human code (arXiv:2603.15911) — the contract exists to spend scarce human judgment where it pays, not to add mass.

The theory this runbook distils: H1 for the six-component anatomy and T0–T2 sizing, H2 for the story → EARS → Gherkin → tests ladder, H3 for the id scheme, the suspect gate and the spec-mode ladder (Böckeler's spec-first → spec-anchored → spec-as-source: PB4 installs spec-first, and the registry plus gates make it spec-anchored — spec-as-source is deliberately not the target), and H4 for the return channel and the four anti-patterns; E8 places all of it in the SDD landscape.

1How to run it

Run PB4 with one person plus the agent, the requirement owner (PB1's editor/curator) within reach, and the module owner reachable for the spec review. Scope: the PB3 module and one real feature — the smallest one somebody genuinely wants shipped, not a toy (picking it is this topic's open question; a feature nobody wants teaches nothing about the loop). Steps are written as ČO (the action), PREČO (the theory link) and DONE (the exit criterion), and the sequence is B1's discipline unchanged: templates before the first contract, the contract before the tests, the tests before delivery — never verify a step with a later step.

PB3's probe rule carries over wherever a criterion is deterministic: a deliberate probe fires or the thing is not installed. A PR without a REQ id must bounce off the gate; a drifting PR must get flagged; a seeded ambiguity must produce a filed question, not a guess. What cannot be probed here — the quality of an intent paragraph, the honesty of a triage — is exactly the part that stays human, which is the point of the whole module.

In the workbook version of this runbook, progress is saved locally (same store as topic progress). This standalone copy does not persist checkbox state.

2Phase A — Contract discipline: owners, template, ids

Phase A is pure installation — no feature work yet, and nothing PB3 already shipped gets rebuilt. Everything here is a committed template, an append to PB3's steering file, or a grep in PB3's CI: cheap by design, because this part of the theory is deterministic where it matters and the judgment parts are one page of names.

Publish a one-page owner map: for every requirement area the pilot touches, one person entitled to declare the requirement wrong — at this client that is PB1's named editor/curator unless the map says otherwise. Record the three lenses that must see every contract — business (intent, scope), developer (constraints, context links), tester (boundary and failure scenarios) — and the standing rule that the agent is the fourth amigo: it drafts and it asks; it never ratifies. Why: H4's Move 1 — a triage without an owner is a discussion, not a decision; and H1's anatomy assigns every contract component an owner, so the names must exist before the first contract does (P4).

DONE = owner map committed; zero requirement areas without a named owner; the PB1 editor confirmed as triage authority.
# owners.md — who can declare a requirement wrong (H4 Move 1)
| REQ area              | owner (declares it wrong) | business lens | dev lens | tester lens |
|-----------------------|---------------------------|---------------|----------|-------------|
| e.g. checkout / promo | <PM — PB1's editor>       | <name>        | <name>   | <name>      |

# Standing rule — the fourth amigo:
# the agent DRAFTS candidate EARS rules and Gherkin scenarios and ASKS the
# boundary questions. It never ratifies. Ratification is human, per lens.

Put the six-component skeleton (template in §6) into the repo as specs/CONTRACT-TEMPLATE.md, the T0/T1/T2 sizing rule at its head, and record the registry on PB3's write-permission matrix as durable — decisions: agent proposes, human ratifies, changed only by PR. Why: H1 — the unit of delegation is a contract, not a prompt, and the two components humans must write themselves are intent and non-goals: rationale is precisely what generated requirements are weakest at (Rationale Clarity 1.54 for GPT-4 against a 2.66 human ground truth across 153 stories, arXiv:2507.15157), and non-goals are where agents measurably wander. Sizing is the HAP1 guard — a full contract on a fifteen-minute task is process theater, which is how spec discipline dies at clients.

DONE = template with T0–T2 header committed under specs/; the registry recorded on the PB3 matrix's durable row.

Stand up H3's five-move scheme in one step: create specs/ (move 1); append the stamping section to PB3's AGENTS.md master — branch req/NNN-*, commit trailer Refs: REQ-NNN, provenance trailer, tests named test_reqNNN_* (move 3); and install the first of the two CI greps — every PR carries a REQ id that resolves to a file in specs/ (move 4; the second grep is C2). Why: at agent speed the link is the memory. Inference-based trace recovery tops out around ~85% recovery accuracy in the best RAG setups (RE:FSQ 2025) — structural ids are deterministic and cost an agent nothing; and unstamped provenance is unrecoverable after the fact: bot-detection methods recovered 3.3% of Claude Code's actual commits, a 30× undercount (arXiv:2606.24429). On the provenance trailer, one decision to make explicit: PB3's matrix introduced it as a team convention — this step makes it enforced. The cross-project standard has settled on Assisted-by: (Linux kernel, Fedora, LLVM — as of May 2026; not universal: QEMU still bans AI-generated contributions and floated its own AI-used-for: trailer instead), while Claude Code stamps Co-Authored-By: by default; pick one form per repo and grep for that form (point-in-time; EU AI Act disclosure duties from Aug 2026 are the regulatory tailwind).

DONE = probe PR without a REQ id fails the gate and a stamped PR passes, verified once; stamping section live in AGENTS.md.
# 1) APPEND to AGENTS.md (PB3's master — do not fork a second steering file)
## Traceability (H3 five-move scheme)
Every delegated task starts from a contract in specs/REQ-NNN-<slug>.md.
- Branch:          req/NNN-<slug>
- Commit trailer:  Refs: REQ-NNN     (merge PR body: "Closes REQ-NNN")
- Provenance:      Assisted-by: <agent>:<model>   on every commit an agent touched
- Tests:           name acceptance tests test_reqNNN_*
T0 work (< ~15 min, pairing) needs no REQ id. Anything delegated does —
if the task has none, ask for one; never invent one.

# 2) CI gate — .github/workflows/req-id-gate.yml (adapt to the client's CI)
name: req-id-gate
on: pull_request
jobs:
  req-id:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: PR carries a REQ id that exists in specs/
        env:
          TITLE:  ${{ github.event.pull_request.title }}
          BODY:   ${{ github.event.pull_request.body }}
          BRANCH: ${{ github.head_ref }}
        run: |
          ID=$(printf '%s\n%s\n%s\n' "$TITLE" "$BODY" "$BRANCH" \
               | grep -oE 'REQ-[0-9]+' | head -n1) || true
          if [ -z "$ID" ]; then
            echo "::error::No REQ-NNN in PR title/body/branch (T0 work: label the PR t0)"; exit 1; fi
          ls specs/ | grep -qi "^${ID}-" \
            || { echo "::error::${ID} has no file in specs/"; exit 1; }
# Exemption: skip the job when the PR carries the label t0 or chore —
# wire that as an `if:` on the job; keep the exemption list short and visible.

3Phase B — Machine-verifiable criteria: the ladder and the inversion

Phase B runs the ladder on the one real feature. The order inside the phase is the handoff: contract → criteria → red tests → review — each artifact ratified before the next is built from it, B1's never-verify-a-step-with-a-later-step applied to specs. The output is the strange-looking artifact that makes agent delegation work: a committed set of acceptance tests that fail.

Author the feature's T2 contract on the committed template. The ladder is H2's: the story states intent; each requirement becomes an EARS rule (six patterns — card below); each rule that can execute becomes a Gherkin scenario — declarative, real data, and failure paths are half the set (empty result, timeout, unavailable dependency). Division of labour per A1: the agent drafts candidate rules and scenarios and asks the boundary questions — drafting is where measured gains live (grounding requirement generation in project context: +23.2 points case-level in RAGcceptance, arXiv:2508.06888) — and the humans ratify. EARS is not academic: Kiro writes its requirements.md in EARS by default and ships an "Analyze Requirements" ambiguity gate, and across 1,400+ acceptance criteria in 35 AWS projects, roughly 60% of first-draft requirements needed refinement in AWS's own testing (vendor figure) — first drafts are drafts, which is why ratification is a step and not a courtesy.

DONE = contract committed as specs/REQ-NNN-<slug>.md with at least one failure-path scenario and every criterion falsifiable.
# Acceptance criteria — EARS first, Gherkin where executable (H2)
# EARS patterns (Mavin et al., IEEE RE'09 — Rolls-Royce):
Ubiquitous          The <system> shall <response>
Event-driven        WHEN <trigger>, the <system> shall <response>
State-driven        WHILE <state>, the <system> shall <response>
Optional feature    WHERE <feature is included>, the <system> shall <response>
Unwanted behaviour  IF <trigger>, THEN the <system> shall <response>
Complex             WHILE <state>, WHEN <trigger>, the <system> shall <response>

# Gherkin — declarative, real data, intention-revealing (BRIEF); example from H2:
Feature: password-reset throttling
  Scenario: sixth request in the window is throttled
    Given account "ada@example.com" has made 5 reset requests in the last 15 minutes
    When the client submits another reset request for "ada@example.com"
    Then the response status is 429
    And no reset email is sent

# Craft rules: one behaviour per scenario; no UI mechanics ("click", "navigate");
# failure paths are half the set — empty result, timeout, unavailable dependency.

Have the agent do the mechanical half: generate step definitions and test skeletons for the ratified scenarios and commit them red, wired into PB3's CI gate alongside PB2's net. The failing suite is the handoff artifact — H2 §4's point, and the industry's revealed preference: SWE-bench Verified defines "resolved" purely as designated FAIL_TO_PASS tests going green, and its authors had to prune under-specified issues to make that grading fair — spec quality visibly changes measured success. The reason the tests must exist before implementation is SWE-Bench+'s false-green data (31.08% weak-test passes): criteria that arrive after the code describe the code, not the requirement (E4). The acceptance tests inherit PB3's protection from day one — agent-read-only under the guard hook, so the reward signal cannot be edited by the thing it rewards.

DONE = acceptance tests run red in CI end-to-end, verified once; an agent edit of an acceptance test bounces off PB3's hook (probe).

The inversion H1 argues for: move the scarce human minutes from after the diff to before it. Review the contract with the seven questions (checklist in §6), ratify rules and scenarios per lens, and keep it to ten minutes — a contract that needs an hour of review is too big; split it and re-size (T-tiers). The gate only pays while contracts stay at 10–30 lines; past that you have rebuilt the pile of markdown H1 quotes the counter-view against ("To be honest, I'd rather review code than all these markdown files") — the check is falsifiability, not prose volume. This is also where review economics get set: intent settled on paper is what lets D1's code review run against green acceptance tests instead of reconstructing intent from a cold diff (E8).

DONE = seven-question checklist filed on the contract PR with every question answered; contract ≤30 lines; sign-off recorded per lens.

4Phase C — The return channel and the drift loop

Phase C is the loop. A contract that cannot push back is HAP4 — thrown over the wall; a contract nobody re-reads is HAP3 — silently stale. These three steps install the return path: the instruction to stop, the place questions land, the sensor that catches drift, and the ritual that answers both. The loop is what upgrades the spec from spec-first to spec-anchored — an anchored spec is allowed to be wrong and gets corrected by implementation learning (H3).

Append one paragraph to the steering file (template below): on a contradictory, impossible or two-way-interpretable criterion, stop and file a question — never pick a branch silently (H4's return-channel rule 1, verbatim). Create the spec-question label routing to the requirement owner, with the SLA split: blockers get an async answer, everything else queues for the weekly triage. Why: asking is not the default — that is the HumanEvalComm >60% and the HiL-Bench 67–88% → 1–9% from §0 — but the inversion is cheap, because an instructed stop is advisory-layer work agents comply with far more reliably than they volunteer it, and the question itself is an artifact carrying the REQ id, not a chat line that evaporates. The practitioner canon converged on the same loop: SPDD (martinfowler.com, Apr 2026) mandates fix the prompt/spec first, then the code when reality diverges.

DONE = a seeded ambiguous criterion makes the agent file a spec-question carrying the REQ id — not a guess — verified once; the label routes to the named owner.
# APPEND to AGENTS.md (PB3's master)
## Spec questions — stop, don't guess
If an acceptance criterion is contradictory, impossible, or allows two
reasonable readings: STOP. Do not pick a branch silently.
File the question as an artifact, not a chat line: an issue / PR comment
labelled spec-question, carrying the REQ id and the readings you considered.
Blocked on the answer? The requirement owner (owners.md) answers async.
Not blocked? Park the question, continue other criteria, and it will be
answered at the weekly triage. Resume the parked criterion only when the
answer has landed in the spec itself.

# Label definition (issue tracker):
# spec-question — routes to the requirement owner from owners.md.
# SLA: blocker = async same-day; otherwise = next weekly triage.

Install the second CI grep, H3 §4's suspect gate: code-side — a PR that touches source paths a spec's context links name, but not the spec, gets labelled suspect-spec; spec-side — a spec change with no linked code or test change (and no spec-only label) gets flagged in reverse. Run it advisory (a label) first and promote it to a failing check once the team trusts it. Why: no shipping SDD tool detects silent drift for you (Kiro's spec sync is file synchronisation, not drift detection — point-in-time), so the sensor is a grep over the diff and the spec's own link list, not a product — E5's litmus applied to the loop itself. HAP3 is the anti-pattern this catches: a spec the team quietly stopped maintaining keeps steering every agent that reads it.

DONE = probe PR editing a linked source path without its spec gets labelled, and the reverse probe gets flagged — each verified once.
#!/usr/bin/env bash
# suspect-gate.sh — flag PRs whose diff and specs disagree (H3 §4).
# Convention: each specs/REQ-*.md lists its context links as backtick-quoted
# repo paths (component E). Advisory first: emit labels, exit 0; flip the
# exit codes once the team trusts the signal.
set -euo pipefail
BASE="${BASE_BRANCH:-origin/main}"
CHANGED=$(git diff --name-only "$BASE"...HEAD)
FLAGGED=0
for spec in specs/REQ-*.md; do
  [ -e "$spec" ] || continue
  LINKS=$(grep -oE '`[^`]+`' "$spec" | tr -d '`' | grep -E '^(src|app|lib|tests)/' || true)
  [ -z "$LINKS" ] && continue
  HIT=$(printf '%s\n' "$CHANGED" | grep -xF -f <(printf '%s\n' "$LINKS") || true)
  SPEC_TOUCHED=$(printf '%s\n' "$CHANGED" | grep -xF "$spec" || true)
  if [ -n "$HIT" ] && [ -z "$SPEC_TOUCHED" ]; then
    echo "suspect-spec: $spec names changed path(s):"; echo "$HIT"; FLAGGED=1
  fi
  if [ -n "$SPEC_TOUCHED" ] && [ -z "$HIT" ]; then
    echo "suspect-code: $spec changed but no linked code/test changed (label spec-only if intended)"
    FLAGGED=1
  fi
done
# advisory mode: report only. Enforced mode: exit $FLAGGED
exit 0

Fifteen minutes weekly over everything labelled spec-question or suspect-spec, the requirement owner in the chair, one row per item. Every row exits through H4's triage question — "is the code wrong, is the requirement wrong, or did we learn something?" — and every exit is an artifact change: code-wrong turns the criterion into a failing test and a fix; requirement-wrong is a spec PR whose dependents flip suspect; we-learned updates the spec and logs the learning — and it is a normal exit, not a failure. A verbal "we aligned" is not an exit: the answer lands in the artifact, not the chat (H4 rule 4 — the same closed loop SPDD mandates). Until a flagged item has been through this question, it is the process that is broken, not just the spec.

DONE = first triage held and its log committed; every entry exited as a spec PR, a criterion rewrite or a logged learning — none verbal-only.
# triage.md — weekly, 15 min, requirement owner chairs (H4 §3)
| date | item (issue/PR) | REQ | exit: code-wrong / req-wrong / learned | owner | artifact changed (PR/commit) |
|------|-----------------|-----|----------------------------------------|-------|------------------------------|

# Rules:
# - one row per spec-question / suspect-spec item; blockers were answered async
#   but still get their row (the answer must be IN the spec by triage time).
# - every row ends in an artifact change. "We aligned on a call" is not an exit.
# - "learned" is a normal exit — count them; zero learnings for weeks means
#   the questions are not being asked, not that the specs are perfect.

5Phase D — The delivery and the exit gate

Phase D cashes the cheque: the discipline exists so that one real feature can cross it end-to-end.

Run the delivery at PB3's granted autonomy level, not above it (E6 — the level was earned by probes, not enthusiasm): the agent takes the contract, implements until the red acceptance tests are green, files spec-questions instead of guessing, and opens the PR with Closes REQ-NNN and the provenance trailer. Review runs against passing acceptance tests: the reviewer reads the diff knowing intent was settled in B3 and the criteria already executed — that is where B3's ten minutes pay out. The merge tail is the risk this closes: 26.1% of 8,106 studied fix PRs closed unmerged, and reviewer engagement is the strongest merge predictor (MSR 2026) — green acceptance tests give the reviewer something concrete to engage with instead of a cold diff.

DONE = feature merged with AC green in CI; PR carries Closes REQ-NNN + provenance trailer; the return channel fired at least once — or its silence is explained by clarity on a session-log read, not by silent guessing.

Walk §7's exit table and fill the delivery report (template in §6): the installed templates, the contract, the spec-review record, the four probes, the triage log, and the loop-health read against PB1's baseline — rework-rate direction and review latency, measured, not felt: the METR RCT's perception gap (19% slower while believing ~20% faster, E7) is the standing reason self-report is inadmissible here. Why: the report is PB5's input — B6's gates need evidence, and PB5 measures at scale what PB4 proved once.

DONE = report delivered; go/no-go recorded with date, criteria and signatories.

Exit routing

If GO → proceed to PB5 Pilot: the discipline has run end-to-end exactly once, which proves it works and proves nothing about scale — PB5 turns the one-off into a measured pilot against the baseline PB1 captured, with B6's gates and STOP criteria deciding what happens next. The one failure that is not small: if the triage keeps exiting verbally, the client has a culture problem no template fixes — name it in the report rather than shipping a green row you do not believe.

Three sentences to remember PB4 by

The contract, not the prompt, is the unit of delegation — six components, two of them permanently human (intent and non-goals), sized T0–T2 so the discipline never costs more than the work. Acceptance criteria are the agent's reward signal, and the evidence says defaults fail on both sides — agents code instead of asking in >60% of cases that warrant a question, and ~31% of benchmark passes traced to weak tests — so criteria ship as red tests the agent cannot edit, and ambiguity ships as a stop-and-ask rule with a place for questions to land. The loop is the product: ids make drift visible, the suspect gate flags it, and the weekly triage's only legal exits are code-wrong, requirement-wrong or we-learned — with every answer landing in the artifact, never in the chat.
