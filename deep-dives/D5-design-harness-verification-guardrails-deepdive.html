0Why it matters

Every module so far has converged on the same sentence: an agent may do any work it can be made to verify. E4 built that into the engineering harness as the red/green loop; D1 argued the design system is the third artifact because it feeds the same loop on the visual layer. This chapter is the loop itself — the concrete checks that turn "the agent composed a screen" into "the screen is proven to conform," run in an order that spends machine time before it spends human attention. Without it, the design layer is exactly where agent output is least trustworthy: unconstrained generation converges measurably to slop (D1's audit), and accessibility is the sharpest known gap — the CHI 2025 CodeA11y study found that developers without accessibility training fail to prompt AI for accessibility, omit the manual steps the AI flags, and cannot verify compliance (16-developer formative study; Mowar et al., DOI 10.1145/3706598.3713335). The follow-up evidence says the fix is not a better prompt but a loop: FeedA11y (Suh & Malek, arXiv:2503.15885) showed re-prompting agents with machine-generated a11y reports significantly outperforms prompting alone. That is the design harness in one study.

The stakes are not aesthetic. The 2026 WebAIM Million found 95.9% of home pages have automatically detectable WCAG failures — worse than 2025's 94.8%, reversing years of slow improvement, before agent-generated UI has even fully arrived in the sample. And since 28 June 2025 the European Accessibility Act applies to e-commerce, banking, transport and e-book services sold in the EU, with WCAG-based conformance (via EN 301 549) as the de-facto bar. A team that lets agents ship unverified UI is accumulating legal and brand risk at machine speed. The harness is how you get the speed without the bill: everything checkable is checked by machines, in CI, before a designer spends a minute — and the designer's scarce judgment is reserved for the one thing no gate can decide, which is whether the result is good (§5).

1The stack — six machine checks, in the order they should run

The design harness is not one tool but a stack, and the order is a design decision: cheapest and fastest first, most expensive last, human gate at the end. Each layer catches a different failure class, so skipping one leaves a shaped hole; each layer is also a filter, so running them out of order wastes the expensive checks on failures the cheap ones would have caught. Six layers, all pre-dating agents, all now load-bearing: (1) Style & token lint — Stylelint/ESLint rules that reject hardcoded values and off-system imports in milliseconds (§4). (2) The component API contract — TypeScript prop types, and increasingly machine-readable schemas per component: Red Hat's PatternFly published JSON Schema + Zod bundles for exactly this purpose, validating LLM-generated prop payloads at runtime (ButtonSchema.parse(...)) — a contract test for component usage, vendor-OSS but the clearest artifact of its kind. (3) Interaction tests — Storybook play functions (userEvent/expect, imported from storybook/test since Storybook 9) executed in CI via the Vitest addon — or the test-runner on Webpack projects — proving the component behaves, not just renders. (4) The a11y gate — axe-core, ~90 rules enabled by default, zero-false-positives as a design policy (§3). (5) Visual regression — pixel or DOM snapshots against CI-built baselines; the slowest machine check and the one that costs money per snapshot (§2). (6) Drift & usage analytics — asynchronous: token-drift CLIs and adoption dashboards that open PRs instead of blocking them (§6).

2Visual regression in practice — baselines, noise, and the false-positive problem

Pixel-diffing is the oldest and noisiest check in the stack, and its mechanics deserve precision. Playwright's toHaveScreenshot() uses the pixelmatch library: threshold is per-pixel perceptual tolerance in YIQ color space (default 0.2 — do not tighten it to 0 on real UIs, anti-aliasing noise will fail everything), and maxDiffPixels/maxDiffPixelRatio are explicit noise budgets, unset by default. Stabilization is largely built in: the assertion waits for two consecutive identical screenshots before comparing, animations default to disabled, the caret is hidden, and volatile regions (timestamps, ads, live data) are covered with mask locators or a stylePath stylesheet. The baseline discipline is the part teams get wrong: Playwright's docs warn that rendering varies with "host OS, version, settings, hardware, power source, headless mode" — so baselines are committed to git and produced in the same environment that verifies them, in practice a pinned Docker image matching CI, and --update-snapshots is a deliberate, reviewed act, never a CI step. Environment consistency beats tolerance: widening thresholds to absorb flake is how gates quietly stop meaning anything.

Above Playwright sit the managed platforms, and their differences matter for procurement. Chromatic (built by Chroma, the company maintaining Storybook) snapshots every story per browser and viewport — component-level by construction — and splits its product into two workflows that map exactly onto this chapter's thesis: UI Tests is the deterministic regression gate (visual + interaction + accessibility on every commit, TurboSnap trimming runs to tests affected by changed files at a fraction of snapshot cost), while UI Review is the human sign-off lane — designers and PMs assigned to the PR, approval as a requirable status check (§6). Its flake handling is algorithmic, not AI: SteadySnap freezes dynamic content and flags unstable tests without blocking the build; the docs list the causes verbatim — "an animation might be caught mid-frame, a font might load late, dynamic data might change, or a network request might not finish in time." Percy (BrowserStack) takes a different bet: it ships DOM snapshots rendered in its own infrastructure, which removes cross-OS rendering noise by construction, and it is where the AI claims live — a Visual AI Engine (OCR-based text-shift tolerance), Intelli Ignore for dynamic regions, and a Visual Review Agent that draws bounding boxes around "meaningful" changes with natural-language summaries (launched with BrowserStack AI, June 2025; "3× faster reviews" is a vendor metric). Applitools markets Visual AI at "99.9999% accuracy" across "one billion images analyzed" — a 2020 press release; cite it as a vendor claim or not at all. The open-source tier needs a maintenance caveat as of mid-2026: reg-suit and Argos are actively maintained (Argos is the credible OSS Chromatic alternative), while BackstopJS and Loki have been effectively unmaintained since 2024 and Lost Pixel was archived in April 2026 (team joined Figma) — a cautionary tale about building a gate on an OSS tool without a maintenance plan.

The topic's open question — how do we tame false positives so the gate doesn't drown the designer? — now has a documented answer, and it is a sequence, not a knob. First kill variance at the source (pinned render environment, fonts awaited, animations off, randomness seeded, network settled). Then exclude what must vary (mask volatile regions, Intelli Ignore-style dynamic-area config). Only then budget what remains (threshold + maxDiffPixels, set globally so the budget is one reviewed decision, not fifty local ones). Finally, track flake as data: Chromatic's model — flag unstable tests, quarantine, don't block — keeps the gate credible, because a gate that cries wolf gets overridden, and an overridden gate is worse than none (E7's anti-pattern catalog applies verbatim). And there is a second, quieter answer: stop relying on the happy path. Playwright's page.emulateMedia() natively emulates forcedColors:'active', colorScheme:'dark', contrast:'more' and reducedMotion:'reduce'; Chromatic modes do the same for viewport, locale and RTL. Snapshotting each story across that stress matrix turns "works in the demo state" into six cheap deterministic checks — forced-colors breakage, dark-mode token gaps and motion-sensitivity failures are exactly the bugs a happy-path baseline never sees and a designer shouldn't have to catch by hand.

3Accessibility as a gate — what axe proves, and what it can't

Accessibility is where the deterministic gate is most mature, most standardized, and most legally non-optional — and still the most oversold. The engine is axe-core (Deque): an open-source rules library, ~105 rules in the 4.12 line with roughly 90 enabled by default, designed under an explicit zero-false-positives policy ("bugs notwithstanding") — which is precisely the property a blocking CI gate needs, because a gate that sometimes cries wolf teaches teams to ignore it. It is also already everywhere, which matters for the client pitch: Google Lighthouse has run axe-core for its accessibility audit since 2018, Microsoft's Accessibility Insights is built on it, and Storybook's a11y addon "uses axe-core to run the tests" — so wiring axe into CI (Playwright + @axe-core/playwright, or the Storybook 9/10 test integration) adds a gate whose rule set the industry already treats as canonical.

Now the honesty. Deque's widely quoted figure — automated testing finds 57% of accessibility issues — comes from Deque's own 2021 study (~2,000 audits, 13,000+ pages, ~300,000 issues) and measures issue volume, not WCAG success criteria; by that stricter denominator only ~30% of criteria are fully automatable. W3C WAI's position is blunt: tools "can not determine accessibility, they can only assist in doing so" — alt-text quality, focus-order logic, keyboard-trap nuances and screen-reader UX all require human judgment. The classic demonstration is GOV.UK's 2017 "world's least accessible webpage" experiment: 143 deliberately planted barriers, ten tools, 29% of barriers missed by every tool, the best single tool finding 37–41%. So the gate is necessary and nowhere near sufficient: axe in CI catches the agent's mechanical failures before a human looks; a periodic manual audit and assistive-technology testing catch the rest. Both halves are policy, and the second cannot be delegated.

Three 2025–26 developments make the gate more urgent, not less. (1) The AI-code evidence. CodeA11y (§0) showed novices don't prompt for accessibility and can't verify it; Suh & Malek found LLM-generated UI beats human code on the basics (contrast, alt text) yet systematically fails ARIA semantics — and that a feedback loop with a11y reports (FeedA11y) closes the gap where prompting doesn't; Aljedaani et al. (2024) measured perceivability violations in 84% of ChatGPT-generated websites. Agent output needs the gate more than human output does, and the gate's output is best fed back to the agent. (2) Regulation. WCAG 2.2 has been a W3C Recommendation since 5 October 2023 (now also ISO/IEC 40500:2025); the European Accessibility Act has applied since 28 June 2025 to e-commerce, consumer banking, passenger transport and e-books, with EN 301 549 (effectively WCAG 2.1 AA) as the conformance route. (3) The anti-pattern got named and fined. Accessibility overlays — the JavaScript widgets promising compliance without gates — drew an FTC consent order against accessiBe (complaint January 2025, $1M, final order April 2025) for deceptive WCAG-compliance claims; the Overlay Fact Sheet carries 1,000+ professional signatories against the category, and UsableNet data shows 22.6% of H1-2025 accessibility lawsuits targeted sites with overlays installed. The contrast is the lesson: an overlay is a cosmetic layer claiming to remove the problem; a gate is a build-time proof that part of the problem never shipped. And one forward marker: the WCAG 3.0 Working Draft (refreshed March 2026) formalizes graded conformance — Bronze/Silver/Gold — the standards world itself conceding that binary pass/fail is not the whole story, which is §5's argument arriving from a second direction.

4Lint as law — the cheapest gate, installed first

The lint layer is where E5's advisory-vs-deterministic line becomes concrete, because every rule here replaces a sentence somebody used to write in a style guide. The Stylelint core carries the two workhorses: color-no-hex (disallow hex colors outright — every color becomes a token reference) and declaration-property-value-allowed-list (per-property allow-lists, regex-capable, unlisted properties unrestricted). The most token-shaped rule in the ecosystem is the plugin scale-unlimited/declaration-strict-value: for named properties the value must be a variable, function or keyword, with autofix support — "use tokens" enforced at build time, not requested in a prompt. On the JS side, ESLint's no-restricted-imports (core, with per-entry custom messages — "Use <DsButton> from @acme/ds instead") keeps agents off raw UI libraries and on the system. One correction to the folklore: no-hardcoded-styles is not a standard ESLint rule — it is a custom-rule example (one niche MUI-specific plugin ships it) — so treat it as a small rule you write once your tokens land, which is itself a useful exercise in what a gate actually is.

The design-system vendors encode their systems the same way, which is the strongest evidence the pattern is real practice rather than our theory: GitHub's @primer/stylelint-config ships primer/colors, primer/spacing, primer/no-override and primer/no-unused-vars; Shopify's stylelint-polaris rules are explicitly "grouped into categories that help us measure the Polaris design system's coverage" — lint doubles as the adoption metric; Salesforce's current slds-linter (SLDS 2) ships Stylelint and ESLint rules that flag raw values and suggest styling hooks; Carbon's stylelint-plugin-carbon-tokens enforces theme/layout/type/motion tokens with three severity presets and autofix. The last piece of the layer watches the boundary between PRs: the open-source CLI Dembrandt extracts a deployed site's live tokens via Playwright, and --compare baseline.json exits 1 on drift with a machine-readable report — the golden-master move from B2/D7, run as a cron gate, opening a PR instead of blocking one.

5Conformance vs quality — the evidence for where the human gate sits

This is the chapter's load-bearing distinction, and the verification pass corrected its key citation — the correction makes the argument stronger. FrontendBench (arXiv:2506.13832; 148 prompt–test pairs across five component levels) does not use an LLM judge: it executes generated UI in a Puppeteer/Jest sandbox and scores it with predefined test scripts, reaching 90.54% average agreement with human experts. Read correctly, that is the best available evidence for this chapter: scripted, harness-style verification of specific UI behavior agrees with experts ~90% of the time — the deterministic stack works. The flip side is equally well measured. UI-Bench (arXiv:2508.20410) ranks text-to-app tools on holistic visual excellence and had to use expert human pairwise comparison (4,000+ judgments, TrueSkill) — the field built a human panel because no machine scorer was trusted. WebDevJudge (arXiv:2510.18560) tested the alternative directly: the best LLM judge reaches 70.34% agreement with experts on holistic web quality and plateaus there — "LLM-as-a-judge falls short of human-level reliability." Adobe's MLLM-as-a-UI-Judge study (arXiv:2510.08783) finds partial perceptual alignment at best, positioned as a supplement to early UX research; UICrit (UIST 2024) found human design critiques still ranked above the best LLM-generated ones. As of July 2026 there is no solid evidence that an LLM can judge holistic design quality at human level — and the burden of proof is on the claim that it can.

That evidence fixes the architecture: the machine gates verify conformance — inside the vocabulary, on real components, no broken baseline, WCAG rules met, behavior as scripted. The designer gate judges quality — hierarchy, brand fit, taste, whether this was the right design at all. The designer sits at PR level, exactly L3 on the E6 scale, reviewing work whose conformance is already machine-proven; tooling like Chromatic UI Review operationalizes that position (design/product/engineering sign-off as a required PR check), but a checklist and CODEOWNERS do the same job. WCAG 3.0's draft move from binary criteria to graded Bronze/Silver/Gold conformance is the standards world arriving at the same shape: pass/fail for what can be pass/fail, graded human judgment for the rest. And the failure mode of getting this wrong is named in D6: remove the designer gate and you have D5-on-the-D-scale — drift, slop and entropy at agent speed — because a harness that proves conformance says nothing about whether the conforming thing was worth building.

6Operating the gate — who approves, what it costs, what erodes between PRs

Three operational truths finish the chapter. First, the human gate is a workflow artifact, not a meeting. The verifiable mechanics: Chromatic UI Review assigns designers/PMs to the PR and exposes sign-off as a requirable status check; CODEOWNERS routes durable-tier changes (tokens, APIs, new components — D4's classes) to the DS team; the deterministic checks run as required CI checks before either. The much-cited "Uber pattern" — CI blocks the merge and auto-files a Jira ticket — traces to a single Builder.io marketing post with no primary source; treat it as a vendor anecdote. The same post's "Shopify found 14% of admin UI off the Polaris mainline" is likewise vendor-only, though the mechanism underneath (stylelint-polaris's coverage-measuring rule categories) is documented and citable; and "GitHub runs a design check on every PR" is better replaced by the verifiable artifact — GitHub's public AI-powered accessibility-scanner Action (preview, October 2025), which scans, files issues and drafts Copilot fixes. Second, the gate has a budget. Managed snapshots cost money (Chromatic's free tier is 5,000 snapshots/month, Chrome-only — secondary sources; TurboSnap exists because snapshot volume is the cost driver), and review minutes cost more; the funnel order in §1 is how you spend both on failures that survived cheaper filters. Third, systems erode between PRs. The async lane — Dembrandt drift compares on a schedule, Omlet scanning codebases for off-system components and version lag, Figma Library Analytics tracking adoption (extended to styles and variables in 2025) — exists to catch the slow failures the per-PR gate never sees: shadow components, forked patterns, a deployed site drifting from its token baseline. All three open work; none of them approves it. Self-healing changes who opens the PR, never who approves it — D4's rule, and the audit trail from I3 still records why.

Vendor or self-published claims — label every time: Deque's 57% (2021 vendor study, issue volume not criteria); Applitools' 99.9999%/"one billion images" (2020 press release); Percy's Visual Review Agent "3× faster" (launch blog); Chromatic's free-tier numbers (secondary sources); FigmaLint's 26/100→100/100 example (conference-session writeup; the tool itself is Southleft's, Brad and Ian Frost co-present — don't attribute it to Frost); the Builder.io anecdotes (Uber's block-and-ticket, Shopify's 14%, GitHub's per-PR checks — no primary sources found; the Uber one has none at all); Omlet and Figma Library Analytics as adoption signals. Synthesis — ours, flagged: the six-layer stack and its cost ordering; the stress-state matrix as standard harness practice; "a gate is not an overlay"; the advisory-vs-deterministic row mapping. These are Report 5's framings argued from cited practice, not published findings.

Do not cite: "FrontendBench shows LLM-as-judge reaches 90.54% agreement" — the paper has no LLM judge; 90.54% is the scripted harness's agreement with experts (this exact sentence lived in our notes and is now corrected workbook-wide); "ds-lint combines token enforcement with 233 WCAG rules" — no such tool could be found; the figure matches nothing (WCAG 2.2 has 87 success criteria, axe-core ~105 rules) and the claim was retired from Report 5's notes; "ESLint no-hardcoded-styles" as a standard rule — it's a custom-rule example (one niche MUI plugin ships that id); "automated tools catch 57% of WCAG criteria" — wrong denominator, it's 57% of issue volume; the GOV.UK study as "142 barriers, 13 tools, 40%" — actual: 143, 10, 37–41%; "animations: 'disabled' must be switched on in Playwright" — it's the default; "Storybook 9, May 2025" — officially June 2025, and in SB9+ the import is storybook/test, not @storybook/test; any implication that Chromatic has AI diff-review (it doesn't — its flake handling is algorithmic; Percy and Applitools are where the AI claims live); "accessiBe was fined in 2024" — the FTC complaint is January 2025, the final order April 2025; and BackstopJS, Loki or Lost Pixel recommended without a maintenance caveat (stale since 2024, and Lost Pixel archived April 2026).

8Takeaways for our engagement

Assess (first client conversations)
  • Which of the six layers exist in CI today? Most clients have types and maybe lint; the gap is almost always the a11y gate and visual regression — price the missing layers, not the present ones.
  • Where do visual baselines come from — a pinned CI image or someone's laptop? The answer predicts the flake rate better than any tool choice.
  • Who may update baselines, and is that act reviewed? An unreviewed --update-snapshots path is a gate that approves itself.
  • Does any accessibility check block a merge today — and what does the team believe it covers? (The 57%-of-issues vs ~30%-of-criteria conversation is the fastest maturity diagnostic in this chapter.)
  • Who holds the designer gate, and how many hours a week do they have? That number, not model quality, caps the D-level (D6).
  • Is there any drift/adoption telemetry between PRs — or does the system only get checked at the moment someone touches it?
Commit (the defaults we bring)
  • Six layers in cost order; the agent iterates on red before any human review; lint + types + axe land in F1, visual regression as soon as baselines are CI-built (D7 sequencing).
  • Baseline discipline as policy: pinned render image, baselines in git, deliberate and reviewed updates only; mask/seed/freeze first, thresholds last, flake tracked without blocking.
  • axe-core in CI plus a scheduled manual audit — and no overlays, ever; the FTC order is the client-friendly citation.
  • The first three lint rules are color-no-hex, declaration-strict-value and no-restricted-imports with a message pointing at the DS; a custom no-hardcoded-styles rule once tokens land.
  • The stress matrix (forced-colors, dark, contrast, reduced-motion, RTL/long-text, 320px) as standard story coverage, not a quarterly audit.
  • Designer gate at L3 with conformance machine-proven before review; drift and adoption telemetry on the async lane, opening PRs — never approving them.
Three sentences to remember D5 by

The design harness is E4's verification-first loop pointed at the visual layer: six checks in cost order — style lint, API contract, interaction tests, axe-core, visual regression across a stress matrix, and the human gate last — so the agent iterates against red output before a designer spends a minute. The evidence says the split is real and where to draw it: scripted verification agrees with experts ~90% on specific UI behavior (FrontendBench), while LLM judges of holistic quality plateau near 70% (WebDevJudge) and the field still scores visual excellence with human panels (UI-Bench) — so machines prove conformance, and the designer at L3 judges quality. Everything in the stack predates agents and is documented practice; what agents change is the urgency — they ship UI faster than any review scales, they omit accessibility unless a loop forces it (CodeA11y, FeedA11y), and since June 2025 the EU prices that omission in law.
