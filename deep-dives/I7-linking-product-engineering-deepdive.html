0Why it matters

Every engagement has two databases that both claim to describe the same work. The product plane — Jira, Linear, Notion — holds intent: epics, stories, acceptance criteria, the priority call that made this ticket matter. The engineering plane — git and the forge — holds change: branches, commits, PRs, tests, releases. Neither owns the other, and nothing keeps them telling the same story except the discipline of linking. How weak that discipline is by default has been measured for seventeen years: Bird et al.'s classic bias study (ESEC/FSE 2009) counted how many fixed bugs could be connected to the commit that fixed them and got 49.6% in Apache, 41.5% in Eclipse — and 8.1% in OpenOffice. Worse than the averages was the direction of the bias: in Apache, 63% of minor-severity bugs were linked but only 15% of blockers — the more severe the work, the less traceable it was, a trend that held across the projects with manually verified data. Agents turn this from a research-dataset nuisance into an operational problem, because the link now carries live traffic in both directions: downward, the ticket has become the agent's prompt — Copilot, Devin and Cursor all accept work by ticket assignment and read the issue text as the task spec (§2); upward, the agent's branches, commits and PRs are the only progress signal the product side ever sees, and an unlinked agent commit is unattributable archaeology within a week (§6). This chapter is the plumbing manual for that link: one key stamped everywhere and enforced deterministically (§1), a system map so agents and humans can traverse from product concept to code location (§3), one master per artifact so the two planes stop forking the truth (§4), MCP as the runtime bridge — priced and threat-modeled, because the link itself is now an attack surface (§5), and provenance so the upward direction survives audit (§6).

1The two planes and the one key

The cheapest piece of information architecture in this whole module is a string: the issue key. PROJ-123 in the branch name, the commit message, the PR description and the test name is a foreign key joining two databases that share no schema — and every tracker and forge on the market already knows how to dereference it. H3 built the in-repo half of this chain (the five-move ID scheme: registry, contract, stamping, gates, bootstrap); this section extends it across the tool boundary, where the mechanics differ per platform and the defaults are weaker than most teams assume.

Two asymmetries in that table decide real rollouts. First, Linear links from the branch name; Jira does not. A team that standardizes type/PROJ-123-slug branch naming gets Linear's bidirectional links for free, but on Jira still needs the key repeated in the commit message — which is exactly what a prepare-commit-msg hook is for. Second, GitHub's auto-close stops at the repo boundary — in a multi-repo estate the "Closes" convention silently degrades to a plain mention, and ticket state stops advancing; teams discover this months later as a pile of "done but open" issues.

The enforcement ladder

E5's advisory-vs-deterministic split applies verbatim. A convention documented in CLAUDE.md ("branch req/012-*, trailer Refs: PROJ-123") is advisory — agents follow it more reliably than humans do, which is precisely why H3 calls stamping nearly free at agent speed, but nothing catches the exceptions. Client-side hooks are the next rung: prepare-commit-msg runs before the editor opens and can auto-fill the key parsed from the branch name (git's docs: it may "edit the message file in place"); commit-msg can reject — "exiting with a non-zero status causes the command to abort." Both are per-developer opt-in, not distributed with the repo, and commit-msg is one --no-verify away from bypassed — so hooks are convenience, not governance. The deterministic rungs live server-side and in CI: GitHub repository rulesets carry metadata restrictions that enforce commit-message patterns at push time; GitLab's push rules do the same on Premium; and a forge-agnostic CI gate (commitlint for Conventional-Commits shapes, a Danger rule or a plain grep for "PR body carries a valid key") is the safety net that works everywhere — the commitlint ecosystem's own framing is that the CI check exists to hold "even when local enforcement fails." The pattern to sell the client is the layering, not any single rung: convention for intent, hook for ergonomics, server rule + CI grep for the guarantee.

Measure it like E7 measures everything: % of commits (or PRs) carrying a valid key, baselined in week one, watched as a curve. There is industry precedent for the metric — LinearB surfaces unlinked branches as shadow work, and its published figure is blunt: "In 75% of teams, 31% of branches are unlinked" — so the client's likely starting point is far from 100%, and the severity inversion from Bird's study says the gap will be worst on the urgent work. The metric is also the honest test of the whole section: if the ladder works, the curve moves without anyone being nagged.

Keep the key boring

Everything in this section assumes the key is short, stable and greppable: PROJ-123, not a URL, not a title. Two corollaries clients trip on. Renaming the Jira project key breaks every historical link — treat the key namespace as append-only (I2's immutability instinct). And the key must survive copy-paste into plain text, because half its life is spent in commit messages, changelogs and agent prompts where markup dies; this is also why autolink references (turning bare keys into tracker links) beat pasting full URLs into code comments.

2The ticket is the prompt

The downward direction of the link stopped being metadata in 2025–26: for a growing class of work, the ticket is the input the agent executes. All three flagship integrations follow the same shape — everything in this paragraph is vendor documentation about the vendor's own product, point-in-time mid-2026. GitHub Copilot coding agent ↔ Jira (public preview March 5, 2026; GA June 25, 2026): assign a Jira issue to Copilot from the assignee dropdown and the agent reads the title, description, labels and comments, works in a GitHub Actions sandbox, opens a draft PR, and streams status back into the Jira ticket; the GA release added Confluence context via MCP. Devin ↔ Linear: assign the ticket to Devin and it reads the description, acceptance criteria and linked issues; playbook labels (!plan, !implement, !triage, !review) select behaviour, and ticket status auto-advances In Progress → In Review → Done through Linear's agent-session interface. Cursor ↔ Linear (August 2025): @-mention or assign @Cursor and a cloud agent pulls the issue description, comments and linked references, produces a branch and draft PR, and posts the link back to the issue. The pattern worth naming — this workbook's phrasing, not any vendor's — is that the product plane has become the agent's input queue and its output log: intent enters as issue text, progress exits as links and state transitions, and the quality of both rides on §1's key discipline.

Which makes ticket quality an engineering input with a measurable failure mode. The sharpest independent evidence comes from the SWE-bench audit line of work: Wang, Pradel and Liu (arXiv:2503.15223, ICSE 2026) re-examined "solved" issues on SWE-bench Verified and found that 29.6% of plausible, test-passing patches behaviourally diverge from the developer's actual fix — and when they traced the suspicious cases back, 66.2% of them were uncertain precisely because the issue statement under-specified the requirement. The agent did not fail to read the ticket; the ticket did not contain the answer, so the agent guessed, and a guess that passes the visible tests is worse than a failure because it merges. A human developer routes around a vague ticket with hallway context and a question in standup; H4 measured how rarely agents ask (HumanEvalComm: over 60% answer with code instead of a clarifying question). The practitioner discourse has converged on the same point in plainer language — the ticket-as-spec essays of the SDD wave all orbit the observation that vague tickets survived only because humans could infer intent from ambient context, and agents reading the same tickets inherit the vagueness without the context. The fix is not "write novels in Jira": it is H1's contract anatomy and H2's machine-checkable acceptance criteria applied at the ticket boundary, plus §1's links so the agent can traverse to the PRD, the spec and the ADRs when the ticket legitimately compresses (§4's excerpt-with-link rule).

Assignment is an autonomy decision

Assigning a ticket to an agent is not a workflow convenience — it is granting E6 autonomy with the product tool as the control surface. Two governance defaults follow. Restrict who can assign or @-mention the agent (and, where supported, whose issues it will act on — §5's GitLost incident turned exactly this knob's absence into a data leak). And treat the agent's ticket-side output — status transitions, comments, closing actions — as write operations under I2's permission matrix: an agent that can close tickets can mark its own guesses as Done.

3The system map

The key joins work items to changes. It cannot answer the prior question every traverse starts with: where in this system does the product concept live? That is the map's job, and the two standards worth teaching a client have conveniently opposite shapes. C4 (Simon Brown) is four zoom levels — system context → container → component → code — and its official framing is exactly the traverse metaphor: "a way to create 'maps of your code', at various levels of detail, in the same way you would use something like Google Maps to zoom in and out of an area you are interested in." arc42 (Gernot Starke & Peter Hruschka, CC BY-SA) is a twelve-section documentation template — context, constraints, solution strategy, building-block view, runtime view, deployment, cross-cutting concepts, decisions, quality, risks, glossary — that answers "where does each kind of architectural fact go"; in practice C4 diagrams slot into its building-block view (a pairing that is practitioner convention, not something either site mandates). They compose rather than compete: C4 is the picture language, arc42 the filing cabinet.

Three rules make the map agent-ready rather than shelf-ware. First, the entry point is a file, not a wiki page: ARCHITECTURE.md at the repo root, in matklad's original argument, because "the biggest difference between an occasional contributor and a core developer lies in the knowledge about the physical architecture of the project" — I5 already established this file as the map layer of progressive disclosure, and it is the first thing an agent's context assembly should surface. Second, the code level is derived (I1): C4's own guidance calls Level 4 "very much an optional level of detail" and recommends against hand-drawing it "because most IDEs can generate this level of detail on demand" — hand-drawn code diagrams are I3 drift with arrows. Generate what is derivable (docs-as-code toolchains: Structurizr DSL, PlantUML, docToolchain for arc42 — tool status point-in-time and currently shifting; Structurizr's cloud service is winding down toward self-hosted), hand-write only the two or three levels where human judgment decides what matters. Third, decisions live next to code: ADRs in source control — the ThoughtWorks Radar's long-standing recommendation ("a record that remains in sync with the code itself"), and I2's immutability rules — so the map explains not just what the containers are but why they are shaped that way, and the agent quoting an ADR quotes the current status line.

The map is for the traverse, not the wall

The test of a system map in an agentic SDLC is operational: given ticket text, can the agent (or the new hire) get from the product noun to the owning container and the entry-point files in two hops? An ARCHITECTURE.md that names the containers, their responsibilities and their top-level directories passes; a beautiful diagram exported to a wiki the agent cannot fetch does not. This is I6's lesson wearing map clothes — the map makes search cheap; it does not replace it.

4One master per artifact

Once traffic flows both ways across the link, the failure mode changes name: it stops being "missing link" and becomes "two truths." The PRD lives in Notion and a stale copy lives in docs/; the acceptance criteria live in Jira and a diverged paraphrase lives in the spec; the architecture lives in Confluence and a five-container sketch from 2023 lives in the README. Every copy is an I3 drift pair waiting to disagree, and an agent — which reads faster and wider than any human — is more likely to find the stale copy, not less. The rule this workbook recommends (our Report-3 synthesis, flagged as such) is one master per artifact, everything else a reference: each artifact class gets exactly one living home, chosen by who edits it, and every other appearance is a link — or, where the consumer genuinely needs inline text (a task spec an agent will execute offline), a summarized excerpt with a link to master, dated, so staleness is at least visible.

The row that surprises clients is status: in a linked estate, status is not written, it is computed — the branch exists, the PR is open, the checks are green, the deploy shipped; the tracker's job is to render that, which is exactly what the §1 integrations do with state transitions. Any hand-authored "status page" is a derived artifact pretending to be durable (I1) and will be wrong by Friday. The counter-habit to coach is the copy-paste reflex: the PM who pastes the full PRD into the epic description, the engineer who pastes the AC table into the PR — each paste feels helpful and creates a fork. "Reference, don't copy" is cheap to say and hard to live; the excerpt-with-link compromise (short, dated, linked) is what makes it liveable, and it is precisely the pattern E2 wants for context anyway: the agent's context window should hold the excerpt and fetch the master on demand, not carry three stale copies of the same intent.

5The MCP bridge

Links let humans traverse; MCP lets the agent traverse. All three major trackers now ship official hosted MCP servers — Atlassian's Rovo MCP Server, Linear's at mcp.linear.app (May 2025), Notion's hosted server — all OAuth-based, and Atlassian's identity model is the one to hold up in the I2 conversation: the agent inherits the invoking user's permissions ("all actions respect users' existing access controls and permissions"), so the ticket the agent can read is the ticket you can read, and admin-side toggles gate read/write/search separately. Roster and auth details are point-in-time mid-2026; re-verify before a client meeting. Two costs arrive with the bridge, and both are now measurable rather than folklore.

The token bill. Every connected server pre-loads tool definitions, and every tool result arrives as raw JSON in the context window; tracker payloads are notoriously padded (a Jira issue is kilobytes of custom fields around a paragraph of intent). Anthropic's own engineering numbers — vendor-measured, flagged as such — put the ceiling case vividly: a multi-server agent burning 150,000 tokens on tool definitions and intermediate results dropped to 2,000 when tool calls moved into code execution, "a time and cost saving of 98.7%," and the companion Tool Search work showed that fewer tool definitions in context improved accuracy while cutting tokens 85%. The independent corroboration is directional rather than threshold-shaped: on MCP-Universe, adding unrelated servers as distractors measurably degraded success rates. The practical rules for the product↔engineering bridge are modest: connect the servers the task actually needs, prefer summarized excerpts over full-payload dumps in anything durable (§4), and record project defaults (keys, board IDs, base URLs) in AGENTS.md so the agent does not re-discover them every session — a practitioner economy, not a benchmark result.

The injection surface. Ticket text is the textbook instance of Simon Willison's lethal trifecta (June 2025): an agent with "access to your private data," "exposure to untrusted content," and "the ability to externally communicate" can be steered by anyone who can write text it will read — and a public issue tracker is an open submission box for exactly that text. This is not hypothetical; the incident list is now three deep and all three are product↔engineering-link-shaped. Invariant Labs, May 2025 ("toxic agent flows"): a malicious issue in a public GitHub repo steers an agent connected to the GitHub MCP server into pulling private-repo data and publishing it in a PR — an architectural flaw of the tool combination, not a bug in any one server. General Analysis on Supabase's MCP, July 2025: injected support-ticket text made a Cursor agent with service_role credentials read the integration_tokens table and paste the secrets back into the ticket thread; Supabase's response — read-only mode, wrapping untrusted content, MCP off by default — is a defense catalogue in miniature. Noma Security's "GitLost", July 2026: GitHub's Agentic Workflows, configured to act on assigned issues, was steered by a public issue — filed by an unauthenticated attacker — into leaking a private repo's README into a public comment, with the built-in guardrails bypassed by prefixing the injected instructions with the single word "additionally," which the model parsed as a follow-on task rather than a new directive. Three incidents, one lesson: the link is the attack surface — the same channel that makes the ticket a prompt makes the ticket an exploit.

Defense defaults for the bridge

Write tools on "always ask"; read-only server modes until the workflow has earned writes (I2's matrix applied to MCP). Restrict which principals' tickets the agent acts on — GitLost's core mitigation — and scope tokens to single repos. Treat fetched ticket text as data, not instructions: wrap or quarantine it where the harness supports it. And audit the combination, not the servers: Invariant's toxic-flow analysis (and its mcp-scan tooling) exists precisely because each server can be individually safe while the trifecta emerges from their union (M7). None of this is exotic — it is the same E5 move as always: the defense that holds is the one enforced outside the context window.

6Provenance and the outward layer

The upward direction of the link has a new passenger: who — or what — did the work. The measurement that should scare any audit-minded client comes from the multi-method agent census (Khosravani & Mockus, arXiv:2606.24429, June 2026): across ~180 million repositories, multi-method detection identified 850,157 Claude Code commits in one snapshot, of which bot-account lookup — the signal most adoption studies rely on — recovered only 28,154, a 3.3% recall and a 30× gap; H3 drew the conclusion and it stands: if provenance is not stamped at write time, it cannot be reconstructed later, even by researchers trying hard. The stamping mechanics are pleasantly boring — git already has trailers (RFC-822-style lines that git interpret-trailers parses natively, the same footer position Conventional Commits formalizes) — but the convention is genuinely contested: Co-authored-by: makes the agent visible in forge UIs but implies co-authorship, which critics argue muddies CLA chains and personhood; the Assisted-by: camp keeps the human as sole author with the tool as annotation. The contest produced a real reversal: VS Code briefly flipped a default that added "Co-authored-by: Copilot" to commits (April 2026), drew immediate backlash — the trailer appeared even for users with Copilot disabled — and reverted to opt-in within weeks. Pick one convention per H3, write it into CLAUDE.md, enforce with a CI grep, and do not let the default of whatever tool arrives next pick it for you.

Above trailers, a trace layer is consolidating — point-in-time mid-2026, moving fast. Agent Trace (agent-trace.dev, v0.1.0 RFC, January 2026) is a Cursor-led, vendor-neutral spec — with Amp, Cognition, Cline and Cloudflare among the named collaborators — for JSON trace records linking code changes to the conversation and line ranges that produced them, storage-agnostic (files, git notes, a database), with explicit non-goals: it does not adjudicate ownership or code quality. GitHub shipped the same idea productized: every Copilot coding-agent commit carries a permanent link back to its full session logs (March 2026), and agent sessions can stream prompts, responses and tool calls to SIEM endpoints including Microsoft Purview (public preview, July 2026) — the session trace becoming a first-class, exportable SDLC artifact, which is exactly the tiered-artifact story I1 and H3 told: a durable decision-provenance spine with disposable payloads. For the EU-facing client, traj-compliance's clock applies — the AI Act's logging and record-keeping articles begin to bite in August 2026 — so "which sessions produced this release" is about to become a question with a legal deadline attached.

Last, the outward-facing corner of the link layer, kept honest: llms.txt (Jeremy Howard, September 2024) proposed a curated markdown index so LLMs can navigate a site's documentation. Mid-2026 status: publication is widespread because docs platforms (Mintlify, Fern, GitBook, Vercel) emit it by default — Anthropic's and Cursor's own docs carry one — but consumption has not materialized; Google's John Mueller: "FWIW no AI system currently uses llms.txt," and I1's Ahrefs study found 97% of published files received zero requests of any kind. The honest framing for a client: an llms.txt for your public developer docs is cheap and harmless, might help agent-fetched documentation at the margin (I5), and is not a discoverability strategy; inside the repo, AGENTS.md/CLAUDE.md and ARCHITECTURE.md do the real indexing work.

7Week one at the client

The linking audit runs alongside I5's disclosure audit and I6's retrieval audit — same week, same interviews, and the cheapest of the three to act on.

Do — in this order
  • 1 · Baseline the link rate. One afternoon of git + tracker API: % of the last 500 commits/PRs carrying a valid issue key. Expect real gaps (LinearB: in 75% of teams, 31% of branches are unlinked); expect them to be worst on urgent work (Bird's severity inversion). This number is the E7 baseline the whole chapter improves.
  • 2 · Pick the key and ladder it. Branch pattern + trailer convention in CLAUDE.md; prepare-commit-msg auto-fill; then the deterministic rung — GitLab push rule, GitHub ruleset metadata restriction, or a CI grep — advisory first, enforced once the team stops noticing it.
  • 3 · Turn on what is free. Linear branch-name autolinks / Jira smart commits + development panel / GitHub custom autolinks for tracker keys. Ten minutes each; most estates have half of it off.
  • 4 · Establish masters. The §4 table as a one-hour workshop: for each artifact class, name the master, delete or link-ify the known twins (I3's two-truths pairs).
  • 5 · Bridge read-only. Connect the tracker MCP with reads only, writes "always ask," author restrictions on; record project defaults in AGENTS.md.
Don't
  • let anyone paste a PRD into the repo (or a spec into a ticket) — link it; excerpt-with-link where inline text is genuinely needed (§4)
  • trust "Closes PROJ-123" across GitHub repos — auto-close is same-repo only; multi-repo estates need the CI gate (§1)
  • let the agent act on tickets filed by arbitrary principals — GitLost is three weeks old at time of writing (§5)
  • cite the vendor integration claims (Copilot/Devin/Cursor §2) as measured outcomes — they are product documentation (E7)
  • hand-draw C4 Level 4 or hand-author status pages — both are derived artifacts (I1, §3–§4)
  • ship llms.txt as a "discoverability" line item — publish it if the docs platform emits it anyway, and expect nothing (§6)

9Applying it — decision guide

Three sentences

The product and engineering planes are two databases joined by one boring foreign key — the issue key in branches, commits, PRs and tests — and the default link rate is bad in the direction that hurts most (Bird 2009: the more severe the work, the less likely it is linked), so the key must be laddered from convention to hook to server-side rule and measured as % commits linked (E7). Agents raise the stakes in both directions: downward the ticket is now the prompt — and under-specified tickets measurably produce plausible-but-wrong patches — while upward, unstamped agent work is unrecoverable (the 30× census), so provenance trailers and session-trace links are stamped at write time or never. The bridge that makes this live — MCP into Jira/Linear/Notion — is paid in tokens and threatened through the very tickets it reads (three named incidents in fourteen months), so it ships read-only first, writes on always-ask, with one master per artifact and "reference, don't copy" keeping the two planes from forking the truth (our synthesis, flagged).
