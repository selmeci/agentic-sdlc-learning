0Why it matters

Every artifact M4 has governed so far — specs, ADRs, context files, generated docs — shares one property: a human can point at it. This chapter is about the artifact class that grows while nobody is pointing. An agent's defining handicap is amnesia: the context window is working memory, and when the session ends, everything in it is gone. A human engineer compounds knowledge — every debugging session leaves a residue of understanding; a vanilla agent re-learns the codebase every morning, forever. Whatever persists across sessions persists because something wrote it somewhere — a progress file, a memory directory, a managed store — which means memory is not a model property you enable but an information-architecture decision you govern. That framing is the chapter. I1 filed memory under its grey zones (durable in effect, agent-written by definition); I2 gave it a row of its own in the write-permission matrix — the fourth write-mode, "write through validation" — and promised the depth here. The depth has two sides. The capability side: memory is what makes long-horizon work coherent at all — an agent that tracks "for the last 1,234 steps I've been training my Pokémon in Route 1" can pursue a goal for thousands of steps; one that cannot, loops. The liability side: a memory file is the one input the agent trusts completely, wrote itself, and no human reviewed — which is why memory poisoning is a named entry in OWASP's agentic Top 10 (ASI06) and why the write-mode exists. Both sides land on the same desk: whoever owns the harness owns what the agent is allowed to remember.

1What agent memory is — a working typology

Strip the vendor language and agent memory sorts into four operational layers, distinguished by lifetime and by who can see them:

SESSION

Working memory

one session · the window itself

The context window: the conversation, tool results, scratch reasoning. Lives for one session, dies with it. E2's territory — managed by retrieval and shedding, not by storage.

FILES

Persistent memory files

agent-written · governed (§4)

A directory the agent reads and writes across sessions — Anthropic's /memories, Claude Code's memory directory, a Memory Bank. Survives resets; steers every future session that loads it.

PROJECT

Cross-session project state

progress files · git history

Progress files, plans, NOTES.md — plus git history itself, the memory system every repo already has: every commit is a remembered decision with provenance for free.

ORG

Organizational stores

shared · audited · scale (§6)

Managed stores shared across agents, repos and teams — vector databases, knowledge graphs, platform memory. The layer where scale arrives and where governance gets hardest (§6).

The academic frame underneath is CoALA — "Cognitive Architectures for Language Agents" (Sumers, Yao, Narasimhan & Griffiths, arXiv:2309.02427, TMLR 2024) — which gives language agents four memory types borrowed from cognitive science: working memory ("active and readily available information as symbolic variables for the current decision cycle"), episodic ("experience from earlier decision cycles"), semantic ("an agent's knowledge about the world and itself"), and procedural — which, in CoALA's sharpest observation, comes in two forms: "implicit knowledge stored in the LLM weights, and explicit knowledge written in the agent's code." Read that against the harness and the mapping is direct: a session summary is episodic; a memory file of project facts is semantic; and CLAUDE.md rules, hooks and Skills are explicit procedural memory — the harness itself is a memory layer, which is why I2 guards writes to it so hard. The research lineage behind each type is short and worth knowing: Generative Agents (Park et al., arXiv:2304.03442, UIST 2023) built the canonical episodic design — a "memory stream" of timestamped observations, retrieved by a weighted score of recency, importance and relevance, periodically synthesized into higher-level "reflections"; Reflexion (Shinn et al., arXiv:2303.11366, NeurIPS 2023) showed episodic memory driving capability — agents "verbally reflect on task feedback signals, then maintain their own reflective text in an episodic memory buffer," lifting HumanEval pass@1 to 91% against GPT-4's then-state-of-the-art 80%; and Voyager (Wang et al., arXiv:2305.16291, 2023) built procedural memory as "an ever-growing skill library of executable code" whose skills are "temporally extended, interpretable, and compositional" and — the phrase that matters for harness design — alleviate catastrophic forgetting. Voyager's skill library is the direct ancestor of Skills (I5): remembered procedure stored as reviewable code, not as weights.

The engineering frame is MemGPT (Packer et al., arXiv:2310.08560, October 2023) — "Towards LLMs as Operating Systems" — which treated the context window as RAM and everything else as disk, "virtual context management" as paging. Its two tiers: main context (system instructions, a self-editable working context, and a FIFO message queue) and external context (searchable recall storage and archival storage). The design's real contribution is agency over memory itself: "Memory edits and retrieval are entirely self-directed: MemGPT autonomously updates and searches through its own memory." A terminology note that keeps citations honest: the widely-repeated triad "core / recall / archival memory" is the product vocabulary of Letta — the company the MemGPT project became (UC Berkeley spinout, September 2024) — not the paper's; the paper says "working context" and calls the outer tiers storage. Small thing, but this workbook keeps paper terms with papers and product terms with products.

Three measurements calibrate expectations before any architecture is chosen. Capability: memory demonstrably works — Reflexion's 11-point HumanEval lift is memory and nothing else. Reliability: sustained memory is hard — LongMemEval (Wu et al., arXiv:2410.10813, ICLR 2025) benchmarked five core abilities (information extraction, multi-session reasoning, temporal reasoning, knowledge updates, abstention) and found "commercial chat assistants and long-context LLMs showing a 30% accuracy drop on memorizing information across sustained interactions." And economics, from the one controlled coding-specific experiment we know of (Markus Sandelin, March 2026 — practitioner-grade: one run per condition, and the persistent-memory tool tested was the author's own, so read it as directional): across three tasks of rising complexity on a real FastAPI codebase, memory left code quality essentially flat, made the simple task slower and costlier than no memory at all, and paid off only as complexity rose — 22–32% cost and 28–40% fewer turns on the hardest task, ~14–16% averaged. Memory is an investment with a break-even point, not a free upgrade — which is also the honest answer to "should every task load everything we know?" The gaps between these results — memory helps, memory systems fail, payoff depends on task shape — are exactly the space §3's engineering and §4's governance exist to close.

2The landscape — three families, and a cautionary benchmark war

Everything on the market in mid-2026 falls into three families, and the sorting matters more than the logos because each family implies a different governance story. All of this section is point-in-time and moves fast; re-verify before a client meeting.

File-based, repo-resident. The family the rest of this chapter defaults to, because its artifacts are already inside the team's git/PR/review machinery. Anthropic's memory tool (type memory_20250818, generally available on the Messages API — "no beta header is required") is deliberately minimal: purely file-based storage under /memories, six operations (view, create, str_replace, insert, delete, rename), executed client-side — "Claude requests file operations, and your application executes them" — so the application owns storage, and the harness owns governance. Claude Code's auto memory applies the shape inside the IDE-agent: a per-project directory with a MEMORY.md index and topic files (§4 takes the layout apart). Cline's Memory Bank is the same idea as a methodology rather than a tool: six prescribed markdown files with distinct lifecycle roles (projectbrief, productContext, systemPatterns, techContext, activeContext, progress) and a documented dependency hierarchy — I1 noted it as the triad reinvented under other names. Cursor's Memories (beta with 1.0 in June 2025, GA in 1.2 that July) adds the missing governance verb: a sidecar model proposes memories from chat, and the user approves — "user approvals for background-generated memories to preserve trust," in the changelog's words. Windsurf's Cascade auto-generates memories to a local per-workspace store. The through-line of the whole family: memory as reviewable text, close to the code it describes.

The coding-agent platforms are converging on the same governance, independently. GitHub's Copilot Memory (public preview, 2026) is two-tier — repository-level facts shared with everyone on the repo, private user-level preferences — and it ships two of this chapter's arguments as product mechanics: facts "are stored with citations pointing to the code that supports them," and when one becomes relevant Copilot "checks those citations against the current branch to confirm the information is still accurate" — I3's staleness check, run at retrieval time — while "any stored fact or preference that goes unused is automatically deleted after 28 days" — TTL as the default, with repo owners reviewing facts and enterprise admins holding export and bulk-delete. Cognition's Devin has Knowledge: items carry a trigger description for retrieval, pin to no repo, one repo or all repos, and scope to the organization — with an explicit promotion step ("if an organization-level knowledge item proves useful enough to share across your entire enterprise, you can promote it directly"), a promotion ladder shipped as UI, rhyming with §4's. Both are vendor designs and point-in-time; the convergence — provenance, re-validation, TTL, scoped promotion — is the signal.

Database and graph engines. Where the file family stores what the agent wrote, this family processes it: extraction, deduplication, retrieval, temporal reasoning. Letta — the company the MemGPT project became — maintains the self-editing memory-blocks design and added the most interesting recent idea, sleep-time compute (Lin et al., arXiv:2504.13171, April 2025): spend model time between sessions — the paper shows offline pre-reasoning cutting test-time compute ~5×; the product frames it as a background agent revising memory while the main agent is idle. Memory that is curated offline instead of mid-task is a genuinely new point in the design space (and, note, a second unattended write path for §5 to worry about). Mem0 (paper: arXiv:2504.19413) is an extraction pipeline over a store; Zep builds on Graphiti, "a temporally-aware knowledge graph engine," so facts carry validity intervals and a query can ask what was true then. LangChain's LangMem SDK (February 2025) ships the CoALA vocabulary as an API — semantic facts, episodic few-shot examples, procedural prompt-updates. All are credible engineering; all quote benchmark numbers, which brings us to the war.

When the vendors benchmark each other — the Mem0/Zep dispute

Mem0's paper claims, on the LoCoMo benchmark, "26% relative improvements in the LLM-as-a-Judge metric over OpenAI" plus "91% lower p95 latency" and >90% token savings versus full-context — and it also benchmarked Zep, unfavorably. Zep's team answered with a rebuttal ("Is Mem0 Really SOTA in Agent Memory?") arguing the comparison misconfigured Zep — timestamps never passed, searches serialized — and re-reporting its own score around 75%; a counter-analysis on Zep's own tracker then argued Zep's headline 84% LoCoMo claim drops to 58.44% once a category designated for exclusion is removed. We do not adjudicate this: the point is that every number in it comes from a party selling the thing measured, the benchmark itself bends under configuration choices, and the honest reading is E7's: memory-vendor benchmarks are vendor claims until a disinterested replication exists. Two later footnotes sharpen the lesson. Sandelin's coding benchmark (§1) names the metric slippage directly: headline "80–95% token savings" figures measure compression of the memory footprint — "not whether a downstream task completes faster." And the benchmark frontier keeps moving — BEAM (third-party, 2026) tests memory at 1M- and 10M-token scales, where every published score so far is, again, vendor-quoted. The independently useful datum stands apart: LongMemEval's ~30% sustained-memory drop (§1) — the problem is real; the leaderboard is marketing.

Platform memory. ChatGPT's memory is the consumer-scale reference: user-directed "saved memories" (February 2024), then automatic "reference chat history" (April 2025) — memory the platform writes for you, stored outside your governance perimeter entirely. For engineering teams it is mostly a contrast object: it demonstrates both the product value of remembering and the governance ceiling of memory you cannot diff, review or gate. The enterprise questions it raises — whose data, what retention, which audit — are §6's.

3The three long-horizon techniques, in depth

E2 introduced the toolkit for work that outlives a context window; this section is the practitioner detail. The source is Anthropic's engineering guidance ("Effective context engineering for AI agents," September 29, 2025 — published, not coincidentally, the same day as the memory tool), and the techniques are three, chosen by task shape.

Compaction summarizes a conversation approaching the window limit and reinitializes with the summary — and the published detail is what makes it copyable: "The model preserves architectural decisions, unresolved bugs, and implementation details while discarding redundant tool outputs or messages. The agent can then continue with this compressed context plus the five most recently accessed files." Note what that list is: a priority order for what deserves to survive a reset — decisions first, open problems second, mechanics last, plus a small working set. Compaction's cost is cache-shaped (E2): it deliberately rewrites history, which invalidates the prompt cache from the first changed byte, so it batches best at milestones rather than eagerly. Its failure mode is subtle — a bad summary is a small memory poisoning of your own making, confidently wrong in exactly the way §5 warns about, which is why milestone compaction plus a durable progress file beats aggressive automatic compression. The cache economics are measured now rather than folklore: across 500+ agent sessions on three providers, prompt caching "reduces API costs by 41-80% and improves time to first token by 13-31%" — but naive full-context caching "can paradoxically increase latency," and the working rule that falls out is one of placement: keep the stable prefix byte-stable, put per-turn-mutating content — memory digests included — at the end of the prompt, and exclude volatile tool results from the cached region (Lumer et al., arXiv:2601.06007, January 2026; E2's Manus rules, now with independent numbers).

Structured note-taking moves state out of the window entirely: the agent writes progress to a file, drops it from attention, and re-reads it when needed. The canonical demonstration is Claude playing Pokémon: the agent "maintains precise tallies across thousands of game steps — tracking objectives like 'for the last 1,234 steps I've been training my Pokémon in Route 1'" — coherence no context window could hold. In engineering work the note file is the progress/plan disposables I1 already classified; what the memory tool adds is a protocol, injected as system-prompt text whenever the tool is enabled, and its two commandments are worth quoting because they encode the whole discipline: "IMPORTANT: ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE." — read before you act, so the session builds on recorded state instead of re-deriving it — and "ASSUME INTERRUPTION: Your context window might be reset at any moment, so you risk losing any progress that is not recorded in your memory directory." — write early, because anything un-written is already lost. The curation clause completes it: keep contents "up-to-date, coherent and organized," rename or delete files that are no longer relevant, "do not create new files unless necessary" — I1's pruning discipline, stated by the vendor as agent instructions.

Sub-agent architectures shed context by delegation: "Each subagent might explore extensively, using tens of thousands of tokens or more, but returns only a condensed, distilled summary of its work (often 1,000-2,000 tokens)." The main thread stays clean; the exploration cost is paid in a disposable window. The discipline that makes it work is the summary contract — the subagent's report is the only thing that survives, so its quality bounds the whole pattern (E3's harness patterns formalize this).

Choosing: compaction for a long conversational flow without natural milestones; note-taking for milestone-based iteration, where the progress file survives any reset; sub-agents for parallel exploration and read-heavy research where only conclusions need to return. They compose — a long project typically runs note-taking as its spine, sub-agents for its research bursts, and compaction as the emergency valve — and all three exist for the same reason: the window is the scarcest resource in the system (E2), and memory is how you stop paying window-rent on things that are true tomorrow.

4The governed write-mode — provenance, validation, promotion

I2's matrix gave memory its row: write through validation — agent-writable like a disposable, durable in effect like a spec, therefore governed like neither. This section is that row unpacked into machinery, and the shape to keep in mind is a pipeline with gates, not a directory with rules.

The substrate: files in the repo, versioned in git. The practical default for engineering teams is the simplest one: the memory directory is plain files, and it is committed. Git-versioning memory buys, for free, everything §5's defenses will ask for — every memory write is diffable (reviewable in a PR like any other change), attributable (the commit says which session wrote it), and revertible (rollback to a pre-compromise state is git revert, machinery the team already trusts). This default is our synthesis, not vendor doctrine — Anthropic's memory tool deliberately leaves storage to the application — but every component is standard practice, and the alternative (an opaque store with none of those properties) has to justify itself against it (§6).

The index discipline. Claude Code's documented memory layout is the reference shape for keeping persistent memory loadable: a per-project memory directory containing "a MEMORY.md entrypoint and optional topic files." The entrypoint is a concise index loaded into every session — with a hard cutoff: "The first 200 lines of MEMORY.md, or the first 25KB, whichever comes first, are loaded at the start of every conversation" — while detail lives in topic files (debugging.md, api-conventions.md) read on demand. That is I5's progressive disclosure applied to memory: pay for the index always, pay for the body only when relevant. The cutoff is worth teaching precisely — it is a load limit, not a style suggestion; the 201st line is not "discouraged," it is silently never seen again, the failure mode I3 would call drift by design. (Keep this distinct from the adjacent guidance to target CLAUDE.md files under 200 lines — that one is a style target for a file humans write; this one is a mechanical truncation of a file the agent writes.)

Provenance and validation at write time. The memory tool's documentation prescribes the mechanical layer (§5 gives the threat it answers): validate every path against escape from the memory directory, strip sensitive data before the handler writes, "track memory file sizes and cap how large a file can grow," and "periodically delete memory files that haven't been accessed in a long time." On top of that mechanical floor, the write-mode adds provenance: a memory entry should say where it came from — which session, which task, from whose instruction — because §5's whole threat model turns on the difference between "the user told me" and "a webpage told me." Frontmatter with a source field costs nothing at write time and is the only thing that makes a poisoned entry findable later. This layering — vendor-documented mechanics, provenance and review as the loop around them — is our synthesis, flagged as such.

The promotion path. The most consequential rule in this section: memory files steer sessions, but steering files rule the repo — and the path between them is a one-way gate. An observation born in a session (disposable) may be recorded into the memory directory (governed, validated write); a memory entry that proves durable — a convention, a hard-won constraint, a decision — may be proposed for CLAUDE.md, AGENTS.md or an ADR; but that promotion is a PR a human merges, never a write the agent performs. The gradient runs from cheap-and-free toward reviewed-and-durable, and each step up adds a gate. Run backwards it also defines demotion: a memory entry contradicted by the codebase is deleted, not debated — I3's staleness logic applied to the agent's own notes, and the multi-session benchmarks say demotion is not hygiene but the main event: agents "fail primarily through misestimation of user state, treating prior session history as a reliable proxy for current context rather than stale information requiring re-validation" (Momento — Merin, Anugraha, Purwarianti & Winata, arXiv:2606.00832, mid-2026). The dominant failure mode of memory is over-trusting it, which reframes the whole discipline: govern re-validation, not just capture. The promotion path is our synthesis; its parts (I1's categories, I2's gates, the vendor's validation floor) are individually attested.

Run forward at team scale, the promotion path also has a practitioner name: compound engineering. Every's engineering team closes each cycle with a "compound" step they call the money step — "after we do a code review… we'll ask the agent to look at the comments, summarize them, and store them for later," as prompt-based rules living in the codebase and plugins, automatically distributed to the whole team (Shipper & Klaassen, December 2025; they ship the workflow as a Claude Code plugin). Strip the branding and it is exactly §4's pipeline with the loop closed: session experience → reviewed rule → shared steering asset — memory as the mechanism by which the team's harness appreciates, not just the agent's recall.

5The threat model — memory poisoning and its defenses

The reason memory gets a governed write-mode instead of a convention is that it is the one artifact class whose corruption is designed to outlive the attack. OWASP's Top 10 for Agentic Applications (GenAI Security Project, December 9, 2025) names it ASI06 — Memory & Context Poisoning: poisoning "occurs when persistent memory, retrieval, or context is shaped to mislead future steps," and it is "the agentic version of data poisoning, but consequences appear at runtime rather than training time." Three properties separate it from ordinary prompt injection, and each defeats a familiar defense. It is persistent — session-scoped review never sees it, because the malicious line is already in storage when the session starts. It is temporally decoupled — planted today, triggered weeks later, so incident response looks in the wrong time window. And it spreads through ordinary reads — any agent that loads the store inherits the payload, so one write can steer a fleet.

The research attacks put numbers on feasibility. AgentPoison (Chen, Xiang, Xiao, Song & Li, arXiv:2407.12784, NeurIPS 2024) — "the first backdoor attack targeting generic and RAG-based LLM agents" — poisons the agent's long-term memory or knowledge base with a handful of trigger-optimized entries and reports "an average attack success rate higher than 80%" at "less than 0.1% poison rate," demonstrated against an autonomous-driving agent, a knowledge-intensive QA agent and a healthcare EHR agent. One entry in a thousand is not a needle a review will find. MINJA (Dong et al., arXiv:2503.03704, 2025) removes the last comfort — that the attacker at least needs write access: "The attacker injects malicious records into the memory bank by only interacting with the agent via queries and output observations." Crafted queries walk the agent into writing the attacker's "bridging steps" into its own memory (an injection success rate of 98.2% and attack success of 76.8% on average, across GPT-4/GPT-4o agents including an EHR agent). The attack surface is the conversation itself — if the agent writes memory from interactions, every interaction is a potential write. The systematization arrived in mid-2026: "From Untrusted Input to Trusted Memory" (Dash, Ge, Jain, Shah & Shang, arXiv:2606.04329) maps four memory write channels and nine structural vulnerabilities into a taxonomy of six attack classes with a dedicated benchmark (MPBench), and hands designers two conclusions this chapter builds on: "agents designed to write and retrieve memory more aggressively are more exploitable" — §3's auto-capture convenience has a security price — and "existing prompt injection defenses fail to cover memory poisoning attacks," so the defense has to live on the write path, not in the prompt.

And the pattern has left the lab twice, both times documented by Johann Rehberger. SpAIware (September 2024): indirect prompt injection planted persistent instructions in ChatGPT's memory via the macOS app, turning memory into spyware — "the data exfiltration vulnerability became a lot more dangerous as it now spawns across chat conversations," every future session exfiltrating to the attacker until OpenAI patched it (app version 1.2024.247). Gemini memory manipulation (February 2025): a poisoned document plants instructions that fire on the user's own trigger words ("yes," "sure") — the technique Rehberger named delayed tool invocation — writing false long-term memories that Google assessed as "an abuse-related risk with low likelihood and low impact." OWASP's ASI06 write-up cites the Gemini case by name; the loop from lab to standard closed in under a year.

The defense stack is E5's ladder applied to the memory pipeline, and it is honest to say up front that only its bottom layer is vendor-documented; the loop above it is assembled practice (I2 flagged the same). Mechanical (deterministic, vendor-documented): path containment — "validate that all paths start with /memories," canonicalize, reject traversal including URL-encoded forms — plus sensitive-data stripping before the handler writes, file-size caps, and expiry of long-unread files. Structural (deterministic, from I2): the memory directory is the only agent-writable persistent path (deny-rules on steering files), memory is git-versioned so every write is diffable and revertible, and the promotion gate (§4) means nothing memory-resident reaches a durable steering file without a human merge — which converts a successful poisoning from "steers the fleet" to "sits in a reviewable diff." Process (advisory, ours by synthesis): provenance on every entry so an audit can ask "where did this belief come from"; review of memory diffs at promotion time, not per-write; and rollback drills — since memory is in git, recovering from a poisoned week is a revert, provided someone can name the last clean commit. The asymmetry to teach: the attacker needs one unvalidated write; the defense needs the write path to be narrow, the storage to be diffable, and the blast radius to stop at the PR gate.

For teams that want the deterministic version off the shelf, OWASP itself ships one: the Agent Memory Guard project — a separate, official OWASP project positioned as the reference implementation for ASI06 — runs inline on every memory read/write and, in its own words, "validates memory integrity using cryptographic baselines (SHA-256 hashing)," "detects injection attempts, sensitive data leakage, protected key modifications, rapid changes, and size anomalies," "enforces declarative YAML security policies on memory read/write operations," and "captures snapshots for forensic analysis and enables rollback to known-good states" (middleware for LangChain, LlamaIndex and CrewAI; point-in-time mid-2026). Note it is the same four moves this section derived from first principles — integrity baselines, anomaly detection on the write path, policy as configuration, snapshot-and-rollback — which is the reassuring kind of convergence. One honesty note on attribution: the canonical ASI06 entry's own mitigation list was not verbatim-reachable when this chapter was written, so concrete defenses here are attributed to Anthropic's tool docs, the Agent Memory Guard project, and this workbook's I2 machinery — not quoted from the Top 10 entry itself.

6Files in the repo or a managed store — the decision

The open question every client asks once agents multiply: keep memory as plain files in the repo, or adopt a managed store (§2's second family)? The criterion this workbook proposes — our synthesis, assembled from I1–I3's machinery — is a default plus three tests.

The default is files, and it is not a compromise. Files in the repo inherit, at zero integration cost, every property this chapter has needed: diffable writes, provenance by commit, review by PR, rollback by revert, deny-rules by path, staleness checks by grep (I3), and retrieval by the agentic search the coding agent already does best (I6). A managed store starts from zero on each of those and must rebuild them as product features — audit logs for provenance, snapshots for rollback, admin roles for gates — inside a second governance system the team must learn, configure and trust. For a single team on a single repo, the store solves problems the team does not have, at the price of problems it did not have either: a new privileged attack surface (§5, at organizational blast radius), a staleness regime with no same-PR rule (I3's problem, minus its cheapest fix), and vendor numbers from a benchmark war (§2).

The three tests that flip the default. First, the boundary test: does the memory have to cross repos, teams or agent runtimes? Files are repo-scoped by construction; the moment "what we learned about the payments API" must reach five repos and a support bot, a shared store is the honest architecture (the alternative — copy-paste across repos — is I1's unowned-copy drift). Second, the scale test: has retrieval outgrown lexical search — thousands of entries, conceptual queries, cross-referencing — so that I6's hybrid argument applies to memory itself? Third, the compliance test: does memory content fall under retention, audit or privacy regimes (it will, the moment it contains customer or incident data) that demand queryable audit and enforced retention beyond what git provides? Two or more tests positive, evaluate a store; none, keep files and revisit quarterly. And if a store does come in, the write-mode comes with it: I2's rule that where path-gates do not exist, the gate moves to identity — a scoped service account per agent, native audit streams, a named human reading them — plus §4's promotion rule intact: nothing flows from the store into a steering file without a human merge. The store changes where memory lives, never who may promote it.

One compliance wrinkle belongs on the table before any store decision, because it cuts both ways. The moment agent memory contains personal data — a customer name in a debugging note is enough — GDPR's right to erasure applies to it, and not only to raw session logs: a derived memory ("user prefers X," extracted from a deleted conversation) is personal data the erasure must reach, which is an awkward query to run against a memory store that was never designed to answer "everything derived from this person." Pulling the other way, the EU AI Act's logging provisions (Art. 12/19 — automatic recording of events, retention of at least six months for high-risk systems; T4 covers the detail) want agent activity records kept. Classify memory content at write time — personal vs technical, log vs derived fact — or inherit a delete-vs-retain conflict with no per-entry way to resolve it. Framing this as a memory-design requirement is our synthesis; the two regulations it derives from are real and already in force on the timeline traj-compliance tracks.

7Week one at the client

Memory setup at a brownfield client is a half-day of decisions and an hour of mechanics, and it should happen in week one — because the alternative is not "no memory," it is unguided memory: capitalized Markdown files accumulating at the repo root (I1 measured the reflex) and agent-written state nobody classified, versioned or gated. The sequence:

Decide (half a day, with the tech lead)
  • one named memory home per repo — the tool's default if there is one, else .claude/memory/ or equivalent; nothing memory-like lives anywhere else
  • git-versioned: memory commits with the work, so writes are diffable, attributable, revertible (§4)
  • the promotion rule, written down: memory → CLAUDE.md/AGENTS.md/ADR only via PR a human merges; never a direct agent write (I2's deny-rules enforce it)
  • the org-store question answered with §6's criterion — for week one the answer is almost always "files, revisit at scale"
Wire (an hour, in the harness)
  • index + topic-file layout (MEMORY.md entrypoint; detail on demand — §4); mind the load cutoff
  • the validation floor in the memory handler: path containment, sensitive-data strip, size caps, expiry (§5, vendor-documented)
  • provenance frontmatter on new entries: source session, task, instruction origin
  • TTL/pruning for session disposables (I1); a quarterly "memory gardening" pass on what remains
  • a rollback drill on the calendar: can the team name and revert to a last-known-clean memory state?

The check that closes the week is the same one I2 and I3 use: try it. Have a session write a memory entry, watch it arrive as a diffable commit with provenance; have the agent propose a CLAUDE.md change from memory, watch it arrive as a PR and not a write; plant a deliberately wrong memory line in a branch, and confirm a later session acts on it — the last one is the demonstration that makes §5 land with a client audience better than any slide.

9Applying it — decision guide

Three sentences to keep

Agent memory is an information-architecture decision, not a model feature: four layers by lifetime — window, persistent files, cross-session state, organizational store — with the harness itself as procedural memory, and the research base (Reflexion's 11-point HumanEval lift on one side, LongMemEval's 30% sustained-memory drop on the other) saying memory pays and memory systems fail, which is why it gets engineering and governance rather than trust. The working default is files in the repo, git-versioned, with an index kept loadable, a validated write path, provenance on entries, and one non-negotiable rule — memory may propose, only a human merge promotes anything into a steering file — because the threat is measured (poisoning at 80% success from 0.1% poison rates, query-only injection, and two documented in-the-wild demos), persistent, and designed to outlive the session that planted it. Buy a managed store only when memory must cross repo boundaries, outgrow lexical retrieval, or meet compliance-grade retention — and when you do, move the gate from paths to identity and keep the promotion rule intact, because the store changes where memory lives, never who may promote it.
