0Why it matters

D1 argued that agent handoff needs a third artifact — the design system as machine-readable infrastructure — and was honest that most of that artifact is only partially checkable. This chapter zooms into the one layer where the checkability is total: design tokens. A token is a named design decision serialized as data — color.action.primary instead of #2A4AD0 — and that single move changes its nature: a hex value in a stylesheet is an anonymous fact, while a token reference is a citable commitment that a linter can verify, a build can resolve, and an agent can be held to. That makes tokens the visual layer's closest counterpart to EARS/Gherkin acceptance criteria (H2): a constrained, named vocabulary that turns "follow the brand" from advisory prose into a deterministic pass/fail gate (E5). The W3C Design Tokens Community Group gave this contract a vendor-neutral serialization — the DTCG format, first stable version October 2025 — and an ecosystem of translators, sync bridges and lint rules grew around it. This chapter covers the contract itself: what a token is (§1), what the spec actually stabilized and what it only promises (§2), the pipeline that moves tokens from design tool to production (§3), the sync-direction decision that quietly determines governance (§4), how the contract binds agents specifically (§5), what the evidence says (§6), and how a brownfield client's pile of ad-hoc SCSS variables becomes a token contract without a big-bang rewrite (§7).

1Anatomy of a token — and the tier that carries the contract

In the DTCG format a token is a JSON object whose reserved properties carry a $ prefix: $value (required), $type (color, dimension, fontWeight, duration…), optional $description. The dollar sign is deliberate namespace hygiene — the spec reserves it for its own properties, present and future, and forbids token or group names from starting with it, so user vocabulary and spec vocabulary can never collide. Tokens nest inside groups ("a set of tokens belonging to a specific category"), and one token can reference another by alias: a $value of "{color.blue.600}" resolves to the target token's $value. Files carry the recommended extensions .tokens or .tokens.json and the media type application/design-tokens+json. That is nearly the whole format — and the smallness is the point: it is a data interchange for named design decisions, not a styling language.

The structure teams build with aliases is not in the spec, and it matters more than the spec. Community practice organizes tokens into tiers: primitives (raw values with no meaning attached — color.blue.600), semantic tokens (intent — color.action.primary, aliased to a primitive), and optionally component tokens (button.primary.background, aliased to a semantic). The lineage is worth crediting precisely: Nathan Curtis's 2016 "Tokens in Design Systems" drew the original two-tier line between options (the values you could use) and decisions (the values you did choose, with context attached); the three-tier form is a later community elaboration, standard in Tokens Studio's docs and most 2026 practice — but it is convention, not DTCG normative text. Attribute it accordingly. In zeroheight's 2026 survey data the convention is visibly mainstream: only around one team in ten still runs a single flat layer, and just over half run the full three tiers.

Why does the middle tier carry the contract? Because it is the only tier whose names encode intent, and intent is what survives change. A rebrand edits primitives and touches nothing else; a dark theme re-resolves the semantic tier against a different primitive set; a component redesign moves component tokens. Code and agents that cite semantic names are insulated from all three. This is also the reason the naming discipline matters more than the JSON syntax: Curtis's companion piece "Naming Tokens in Design Systems" (2020) decomposes names into ordered levels — namespace, category, concept, property, variant, scale — and that grammar, not the file format, is what makes ten thousand tokens navigable by a human or retrievable by an agent.

2The spec — what 2025.10 stabilized, and what it only promises

The vocabulary is a decade older than the standard. Jina Anne coined "design tokens" at Salesforce around 2014, where the Lightning Design System distributed brand decisions across platforms through the open-source tool Theo; Curtis's 2016 article gave the idea its conceptual frame; and in 2019 Anne and Kaelig Deloumeau-Prigent founded the W3C Design Tokens Community Group to standardize the serialization. The group took six years to ship — a running joke in the community until it stopped being one: on 28 October 2025 the DTCG announced the first stable version, 2025.10, developed by (verbatim) "more than 20 editors and authors" with a roster spanning 24 named organizations — Adobe, Amazon, Google, Microsoft, Meta, Figma, Sketch, Salesforce, Shopify, Tokens Studio, Penpot, Supernova, zeroheight and others (the 24 is our count of the announcement's list, not a figure the group published). One governance note for anyone tracking the group itself: in June 2026 Anne and Val Head stepped back to chairs emerita, leaving Deloumeau-Prigent as chair.

Precision about what "stable" covers, because the announcement itself blurs it. The 2025.10 release comprises two stable modules: the Format Module (tokens, groups, types, aliases — everything in §1) and the Color Module (modern color spaces: Display P3, Oklch, CSS Color 4 alignment). Both self-describe as "considered stable." Theming, modes and multi-brand are not in either. They live in a third document, the Resolver Module — sets of base tokens plus modifiers for contexts like light/dark, composed via .resolver.json files — and that document is a draft carrying an unusually blunt warning: "Do not attempt to implement this version of the specification. Do not reference this version as authoritative in any way." Yet the W3C announcement headlines "theming and multi-brand support" as a 2025.10 capability. Both statements come from the same organization; only one is normative.

Read the status line, not the press release

The DTCG's own announcement markets theming as shipped while the module that specifies it says "do not implement." This is the E5 lesson in standards form: the press release is advisory, the status-of-this-document section is deterministic. The same discipline applies one level up — the spec itself, verbatim, "is not a W3C Standard nor is it on the W3C Standards Track"; it is a Community Group Candidate Recommendation published under the W3C Community Final Specification Agreement. That is a real, useful thing — a patent-licensed, multi-vendor stable format — but a client slide that says "the W3C standard for design tokens" is overclaiming, and a theming architecture built on the Resolver draft is building on text its own authors disown. As of July 2026 there is no 2026.x release; 2025.10 remains current and the Resolver Module remains the active frontier.

The spec also has a citable dissenter, and the disagreement is instructive rather than dismissable. Donnie D'Amato — design-systems practitioner and author of the "Avoiding tokens" essay — argues the group is "scope-creeping into territories that will make it difficult for it to ever be finished" and worries that tool-oriented formats invite "vendor lock-in where you might only be able to edit design tokens from tools"; his deeper point is that the nested-JSON format optimizes for machine parsing over human authorship, and that CSS custom properties already provide a web-native token medium. For this workbook's purposes the criticism sharpens rather than undermines the thesis: we want tokens precisely because they are machine-first — the agent and the linter are the new readers — but D'Amato is the reminder that the format is contested, that Figma conspicuously chose "variables" over "tokens" as its product vocabulary, and that betting a client's architecture on any single serialization deserves the same point-in-time humility as every tool fact in this module.

3The pipeline — from decision to production, tool by tool

A token file does nothing by itself; the contract is only as real as the pipeline that resolves it into every surface the product ships. The reference shape, mid-2026: decisions serialized as DTCG JSON in git; a translator resolves aliases and emits platform outputs (CSS custom properties, SCSS, JS/TS, iOS, Android) as a CI build step; design tools consume or contribute through a sync bridge. Every tool fact below is point-in-time — this is the fastest-moving layer of module M6, and each item should be re-verified before a client meeting.

TRANSLATOR

Style Dictionary

the default

Created at Amazon by Danny Banks; since a 2023 handover co-maintained with Tokens Studio, with Joren Broekema as core maintainer (the canonical repo now lives at style-dictionary/style-dictionary). v4 (April 2024) brought first-class DTCG-format support. v5 is released and stable — v5.5.0 shipped June 2026 — but full compliance with the 2025.10 spec is still an open workstream (tracked in issue #1590: color, border and shadow types done; gradients and durations in progress; the Resolver draft deliberately not implemented). The one-line status a client should hear: the tool is production-grade, the spec coverage is converging, and nobody serious implements the theming draft yet.

TRANSLATOR

Terrazzo

the alternative

Cobalt UI 2.0, renamed. DTCG-native from the start rather than retrofitted, MIT-licensed, with a CLI that emits CSS, Sass, JS/TS, JSON and Tailwind presets and a plugin API pitched as cleaner than Style Dictionary's. Worth shortlisting for greenfield token builds; references to "Cobalt UI" in older material are this project.

BRIDGE

Tokens Studio

Figma ↔ git

The de-facto bridge: a Figma plugin (now also a standalone platform) that stores tokens as JSON and syncs them to GitHub/GitLab, including via GitHub Actions — the mechanism that makes "Figma proposes, git disposes" practical. Note the commercial reality: git sync at team grade (branching, multi-file) sits in the paid tier (≈39€/month per editor at mid-2026 pricing — point-in-time), and teammates without a licence read tokens but cannot write. Budget it as infrastructure, not as a free plugin.

DESIGN TOOLS

Figma variables · Penpot

the endpoints

Figma's variables are the design-side representation, but two constraints define mid-2026 practice: the Variables REST API "requires an Enterprise plan" (verbatim from Figma's docs), and native export is not DTCG-clean — alias resolution generally needs a custom parser, and both the REST path and the MCP tools return only default mode values. Figma has announced native DTCG-conformant import/export, reported for late 2026 — not GA as of this writing, so any "Figma can/cannot export DTCG" sentence needs a date on it. Penpot, by contrast, shipped native DTCG tokens in 2025 in collaboration with Tokens Studio — the first open-source design tool to do so, and the existence proof that the format works as a native design-tool primitive rather than a plugin afterthought.

4Sync direction is the governance question

Teams argue about translators and bridges; the decision that actually shapes behavior is which copy is allowed to win. Three models exist. Design-first treats Figma as master and exports toward code — the default posture of design teams, and the drift machine: overrides and detached styles accumulate in the canvas with no gate, exports lag, and the "source of truth" quietly becomes "whatever the last export said." Bidirectional sync promises both sides an editable copy and costs the most machinery while producing the most conflicts — a team must earn it with mature review discipline on both sides, and few have. Code-first — tokens as versioned DTCG JSON in the repo, Figma importing as a consumer — is the recommended default of this workbook and of a converging practitioner line: Artur Sopelnik's February 2026 essay states it as a mental shift, verbatim — "design tokens are code, not design files" — and draws the consequence that "Figma should not be the source of truth for design tokens. Instead, it should be a consumer." Builder.io's governance essay (Steve Sewell, September 2025) makes the same call from the vendor side under the title "Code is the source of truth (not Figma)" — "Figma is, at best, a set of suggestions"; "All that matters is what makes it into code."

The reasons are the unglamorous ones that decide governance questions everywhere in this workbook. A git-resident token file gets CI validation (schema, alias resolution, lint), a reviewable diff, a PR gate with a named approver, and tool independence — the DTCG file outlives any particular design tool's export format. In I1's taxonomy: the JSON is durable, every generated output is derived, and canvas experiments are disposable; in I2's terms, the write-permission matrix has one master cell and it sits in the repo. And code-first does not demote designers to consumers of engineering — the strongest counterexample-shaped confirmation is Adobe Spectrum, whose token data is a public git repository explicitly described as the canonical dataset, while (per Adobe's Garth Braithwaite, verbatim) "we have some authoring experience that we allow the designers to author these design tokens in figma, and we use some plugins and some custom workflows to then bring that into our public GitHub repository". Designer-first authoring, code-resident truth: the author experience and the system of record are separate decisions, and only the second one is the contract.

For the agentic thesis the choice is even less symmetric than for humans: an agent can read, diff, validate and cite a JSON file in its own repository natively, while a design canvas is reachable only through rate-limited, plan-gated APIs that flatten modes and aliases (§3). Putting the master where the agent's hands already are is not a style preference; it is what makes §5 possible at all.

5The contract agents can be held to

Why do tokens matter more in agentic development than they did before it? Because generation without constraint samples the training-data mean — D1's slop evidence — and a token vocabulary is the cheapest effective constraint: it replaces an open universe of values with a closed, named set. Practitioner Hardik Pandya's July 2026 write-up states the mechanism in one line: the token layer gives the model "a closed set of values instead of fabricating new ones." The agent does not need taste to pick color.action.primary; it needs the vocabulary in context and a gate that rejects everything outside it. That splits the work into the two halves this module keeps insisting on: supply (make the vocabulary readable) and enforcement (make violations fail deterministically).

Supply — how the vocabulary reaches the agent. The baseline channel is the one §4 already built: the DTCG JSON and its generated outputs live in the repo the agent works in, referenced from the project's context file ("use tokens from tokens/; never hardcode a color") — advisory, but cheap and always present. Above that, mid-2026 practice offers three sharper channels, all point-in-time. Figma's Dev Mode MCP server exposes get_variable_defs, returning the variables and styles used in a selection so generated code can reference the real token names — launched in beta, and it currently returns only default mode values. Vibe-tool constraint files — Lovable's design-system.json (D3) and the theming registries of the shadcn ecosystem — are the same idea in tool-specific dress. And in April 2026 Google Labs open-sourced DESIGN.md, the most direct instantiation of this chapter's thesis to date: a repo-root file combining machine-readable design tokens in YAML front matter with human-readable design rationale in markdown prose, a lint CLI that validates structure and checks WCAG contrast and emits agent-actionable structured JSON, and exporters to Tailwind config and W3C DTCG. It is deliberately tool-agnostic — the spec addresses "coding agents" generically, and the vendor lists in circulating write-ups are third-party commentary, not spec text — and uptake requests appeared in other vendors' ecosystems within a day of release. Its spec is honestly versioned alpha, which earns it the same flag as the Resolver draft: watch it, pilot it, do not architect on it yet.

DESIGN.md, read through this workbook's lenses

What makes DESIGN.md interesting is not novelty — it is convergence. It is tokens-as-code (§4) with the durable/derived split intact (the DTCG and Tailwind exports are derived); it is progressive disclosure (I5) — front-matter data for machines, prose rationale for the humans and the judgment calls; and it ships with its own deterministic gate (the lint CLI) rather than trusting the prose — E5 applied by a vendor who never read Report 1. A Google-shipped spec being requested in Anthropic's skills ecosystem within a day is also the clearest signal yet that the "agent-readable design contract" is becoming a layer every vendor expects to exist. Point-in-time, alpha, re-verify — but this is the direction of travel.

Enforcement — the deterministic half. The advisory instruction "use tokens" becomes law through a short, boring stack, every piece of which predates agents: Stylelint's built-in color-no-hex (disallow hex colors outright) and declaration-property-value-allowed-list (allow-list var(--…) patterns per property); the purpose-built stylelint-declaration-strict-value plugin (rule id scale-unlimited/declaration-strict-value), which forces a variable, function or keyword for named properties — the single most token-shaped lint rule in the ecosystem; ESLint's no-restricted-imports with a message pointing at the design-system component, keeping agents off raw UI libraries; and a drift check comparing extracted reality against the committed baseline — the young open-source CLI Dembrandt is built exactly for this, failing CI on drift with a machine-readable JSON report and honest exit codes. CI blocks the merge; the agent iterates against the failure output before a human ever reviews (E4's loop, D5's gate). One security note that costs a sentence and buys a habit: token files, DESIGN.md and their kin are agent-read context, so they inherit the M7 rule — provenance and review before an agent consumes them; there is no known token-file injection incident as of July 2026, and the write-gate from I2 is precisely what keeps it that way.

And the boundary, restated once because every chapter in this module must: the gate verifies conformance — the value came from the vocabulary, the import came from the system, the contrast clears the threshold — never quality. Whether color.action.primary was the right token for this button in this flow is D1's judgment question, and it stays behind the designer's gate (D5, D6). Tokens are the Gherkin of the visual layer precisely this far and no further.

6What the evidence shows — and the do-not-cite list

The honest evidence file for tokens is thinner than the confidence of the ecosystem suggests, and the gap itself is worth showing a client. What holds up: adoption is real and steep. zeroheight's Design Systems Report 2025 (n=294 practitioners, surveyed late 2024) reports, verbatim, "Design tokens have gone from 56% in 2024 to 84% this year, which shows that design tokens are at a point of mass adoption in design systems"; the 2026 edition (n=147 — note the smaller sample) has tokens in 86% of systems — "almost ubiquitous", said jointly of tokens and documentation. Cite the jump with the 2025 report's dates, not as a 2026 number — and carry the same report's cold shower alongside it: only about 40% of teams report any token pipeline, so for the majority the contract exists but the plumbing of §3 does not.

On agents specifically, the rigorous evidence sits one level up from tokens. A CHI 2026 extended abstract ("Design System-Compliant User Interface Generation with LLM Agents") compares strategies for feeding a design system to an agent and finds registry-based supply — the agent assembles pre-built, system-compliant components rather than generating free-form — reached the highest design-system compliance, 95.08%, with a small pilot (n=9) reporting the work felt easier. Read it for the direction, not the decimals: it is an extended abstract with a tiny pilot, its metric is compliance rather than general code quality, and it constrains at the component level, not the token level. The sharper honest statement: as of July 2026 no rigorous study isolates design tokens as the causal variable in agent output quality. The mechanism argument of §5 stands on the D1 slop evidence plus the general constraint literature, and this workbook says so rather than borrowing decimals from adjacent studies. One practical trap for anyone searching this literature: "token" in the LLM papers means context tokens — the arXiv results for "design tokens + LLM" are dominated by token-economics work that has nothing to do with design, and at least one plausible-looking citation in circulation is exactly this collision.

The do-not-cite list for this topic

Circulating numbers we could not pin to a primary source, mostly traceable to third-hand agency aggregations: Atlassian's "67% reduction in duplicate UI code" and "consistency 58→89"; GitHub Primer's "WCAG compliance 71→96%"; "31–47% development time saved with tokens"; and the Spotify rebrand folklore ("months before tokens, minutes after") — repeated everywhere, sourced nowhere we could find. If a deck needs a rebrand number, the best available is vendor self-report with the flag showing: Netguru's Silk case study claims its 2024 rebrand's core redesigns completed in two days, the full process in about three working days, and estimates it would have taken an additional four days without the system — their own design system, their own estimate, tokens not isolated as the variable. Weak, and honestly labelled beats strong and unpinned.

7Brownfield: from ad-hoc SCSS to a token contract without a big-bang

The client in this engagement does not have a token file; they have nine years of SCSS variables, four near-identical grays and a $blue-new-final-2. The move to DTCG is a strangler-fig migration (B5) with a characterization step (B2), and it never requires freezing feature work. Six steps, each with an exit condition — the sequence is this workbook's synthesis, assembled from the named tools and the module's own gates:

stepwhat happensexit condition
1 · inventoryextract the de-facto tokens from shipped CSS — Project Wallace's CSS Design Tokens Analyzer emits a spec-compliant token set with per-value usage counts (you might find 47 colors with 80% of usage in six of them — the counts are the consolidation map); CSS Stats and Superposition triangulatede-facto token report reviewed with the designer
2 · characterizegolden-master the current UI: visual-regression snapshots of key screens as the baseline (B2's move, D5's tooling) — the migration must change nothing visually until a human chooses tobaseline green in CI
3 · consolidatedesigner collapses the inventory into primitives + a semantic tier (§1), naming per Curtis's grammar; every collapse (four grays → one) is a reviewed, intentional diff against step 2's baselinetokens.json in git, PR-gated (I2)
4 · bridgetranslator emits CSS custom properties; legacy SCSS variables are re-pointed at them ($brand-blue: var(--color-action-primary)) so every existing stylesheet compiles unchanged — the CMS Design System's Sass-to-CSS guide documents this incremental path, and Halodoc's engineering blog shows the same JSON→SCSS bridge in productionold code compiles against new tokens; baseline still green
5 · ratchetturn on the §5 lint stack one property category at a time — colors first, then spacing, then type — new violations fail CI while the legacy count only goes down (the same one-way ratchet as B2's coverage exit criteria)zero hardcoded values in the gated categories
6 · open the gateonly now do agents generate UI against the vocabulary, and Figma imports the tokens as a consumer (Tokens Studio bridge) — supply and enforcement both exist, so D2-level generation (D6 scale) is safe to pilotagent PRs passing the token gate unassisted

The order is the load-bearing part, and it is B1's dependency-chain argument in the design layer: enforcement before generation, characterization before change. A team that starts by letting an agent "clean up the styles" runs the bootstrap paradox again — changes with no baseline to verify against; a team that starts by writing the perfect token taxonomy in a workshop ships a document, not a contract. Start from shipped CSS (what is), collapse toward intent (what should be), keep the golden master green throughout, and let the ratchet — not a mandate — retire the legacy values.

8Takeaways for our engagement

Assess (first client conversations)
  • Where do design decisions live today — SCSS variables, a Figma library, someone's memory? Run the Wallace inventory before proposing anything.
  • Which sync direction is implicitly in force, and who currently wins a conflict between Figma and code?
  • Does any CI gate reference design at all? If not, §5's lint stack is the cheapest first win in the whole design module.
  • Figma plan tier — the Variables API is Enterprise-gated, which changes the available bridges.
Commit (the defaults we bring)
  • Code-first: DTCG JSON in git as the single master, PR + human gate, everything else derived.
  • Stable-spec only: Format + Color modules; treat the Resolver draft and DESIGN.md as watch-list, not foundation.
  • Enforcement before generation: the §7 ratchet, colors first; agents pilot only after the gate holds.
  • Evidence honesty: the 56→84 jump with its real dates, the 40% pipeline gap, the do-not-cite list — and the admission that token-specific causal evidence does not yet exist.
Three sentences to remember D2 by

A design token turns a design decision into a named, machine-checkable commitment, and the DTCG's 2025.10 release gave that commitment a stable, vendor-neutral serialization — Format and Color modules only, with theming still an explicitly do-not-implement draft and the whole thing a Community Group report, not a W3C standard. Put the tokens in git as the master copy and they become the one layer of the design system an agent can be held to deterministically: a closed vocabulary supplied through context and MCP, enforced by a lint-and-drift gate that CI runs before any human reviews — conformance always, judgment never. Adoption is genuinely mainstream (56%→84% across zeroheight's 2024→2025 surveys, ~86% in 2026) but the plumbing is not (~40% have pipelines), which is exactly the gap a brownfield engagement fills with the six-step inventory→characterize→consolidate→bridge→ratchet→generate migration — enforcement before generation, always.
