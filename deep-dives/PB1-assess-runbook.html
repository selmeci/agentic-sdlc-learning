0Why this runbook exists

PB1 is the entry point of every engagement: you assess the surrounding system before building anything on it, because AI amplifies that system rather than fixing it. The evidence is independent and directional — DORA 2024 associates a 25% increase in AI adoption with −7.2% delivery stability, and a METR randomised controlled trial found experienced developers were 19% slower while believing they were about 20% faster. PB1 exists to replace optimism with measured facts before the F0 gate.

What this is not

This is not an audit produced for the client's benefit alone. The output decides our entry decision at the F0 gate — whether we proceed, what preconditions we attach, and where the first real risk lives.

The theory behind the checklist lives in the workbook topics this runbook distils — especially E7 Metrics & anti-patterns for the delivery baseline and the review-capacity argument, and B1 Safe ordering for why verification must come before delegation.

1How to run it

Run PB1 with one person, 2–4 focused days, read access to the repository, CI, deploy logs and the incident system, one team interview, and a client sponsor empowered to sign the F0 decision. Every step below is written as ČO (the action), PREČO (the theory link), and DONE (the exit criterion). Check steps off as you go; the progress chip updates automatically.

In the workbook version of this runbook, progress is saved locally (same store as topic progress). This standalone copy does not persist checkbox state.

2Phase A — Verification base & delivery baseline

Run the full suite three times from a clean checkout. Record: one-command invocation, wall-clock runtime, pass rate, whether it finishes under ~15 min, and the list of flaky tests. Why: the suite is the external truth the whole loop stands on — without a trusted, fast signal there is nothing for an agent to converge on (E4 Verification-first, B1 Safe ordering).

DONE = runtime, pass rate, under-15-min status and flaky list recorded in report §2; headless one-command status recorded as a fact (feeds questionnaire A2).

Insert a deliberately broken test and let CI run. Confirm the failure is machine-readable: a non-zero exit code and a parseable log or status payload. Why: an agent that cannot read CI cannot self-correct; a red/green dot is not enough (E4 Verification-first, E7 Metrics & anti-patterns).

DONE = broken-test experiment run; failure artifact linked in report §2; whether the red build blocks merge recorded (feeds questionnaire A3).

Compute the DORA four keys plus rework rate retroactively from git and CI history, add cost per merged PR (review hours, CI minutes, seat licences — the token column is added by PB5), agree the before/after window with the client, and confirm access to git history, deploy logs, and the incident system. The baseline is captured now, at assess time — a pilot without it can be neither proven nor disproven (the METR perception gap), so it cannot wait for PB5 (E7 day-0 DORA checklist).

DONE = baseline numbers recorded in report §3; window + data access confirmed in writing.

3Phase B — Information architecture

List architecture docs, ADRs, specs, READMEs, and runbooks; record owner, last-touched date, and whether a newcomer can retrieve each from the repo. Why: an agent cannot compensate for missing or buried truth; the loop will hallucinate consensus where none exists (I1 Artifact taxonomy, I3 Consistency drift).

DONE = artifact table in report §4 with gaps explicitly listed.

4Phase C — Governance minimum

Prove production credentials are unreachable from the development environment, then run a secrets scan across the repo history. Why: environment separation and secrets hygiene are hard boundaries, not maturity shades — the Replit production-DB incident (July 2025) showed what happens when they fail. GitGuardian 2026 vendor telemetry reports AI-assisted commits leak secrets at 3.2% versus a 1.5% baseline; treat that as a vendor claim, not independent evidence (E5 Governance, B6 Roadmap F0–F3; see also module M7 Security).

DONE = reachability tested and result recorded; scan report attached; if prod credentials are reachable → questionnaire C1 scored 0 and logged as an F0 precondition.

Document branch protection, CODEOWNERS, who and what can merge, and where an agent's write access would stop today. Why: delegation without a permission boundary is just automation of mistakes; the write-permission matrix is the deterministic half of governance and the RBAC/least-privilege check (I2 Lifecycle & write permissions, E5 Governance).

DONE = draft write-permission matrix in report §5 (skeleton below).

5Phase D — Team readiness

Identify who will curate agent config and review agent PRs, and secure a concrete weekly review budget in hours. Why: agent output scales faster than review; Faros AI vendor telemetry shows +98% PR volume and +91% review time when assistive tools land without capacity planning (P4 The PM role, H4 Feedback-loop anti-patterns, E7 Metrics & anti-patterns).

DONE = named owner(s) + weekly review budget recorded in report §6, or absence recorded as questionnaire D2 = 0.

6Phase E — Evaluation & the F0 gate

Fill the 24-point questionnaire below, mark the four [GATE] items, and map the result to the recommendation table. Why: the score is a structured way to surface gaps, not a disguise for gut feel; the thresholds are our synthesis, calibrated against the theory in E6 Autonomy levels.

DONE = filled questionnaire archived with the engagement notes; recommended level written into report §7.

Produce the assessment report from the template, convene the F0 gate, and record a go / no-go / go-with-preconditions decision together with the STOP criteria that would pull the cord. Why: the gate is the only thing that turns assessment into action; without a recorded decision and triggers, the report becomes shelfware (B6 Roadmap F0–F3).

DONE = decision (go / no-go / go-with-preconditions) recorded with date, criteria and signatories.

The questionnaire stays at 24 points. The delivery baseline captured in step A3 is measurement data, not a scored maturity dimension — scoring it would double-count evidence the gates already cover and would break the calibrated thresholds in the recommendation table.

The mapping below is our synthesis, not an industry standard. It converts the questionnaire into an entry recommendation; the non-compensable gate criteria are what make it conservative.

ScoreGatesRecommendation
< 10anyL1 only; engagement starts by fixing the base (PB2 becomes the deliverable, not a prerequisite)
10–164/4L2 entry; F0 read-only archaeology can start (B4)
≥ 174/4L3 pilot candidate; proceed to PB2 with the F0→F1 exit criteria from B6
any< 4/4no L2+ autonomy until the failed gate is fixed — non-compensable

The table tops out at L3 deliberately — L4 is a per-slice decision for narrow deterministic tasks and L5 is not achieved (E6); a high score buys a pilot, not a higher ceiling.

STOP criteria for F0 are taken from B6 and adapted to the client's context. Typical triggers include a delivery-stability drop beyond the agreed threshold, a secrets leak or production credential reachable from dev, and any unreviewed merge. Each trigger names who pulls the cord and how.

7Exit gate & takeaways

Exit criteria

Report delivered; delivery baseline recorded (report §3 is the pilot's "before" snapshot — PB5 compares against it); go/no-go recorded; if GO → proceed to PB2 with the report's §2 numbers as the bootstrap baseline.

Three sentences to remember PB1 by

Assess before build, because AI amplifies the system it lands on — good or bad. The delivery baseline and the four non-compensable gates are the guardrails that keep the assessment honest; a high average with one gate failed is exactly the false-confidence case PB1 exists to catch. The F0 decision is an action, not a report: go, no-go, or go-with-preconditions, each with a named owner and STOP criteria.

Sources and caveats for the empirical claims: DORA 2024/2025 reports (dora.dev) — note the 2025 update partially reverses the throughput finding while instability stays elevated; METR RCT (arXiv:2507.09089) and the 2026 follow-up (metr.org) — the follow-up weakens the slowdown estimate but the perception-gap lesson stands; GitGuardian State of Secrets Sprawl 2026 — vendor telemetry, not independent evidence; Replit incident — July 2025.
