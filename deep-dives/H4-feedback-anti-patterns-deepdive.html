0Why it matters

Module M3 built the forward path of delegation: H1 the contract, H2 its machine-verifiable criteria, H3 the chain of IDs that keeps the artifacts connected. H4 is the return path — what happens when the implementation contradicts the contract, when a criterion turns out to be impossible, ambiguous or wrong. A contract with no return channel is not a contract; it is a memo thrown over a wall. And the data says the wall is real: when requirements are ambiguous, more than 60% of code-LLM responses generate code anyway rather than ask a clarifying question (HumanEvalComm, ACM TOSEM 2025), and roughly a quarter of agent-authored fix PRs in the wild end up closed without merging — the loop failing at its last hop (arXiv:2602.00164). The feedback loop is where a handoff process either becomes self-correcting or quietly rots.

There is also an asymmetry worth naming up front. The engineering layer is structurally the best-placed detector of bad requirements in the whole SDLC: implementation is where vagueness becomes a concrete fork in the road ("should the discount apply before or after tax?" cannot be deferred once you are writing the line). Agents hit those forks earlier, faster and in greater volume than any human team ever did — the question is whether the discovery travels back to the requirement's owner or dies in a session log. M2 ended with the PM as editor and curator (P4); H4 supplies the mechanism that keeps feeding that editor the one input only implementation can produce: evidence about the requirements themselves.

The division of labour with E7

E7's anti-pattern catalogue (AP1–AP6) covers rollout failures — autonomy without guardrails, review bottlenecks, perception-driven decisions. H4's catalogue covers handoff failures: the four ways the contract itself goes wrong between product and engineering. Same discipline — every named failure maps to a specific control — different layer of the stack.

1The silence is the default — agents don't ask

The naïve assumption behind throw-over-the-wall delegation is that an agent facing a broken requirement will stop and say so. Measured behaviour says otherwise, consistently, across independent benchmarks. HumanEvalComm (Wu et al., ACM TOSEM 2025, arXiv:2406.00215) rewrote 164 HumanEval problems into 762 variants with injected ambiguity, inconsistency or missing information: over 60% of code-LLM responses produced code instead of a clarifying question. "Learning to Ask" (Wang et al., EMNLP 2025) found the same reflex in tool-using agents — faced with an instruction missing a required argument, models hallucinate the missing value rather than ask, a behaviour the authors trace to next-token training itself. And MAST, the UC Berkeley-led failure taxonomy of multi-agent systems (arXiv:2503.13657, preprint), puts a number on it in the wild: "fail to ask for clarification" alone accounts for roughly 12% of observed failures.

The sharpest result is the newest. HiL-Bench (Scale AI, arXiv:2604.09408, May 2026 — preprint, re-verify) gives agents an explicit ask_human() tool and withholds one piece of task-critical information: on software-engineering tasks, pass rates (Pass@3) that sit at 67–88% with full information collapse to 1–9% when success requires recognising the gap and asking. On those same tasks even the best models detected fewer than half of the blockers — the dominant failure signature is confident execution on wrong beliefs. The capability gap is not writing code; it is knowing when not to. The cost of the silence is also quantified: on ambiguous requirements, Pass@1 drops by ~7.2 percentage points on average and by 31 for the worst ambiguity types, and ambiguity nearly doubles a model's internal self-contradiction rate (Orchid benchmark, arXiv:2604.21505 — preprint, re-verify). A newer interactive line measures the questions themselves and finds detection wildly uneven by defect class — some ambiguity types get caught nearly every time, others barely at all (ClarifyCodeBench, arXiv:2607.00711 — preprint).

What the benchmarks establish
  • default behaviour is to proceed, not ask — >60% code-instead-of-question on ambiguous specs (TOSEM 2025)
  • the gap is judgment, not skill: 67–88% → 1–9% when asking is required (HiL-Bench, preprint)
  • detection is uneven by defect class — some ambiguity types are barely caught (ClarifyCodeBench, preprint)
  • guided clarification measurably pays: +11.52% relative from targeted questions (ClarifyGPT, H2)
What follows for the harness
  • never rely on the agent volunteering doubt — instruct it to stop on ambiguity (§5, §8)
  • move contradiction-detection upstream into tooling (§2)
  • make the escalation path cheaper than the guess: a named channel, a label, an owner (§5)
  • treat "no questions asked" on a vague contract as a smell, not a compliment

One caution for client conversations: this is a fast-moving capability, and vendors are attacking it directly — Devin ships confidence-gated question-asking (§5), and 2026 work shows help-seeking is trainable. The load-bearing claim is not "agents will never ask"; it is that asking is not the default, so a process that assumes it is has no floor under it. Point-in-time, mid-2026.

2The agent as detector — verification moves upstream

The same engineering layer that stays silent by default can be made loud by design — and the most concrete existing exhibit is Kiro's Requirements Analysis (announced May 12, 2026, Kiro Applied Science). It runs on the spec before any code is generated, and it is genuinely neurosymbolic rather than "more LLM": in the team's own words, "the analysis automatically encodes the specification into formal logic using LLMs and uses SMT solvers to generate the findings." The pipeline has three stages — refinement rewrites thesis-level sentences into testable ones (emitting new EARS acceptance criteria, the H2 format); auto-formalization translates them into a formal model; logical analysis then finds contradictions and gaps and enumerates concrete counter-examples. This is E4's verification-first principle applied one artifact earlier: the requirement itself gets a failing test.

The headline number travels with a caveat: AWS reports — per The New Stack's coverage, not Kiro's own post — that in internal testing across 35 projects with 1,400+ acceptance criteria, roughly 60% of first-draft requirements needed refinement before they could be reliably implemented. Unaudited vendor internal testing; cite the mechanism as fact and the percentage as a vendor claim. The direction, though, matches the independent research: ClarifyGPT (H2) showed LLMs can profitably interrogate ambiguous prompts (+11.52% relative), and §1's benchmarks show unaided detection is unreliable and uneven by defect class — which is why delegating contradiction-hunting to an SMT solver rather than to the agent's judgment is the architecturally honest move (E5's advisory→deterministic principle, applied to requirements).

The old software-engineering intuition behind all this — defects are cheapest to fix at requirements time — is directionally sound but numerically folkloric: the oft-quoted 100:1 cost-escalation ratio traces, per Laurent Bossavit's The Leprechauns of Software Engineering, to secondary sources citing secondary sources, with reported ratios ranging from ~5:1 to ~100:1. Use the direction, never the multiplier. What has actually changed since Boehm's era is the cost side: a requirements-level check used to mean a review meeting; now it is a solver run that costs cents and minutes. Cheap upstream verification, not a scary multiplier, is the honest argument.

3Drift triage — the three-way question

When implementation and contract disagree — an agent reports an impossible criterion, a suspect label lands (H3's gate), a red test contradicts the spec — the process needs one disciplined move before any fix: deciding which artifact is wrong. The cleanest formulation is Leonid Bugaev's (reqproof blog, "Source of truth: Code, Spec, or Requirement?", May 2026 — a practitioner building a traceability product, so note the commercial interest in the thesis): "is the code wrong, is the requirement wrong, or did we learn something?" — and, crucially, "until that is resolved, something is broken in the process." The question is the whole protocol in one line: every drift event has exactly three exits, each with a different owner and a different artifact to change.

The three exits, operationally. Code wrong — the requirement stands; the criterion becomes or remains a failing test (E4) and the fix is an engineering matter; no product decision needed. Requirement wrong — the code faithfully implements a mistake; the fix is a spec PR owned by the PM (P4's editor role), with the criteria rewritten in H2's formats and every artifact tracing to the requirement flipped suspect (H3). We learned something — the most common and least honoured exit: neither artifact was "wrong" at authoring time, implementation produced knowledge nobody had. This is Kent Beck's point against whole-spec-first development — it "encodes the (to me bizarre) assumption that you aren't going to learn anything during implementation that would change the specification" (Dec 2025). A process that treats every learning as someone's error will hide its learnings; the spec gets updated, the reason gets logged, nobody gets blamed.

Two properties make triage work at agent speed. It must be fast — the queue grows at generation speed, so the decision is a ten-minute conversation at most, default weekly ritual plus an async lane for blockers (§8). And it must be owned — the single most common failure is nobody having the authority to say "the requirement is wrong": engineers patch the code to match a broken spec, or quietly patch the spec to match the code, and both artifacts stay plausible while the drift compounds. Until the question is answered, Bugaev's line applies verbatim: something is broken in the process, not merely in the artifacts.

4The anti-pattern catalogue

Report 2's four handoff anti-patterns, each with its detection smell and its control. The symmetry worth teaching: the first two are the two ways to get the contract wrong (too much, too little), the second two are the two ways to lose the loop (spec abandoned, signal never returned).

HAP1

Over-specification

controlled by H1 sizing · P3

The spec outgrows the code and the review burden is relocated, not removed — the tech lead now reviews markdown instead of diffs. The mechanism is measurable: AI-generated review commentary runs ~29.6 tokens per line of code against a human baseline of 4.1 (arXiv:2603.15911) — density, not quality, and someone must read it. Smells: contracts routinely above H1's 10–30 lines; spec-writing time exceeding implementation time on T0/T1 tasks; PMs "spending more time on markdown than on the product." Control: H1's sizing tiers (T0 micro-tasks need an issue ID, not a contract), P3's decomposition heuristic, and Fowler's skeptical question as the standing review filter: would you rather review this pile of markdown, or the code?

HAP2

Under-specification

controlled by H1/H2 · §1's instruction rule

The contract leaves the fork unresolved, and §1 says what happens next: the agent picks a branch silently and executes it with confidence. On ambiguous requirements Pass@1 drops ~7 points on average and 30+ for the worst ambiguity types (Orchid, preprint); practitioners' war stories are all variations of a quick task ballooning into hours of thrash as the agent rewrites around a misread intent (widely reported; no pinnable canonical source — we cite it as an unattributed pattern, not a statistic). Smells: "confidently wrong" green PRs solving a different problem; follow-up-question count rising after merge instead of before work. Control: intent and non-goals written by humans (H1's two irreplaceable components), EARS-formatted criteria (H2), and the standing stop-on-ambiguity instruction from §5.

HAP3

Stale spec — silent drift

controlled by H3's suspect gate

The spec is written, the feature ships, the spec is never touched again — and six weeks later an agent faithfully implements against a document describing a system that no longer exists. The 2026 SDD literature names this "silent spec-code drift": code evolves, the specification does not, and the divergence stays invisible until it is expensive. No mainstream SDD tool detects it today (H3 §4 — Kiro's Sync Files checks task completion, not drift; point-in-time, mid-2026). Smells: spec directories with no commits since the feature merged; agents citing spec lines that contradict current code. Control: H3's git-native suspect gate — a CI grep, not a product — plus M4's rule that specs are durable artifacts riding the same review as code.

HAP4

Throw over the wall

controlled by §5's return channel

The contract is handed off and nobody is listening for the return signal. Downstream, the loop's failure is visible in the merge data: in a study of 8,106 agent-authored fix PRs, 26.1% were closed unmerged — with test failures (18.1%) and incorrect or incomplete fixes (15.3%) leading the study's qualitative sample of closures — "syntactically plausible fixes alone are often insufficient for successful integration" (arXiv:2602.00164, preprint). The mirror finding: across 33,596 agent PRs, reviewer engagement was the strongest merge predictor, and success concentrated in "actionable review loops" where concrete human feedback let the agent converge — iteration volume alone did not (arXiv:2602.19441, MSR 2026). The wall fails in both directions: unreviewed agent output dies, and agent-discovered requirement bugs drown in session logs. Control: the return channel, formalized — §5.

5The return channel — from session log to owned artifact

The topic's open question — how does an agent-discovered ambiguity reach the PM without a human courier? — now has partial product answers, worth surveying precisely because they disagree (all point-in-time, mid-2026; re-verify). Devin is the most structured: since the 2.1 release it self-assesses confidence (🟢🟡🔴), and "when Devin doesn't have 🟢 confidence, it will ask clarifying questions" and wait for approval before proceeding — Cognition reports green-confidence tasks are twice as likely to end in a merged PR as red (vendor-reported). Its docs also define an escalation ladder for blockers: investigate, fix if adjacent, note and continue, or escalate with an explanation. GitHub Copilot's coding agent and OpenAI's Codex both concentrate clarification in their planning modes — Copilot's plan step emits "open questions for clarification", Codex's plan mode "lets Codex gather context, ask clarifying questions" — but in autonomous execution both largely proceed without asking, and both trackers carry open requests from users asking for ask-by-default. The industry state, compressed: asking exists, but mostly as a mode you must enter, not a reflex you can rely on — which returns the burden to the harness.

SYNTHESIS

A minimal return-channel protocol

ours — flag it as such to the client

Four rules, tool-agnostic, cheap enough for week one. (1) A standing instruction in the agent steering file (CLAUDE.md / AGENTS.md): on a contradictory, impossible or two-way-interpretable criterion, stop and file a question — never pick a branch silently. §1's evidence says this must be instructed; the same evidence (and H3's ID-discipline experience) says instructed agents comply deterministically. (2) A structured landing place: the question is an artifact, not a chat line — an issue comment or PR label (spec-question, carrying the REQ-ID from H3's chain) that routes to the requirement's owner, so it survives the session that raised it. (3) A named owner and an SLA: every requirement has exactly one person entitled to answer "the requirement is wrong" (P4's editor); blockers get an async answer, the rest queue for the weekly triage (§3). (4) The answer lands in the artifact, not the chat: the resolution is a spec edit, a criterion rewrite or a logged learning — §6's rule that anything decided in a disposable channel gets promoted to a durable one, or it never happened.

Where should the learning exits of triage land? The emerging convention worth watching is the Agent Decision Record — ADRs extended to decisions an agent made autonomously (library choice, pattern choice, interpretation of an ambiguous criterion), so agent reasoning gets the same traceability as human architecture decisions; early-stage, a proposed convention rather than a standard, but it slots exactly into the gap. The division of labour it implies is clean: the steering file says what must never happen; the decision record says what was decided and why; the spec says what must be true. Everything else — the session transcript, the chat thread — is M4's disposable tier, and treating it as a system of record is how ambiguities drown.

6Multi-agent pipelines — drift compounds

Everything above assumed one agent and one contract. Chain agents — a planner feeding an implementer feeding a reviewer — and the feedback problem multiplies, because each hop is a fresh interpretation with no human gate between hops. The best empirical map is MAST (arXiv:2503.13657, preprint — a UC Berkeley-led study of 200+ traces across 7 frameworks, 14 failure modes): roughly 42% of failures are specification issues, 37% inter-agent misalignment, 21% weak verification (v2 figures; version-dependent, cite rounded). Read against M3, that distribution is the whole module restated as a failure budget: bad contracts, broken handoffs between workers, missing gates. Structural work on error propagation adds that failures at hub nodes of an agent graph infect most of the pipeline while leaf failures stay local (arXiv:2603.04474 — measured with injected faults, so cite it for topology, not for natural drift rates).

The practitioner conclusion has not changed since Cognition's "Don't Build Multi-Agents" (E3): keep writes single-threaded, let other agents contribute intelligence rather than concurrent action. H4 adds the contract-shaped corollary: every agent→agent handoff inherits the full H1 discipline — explicit contract, machine-verifiable criteria, IDs, and a return path — because the receiving agent is precisely the "new team member with amnesia" P3 planned for, minus the ability to bump into you at the coffee machine. If a pipeline cannot afford a human triage gate between hops, it must at least afford deterministic verification there (E4/E5); interpretive drift plus seemingly-coherent output is the failure mode with no smell until the end.

7Loop-health metrics — is the feedback loop working?

The loop is invisible in throughput metrics — you inspect it through rework. The anchor is DORA's Rework Rate, the fifth software-delivery metric (introduced in the 2024 report, carried into the 2025 benchmarks): the share of deployments that are unplanned fixes for something that did not work the first time — added precisely because AI-accelerated throughput can look healthy while instability accumulates underneath (strong independent industry research). Around it, three supporting signals, each honestly labelled. GitClear — a code-analytics vendor — reports churn (code revised within two weeks) rising from 3.1% to a projected 5.7% between 2020 and 2024 and duplicated blocks rising ~8×; correlational, no control group, and churn conflates healthy iteration with defect rework, but the direction rhymes with DORA's concern (E7 cites the same data). New Relic's commissioned survey (Hanover Research, June 2026; n=200 US decision-makers — self-reported perception, not telemetry) captures the loop failing in one contradiction: 94% of leaders rate AI code higher-quality at review time, yet 78% report more production incidents and 74% say at least a quarter of AI code needed significant rework within a year — code that grades well at the gate and fails downstream is exactly what a missing feedback loop looks like from above. And the merge-outcome studies from §4 (26.1% of fix PRs closed unmerged; reviewer engagement the strongest merge predictor) give the PR-level view of the same health check.

Track from day 0 (feeds E7's baseline & T2's pilot)
  • rework rate (DORA definition; fifth metric since 2024) — the headline loop metric
  • agent-PR outcomes: merged / closed-unmerged / abandoned, per agent and per team
  • spec-question count and time-to-answer (the §5 channel's own health)
  • spec-change frequency after merge — the "we learned" exits actually landing
Read them together, not alone
  • zero spec-questions on a vague backlog = §1's silence, not clarity
  • rework up + churn up + review praise up = the New Relic contradiction — audit the gate (E4)
  • high closed-unmerged share = the wall (HAP4), not "bad agent"
  • every number here is a trend instrument, not a target — Goodhart applies (E7)

8Five moves for a brownfield client

The loop is the cheapest part of M3 to install — it is mostly agreements, labels and one CI rule. In order:

Move 1 — name the owners. Every requirement gets exactly one person entitled to declare it wrong (usually the PM, P4). Publish the mapping; triage without an owner is a discussion, not a decision.

Move 2 — instruct the agent to stop and ask. One paragraph in the steering file: on contradiction, impossibility or two-way interpretation, file a spec-question with the REQ-ID and stop. This converts §1's measured silence into a deterministic behaviour — the same inversion H3 found for ID discipline: what humans find effortful, an instructed agent does every time.

Move 3 — wire the suspect gate. H3's two CI greps, so drift surfaces as a PR label while the diff is open. The gate is the loop's sensor; without it, triage only ever sees what the agent volunteers.

Move 4 — install the triage ritual. Fifteen minutes weekly over two queues (spec-question, suspect-spec), one agenda item per entry: code wrong, requirement wrong, or did we learn something? Async lane for blockers. The output is always an artifact change — spec PR, criterion rewrite, logged learning — never just a verbal answer.

Move 5 — baseline rework before the pilot. Rework rate, agent-PR merge outcomes, reopened tickets — captured before the first agent task (E7's discipline, T2's pilot design), so six weeks later "the loop works" is a measured claim, not a feeling.

10Takeaways for our engagement

Three sentences to keep

Agents do not volunteer doubt — across independent benchmarks the default on an ambiguous or contradictory contract is confident silent execution, so the return channel must be instructed, structured and owned, never assumed. Every drift event has exactly three exits — code wrong, requirement wrong, or we learned something — and until someone with authority picks one, the process (not the artifact) is what's broken; the third exit is normal engineering, not failure, and a loop that punishes learning will hide it. The whole apparatus is cheap — a steering-file rule, a label with an owner, two CI greps, fifteen minutes a week — and its health is measurable from day 0 through DORA's rework rate and agent-PR merge outcomes, which is what turns "the handoff works" from a feeling into a number.
