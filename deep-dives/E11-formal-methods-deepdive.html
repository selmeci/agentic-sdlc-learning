0Why it matters

E4 built this workbook's verification hierarchy on tests — code that checks code, sampled one input at a time. Formal methods sit one rung higher: they check all inputs at once, by proof or exhaustive model checking, and for thirty years they stayed in avionics and hyperscaler infrastructure because writing the specifications cost too much. Two 2026 developments put them in this workbook. First, the economics flipped: when agents write both the code and the formal spec, the historically prohibitive authoring cost collapses — that is the argument of the antfly.io essay from the boss's reading list ("Cheap code means formal verification is reasonable now"), and it lands on prepared ground, because AWS documented the payoff back in 2015: "in every case TLA+ has added significant value, either finding subtle bugs that we are sure we would not have found by other means" — including a DynamoDB bug whose shortest error trace was 35 steps long. A benchmark now measures the practice — "vericoding," spec-to-verified-code — at 12,504 formal specifications, with off-the-shelf LLMs succeeding on 82% of Dafny tasks. Second, and separately, a research wave is applying formal methods to the agent itself: runtime enforcement frameworks (AgentSpec, ICSE 2026 — the only peer-reviewed source in this chapter) that watch an agent's actions against declarative safety rules, and predictive extensions that model-check where a trajectory is heading. The honest headline for both directions is the same: real prototypes, quantified trade-offs, and a state of the art that is runtime monitoring, not proofs of agent correctness — the December 2024 roadmap's verdict that "a fully comprehensive methodology for integrating formal methods into LLMs remains elusive" still stands. This chapter keeps the two directions rigorously apart, because conflating them is how formal verification gets oversold to clients.

1Two directions, one label — keep them apart

"Formal verification of agents" is used in the wild for two nearly unrelated practices, and the first discipline this chapter teaches is to ask which one a source means. Direction A — formal methods on the agent's output: the agent (or the human) writes a machine-checkable specification, and a verifier — Dafny's SMT backend, Lean's kernel, TLA+'s model checker TLC — proves the code or design meets it. The agent's role is authorship; the guarantee attaches to the artifact. The antfly essay lives here: its agent writes TLA+ models of a distributed key-value store and hunts race conditions. Direction B — formal methods on the agent's behaviour: the agent itself is treated as an opaque, stochastic component whose actions at runtime are checked against declarative rules — stop it before it deletes the database, whatever the model weights think. AgentSpec and its successors live here; so, at the deployed-practice end, do the learned guardrail filters (NeMo Guardrails, Llama Guard) that these formal approaches aim to outdo with provable semantics. The academic framing for why Direction B is runtime-only is Ferrando's FMAS 2025 vision paper ("Watchdogs and Oracles"): LLMs are "stochastic, non-deterministic, and continually updated," so static pre-deployment verification of the model's internals is impractical — a widely held position in the formal-methods literature, and the reason every serious system in §5 is a monitor, not a proof. Note what this means for vocabulary discipline: nothing in this chapter certifies "the agent is correct." Direction A certifies artifacts; Direction B bounds behaviours.

2The economics that changed

The cost objection is real and citable. The AWS team's CACM paper ("How Amazon Web Services Uses Formal Methods," Newcombe, Rath, Zhang, Munteanu, Brooker & Deardeuff, April 2015) states the industry's prior plainly: "formal methods have a reputation of requiring a huge amount of training and effort to verify a tiny piece of relatively straightforward code, so the return on investment is only justified in safety-critical domains" — and then reports that seven Amazon teams found "that perception to be quite wrong," with TLA+ surfacing bugs no other technique caught (the DynamoDB replication bug's shortest error trace: 35 steps). They even avoided the words "formal," "verification" and "proof" internally, marketing the practice as "debugging designs." Ten years later the residual cost — a human sitting down to write the spec — is exactly the line item agents attack. Rowan Copley's antfly essay (May 6, 2026) makes the economic argument for the agent era: if generated code is nearly free, the historically expensive step of formal modelling is nearly free too, so why wouldn't we point an agent at TLA+ for anything concurrency-shaped? His workflow is worth keeping precisely because it is so disciplined about ownership: the human scopes hypotheses and boundaries in two files (assumptions.md, boundaries.md); the agent writes and runs the TLA+ models; every bug the model checker finds must be reproduced as a failing unit test before it counts; the fix and the test are kept, and the intermediate TLA+ models are discarded — the deliverable is small, the modelling is scaffolding (I1's disposable tier, applied to proofs).

Tier it honestly: Antfly is a vendor blog (the company sells agent retrieval infrastructure, not verification tooling — the technique is the pitch, not a product), the essay contains one validation anchor and no quantified metrics. And the anchor is weaker than its retellings: Copley took a real race-condition PR from the Pebble repo (CockroachDB's key-value store) recent enough to postdate the model's training data, checked out the commit before the merge — and his workflow "actually homed in on another bug, which I was then able to verify was also fixed in another Pebble PR later." It found a real, independently-confirmed bug; it did not rediscover the bug. That is still a genuine result — and precisely the kind of nuance a client deck will flatten if we let it. The strategic tailwind, meanwhile, is real and primary-sourced: Werner Vogels' February 2026 conversation with Byron Cook (who runs AWS's automated-reasoning group) frames the coupling directly — "When AI systems generate code, make decisions, or provide information, we need efficient ways to verify that their outputs are correct" — with Automated Reasoning checks already shipping in Bedrock Guardrails under the "neuro-symbolic" banner.

3Vericoding — what the numbers actually say

The practice of generating verified code from formal specifications now has a measuring stick: the vericoding benchmark (arXiv:2509.22908; BAIF and MIT authors including Max Tegmark; an arXiv preprint — no peer-reviewed venue, present it as such). It assembles 12,504 formal specifications — 3,029 Dafny, 2,334 Verus/Rust, 7,141 Lean — and coins the term in explicit contrast to its opposite: vericoding generates "formally verified code from specifications… in contrast to vibe coding, which generates potentially buggy code from a natural language description." Off-the-shelf LLM success rates: 82% in Dafny, 44% in Verus/Rust, 27% in Lean — language and toolchain maturity dominate, and the paper notes pure-Dafny verification improving from 68% to 96% over the prior year. The earlier peer-reviewed anchor points the same way: Misu, Lopes, Ma & Noble (FSE 2024, arXiv:2402.00247 — UC Irvine, not Microsoft as often misattributed) got GPT-4 to produce "verified, human-evaluated Dafny methods for 58% of the problems" on 178 MBPP tasks with retrieval-augmented prompting (19% without context) — prompt-engineering, notably, not an agentic repair loop. Re:Form (arXiv:2507.16331) closes the training-side loop: RL with feedback from the Dafny verifier itself, "reducing human annotations" because the verifier is an incorruptible reward signal — the formal-methods version of E4's verifiable-reward argument.

Now the cold water, from the TLA+ side that antfly's essay lives on. A Loyola University Chicago study ("Can LLMs Write Correct TLA+ Specifications?", arXiv:2606.05792, June 2026) tested 30 LLMs across 205 natural-language-to-TLA+ tasks: up to 26.6% syntactic correctness, and only 8.6% semantic correctness — with the sobering conclusion that "current LLMs do not generate reliable TLA+ specifications without expert oversight." SysMoBench (arXiv:2509.23130) reaches the same posture at systems scale, benchmarking AI formal modelling of real concurrent systems (Etcd, Redis, ZooKeeper) and documenting real limitations. The apparent contradiction with the 82%-Dafny number dissolves on inspection, and resolving it is the chapter's most useful technical lesson: Dafny vericoding starts from a formal spec and has a mechanical checker in the loop — the LLM iterates against ground truth. NL-to-TLA+ starts from prose and asks the model to invent the formalization — semantic correctness has no oracle. Generation against a verifier works; unsupervised formalization does not (yet). The antfly workflow quietly respects this: its human scopes the properties, and its bugs must reproduce as failing tests before they count. And the production existence proof for Direction A is AWS again: Cedar, the open-source authorization language, is developed by "verification-guided development" — the model formalized in Lean with mechanized proofs, differentially tested against the Rust implementation across millions of inputs, with the Cedar Symbolic Compiler itself implemented in Lean and proven correct. Formal verification of agent-adjacent infrastructure ships at hyperscaler scale today; what remains research is the agent writing the formalization unsupervised.

4The intent gap — the spec is the bottleneck

If verifiers are mechanical and generation is cheap, everything concentrates in one place: where does the specification come from, and does it say what we meant? Shuvendu Lahiri's grand-challenge paper (Microsoft Research, arXiv:2603.17150) names it: "The gap between informal natural language requirements and precise program behavior — the intent gap — has always plagued software engineering, but AI-generated code amplifies it to an unprecedented scale." His programme, intent formalization, is "the translation of informal user intent into a set of checkable formal specifications" — which readers of this workbook will recognize as H2's EARS/Gherkin ladder extended to its top rung. The ladder view is the practical takeaway: plain-language AC → EARS-structured requirements → Gherkin scenarios that execute → property-based tests → Dafny/TLA+ specifications that verify. Every rung up buys stronger guarantees for more authoring discipline; H2 argued most brownfield teams should climb one rung; this chapter's evidence says the top rungs are becoming climbable because the agent does the transcription while the human owns the intent. The failure mode is climbing the ladder with unowned intent — a formally verified implementation of the wrong requirement is E7's rework problem with a proof attached. That is also the honest frame for E8's spec-driven development: SDD makes the spec the primary artifact; intent formalization makes it checkable; neither makes it correct — only the humans who own the intent do (P-module, H4's clarification loop).

5Direction B — runtime enforcement of agent behaviour

The December 2024 academic roadmap (arXiv:2412.06512 — authors including PAT's creators; toolchain surveyed: Z3, Coq, Isabelle/HOL, PAT, Alloy, PRISM, UPPAAL) is the reference point for expectations: bidirectional ambitions — formal methods to make LLMs reliable, LLMs to make formal methods usable — and the verdict quoted in §0: a comprehensive methodology "remains elusive." What exists eighteen months later is a coherent runtime-enforcement stack, and it is worth teaching because it is the deterministic end-state of E5's advisory/deterministic distinction. Reactive enforcement: AgentSpec (Wang, Poskitt & Sun, SMU — accepted at ICSE 2026, the one peer-reviewed system here) is a lightweight DSL — trigger, predicates, enforcement action — evaluated on every step of the agent loop. The authors report it preventing over 90% of unsafe code-agent executions and eliminating hazardous actions across ten embodied-agent task categories at millisecond overhead. Their most decision-relevant finding is negative: rules auto-generated by an LLM (o1, few-shot) reached 95.56% precision but only 70.96% recall on embodied-agent risks, identified 87.26% of risky code, and prevented rule-breaking in only 5 of 8 autonomous-vehicle scenarios — the guardrail author cannot be the same class of system the guardrail guards, which is E4's evaluator-independence argument arriving in the safety layer. (The paper's "first runtime enforcement framework" claim is hedged "to the best of our knowledge" — NeMo Guardrails and TrustAgent predate it on adjacent ground; never state it as fact.)

Predictive enforcement: ProbGuard — formerly Pro2Guard, and not to be confused with the Java obfuscator ProGuard — extends the idea forward in time (arXiv:2508.00500; same SMU group plus Jiali Wei, XJTU; a preprint whose v3 softened its own title from "enforcement" to "monitoring"). It learns a discrete-time Markov chain from agent execution traces and model-checks PCTL properties with PRISM at runtime, intervening when the probability of reaching an unsafe state crosses a threshold — reactive rules "lack foresight and struggle with long-horizon dependencies," in its framing (which doubles as its critique of the authors' own AgentSpec; a self-comparison, tier accordingly). Its SafeAgentBench numbers quantify the trade every client conversation about guardrails needs: baseline 40.63% of embodied-agent executions unsafe; AgentSpec-style reactive rules cut that to 19.79%; ProbGuard in hard-stop mode (θ=0.1) reaches 2.60% unsafe — but task completion collapses to 10.42%; its reflect mode lands between, 14.07% unsafe at 47.74% completion. Safety is purchasable; the currency is utility; pick the operating point per blast radius (E6's rubric, now with a measured curve). The wave behind these two is real but young — VeriGuard (Google authors, arXiv:2510.05156) verifies synthesized per-agent policies offline and monitors actions against them online; Agent Behavioral Contracts (arXiv:2602.22302) port design-by-contract with runtime-checked preconditions, invariants, governance and recovery clauses at sub-10ms overhead — and the deployed baseline they all aim to beat is the learned filter (NeMo Guardrails, arXiv:2310.10501; Llama Guard, arXiv:2312.06674), which offers no semantics, only a classifier. Every system in this paragraph except AgentSpec is a non-peer-reviewed preprint; the 2025–26 trajectory is steep, the deployed base is thin, and both facts belong in any client sentence about "provable agent safety."

6Week one at the client — what is usable now

Almost nothing in §5 is procurable today, and saying so is part of the value. What a brownfield client can actually deploy this year, in ascending ambition: (1) The deterministic rails we already teach — E5's hooks, I2's write permissions, S8's rules — are runtime enforcement without the formal semantics; frame AgentSpec-style DSLs as where that stack is heading, not what to buy. (2) Direction A on the narrowest, hottest surface: one concurrency-critical or money-critical component, the antfly workflow with its ownership discipline intact — human-scoped properties in reviewed files, agent-written TLA+ or property-based models, every finding reproduced as a failing test before anyone believes it, models discarded, tests kept. This is a contained experiment with mechanical exit criteria (T2), and the 8.6% semantic-correctness number is the reason the human scoping step is non-negotiable. (3) The intent ladder, one rung: where H2 already put EARS/Gherkin in place, pilot property-based tests generated from AC on one module — the cheapest formalization step with a real oracle. (4) Watch, don't buy, Direction B: track AgentSpec (ICSE 2026) and the behavioural-contracts wave; revisit when something ships with support. The reusable client sentence: proofs for the artifact where it is cheapest, rails for the agent where it is riskiest, and nobody certifies "the agent is correct."

8Applying it — decision guide

Three sentences to keep

"Formal verification of agents" means two different things — proving properties of what the agent produced (real today where a mechanical verifier sits in the loop: 82% Dafny vericoding, Cedar/Lean in production) versus bounding what the agent does (a fast-moving research stack — AgentSpec, ICSE 2026 — whose state of the art is runtime monitoring, not proofs) — and every source should be asked which one it means. The economics genuinely flipped: agents collapse the spec-authoring cost that kept formal methods in avionics, but the 8.6% semantic-correctness result for unsupervised NL-to-TLA+ says the human still owns the intent and the properties. Safety from runtime enforcement is purchasable and now priced — 40.6% → 2.6% unsafe at the cost of 59% → 10% task completion — so the blast radius, not the vendor deck, picks the operating point.
