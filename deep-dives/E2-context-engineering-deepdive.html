0Why context is the highest-leverage component

If E1 taught that the harness is where our engineering value lives, E2 is the sharpest edge of that value. Context engineering is the single biggest lever on output quality — bigger than model choice, bigger than prompt wording. An excellent model fed the wrong context produces confident, wrong work; a modest model fed precisely the right context often outperforms it.

For our engagement this is where a client's existing information architecture (M4) either pays off or bites. A brownfield repo with retrievable, well-structured sources makes context engineering cheap; a repo where knowledge lives in people's heads and stale wikis makes it the first thing we must fix. This document unpacks what context is, the physics that constrain it, how it is assembled, and what we configure.

The one idea to internalise

Context is not "everything the model might need" — it is the smallest set of high-signal tokens that makes the next step succeed. More context is not better; past a point it is actively worse. The discipline is curation under a budget, repeated every turn.

1What "context" actually contains

Context is the full working set handed to the model at a given step — not just the user's message. In a coding agent it is assembled from several distinct sources, each with a different trust level and lifespan:

Trusted / durable
  • system & role instructions (CLAUDE.md/AGENTS.md)
  • the task spec (handoff contract, M3)
  • source-of-truth artifacts: code, ADRs, specs (M4)
  • tool definitions — every registered tool's name, description and schema sits in the window before any work starts
Untrusted / tainted
  • retrieved web content
  • ticket & issue text, design-file text (M7 injection channels)
  • tool results from external systems
Ephemeral / working
  • prior turns in the session, tool outputs, scratch reasoning — the conversation so far, which grows every step and must be managed (compaction, summarisation).

The crucial mental model: every token in the window competes for the same finite attention budget. A retrieved file the agent doesn't need isn't neutral — it dilutes attention on the tokens that matter and, if it carries untrusted instructions, is an attack surface. Context assembly is therefore a series of inclusion decisions, each with a cost.

Two consumers are routinely under-budgeted because they arrive silently: tool definitions and accumulated tool results. In brownfield engagements, MCP tool-schema sprawl — every connected server injecting dozens of verbose schemas — is the dominant real-world context bloat we see at clients. Mitigations: a small set of well-described tools beats a large overlapping one; defer or lazily load tool schemas until a task actually needs them; and clear or summarise stale tool results once they have been acted on, rather than letting them ride along every subsequent turn.

2The physics: context rot & the attention budget

The hard constraint that shapes everything is context rot: a model's ability to accurately recall and use information degrades as the context window grows. This is not a bug to be patched; it is an empirical regularity: attention spread across more tokens gets diluted, and models are simply not trained or optimised to perform at those depths — architectures with sub-quadratic attention exhibit the same degradation, so it is not just a compute-cost artifact of full attention. Chroma's research across 18 models showed recall falling well before the advertised window limit, and getting worse as the needle's semantic similarity to the question dropped.

Two consequences we act on. First, precise lexical retrieval over a small context beats semantic search over a huge one — this is why Claude Code uses agentic grep/glob/read rather than pre-indexing the codebase (see E6/M4). Second, long-running work must actively shed context: compaction (summarise and reinitialise), structured note-taking (write state to a file, drop it from the window), and sub-agents (isolate a sub-task in its own window, return only a short summary).

The Goldilocks framing

The system prompt and assembled context should sit in a "Goldilocks zone": specific enough to constrain behaviour, not so bloated that it rots. Neither brittle hard-coded if-else instructions nor a vague "be helpful and use good judgment." The same applies to what you retrieve: enough to succeed, nothing that merely could be relevant.

3How context is assembled — inputs & outputs

Context engineering is a runtime discipline, executed by the harness every turn. Broken into its moving parts:

A

Retrieval

finding the right tokens

The mechanism that pulls candidate information: agentic search (grep/glob/read over the live filesystem) or RAG (embeddings over a pre-built index), or a hybrid. The choice is a real architectural decision covered in M4/E6.

Requires on input
  • a query or navigation strategy
  • source-of-truth artifacts that are actually findable (M4)
  • for RAG: a maintained index; for agentic: fast file tools
Produces on output
  • candidate snippets/files to consider including
  • freshness (agentic reads live state; an index can drift)
Where it sits
  • Implementation runtime — the agent retrieves as it navigates; also onboarding, where retrieval quality decides how fast an agent understands a codebase.

For our rollout: default to agentic search (precision, freshness, no index to maintain); consider hybrid only when a repo is too large to navigate by grep in a reasonable number of steps, and measure the benefit against maintenance cost.

B

Selection & compression

deciding what actually goes in

Given candidates, decide what to include verbatim, what to summarise, and what to leave as a link. This is where the attention budget is spent well or wasted.

Requires on input
  • retrieved candidates + the current goal
  • a budget discipline (target token size)
  • the "reference, don't copy" convention (M4)
Produces on output
  • a curated window: verbatim where needed, summarised excerpts + links elsewhere
  • headroom left for reasoning and tool results
Where it sits
  • Cross-cutting, every turn — the core of "context engineering" as a live activity.

For our rollout: copy summarised excerpts with a link to the master, never paste whole documents (that both rots the window and duplicates a source of truth). This is the M4 principle applied at runtime.

C

Long-horizon management

surviving many turns

For tasks spanning many steps, the window fills with history and rots. Three techniques shed load without losing the thread.

Requires on input
  • a place to persist state (progress file, notes, memory — the E1 memory component / M4)
  • a stop/compact trigger (window pressure, milestone)
Produces on output
  • a reinitialised, small context that preserves decisions & open threads
  • continuity across compaction boundaries
Where it sits
  • Implementation runtime for long tasks; the boundary where context (E2) hands off to memory (the E1 memory component, module M4).

For our rollout: pick the technique by task shape — compaction for a long conversational flow, note-taking for milestone iteration, sub-agents for parallel exploration where only a summary needs to return. Each keeps the main window in the green zone.

Cache economics — the cost dimension of assembly

Assembly choices also decide what you pay. Prompt/KV-cache hit rate is a first-order cost and latency lever: a cache miss costs a multiple of a hit (on current pricing, roughly an order of magnitude — a point-in-time figure, re-verify). Practitioner synthesis (notably the Manus team's published lessons) converges on three rules: keep the system prefix byte-stable (no timestamps or reordered tool lists at the top); assemble context append-only — never mutate or reorder earlier turns, because any edit invalidates the cache from that point forward; and time compaction cache-aware, since compaction deliberately rewrites history and forces a full re-read, so batch it at milestones rather than compacting eagerly.

4Context as an attack surface

Because the model cannot separate trusted instructions from untrusted data in one token stream, context is the primary delivery vector for prompt injection (M7). Any retrieved web page, ticket body, design-file layer name, or tool result can carry hidden instructions that the model may follow.

The rule that follows

Treat every retrieved item as tainted until proven otherwise. Summarise rather than pass raw untrusted content (Claude Code does this for web fetches). Once untrusted content has entered the window, apply the trifecta logic: an agent holding untrusted input must not also hold both sensitive data and an exfiltration path (Meta AI's "Agents Rule of Two", 2025 — M7). Context engineering and security engineering are the same activity seen from two angles.

6Takeaways for our engagement

Assess (feeds E6)
  • Are source-of-truth artifacts retrievable, or tribal knowledge?
  • Is documentation fresh enough to be worth retrieving?
  • Repo size/shape: does grep navigation stay tractable?
  • Where does untrusted content enter (tickets, web, design)?
Build
  • default to agentic search; hybrid only when measured to help
  • enforce "reference, don't copy" — excerpts + links
  • set compaction/notes conventions for long tasks
  • taint untrusted retrieval; summarise web content
Three sentences to remember E2 by

Context is the smallest set of high-signal tokens that makes the next step succeed — more is worse, because attention rots as the window grows. Retrieve precisely, include sparingly (reference don't copy), and shed load on long tasks via compaction, notes and sub-agents. Every retrieved token is also a trust decision, so context engineering and injection defence are one discipline.
