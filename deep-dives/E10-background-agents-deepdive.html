0Why it matters

Everything in M1 so far assumes a developer watching an agent work. This chapter is about what happens when nobody is watching — agents that run in the cloud, decoupled from anyone's laptop, for hours or days, triggered by events rather than prompts. The pattern converged across every major vendor within about a year: OpenAI's Codex cloud tasks (May 2025), Cursor's background agents (GA June 2025), Google's Jules, GitHub's Copilot coding agent, and Ona — the ex-Gitpod platform whose background-agents guide this chapter dissects, and which OpenAI agreed to acquire in June 2026 precisely "to help Codex take on longer-running work." The capability curve says this is not hype timing: METR's time-horizon data — the E7 metric — now puts the frontier at roughly a 12-hour 50%-success task length (Claude Opus 4.6, TH1.1 data), with the doubling time compressed to ~4.3 months since 2023. But the same measurement discipline cuts the other way: a July 2026 long-horizon benchmark built expressly for sustained work found the best frontier model completing only ~15% of 90-minute terminal tasks at a strict threshold — long-horizon execution is precisely where agents still run out of steam. That tension — capability doubling into a regime the benchmarks say is barely solved — is why this topic needs governance-first treatment, not enthusiasm. Our L1–L5 scale (E6) answers how much an agent may do; this chapter adds the orthogonal axes the scale does not cover: where it runs (your machine vs an isolated sandbox), when (interactive vs async), and who initiates (a human prompt vs a PR, a ticket, a CVE feed, a cron job).

1The execution axis — what a background agent is

Ona's definition is the cleanest on record: "Background agents run in their own development environment in the cloud, with a full toolchain, test suite, and access to internal systems, completely decoupled from your device." The contrast class is the laptop assistant — Claude Code or Cursor in a terminal, synchronous, watched. The background agent inverts every default: it gets an ephemeral, isolated environment provisioned per task; it works asynchronously (you review the result, not the process); and increasingly it is event-initiated — a PR comment, a Slack message, a Linear ticket, a webhook, a published CVE. That last inversion is the conceptually new one for our framework: E6's levels grade how much an agent may decide, but they quietly assume a human pressed enter. Event-driven initiation removes that assumption, which is why §5's governance cannot live in the prompt.

The industry convergence is striking and verifiable from primary launch materials, whatever one thinks of any vendor's numbers: OpenAI's Codex cloud (research preview May 16, 2025) runs each task in its own cloud sandbox preloaded with your repo, works many tasks in parallel, and proposes PRs; Cursor's background agents (v0.50 May 2025, GA in 1.0 on June 4, 2025) spin up an isolated Ubuntu VM, clone the repo, work on a dedicated branch and open a PR — extended in February 2026 to full desktop-and-browser "computer use" per agent; Google's Jules is an async coding agent with a CLI and public API (October 2025) for embedding in native workflows; GitHub's Copilot coding agent picks up issues and works them in an Actions-based sandbox into a PR. Five independent product teams, one shape: sandbox → work → verifiable artifact (PR) → human review. The shape is the finding — it encodes the assumption that agents at this autonomy level must be contained (S-module), must leave an auditable trail (§5), and must terminate in the one artifact engineering organizations already know how to gate: a pull request.

Addy Osmani's companion essay ("Long-Running Agents," April 28, 2026 — high-quality practitioner synthesis, our tiering) contributes the taxonomy this chapter uses throughout, separating three things vendors blur: long-horizon reasoning (planning across many dependent steps), long-running execution (a process that survives hours or days and thousands of model calls), and persistent agency (identity and memory outliving any single task — I4's territory, and the only one of the three that raises S6-style poisoning stakes). A background agent needs the second; it benefits from the first; it does not require the third — and conflating them is how pilots end up buying persistent memory infrastructure to solve what was actually a checkpointing problem.

2The long-horizon evidence — both directions at once

The capability line every deck quotes is METR's (arXiv:2503.14499, the E7 method): "the frontier AI time horizon has been doubling approximately every seven months since 2019" — with the abstract's own hedge, "though the trend may have accelerated in 2024." It did. The TH1.1 revision (January 2026) puts the doubling time since 2023 at roughly 4.3 months (130.8 days, CI 107–161), and METR's current raw data (May 2026 update) places Claude Opus 4.6 at a ~12-hour 50%-success horizon, with an early Claude Mythos preview point at ~17.4 hours. Osmani's essay quotes only the conservative seven-month figure — when citing this material, use the accelerated numbers and name the revision, or the client's planning horizon will be off by a factor of two. The 80%-success horizons remain far shorter than the 50% ones, which is the fine print that matters for unattended work: a coin-flip success rate is delegation-with-review; it is nowhere near fire-and-forget.

Now the other direction. The Long-Horizon Terminal-Bench (LHTB, arXiv:2607.08964, July 2026) was built specifically for sustained autonomous work: 46 terminal tasks across 9 categories, a 90-minute budget per task, hidden fake-proof verifiers, and dense partial credit — because binary pass/fail "collapses almost every model to zero." Agents consumed on average 9.9M tokens and ~231 episodes per task; the best frontier model achieved 15.2% pass@1 at a 0.95 reward threshold (10.9% at a perfect 1.0), with a cross-model mean of 4.3%. SWE-Bench Pro (arXiv:2509.16941) points the same way from the task side: its problems "may require hours to days for a professional software engineer to complete," and they were chosen precisely because current agents mostly cannot. Read together, the two lines say: the frontier horizon is growing fast, and the reliability at that horizon is low. Osmani's "three walls" name the mechanisms — finite context (E2's context rot arrives long before the token limit), no persistent state across sessions, and no trustworthy self-verification (models grade their own work too generously — E4's oldest lesson). Everything in §3 and §4 is engineering around those three walls.

Numbers that do NOT exist (yet)

No independent survey publishes a "share of developers delegating async background tasks" figure. DORA 2025 reports ~90% of developers using some form of AI assistance — general assistance, not background delegation; do not stretch it. The adoption evidence for background agents specifically is vendor case studies (§6), and they are tiered accordingly.

3Persistence mechanics — surviving the context window

The primitive underneath every production pattern is embarrassingly small. Geoffrey Huntley's Ralph loop ("Ralph Wiggum as a 'software engineer'," ghuntley.com, July 14, 2025) is literally while :; do cat PROMPT.md | claude-code ; done — an amnesiac agent re-launched forever, with all state externalized to files (a PRD, a progress file, the repo itself). The insight that makes it more than a joke: the agent is amnesiac, but the filesystem is not. Fresh context each lap sidesteps context rot; durable files carry the plan forward; git carries the work. Every serious system since is a refinement of that split (attribute the pattern to Huntley, popularized with Ryan Carson — a practitioner pattern, not a standard).

Anthropic's two engineering posts supply the refined version, and E9 already met them as harness artifacts. The long-running harness (Nov 2025) splits roles: an initializer agent — "the very first agent session uses a specialized prompt that asks the model to set up the initial environment" — builds the feature list, progress notes and scripts; then a coding agent per session — "every subsequent session asks the model to make incremental progress, then leave structured updates." Its most load-bearing rule guards the work ledger itself: "It is unacceptable to remove or edit tests because this could lead to missing or buggy functionality" — which, per this workbook's I2 verification, protects the feature-list entries the harness tracks (Osmani's gloss "test ratchet" names the family of mechanism; I2/ImpossibleBench cover the protect-the-test-files variant). The Managed Agents architecture (Apr 2026) then virtualizes the whole thing: "a session (the append-only log of everything that happened), a harness (the loop that calls Claude and routes Claude's tool calls…), and a sandbox (an execution environment where Claude can run code and edit files)" — brain decoupled from hands. Recovery becomes an API call: "When one fails, a new one can be rebooted with wake(sessionId), use getSession(id) to get back the event log, and resume from the last event." The reported payoff — p50 time-to-first-token down roughly 60%, p95 down over 90% — is a vendor number, but the design lesson is tier-free: the session log, not the agent process, is the unit of persistence. Harvey's summit formulation says the same thing from the audit side: "the run is the persistent record, not the agent."

Two corollaries complete the mechanics. Context resets as a first-class operation: for very long tasks the working pattern is not rolling summarization but a full teardown — write a structured handoff document, kill the session, start clean from the handoff (the same discipline H1's contract imposes between humans and agents, applied agent-to-agent; and exactly what a Ralph lap does implicitly). Checkpoint-and-resume beats heroic sessions: commit early and often to git, treat each green state as a savepoint, and let recovery mean "reset to last savepoint and replay intent" — brownfield teams already know this discipline as trunk-based development; the agent inherits it for free if the harness enforces small increments (Anthropic's "incremental progress" prompt).

4Verification at a distance — roles, reviews, and the fatigue data

Unattended execution concentrates everything E4 says about self-verification. Both source guides converge on the same two rules. First, the evaluator is never the generator: Anthropic's harness-design post builds a GAN-inspired generator/evaluator pair (E9 §4); Ona's summit synthesis reports every adopting org giving agents the means to verify their own work mechanically — tests, telemetry queries, screenshots — rather than by self-assessment; OpenAI's internal Codex teams route agent output through agent-to-agent review before human review. Cursor reportedly goes further and splits three roles — planner / worker / judge — with different models per role, having found GPT models outlasting Opus for extended autonomous work because Opus "tended to stop early"; that detail reaches us only through Osmani's retelling, so treat it as secondhand practitioner report, not a benchmark. Second, delegated approval with escalation: pre-authorize the reversible (formatting, test additions, dependency bumps inside policy) and route the irreversible (schema changes, deletions, anything security-adjacent) to a human — E6's rubric, made operational because nobody is present to click.

The approval-fatigue telemetry explains why per-action human gating cannot be the answer at this scale. Anthropic's auto-mode post (March 2026) opens with the number: "Claude Code users approve 93% of permission prompts." A gate almost always answered yes is behaviourally no gate at all — and usage data shows auto-approve rates rising with experience, from roughly 20% for new users to over 40% by 750 sessions (partly deliberate trust calibration, per Anthropic, not pure fatigue). Even the mitigation carries a caveat worth quoting in any client conversation: Anthropic's own classifier for dangerous actions reports a "17% false-negative rate on real overeager actions… the honest number." The design conclusion is the S-module's, arrived at from the UX side: move enforcement from the approval dialog to the environment — which is §5.

5Governance — the boundary is the infrastructure

The single most reusable sentence in either source is Ona's: "If the guardrail lives inside the same context window as the agent, it's a suggestion, not a boundary. Security enforcement needs to happen at the infrastructure layer." That is E5's advisory-versus-deterministic distinction and S3's rules-file lesson, restated for the async regime — and every serious deployment in the summit material behaves accordingly: network policies, kernel-level controls, scoped short-lived credentials, air-gapped sandboxes for the most sensitive monorepos, ephemeral workers destroyed after each run with exactly one repo, one tool bundle, one artifact path and one audit trail each. Sandboxing is reported as universal — "every adopting organization sandboxes agents, no exceptions" — which we can believe as a design norm even while tiering the surrounding numbers as vendor claims, because it is the same conclusion the S-module reached from the lethal-trifecta side: an event-triggered agent processing untrusted inputs (PR comments, tickets, CVE feeds!) with repo access and an outbound channel is the trifecta by construction unless the infrastructure breaks one leg.

Cost is a governance surface, not an accounting detail. A background fleet can fail by looping, and a looping agent burns money at machine speed with nobody watching — Uber's summit self-report of AI compute costs up 6× since 2024 shows the baseline growing even when everything works. The practitioner consensus pattern is tiered budgets plus a token-velocity circuit breaker: per-session and per-day caps with automatic termination, and a rate-based trip (one published tunable: a healthy agent runs ~3–4k tokens/minute; tripping at ~10k/min catches a runaway loop inside a minute). Cite the pattern as practitioner synthesis; the viral runaway-bill dollar figures circulating with it trace only to blogs citing each other — folklore until a primary incident report exists (the same discipline that retired Boehm's 100:1 in H4). Add the two remaining rails: credential hygiene — long-lived tokens are the async agent's most dangerous convenience; scope them per run, expire them with the sandbox — and observability as a precondition, not an afterthought: Ona's red-flag list is blunt that fleets without per-run cost and outcome visibility are pilots that will not survive contact with a CFO. Harvey's "the run is the persistent record" closes the loop: the same append-only session log that §3 uses for recovery is the audit artifact compliance will ask for (H3's provenance, T4's Article-12-style logging).

6The adoption path — maturity, first use cases, and the vendor numbers

Ona's maturity model is a useful narrative frame precisely because it is self-serving — read it with that light on. Phase one, laptop code assistants, it calls a "false summit": a local maximum that speeds the individual developer without moving organizational throughput (the same individual-vs-system gap E7's measurement section documents from the DORA side). Phase two, background agents — "you can't skip this phase" — is where isolation, verification and progressive autonomy get built. Phase three, the "software factory," is fleets responding to events end-to-end ("when a CVE is published, agents scan your repos and open a PR"), humans reviewing. The vendor incentive is obvious: phase two is what Ona sells. But the engineering claim underneath survives the discount — you cannot operate event-driven fleets without first having built sandboxing, verification, observability and budgets, and teams that try to jump from laptop assistants to the factory are skipping the harness. The trap list is corroborated by everything this workbook has verified independently: too-ambitious first projects, agents treated as junior devs needing chat supervision, laptop or shared-CI execution as a dead end, no observability, security bolted on in-context instead of infra-layer, and "waiting for a better model" instead of engineering the harness (E9's founding argument).

First use cases: "repetitive, well-scoped work with clear success criteria and real business value" — migrations, flaky-test fixes, dependency upgrades, security patching, documentation. Toil, not greenfield features. This matches our brownfield thesis (M5): the highest-value early work is exactly the work with the strongest mechanical verification and the lowest ambiguity. The build-vs-buy guidance — build only with a dedicated platform team, a 12–18-month horizon, and infrastructure as a differentiator; sandboxed environments alone are a 3–6-month build — is vendor-estimated but directionally consistent with T1's economics. The summit case studies, all self-reported at a vendor-hosted event, none independently audited — quote them as existence proofs of the pattern at scale, never as expected returns:

7Week one at the client

Day one — locate the client on the map. Almost certainly phase one; the useful question is which phase-two ingredient is furthest along (usually CI as verification) and which is absent (usually per-run observability and budgets). Day two — pick the first background task from toil, not features: one migration, dependency-upgrade or flaky-test class with mechanical success criteria, run through whichever background-agent surface the client's existing vendor already ships (Copilot coding agent, Codex cloud, Cursor — no new procurement for a pilot). Day three — write the delegation contract before the first run: goal, scope, non-goals, tool permissions, stopping condition, budget cap, escalation rule (H1's contract anatomy, pointed at an agent that cannot ask mid-flight). Day four — wire the three rails: sandbox with scoped credentials (infra-layer, not prompt-layer), token/cost cap with automatic termination, and the PR as the only exit channel. Day five — review the run record, not just the diff: the session log is the audit artifact; if the pilot's runs are not reconstructible from it, fix that before scaling — it is also the E7/T2 measurement substrate. Track E6-style operational metrics from day one: interventions per run, escalation rate, cost per merged change, defect escape rate.

9Applying it — decision guide

Three sentences to keep

Background agents change the axes, not just the amount, of autonomy — where the agent runs, when, and who initiates — so governance must move from the approval dialog (93% approved, effectively unread) into the infrastructure: sandbox, scoped credentials, budgets, and the PR as the only exit. The frontier task horizon is doubling every ~4 months and sits near 12 hours at 50% success, but purpose-built long-horizon benchmarks show even the best models completing ~15% at strict thresholds — delegate toil with mechanical verification, and treat fire-and-forget as unearned. The unit of persistence, recovery and audit is the append-only run record, not the agent process — design for replay, and the same log serves the pilot's measurement and compliance for free.
