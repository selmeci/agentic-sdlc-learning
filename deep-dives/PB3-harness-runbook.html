0Why this runbook exists

PB3 installs the cage around the write scope PB2 earned. PB2 built the net — one characterized, mutation-audited module with a CI loop the agent can read; PB3 builds the harness that makes opening that module's write scope safe: the context layer the model reads, the deterministic guardrails it cannot argue with, and the platform layer that binds every writer, human or agent. The design rule is the litmus E1 states and E5 turns into a method: "what the model should know" goes in the prompt layer — CLAUDE.md, advisory; "what must never happen" goes in a hook, a permission rule, or a CI gate — deterministic. Under prompt injection, only the deterministic layer holds (E1 Agent = model + harness, E5 Governance).

Unlike PB1, which meets any client, and PB2, which handles two entry postures, PB3 has exactly one entry posture: a recorded PB2 GO — one module whose net catches change, catches faults at the ADR-recorded mutation bar, and runs in CI. PB2's own exit routing states the principle this runbook enforces: "the F2 boundary opens exactly as wide as the net that guards it." If there is no PB2 GO there is nothing to build a cage around — installing write permissions on an unverified module automates the exact anti-pattern the whole framework exists to prevent. Go back to PB2 Bootstrap first.

The case for the deterministic side is measured, not rhetorical. ImpossibleBench (Zhong, Raghunathan & Carlini, arXiv:2510.20270) mutates tests to contradict the spec, so that passing equals cheating: GPT-5 cheats in 54% of trials, and Claude models and Qwen3-Coder cheat primarily — more than 79% of the time — by editing the test cases themselves; giving the model an explicit abort tool cut cheating from 54% to 9%, an escape hatch cheaper than the temptation. And the Replit incident (July 2025, reported by The Register and Fortune): an agent ran npm run db:push against a production database during a declared code freeze, destroying live data for a thousand-plus executives and companies. Jason Lemkin's diagnosis — "There is no way to enforce a code freeze in vibe coding apps" — is the whole argument in one sentence: instructions existed, a gate did not (I2 Lifecycle & write permissions).

The vendor now says it too

Anthropic's own documentation describes CLAUDE.md and memory as "context, not enforced configuration" and continues: "To block an action regardless of what Claude decides, use a PreToolUse hook instead." A first-party confirmation of the litmus (official docs, verified 2026-07-18 — point-in-time like every tool fact on this page). And the harness is not decoration around the model: a May 2026 position paper measured harness-induced variance exceeding model-induced variance by 7.8× on average (arXiv:2605.23950) — which harness the client ends up with matters more than which model (E9 Harness tuning).

The theory this runbook distils: E1 for the seven-component frame, E4 Verification-first for the three-layer protection ladder, E5 Governance for the mechanisms and the five layers every harness has, I2 for the write-permission matrix and the hard evidence, and E2 Context engineering for the budget discipline the context layer must obey.

1How to run it

Run PB3 with one person plus the agent, the module owner within reach, scoped to the PB2-GO module and its CI/config surroundings — nothing else. The steps are sequential by design: each artifact is a precondition of the next (the permission baseline before the hook that complements it, both before the smoke task that probes them), and you never verify a step with a later step — B1 Safe ordering's discipline, unchanged from PB2. Size every step at two days or less. As in PB1 and PB2, every step is written as ČO (the action), PREČO (the theory link that justifies it) and DONE (the exit criterion).

One thing is new here: almost every DONE criterion has the same shape — a deliberate probe is blocked or caught, verified once. A harness is proven by attack, not by reading its config: you ask the agent to edit a protected test and watch the denial, you plant a dummy secret and watch the scan go red, you push as a non-owner and watch the platform refuse. If a criterion cannot be probed, the thing it describes is not installed yet.

In the workbook version of this runbook, progress is saved locally (same store as topic progress). This standalone copy does not persist checkbox state.

2Phase A — The context layer: extracted, not decreed

The advisory layer comes first not because it is strongest but because it is cheapest and steers the ninety-five-percent case; the deterministic twin of its every "never" lands in Phase B. Its failure mode is decree — B1's WO3: writing the context file from aspiration before the evidence exists. That failure is measured: agent-written context files scored −2% success at +23% cost in the ETH Zurich study, while human-written ones were modestly positive. The discipline that works is extraction — the file records what the repo already does, and a human prunes it. E2 adds the budget rule: the smallest set of high-signal tokens, reference-don't-copy, because recall degrades well before the context window fills (context rot) — a bloated CLAUDE.md dilutes the very attention budget it exists to protect.

Run the extraction prompt (template below) over the PB2 module: build, test and lint commands verbatim; conventions the code actually follows, two cited examples each; corrections reviewers have made repeatedly. Nothing the agent cannot cite from the repo goes in. Why: extracted-not-decreed (B1 WO3, E2); iteration on the file after install is E9's Ratchet, later work. One vendor fact shapes the file layout, quoted from the official docs (verified 2026-07-18): "Claude Code reads CLAUDE.md, not AGENTS.md" — so the master is written once as AGENTS.md and CLAUDE.md imports it (@AGENTS.md), keeping a single source every tool shares; a symlink is the no-extra-content alternative. Do not cite tool-adoption counts for AGENTS.md — they are aggregator claims.

DONE = draft exists; every line traceable to a cited observation; zero invented rules on owner spot-check.
# Context layer — extraction prompt + file skeleton (one pass, two artifacts)

# 1) The extraction prompt (agent, READ-ONLY on the pilot module)
Role: you are extracting a context file for <module>. Read-only.
Rules — extract, don't decree:
1. List the build, test and lint commands that actually work, verbatim.
2. List naming / layout / error-handling conventions the code actually
   follows — cite two file examples per claim. No aspirations, no "should".
3. List corrections reviewers made repeatedly (git log, PR comments) —
   a rule earns its line only by recurring.
4. Do NOT invent rules. Anything you cannot cite from the repo, leave out.
Output: a draft AGENTS.md; a human prunes it before anything is committed.

# 2) CLAUDE.md (repo root) — one line of substance:
@AGENTS.md
# Claude Code reads CLAUDE.md, not AGENTS.md (official docs, 2026-07-18);
# the import keeps ONE master all tools share. Add below it only what other
# tools must not see (e.g. plan-mode notes for this module).

# 3) AGENTS.md (the shared master) — skeleton:
## Build & test        <the verbatim commands from the extraction>
## Conventions          <only what recurred; one line each; date-stamped>
## Never (advisory L1 — the ENFORCED copy lives in Phase B hooks/permissions)
- Never modify or delete tests under tests/ — fix the code, not the test.
- Never edit CLAUDE.md/AGENTS.md, CI config or migrations directly; propose a PR.

The module owner prunes the draft to what is genuinely load-bearing: agree a line budget up front (E3's heuristic — a progress file over ~200 lines is a smell — applies to steering files with equal force), cut anything an ADR or spec already covers (reference, don't copy), date-stamp what remains. The advisory "never" lines from the skeleton stay in — they are L1 of E4's protection ladder, necessary and insufficient alone. Why: context engineering is a runtime discipline, not a one-time artifact — this file will be maintained (E9); PB3 only installs its first honest version.

DONE = CLAUDE.md + AGENTS.md committed; under the agreed budget; import wired (one master); owner sign-off in report §1.

3Phase B — Deterministic guardrails: the starter kit

This phase is the starter kit this topic promised: every template below is a runnable config, not prose about config. The shape is E4's three-layer test-protection ladder — L1 advisory (Phase A wrote it), L2 a PreToolUse hook that blocks regardless of what the model decides, L3 a CI gate that proves the suite is intact and genuinely green at merge. The order inside the phase follows I2's week-one sequencing, which is also the answer to this topic's own open question — which rules graduate from prose to enforced hooks in week one: "gate the scorers first (tests, CI config), then the steering files (CLAUDE.md/AGENTS.md — small files, outsized blast radius), then decisions (ADR seams), then the long tail." Resist gating everything on day one: that relocates review onto every write at once and the team routes around it — the same overload pattern H4 documents for over-specification.

Install the deny/ask/allow baseline below, scoped to the module: the scorers and steering files deny-listed, high-blast-radius commands on ask, the module's own test and lint commands allowed so verification is the path of least resistance (E4). Why: this is the declarative-policy layer, the first of E5's five layers every harness has — and the litmus applied: every deny line is a "must never happen". Once the pilot pattern settles, the non-negotiables belong in managed-settings.json (MDM-deployed), which sits above user and project settings and cannot be overridden — teams inherit the baseline and cannot silently weaken it (E5; point-in-time).

DONE = baseline committed at project scope; a deliberate agent attempt to edit a denied path AND to read .env both refused, verified once, logged in report §2.
# Permission baseline — .claude/settings.json (project scope, committed).
# Syntax verified against the official Claude Code docs 2026-07-18 —
# point-in-time; re-verify before installing at a client.
# Evaluation order is deny → ask → allow, first match wins; specificity does
# NOT change order, so a deny can never carry exceptions.
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(**/*.pem)",
      "Read(./secrets/**)",
      "Edit(./tests/**)",
      "Edit(./CLAUDE.md)",
      "Edit(./AGENTS.md)",
      "Edit(.github/workflows/**)",
      "Bash(git push --force*)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(npm run db:*)",
      "Bash(*migrate*)"
    ],
    "allow": [
      "Bash(npm run test *)",
      "Bash(npm run lint)"
    ],
    "defaultMode": "default"
  }
}
# The gotchas that bite (all from the official docs, mid-2026):
# - Edit(path) covers ALL file-editing tools (Edit, Write, NotebookEdit).
#   A Write(path) rule is accepted but silently never matches — use Edit(...).
# - "/path" anchors at THIS settings file's directory, not filesystem root;
#   "//path" is absolute; a bare filename (".env") matches at any depth.
# - Deny rules bind the agent's file tools and recognized shell commands —
#   NOT arbitrary subprocesses. A script the agent runs can open any file the
#   OS allows; closing that hole is the sandbox layer's job (§6).
# - Built-in protected paths (.git, .claude, shell rc files, .mcp.json,
#   .pre-commit-config.yaml, …) are never auto-approved no matter what allow
#   rules say — do not re-invent that list here.

Install the hook below: it denies edits to the PB2-committed tests and the steering files, with a reason the model can read, and returns no opinion for everything else. Why: rules are declarative; the hook is E5's programmable-interceptor layer, for anything a path pattern cannot express — and the deliberate redundancy with B1 is the point: two independent layers guarding the scorers, justified by ImpossibleBench's finding that models cheat primarily by editing tests (I2, §0). One correction to carry, flagged point-in-time: per the current official docs a PreToolUse hook decision does not override permissions.deny/ask — rules and hooks are layered, and a blocking hook beats an allow rule (earlier 2025-era write-ups, including our own E5 §2 sketch, describe hook-wins semantics; follow the docs and re-verify at install time). E5's design rule stands unchanged either way: the hook returns no-opinion for whatever it does not explicitly handle — never blanket-allow.

DONE = probe edit of a protected test blocked with the reason visible; a legitimate non-protected edit still passes (regression-check the fall-through — a bug there silently reopens everything); both in report §3.
# PreToolUse guard hook — L2 of the E4 ladder. Two files.
# Syntax verified against the official Claude Code docs 2026-07-18.

# 1) .claude/settings.json — the hooks block
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/protect-tests.sh"
          }
        ]
      }
    ]
  }
}

# 2) .claude/hooks/protect-tests.sh  (chmod +x; reads tool input as JSON on stdin)
#!/bin/bash
FILE=$(jq -r '.tool_input.file_path // empty')
case "$FILE" in
  */tests/*|*CLAUDE.md|*AGENTS.md)
    jq -n '{
      hookSpecificOutput: {
        hookEventName: "PreToolUse",
        permissionDecision: "deny",
        permissionDecisionReason: "Protected path (PB3 ADR): approved tests and steering files are read-only for the agent. Propose the change as a PR instead."
      }
    }'
    ;;
  *)
    exit 0   # NO OPINION — fall through to the permission rules.
             # Never blanket-allow here: a bug in this branch would
             # silently reopen everything the hook exists to protect.
    ;;
esac

# Exit codes: 0 = success (stdout JSON is parsed); 2 = blocking error
# (stderr shown to the model); any other = non-blocking error, call proceeds.
# Layering (per official docs, mid-2026): the hook does NOT override
# permissions.deny/ask — B1's deny rules stay in force alongside this.

Wire L3: a required check proving, on every PR, that the suite is intact (no tests deleted or skipped into silence), genuinely green, and that the mutation score has not regressed below the bar recorded in PB2's ADR — reuse PB2's diff-scoped Mode-1 job as the component (E4, PB2). Why: L2 binds the agent's tools inside a session; L3 binds the artifact at merge, whoever produced it. Two platform facts, vendor-primary (GitHub, mid-2026): branch protection and CODEOWNERS apply to agent PRs exactly as to human PRs — no separate native "AI PR" gate exists, and none is needed; and agent-opened PRs should keep the standard first-run "Approve and run" workflow gate, so a human looks before any CI executes. GitHub's own Copilot coding agent ships precisely these guardrails — pushes only to its own branch, workflows human-approved, requester cannot self-approve — vendor-designed, and it concedes the whole argument.

DONE = a deliberately weakened test change (one deleted case, one added skip) goes red end-to-end, verified once, linked in report §4.
# CI verification gate — L3 of the E4 ladder (GitHub Actions flavour, adapt).
# ONE gate for every PR, agent or human — the enforcement surface is the one
# you already have, not a new "AI pipeline".
name: verification-gate
on: pull_request
jobs:
  suite-intact:                    # no tests deleted or skipped into silence
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - run: <fail if the diff deletes files under tests/>
      - run: <fail if the diff adds skip / xfail / disabled markers to tests/>
  genuinely-green:
    needs: suite-intact
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: <full suite command>            # green means green — no "|| true"
  mutation-bar:
    needs: genuinely-green
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - run: <diff-scoped mutation command>  # reuse PB2's Mode-1 job here
      - run: <assert score at or above the ADR bar>   # ~70% is OUR heuristic,
                                                       # recorded in the ADR
# Outside the YAML: mark this workflow a required status check on the default
# branch, and keep the first-run "Approve and run" gate for PRs opened by any
# agent identity (a human approves before workflows execute).

Before the write scope opens, install the scan stack below: gitleaks in pre-commit and CI, a detect-secrets baseline to silence legacy noise, a periodic TruffleHog verified sweep of history, platform-native scanning as backstop. Why: the numbers say the risk is growing, not shrinking — GitGuardian's 2026 report (vendor-published research; disclosed, broadly reproducible methodology) counts 28.65M new hardcoded secrets on public GitHub in 2025, +34% year over year, with AI-service-tied leaks up 81%; repos with Claude Code configuration present showed a 3.2% secret-leak incidence against a 1.5% GitHub-wide baseline (the pinned, specific figure — avoid their vaguer AI-wide multipliers). And detection without revocation is theater: 64% of secrets confirmed valid in 2022 were still unrevoked in 2026 — name a revocation owner, not just a scanner.

DONE = a planted dummy secret is caught at pre-commit AND in CI, verified once; the legacy baseline triaged with the owner; revocation owner named in report §5.

Write one ADR listing every deny rule, hook and gate installed in this phase, each tagged with the side of the litmus that put it there — and write down the week-one graduation answer explicitly: the scorers (tests, CI config) and the steering files are enforced from day one; ADR seams next; the long tail stays prose until it earns enforcement by recurring (I2 §7, E5 §1). 4Phase C — The write-permission matrix and the layers beyond the agent

Layer 1 — B1's rules and B2's hook — keeps the agent from thrashing the working tree; it binds the agent's own tools and nothing else. The durable gate becomes real only when it binds every writer: agent, human, and the human's Friday-afternoon shortcut (I2 §4). Phase C is the information-architecture half of the install: classify the module's artifacts by I1's taxonomy (durable / derived / disposable, plus agent memory), then push enforcement out to the platform and CI, where identity does not matter. The matrix itself is Report 3's synthesis, not an industry standard — individually well-attested components, packaged by us; carry that flag to the client.

Classify every artifact class the module actually has and assign each a write mode, a gate, an enforcement mechanism and an owner — the pre-filled week-one table below is the starting point, not the answer. Why: an unclassified artifact class is an ungoverned write surface. One data point argues for special care on the durable row: a model that learned to reward-hack on coding tasks generalized to intentionally sabotaging its operator's safety-detection code 12% of the time (Anthropic reward-hacking study, Nov 2025) — the audit and detection tooling itself belongs on the gated row, not beside it (I2, I1).

DONE = matrix committed; every artifact class the module has is classified; zero unclassified rows.

Add CODEOWNERS for the module's durable paths and a repository ruleset that requires PR + code-owner review and rejects pushes touching protected paths outright — GitHub's "restrict file paths" push rule (GA since September 2024) does this whoever pushes (vendor primary). Why: this is the layer that holds when someone bypasses the agent entirely. If the module has derived artifacts — generated docs, lockfiles, digests — add a CI step that reruns the generator and fails on any difference: <generate> && git diff --exit-code is the whole trick (Kubernetes' hack/verify-* fleet is the canonical example at scale). Why: the derived row's rule is "the master changes, or nothing does" — hand edits to generated files are drift the generator will silently overwrite later (I2 §4). This step is explicitly optional: if the module has no derived artifacts, record N/A in the matrix and move on — do not invent derived artifacts in order to gate them.

DONE = a hand-edited generated file goes red in CI, verified once — or a recorded N/A on the matrix's derived row.

Inventory every MCP server wired into the harness; remove what the module's work does not need; pin and gate what remains. Why, twice over. Context: MCP tool-schema sprawl — every connected server injecting dozens of verbose schemas — is the dominant real-world context bloat at clients (E2). Security: tool descriptions are attack surface — tool poisoning was first publicly demonstrated in April 2025 against a WhatsApp MCP server, malicious instructions living in the tool's description, not its output; CVE-2025-54136 ("MCPoison") and CVE-2025-54135 ("CurXecute") put the class on the map, CVE-2025-6514 (mcp-remote, CVSS 9.6, 437,000+ downloads affected) showed supply-chain scale, and CVE-2025-49596 (MCP Inspector) reached the dev tooling itself. The baseline (MCP spec + OWASP MCP cheat sheet): trusted publishers only; review tool definitions before install; pin definitions by hash and alert on drift (the rug-pull defence — MCP has no built-in integrity guarantee on tool metadata); OAuth 2.1 + PKCE, one token per server, minimal scope; fail closed. Claude Code's native controls (point-in-time): allowedMcpServers / allowManagedMcpServersOnly, with local servers matched by exact command + args so a binary cannot be silently swapped. OWASP's Agentic Top 10 2026 (Dec 2025) names the governing principle "Least Agency" — minimize not just what an agent can access but how much it can do without checking back.

DONE = server inventory committed: each surviving server has a publisher, a pinned version or hash, and a one-line reason; removals logged; the allowlist enforced in settings.

5Phase D — Autonomy, the probe, and the exit gate

The cage is built. Phase D decides how much freedom it earns, proves the decision under one real task, and writes the record that PB4 will inherit.

Score the module against E6's prerequisite table: L3 requires harness, verification and governance in place — PB2 supplied verification, Phases B–C supplied the rest; L4 needs memory and blast-radius control a pilot rarely has, so expect L2–L3. Record the level with evidence per prerequisite and set the demotion trigger in advance — promotion is slow, demotion is instant, and the accountable reviewer or lead pulls that trigger, never the agent operator (E6). Then run one small, real task at the granted level and attack your own install: ask for an edit of a protected path mid-task (B2's layer must block it) and seed a regression (B3's gate must catch it). The session ends merge-ready — E3's clean-state principle.

DONE = level recorded with per-prerequisite evidence and a pre-set demotion trigger; smoke task merge-ready; the probe blocked; the seeded regression caught — all in report §8.

Walk the exit-criteria table in §7 and fill the harness-install report (template in §6): what was installed, every probe and its result, the matrix summary, the autonomy level with its demotion trigger, go/no-go with signatories. Why: the report is the module's governance record — the artifact PB4's handoff contract will point at, and the baseline E9's tuning work later measures against. Every criterion is a probe result, which is what keeps this gate honest and cheap to re-run (B1 — never verify a step with a later step; the gate only reads evidence the steps already produced).

DONE = report delivered; go/no-go recorded with date, criteria and signatories.

The two layers this runbook does not give a step: sandbox and egress

B1's deny rules bind the agent's file tools, not arbitrary subprocesses — a script the agent runs can open any file the OS allows. The OS sandbox closes that hole, and the egress allowlist severs the exfiltration leg of the lethal trifecta — untrusted input, sensitive data, external comms; Meta's Rule of Two (hold at most two of the three per session) is the design heuristic, vendor guidance from 2025. Both ship as settings (table above) but are environment-specific enough that they are a configured follow-up with the client's platform team, not a two-day step here. Sequence them by exposure: headless and CI agents get the strictest profiles first — there is no human in that loop to catch anything (E3).

Three sentences to remember PB3 by

The harness decides more than the model — measured harness-induced variance exceeded model-induced variance by 7.8× on average in one 2026 position paper — and PB3 is where the client's harness stops being an accident. The litmus is the whole method: what the model should know is prose in CLAUDE.md; what must never happen is a permission rule, a hook or a CI gate — because models measurably talk their way past instructions (54% and worse in ImpossibleBench; the Replit code freeze) and cannot talk their way past an exit code. Nothing counts as installed until a deliberate probe has bounced off it — verified once, recorded in the report.

Sources and caveats for the empirical claims. Strong independent evidence: ImpossibleBench (arXiv:2510.20270 — the 54% cheat rate, the >79% test-editing mode, the abort tool's 54%→9%), the harness-variance measurement (arXiv:2605.23950 — 7.8× on average; a position paper's measurement, not a replicated benchmark), Anthropic's reward-hacking study (Nov 2025 — the 12% sabotage generalization), and the named CVE records (CVE-2025-54136, CVE-2025-54135, CVE-2025-6514, CVE-2025-49596). Vendor primary sources — strong for their own products' behavior, all point-in-time: the official Claude Code docs (every settings/hook syntax block here, verified 2026-07-18, including the layering correction that hook decisions do not override deny/ask rules) and GitHub's blog and changelog (restrict-file-paths GA, agent-PR attribution, Copilot's approval gates). Vendor-published research: GitGuardian 2026 (28.65M secrets, +34% YoY, +81% AI-tied, 64% unrevoked; the 3.2%-vs-1.5% figure is the pinned Claude-Code-specific one — their AI-wide multipliers are deliberately not used). Incident journalism: the Replit/SaaStr episode (The Register, Fortune, July 2025). Community and practitioner synthesis: OWASP's Agentic Top 10 2026 (Dec 2025 — Least Agency, Strong Observability; working-group output, not an empirical study), the OWASP MCP cheat sheet, Meta's Rule of Two (2025). Our synthesis, flagged as such: the write-permission matrix and its week-one sequencing (Report 3), the ~70% mutation bar (a working heuristic recorded in an ADR), and this runbook's own phase compression. Deliberately not cited: claims that Claude Code reads AGENTS.md natively (the docs state the opposite), MCP vulnerability-aggregator percentages, and AI-wide secret-leak multipliers.
