0Why it matters

D1 argued that the design system is the third handoff artifact; D2 made its most machine-checkable layer — tokens — into a literal contract. Both quietly assumed the agent can actually read the artifact. This chapter is about that assumption: the channel through which design context reaches the model's context window bounds the quality of what comes back, and the bound is usually tighter than the model's own capability. A frontier model fed a flattened screenshot will re-implement your button from pixels; a weaker model fed the button's name, import path and props will reuse it. That asymmetry is why "consumption" deserves its own chapter: it is where E2's context engineering meets the design layer, and it decides whether the token contract of D2 is enforced by the generator itself or left for the D5 gate to clean up after.

Four channels exist in mid-2026 practice, in ascending order of what they carry: the screenshot (pixels, nothing else), tool metadata over MCP (names, variables, generated code), the identity mapping (Code Connect: this Figma component is that code component), and the repository itself — component source, token JSON, DESIGN.md — which the agent reads like any other code. The chapter walks each channel with its evidence, then turns to two findings the swarm of hype tends to omit: structured context is not free (the context bill is real and measurable) and it is not automatically better (naively consumed metadata actively degrades output). Every tool fact is point-in-time, July 2026; this is the fastest-moving surface in module M6.

1The screenshot ceiling

The default channel is still the screenshot: paste a PNG, get markup. What pixels cannot carry is everything that makes production UI production UI — the names of things (is this Button or ActionButton?), token references instead of raw values (color.action.primary, not #2A4AD0), props and their types, component states and variants, and the boundary between what exists in the system and what must be built new. An agent working from pixels faces a perverse choice on every element: reuse a component it cannot see, or hallucinate one that looks right. It usually hallucinates — and the output looks correct while being structurally wrong: hard-coded values, duplicated components, no tokens. This is the failure mode the whole design-harness module exists to prevent, and no amount of visual fidelity in the input fixes it, because the missing information was never visual.

The ceiling has a number, and the number deserves precise citation because it circulates in mangled form. Design2Code (Si, Zhang et al., Stanford SALT with Georgia Tech, Microsoft and Google DeepMind; arXiv:2403.03163, published at NAACL 2025) curated 484 real webpages from the C4 validation set — filtered from 127.9k down to 14k algorithmically, then hand-curated, with all images replaced by placeholders — and asked human annotators whether AI-generated reconstructions from screenshots were "similar enough in appearance and content to be deployed interchangeably" with the original. For GPT-4V with self-revision prompting, 49% were judged interchangeable; in a separate blind pairwise comparison, annotators actually preferred the AI output to the original 64% of the time. Note the attribution: those headline numbers are GPT-4V's — GPT-4o only entered the paper in v2 and tops the model ranking, not the famous human-eval figures. The automatic metrics locate the weakness precisely: models lag most on visual-element recall (Block-Match — missing or hallucinated blocks) and layout position, far more than on text or color.

Three caveats keep the number honest, and they all point the same direction. First, the benchmark is static by construction: one self-contained HTML file, no JavaScript, no responsive behavior, no states — the authors themselves name dynamic pages as future work, and the follow-ups exist precisely to fill that (Interaction2Code, arXiv:2411.03292, 127 pages and 374 interactions in its final version, finds form/select interactions usable over 80% of the time but iframe/progress under 60%). Second, the input is a flattened screenshot of a stranger's page — the benchmark never tests the thing a design-system engagement cares about: reusing your components, honoring your tokens. Third, the 2024-era models it measured are two generations old, and the paper's harder split — Design2Code-HARD, 80 GitHub Pages cases added in the same paper's v2, where models fail to generate 30–40% of content — shows the ceiling drops fast as pages get real. Read 49% as "coin-flip interchangeability on simple static pages with nothing to reuse" — the floor from which structured channels improve, not a verdict on current models.

2The Figma MCP server — metadata as a first-class channel

The structural change of the last year is that design metadata stopped requiring screenshots-plus-API-glue and became a native agent channel. Figma's MCP server — launched as the Dev Mode MCP server beta on 4 June 2025, renamed, and generally available since Schema, 28 October 2025 — exposes the design file as tools rather than pixels. The read core: get_design_context (generated code for a selection; default output is a React + Tailwind "pseudo-representation", configurable by prompt), get_variable_defs (design variables and styles, including the code syntax authors attach to each variable — the D2 contract arriving through the tool), get_code_connect_map (§3), get_metadata (an XML tree of the selection's structure) and get_screenshot (explicitly the supplement, not the payload). Text, SVG, images, layer names and annotations all flow into the context — remember that sentence in §7, because everything that flows in is an injection surface. Tooling moved fast after GA: a remote server at mcp.figma.com (no desktop app required, the preferred path since September 2025), Make-file code resources, and — still beta, remote-only, usage-priced-to-come — write tools that let an agent modify the canvas, turning a read channel into a two-way trust boundary.

Access is a commercial decision, and it shapes engagements. Starter-plan and View/Collab seats are throttled to six tool calls per month — a demo, not a workflow; Dev or Full seats get Tier-1 REST-API rate limits. Code Connect (§3) is gated harder: Organization or Enterprise plan with a paid seat. And the client list is curated — only clients in Figma's MCP Catalog can connect to the remote server (Claude Code, Cursor, VS Code, Codex, Gemini CLI among them; a June 2026 regression in the remote server's tool schemas broke OpenAI-compatible validators outright — point-in-time, but a useful reminder that this plumbing is young).

The constraint nobody puts in the keynote is the context bill. Figma's own troubleshooting docs show a real error: a single get_design_context call on a large frame returned 351,378 tokens against Claude Code's default 25,000-token MCP output cap — fourteen times the limit. The documented failure modes are exactly what E2 predicts: agents told to "build this page" enter retry loops re-fetching a payload that can never fit (observed in the wild on the community Framelink server), and tool definitions plus server instructions tax the window before the first user message (a user-measured ~24k tokens in one Claude Code report). The official mitigations read like a context-engineering checklist: select small frames or individual components, never whole pages; on overflow, fetch get_metadata first and re-fetch only the nodes you need; keep the screenshot as supplement. The community server ecosystem (Framelink's Figma-Context-MCP, which predates the official one) exists largely to slim this payload — its June 2026 "payload diet" releases remove duplicated auto-generated layer titles. §5 shows what happens when you skip the diet and feed the metadata raw.

3Code Connect — the identity layer that kills hallucinated components

Metadata tells the agent what something looks like; it does not tell the agent what something is. That is the gap Code Connect fills: a published mapping from a Figma component to your actual code component. Once mappings exist, Dev Mode and the MCP server stop emitting autogenerated approximations and return production truth — per selected instance, get_code_connect_map hands the agent the componentName, the source (file path or URL in your repo), a props-resolved example snippet and its snippetImports. This kills the two worst consumption failure modes named in §1: the agent cannot hallucinate a component it has been handed by name, and it cannot re-implement a component whose import statement sits in its context. Figma's own framing stays at "boost output quality by reusing your actual components" — the anti-hallucination claim is ours and the community's, and while the mechanism is verifiable, any quantified benefit is vendor claim or worse (§8).

The mechanics changed in 2026 and older write-ups are stale. The figma.connect() parser API — one call per component, per framework — is now the legacy path; the recommended form is template files (.figma.ts), framework-agnostic TypeScript that declares exactly how the snippet should render, published with the @figma/code-connect CLI (v1.4.9, July 2026) from a figma.config.json in the repo root. Mapping source lives in your repo next to the components; the published mapping data lives server-side at Figma, keyed to node IDs. First-class integrations cover React (+ Native), HTML/Web Components, SwiftUI and Jetpack Compose; Angular and Vue ride the HTML integration with auto-detection — second-class in the MCP path, with documented cases of the server returning generic React/Tailwind despite published Angular mappings. A Code Connect UI inside Figma (GA at Schema 2025) adds one-to-many mappings — one design component to React, SwiftUI and Compose implementations at once — and Figma now ships an official agent skill that bootstraps template files from a Figma URL, so the agent can write its own identity layer.

Three limits decide whether Code Connect is viable on an engagement, and none is technical in the interesting sense. First, gating: Organization or Enterprise plan plus a paid Full/Dev seat — the strongest anti-hallucination tool in the stack is unavailable to a startup on Professional. Second, partial coverage degrades silently: nested instances render only if the nested component is also connected, and everything else — text, layers, unconnected instances — is omitted from the snippet, so a half-mapped system produces confident, incomplete output. Third, maintenance is real: snippets are treated as strings, not executed, so drift between the mapping and the component is invisible until an agent emits stale props; Figma's own docs implicitly concede the per-component authoring cost by recommending scripted generation for icon sets. Budget Code Connect as a maintained artifact (I1 durable, PR-gated) — the moment it rots, channel 3 is worse than channel 2, because the lies arrive with import paths attached.

4Storybook MCP — from consumption to a self-healing loop

Storybook's entry flips the direction of the channel. The Figma server answers "what should this look like?"; Storybook MCP — the official @storybook/addon-mcp, experimental since August 2025 and launched as the headline feature of Storybook 10.3 (April 2026), still in Preview at 10.5.2 — also answers "did I build it right?". Its seven tools form three toolsets: docs (a Component Manifest backs list-all-documentation and get-documentation, which returns the props/TypeScript contract plus the first three stories as usage snippets), dev (changed-stories lookup, story-authoring instructions, live story preview embedded in supporting clients) and test (run-story-tests — Vitest-powered component tests plus accessibility violations when addon-a11y is configured, results returned structured). The gating fact for non-React clients: the docs toolset depends on the Component Manifest, which is React-only in Preview; dev and test are less framework-bound, but the headline capability is a React capability.

The combination is what Storybook calls self-healing testing, and it is the design layer's cleanest instance of the E4 loop: the agent reads the manifest (consumption), generates the component, runs story tests (verification), gets structured failures — a missing htmlFor/id pair flagged by a11y rules, say — fixes its own code and re-runs to green, escalating only when judgment is needed. An independent walkthrough (LogRocket, July 2026) watched exactly this happen: generated login form, 3 of 5 story tests failing on accessibility, agent fixed the shared input component, 5 of 5 passed. Notice what the loop consumes: not pictures of the design system but its executable contract — props as types, usage as stories, correctness as tests. That is channel 4 (the repo) with a concierge.

The numbers attached to this need their honest label, because they circulate stripped of it. Storybook's launch blog carries a benchmark chart — "generating UI with the Reshaped component library, with and without Storybook MCP" — showing 12.8% improved code usage (a quality metric, i.e. correct reuse of design-system components — not an acceptance rate, the label it is usually retold with), 2.76× faster generation and 27% fewer tokens. All three are vendor claims: self-published, no methodology paper, no disclosed sample sizes or models, measuring only the docs toolset, and unreplicated by any third party as of July 2026. The direction is plausible and independently observed qualitatively; the decimals are marketing until reproduced. §8 lists exactly what may and may not be said with them.

5The metadata trap — more context is not better context

The naive reading of §1–§4 is a ladder you simply climb: screenshots bad, structured metadata good, give the agent everything. The best counter-evidence arrived in December 2025 with FigmaBench, the first benchmark pairing real Figma structured JSON with screenshots (1,234 curated samples; metrics for visual consistency, structural layout, textual/stylistic fidelity with a hallucination penalty, and responsive quality). Its headline finding is a fidelity–responsiveness paradox with a named cause — the metadata trap: models shortcut layout reasoning by transcribing absolute coordinates straight from the metadata, producing rigid, non-responsive code that scores well on visual fidelity and fails as engineering. More structured context made the output worse on the dimension that matters in production. The trap is E2 in miniature: a context window full of seductive, copyable numbers crowds out the reasoning the task actually required.

The mitigation is not less structure but curated structure — and the ecosystem has converged on what curation means. FigmaBench's own pipeline strips base64 assets to external files, prunes invisible layers and empty containers, and normalizes coordinates — because raw design JSON "causes context length explosion" (their words, and §2's 351k-token error is the industry version of the sentence). Framelink's entire existence is payload reduction. Figma's official choreography — get_metadata for the map, then scoped get_design_context per node, screenshot as supplement — is curation as a fetch policy. The generalizable rule for engagements: the agent should consume design context the way it consumes code — lazily, scoped, on demand, never as a bulk dump. Tokens and component contracts (channels 3–4) are inherently curated: small, named, and bounded. Raw canvas JSON is the opposite. Budget design context like any other context: every node you fetch is paid for twice, once in tokens and once in the rot risk of what it displaces (E2).

6The vibe-tool default — whose design system is it anyway?

The consumption question has a second front: the greenfield generators. Their default is a design system — just not yours. v0 states it verbatim in its docs: new chats start with a Next.js app using shadcn/ui and Tailwind; v0 "is specifically trained on the default implementations of the shadcn/ui components and may struggle with any customizations." Lovable scaffolds Vite/React/Tailwind; Bolt has no single fixed stack but in practice lands on Vite + React + Tailwind per prompt. Unconstrained, all three generate competent generic UI that ignores your tokens, your components and your conventions — the §1 failure mode wearing a nicer demo. The strategically interesting fact of 2025→2026 is that all three vendors shipped gated "teach it your system" features, which is the market conceding the chapter's thesis: consumption of the client's real design context is the difference between a toy and a tool.

V0

Design Systems 2.0

skill-based import

The 2025 mechanism — self-host a shadcn-compatible registry, point v0 at it — is now labeled legacy. The current path imports your system from sources (a DS repo, a consumer app, Figma frames, Storybook links, npm packages incl. private), builds a starter app you review, and saves it as a team- or personal-scoped skill backed by a v0.json; team owners on paid plans can pin a team default applied to new chats. The model still leans shadcn-shaped — closest fit when your system already is.

LOVABLE

design-system.json

the enterprise constraint

The closest thing to an enterprise contract in the category, with a correction to how it is usually described: design-system.json is not a file you hand-author at repo root — it is a generated schema (tokens, component catalog with variants/props/examples, stack constraints) produced on each release of a dedicated design-system project, read on every generation in connected projects, with adherence checks that scan each generation for raw color literals, one-off values and local re-implementations, and retry. Enterprise plans only, React only, one system per project.

BOLT

Design systems (Mar 2026)

Storybook-flavored

The newest entrant: paid Teams upload sources (repo, published Storybook, npm, docs site — GitHub/npm give the best results; docs alone yield "a theme rather than a true design system"), Bolt generates a browsable Storybook inside the tool as the reference, plus free-text agent instructions ("exclude deprecated components"). Attaching a system to an existing project deliberately does not rewrite code — you prompt the update yourself.

Two patterns matter beyond the vendor detail. First, the convergence: everyone is rebuilding the same architecture — a machine-readable package of tokens + components + constraints, injected at generation time, checked afterward. That is D1's third artifact wearing vendor packaging, and it validates the D1–D2 investment: a client with DTCG tokens in git and a documented component catalog can feed any of these tools; a client without one is locked to whatever the tool's default decided. Second, the gate: every one of these features sits behind a paid tier (Lovable's behind Enterprise). Budget lines, not prompts, now decide whether the generator respects your system — worth saying plainly in a proposal.

7Choosing channels — the ladder, the security surface, the pipeline

The channels compose, and the composition an engagement should default to looks like this: repo always (tokens + component source + DESIGN.md are the cheapest, most verifiable context and cost no plan tier), metadata scoped (Figma MCP per component, never per page), identity where it pays (Code Connect on the core component set if the plan allows), verification in the loop (Storybook MCP or equivalent tests so consumption errors are caught deterministically), and screenshots as supplement — visual reference, never the source of truth. The table ranks what each channel carries against what it costs; the security note follows, because every one of these channels is also an attack surface (S4).

The security reading is short and serious. Everything a design channel feeds the agent is untrusted content in the lethal-trifecta sense (S2): layer names, text and annotations enter the context unsanitized, and a malicious collaborator can hide payloads in invisible layers — 0% opacity text, off-canvas nodes, 1px fonts (filed against the Framelink server, March 2026; the official server feeds the same content classes and has published no sanitization story — its mitigation posture is access control, reviewing third-party clients before they may connect). The surface is not hypothetical: CVE-2025-53967, a command injection in the community figma-developer-mcp server, is explicitly exploitable via indirect prompt injection driving the MCP client — design content to remote code execution under the server process. Add the bridge-architecture attacks (local WebSocket bridges that let any local process impersonate the design tool — the pattern Penpot's official beta MCP and several community servers share) and the write-to-canvas beta (§2), and the posture writes itself: least privilege on tokens and scopes, read-only by default, human gate on writes, and treat canvas text like issue-tracker text. No confirmed real-world breach via a design-tool MCP is public as of July 2026 — demonstrated surface, not observed incident; word it exactly that way.

Practitioner consensus, where it exists, matches the ladder: metadata-first-screenshot-second (Figma's own guidance and OpenAI's curated Figma skill); component-library-first — map the system once, then let the channel resolve instances to real components, expecting 40–60% code reuse on complex screens, not 100% (the rest is genuinely new composition, and pretending otherwise sets clients up for disappointment); and tokens-first SSOT (D2) so every channel downstream reads the same named decisions. The open-source flank is worth one line: Penpot ships an official MCP server (beta, browser-tab-bound) — the only non-Figma design tool with a first-party channel, and the only one whose SSOT you can self-host.

8Evidence & do-not-cite

The design-conversation is drowning in vendor numbers, so the file needs explicit shelves. Usable with attribution: Design2Code's 49% interchangeable / 64% preferred (GPT-4V, self-revision, static placeholdered pages, 2024-era — say all of it); Design2Code-HARD's 30–40% missing content on hard pages; Interaction2Code's >80%/<60% split by interaction type; FigmaBench's metadata trap (a December 2025 preprint — peer review pending, directionally strong); Figma's own 351,378-token error as the context-bill illustration; CVE-2025-53967 for the injection surface. Vendor claims, label every time: Storybook's 12.8% / 2.76× / −27% (self-published chart, docs toolset only, unreplicated — and it is "improved code usage", never "acceptance rate"); Figma's token-efficiency and faster-build claims (internal evals, alpha customers); anything any vibe tool says about respecting your system. Do not cite at all: "structured context always beats screenshots" (FigmaBench falsified the naive form); "Code Connect prevents hallucination" as a measured claim (the mechanism is demonstrable, no quantified study exists); "Figma MCP is beta" (read is GA since Oct 2025 — only write-to-canvas remains beta); "DTCG is the wire format of design MCP traffic" (it is not — servers emit tool-specific JSON); and any sentence about these tools without a month attached — this chapter's tool facts have a shelf life measured in weeks.

9Applying it — decision guide

Assess (first client conversations)
  • Which channel does the agent actually use today — screenshots in a chat, or something structured? The answer predicts output quality better than the model choice.
  • Figma plan tier and seats: Code Connect is Org/Enterprise-gated, and anything below a Dev seat gets six MCP calls a month — this alone can veto the §7 pipeline.
  • Is there a Storybook (or equivalent) with real stories? Without it, the self-healing loop has nothing to run.
  • React or not? Storybook MCP's docs toolset is React-only in Preview; Code Connect's non-React path is second-class — a Vue/Angular client gets a thinner chapter.
  • Who can write to the design file? Hidden-layer injection needs only edit access on a shared file.
Commit (the defaults we bring)
  • Repo always, metadata scoped, identity where it pays, verification in the loop, screenshots as supplement — in that order of priority.
  • Curation over volume: no bulk canvas dumps; scoped fetches; small frames; the metadata trap cited whenever someone proposes "just give it the whole file".
  • Evidence honesty: Storybook's figures as labeled vendor claims, Design2Code with its caveats attached, no sentence about these tools without a date.
  • Security by default: read-only channels, least-privilege scopes, human gate on any write-to-canvas path (D7 carries the full treatment).
  • Vibe tools only under constraint: no unconstrained v0/Lovable/Bolt output into a client codebase, ever.
Three sentences to remember D3 by

The channel bounds the output: a screenshot carries pixels and invites hallucinated components, while structured channels — Figma MCP metadata, Code Connect's identity mapping, Storybook's executable contract, and the repository itself — carry the names, tokens and props that let an agent reuse instead of invent, with Code Connect as the single strongest anti-hallucination mechanism and the strictest paywall. Structured context is neither free nor automatically better: a single design-context call can bill fourteen times the default token cap, and FigmaBench's metadata trap shows raw structure degrading engineering quality — so consume design context lazily, scoped and curated, exactly like code. The verified numbers are modest and specific (49% interchangeability at the screenshot ceiling, a self-healing test loop that demonstrably works, one CVE on the community server); everything louder — the 2.76× speedups, the "prevents hallucination" slogans — is vendor claim until replicated, and the whole chapter is point-in-time for July 2026.
