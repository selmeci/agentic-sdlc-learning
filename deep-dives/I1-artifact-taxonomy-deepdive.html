0Why it matters

Module M4 asks how information survives contact with agents. This first chapter supplies the load-bearing distinction everything else in the module governs: what kind of artifact is this? Agents are document factories — every task can emit a plan, a progress log, a summary, generated docs, a scratch analysis — and a brownfield repo six months into agentic development contains more prose than some codebases contain code. Without a classification, all of it is just "documentation": nobody knows what may be edited, what must be regenerated, what can be deleted, and what the agent should trust. With a classification, every file has an owner, a change path and a lifetime — and I2's write-permissions have something concrete to attach to.

The baseline was already bad before agents arrived. Tan, Wagner & Treude measured documentation in the 1,000 most-starred GitHub projects: 28.9% contained at least one outdated code-element reference, stale for 4.7 years on average (EMSE 2023, arXiv:2212.01479) — with humans producing the text at human speed. Agentic development changes both multipliers at once: far more text gets produced, and — the genuinely new failure mode — the producer is also the consumer. An agent reads whatever is lying around in the repo and treats it as context; a stale plan or a drifted copy doesn't just embarrass the wiki, it steers the next generation. And volume itself has a cost even when nothing is wrong: Chroma's "context rot" study (Hong, Troynikov & Huber, July 2025) showed accuracy degrading non-uniformly across 18 frontier models as input grows, even on simple tasks. More surviving text is not more knowledge — past a point it is less.

1The triad — durable, derived, disposable

Flag: this taxonomy is our synthesis

Like the D1–D5 scale, the durable/derived/disposable triad is Report 3's original synthesis, not an industry standard — no external framework packages artifacts this way (checked 2026-07-15). It does have honest ancestors: Charity Majors' durable-vs-disposable split for code (by intended lifespan), records management's vital/important/useful classes (where "important = can be recreated if lost" is exactly our "derived"), and — strongest of all — the never-hand-edit discipline for generated files that the software industry has already encoded in tooling (§3). The three-bucket packaging is ours; present it as such.

The taxonomy classifies every artifact the development system produces or consumes by one question with three answers: if this file changed or vanished, what is the correct response? If the answer is "someone must decide, review and own the change" — it is durable. If the answer is "regenerate it from its master" — it is derived. If the answer is "nothing; it served its purpose" — it is disposable. Owner, change path and lifetime all follow from that one classification.

DURABLE

Sources of truth

owned · reviewed · human gate

The artifacts other artifacts are answerable to: the PRD, the spec (H1's contract), ADRs, the code, the tests, the steering files (CLAUDE.md / AGENTS.md), the design-token JSON (D4). A durable artifact can be wrong — which is precisely why it has a named owner and changes only through review. Losing one is a real loss; duplicating one creates a second truth that will drift (§5).

DERIVED

Regenerable projections

never hand-edit · regenerate

Anything produced from a master by a repeatable process: generated API docs, diagrams-from-code, compiled outputs, digests of product documents prepared for agent context, agent-written documentation of legacy code (§6). Derived artifacts are never edited in place — a change request against a derived artifact is really a change request against its master, followed by regeneration. Hand-editing one forks the truth: the edit either dies at the next regeneration or survives it and drifts.

DISPOSABLE

Working files

free to write · prune

The externalized thinking agentic work produces: plans, progress logs, session transcripts, scratch analyses, one-shot prototypes. Valuable during the run, dead weight after it. Disposables are the one category the agent writes freely — and the one category with a default fate of deletion. Anything in this pile worth keeping gets promoted deliberately (§6), never kept by inertia.

Three decision tests resolve most classification arguments in practice. The disappearance test: if the file vanished, would you restore it from memory and review (durable), rerun a generator (derived), or shrug (disposable)? The repair test: when it is wrong, is fixing it in place ever correct? Only for durables — a wrong derived artifact means a wrong master or a stale generation, and a wrong disposable means nothing. The trust test: does anything downstream — a human decision, an agent's context, a build — rely on this being current? If yes, it is either a durable or a derived artifact with a declared master; if it is neither, you have found an unowned copy, which is drift with a timestamp (I3).

2Durable — the human gate

Durable artifacts are the small set the whole system is answerable to, and their defining property is not importance but ownership: someone named can be wrong about them, and changes flow through review. The archetype predates agents by a decade — Michael Nygard's architecture decision record ("Documenting Architecture Decisions", Cognitect blog, November 2011): a short numbered document per significant decision, with an explicit lifecycle — proposed, accepted, and when a later decision changes course, deprecated or superseded. The crucial move is that an accepted ADR is never edited; change means a new ADR that supersedes the old one. The decision history stays readable and trustworthy — for the agent as much as for the human, because an agent that greps the ADR set sees not just what holds but what used to hold and why it changed. I2 takes the lifecycle and its enforcement from here.

Agentic development adds one genuinely new durable class: agent-facing configuration. CLAUDE.md, AGENTS.md and their siblings are small files with outsized blast radius — they steer every future generation, so an error in one compounds in a way an error in a README never did. The AGENTS.md convention illustrates how quickly this class hardened: it began as a multi-vendor coalition (OpenAI Codex, Sourcegraph Amp, Google Jules, Cursor, Factory), is now stewarded by the Agentic AI Foundation under the Linux Foundation, and self-reports adoption in over 60,000 repositories with 30+ tools reading it natively (as of mid-2026 — point-in-time, re-verify). A file read by every agent on every task is a source of truth by any definition; treat its edits like schema migrations, not like prose. E3's harness discipline and I5's gardening rules both assume this classification.

The spec-driven toolchain is converging on the same classification from the vendor side: durables belong in the repo, versioned next to the code they govern. Kiro stores each feature's spec as a folder under .kiro/specs/ — a requirements.md in EARS notation (H2's format), a design.md, a tasks.md — with the instruction to "store specs directly in your project repository alongside the code they describe"; GitHub's Spec Kit lays numbered feature folders under specs/ (spec.md, plan.md, tasks.md, contracts) beneath a constitution of "non-negotiable principles" at .specify/memory/constitution.md. Even a tool's own config directory encodes the taxonomy: Claude Code documents settings.json as committed and team-shared while settings.local.json is personal and git-ignored — one file family, deliberately split by change path and audience. Tool layouts are point-in-time (mid-2026); the direction — specs as versioned, repo-resident durables — is the stable part.

Durable ≠ large, durable ≠ old

A 40-line CLAUDE.md is durable; a 4,000-line generated API reference is not. The category is about the change path — whether edits require an owner's judgment — not about size, age or format. This is also why "everything is durable" is a real anti-pattern: declaring every document a source of truth relocates the review burden onto all of them at once, and the team responds by reviewing none (the same overload mechanism as H4's over-specification).

3Derived — never hand-edit, and the industry already agrees

The derived category comes with exactly one rule — never edit the projection; change the master and regenerate — and the strongest evidence for the rule is that the software industry converged on it long before agents. Go specifies a canonical machine-readable header for generated files, down to a regular expression in the toolchain docs: ^// Code generated .* DO NOT EDIT\.$ — and the ecosystem's generators (protoc-gen-go, stringer) emit it. GitHub's linguist lets a repo mark files linguist-generated in .gitattributes, with two effects that encode the whole philosophy: the files are excluded from language statistics and hidden by default in pull-request diffs. Read that second effect carefully — the platform deliberately steers review attention away from derived artifacts, because the reviewable object is the master and the generator, not their output.

That reviewing insight transfers directly to agentic artifacts. A digest of the PRD prepared for agent context, a diagram generated from the architecture, API docs generated from code — none of these should consume line-by-line review, and none should ever accept a hand fix, however urgent. The failure mode is always the same: the hand edit either dies silently at the next regeneration (the fix evaporates) or the regeneration stops being run so the edit survives (the artifact quietly becomes an unowned durable — the worst object in the taxonomy, trusted and maintained by nobody). If a derived artifact keeps needing manual corrections, that is not an editing problem; it is a signal that either its master is wrong or its generator is — fix those, or reclassify the artifact as durable and give it an owner honestly.

The lockfile nuance — versioned is not the axis

npm's docs say package-lock.json "is intended to be committed into source repositories" so that teammates, deployments and CI "install exactly the same dependencies" — yarn and pnpm prescribe the same. A fully regenerable artifact, deliberately versioned: derived-but-committed. The lesson is that git-tracked vs not is orthogonal to the taxonomy. The axis is the change path: nobody hand-edits a lockfile; it changes only as a by-product of changing the master (package.json) and rerunning the tool. Committing derived artifacts is fine — often necessary for reproducibility — as long as the no-hand-edit rule survives the commit.

The category carries one more obligation: a derived artifact must have a consumer, because regeneration is a standing cost paid on every master change. The cautionary exhibit is llms.txt — a derived digest of a website prepared for machine readers, adopted on faith. Ahrefs measured 137,210 domains carrying one: 97% of the files received zero traffic in May 2026, and of the requests that did arrive, just 1.1% came from AI retrieval bots — for AI visibility, "an llms.txt file is largely decoration" (Linehan & Guan, Ahrefs, June 2026; I7 reaches the same verdict from the integration side). A projection nobody loads is not information architecture; it is maintenance debt with extra steps. The classification sweep (§7) therefore records two names per derived artifact: its master and its consumer.

4Disposable — the category agents made necessary

Before agents, scratch thinking lived in editors, terminals and heads, and vanished on its own. Agents externalize it: the plan, the progress log, the running commentary all become files, because files are how an agent thinks past its context window. Anthropic's long-running-harness work shows the pattern at its most deliberate: an initializer agent writes an init.sh, makes an initial commit, and expands the prompt into a structured JSON feature list whose entries carry category / description / steps / passes: false — with the agent allowed to flip passes as features verify; alongside it, a claude-progress.txt is read at session start and updated at session end. Even the format choice is a taxonomy decision: JSON over Markdown because, in Anthropic's words, "the model is less likely to inappropriately change or overwrite JSON files compared to Markdown files" — structure as a soft write-permission (E3 covers the harness itself; I2 makes the permission hard).

What the harness post does not supply is an afterlife policy — its bar is a clean state in git terms (committed, mergeable), and the explicit pruning guidance lives in Anthropic's memory-tool docs instead: rename or delete files that are no longer relevant, and "do not create new files unless necessary." The gap matters, because the default fate of working files in the wild is accumulation: PLAN-v2-final.md next to PLAN-v3-actually-final.md in the repo root, session summaries nobody re-read, TODO files three sprints stale. The sprawl is measured behavior, not caricature — "the default behavior for models as of writing this in December 2025 is to create capitalized Markdown files at the repository root" (Onur Solmaz, whose SimpleDoc convention is the neatest practitioner fix: agent-written docs land in docs/ with a YYYY-MM-DD- prefix, lowercase names and frontmatter naming the author, while only timeless files — README, AGENTS.md — keep root-level capitals, so a shouting filename becomes a visible claim to durable status). This is not an aesthetic problem. Agents read what is lying around; every stale plan is a candidate context input, and the context-rot result (§0) says volume alone degrades performance — before you account for the stale plan being wrong. Pruning disposables is context hygiene, not tidiness — and not a new discipline either: records management has ended documents' lives on schedule for decades (retention schedules, disposition); the taxonomy imports the practice without the paperwork.

A disposable policy that works
  • a named home — a scratch/progress directory, gitignored or clearly conventioned, never the repo root
  • a TTL — end of task, end of sprint; expiry is the default, keeping is the decision
  • machine-readable working state (JSON) over prose where the agent must update it
  • promotion is explicit: worth keeping → validated, relocated, owned (§6)
Smells that the policy is missing
  • plans and summaries accumulating in the repo root with version suffixes
  • agents citing last month's plan as if it were the spec
  • nobody can say which progress file is current without asking
  • "we keep everything, storage is cheap" — storage is; context isn't

5The SSOT boundary — two mastering systems, one taxonomy

Single source of truth is an information-systems term of art: every element mastered — edited — in exactly one place. The taxonomy is how SSOT becomes operational per artifact, but a real organization complicates it with a structural fact: product durables and engineering durables live in different mastering systems. The PRD, the roadmap and the tickets are mastered in Notion, Confluence or Jira; the spec, the code, the tests and the steering files are mastered in git. Neither side is wrong — but the boundary between them is where duplication pressure concentrates, because the agents work in the repo while half the truth they need lives outside it.

Hence the canonical M4 anti-pattern: the PM pastes the current PRD into CLAUDE.md so the agent "always has it." Every instinct behind the move is sound — the agent does need product context — and the result is still a fork: the pasted copy is a durable artifact's projection with no regeneration path, it will drift the day the PRD next changes, and the agent will keep trusting it precisely because it sits in the most-trusted file in the repo. The taxonomy names the two correct bridges. Either a pointer — the agent fetches the PRD at need across the boundary (I7's MCP integration), paying freshness's token cost — or a labelled derived digest: a summarized excerpt carrying its source and generation date, regenerated from the master on change, never hand-maintained. What separates a digest from a pasted copy is exactly the derived discipline of §3: a declared master, a repeatable projection, and no hand edits.

The boundary also explains why "just put everything in git" and "just put everything in Notion" both fail: each side's durables need their native review gate (PRs for engineering artifacts, the product tool's ownership model for product artifacts), and each side's practitioners live where their masters are. The realistic posture for a brownfield client is two mastering systems, an explicit list of which durables live where (§7), and derived digests or pointers — never copies — crossing the line. I7 supplies the mechanics: ID schemes across the boundary, MCP as the integration layer, and the reference-don't-copy rule with its context-rot rationale.

6Grey zones — memory, agent-written docs, and the promotion path

Three artifact families genuinely strain the triad, and all three resolve through the same move: an explicit promotion decision instead of an accidental one.

Agent-generated documentation of legacy code — M5's archaeology output — is born derived: produced from the code by a repeatable process, regenerable at will. It stops being derived the moment a human validates it and the team starts relying on it, because reliance is what creates the trust test of §1. The honest options are two: keep it derived — regenerated on code change, never hand-corrected, trusted only as far as its generator — or promote it to durable through explicit validation, an assigned owner and the review gate. The state to refuse is the middle one: half-trusted, unowned, hand-patched — the workbook's open question ("where does agent-generated legacy documentation belong?") answered by deciding, per document, rather than letting the pile decide.

Agent memory files are the sharpest case, because they are durable in effect — they persist across sessions and steer future behavior — while being agent-written by design, with no human gate in the loop. Anthropic's memory tool makes the shape concrete: purely file-based storage under /memories, six operations (view, create, str_replace, insert, delete, rename), generally available on the Messages API with the application controlling the actual storage backend. A file the agent writes today and trusts tomorrow inverts the durable discipline — which is exactly the surface OWASP's memory-poisoning entry describes (I4): a poisoned memory is persistent, temporally decoupled from its cause, and invisible to a per-session review. The taxonomy answer is that memory is a governed fourth write-mode, not an exception: agent-writable like a disposable, persistent like a durable — so it gets what neither category needs alone: provenance on every entry, validation before write where feasible, audit and rollback (I2's machinery, I4's threat model).

Structured memory conventions show teams already reinventing the triad under other names. Cline's Memory Bank — official docs, though methodology rather than enforced tooling — prescribes six markdown files with distinct roles: projectbrief.md and systemPatterns.md are durable in all but name (foundation documents everything else builds on), while activeContext.md and progress.md are working state — the pattern works precisely because the file names encode a lifecycle. The taxonomy's contribution is to make that encoding explicit and reviewable rather than implicit in a convention (advisory today; I2 makes it deterministic).

Promotion and demotion, stated as rules

Promotion (disposable/derived → durable): validate the content, give it an owner, move it to the master's home, put its future changes behind the gate. A plan that turned out to be the design doc earns this — by decision, with a commit that says so. Demotion (durable → derived/disposable): a source of truth nobody owns is not a source of truth; either assign the owner or reclassify honestly and delete or regenerate. Every artifact should survive the question "who owns this, and what happens when it's wrong?" — the ones that can't are the drift inventory (I3).

7The brownfield sweep — classification as a first-week move

At a client, the taxonomy converts into a concrete, low-risk first-week deliverable: the classification sweep. Inventory what exists — the repo tree and the wiki/product tool, an afternoon with an agent doing the listing — and put every artifact class into a five-column table: artifact, category, master, owner, change path. The sweep's value is less the table than the arguments it forces: every "we're not sure who owns this" is a drift site found before it bites, and every "this is generated but we've been patching it by hand" is a fork already in progress.

The adoptable rule set that falls out fits on one page: derived files get the generated-file header and linguist-generated attribute so tooling steers review correctly; disposables get one named directory and a TTL; product durables stay in their tool, with a pointer or a labelled digest — never a paste — crossing the boundary; steering and memory files are reviewed like code because they steer code. All of it is advisory prose until I2 — write-permissions per category, enforced with hooks rather than trust — makes it deterministic, which is the same advisory→deterministic move the framework makes everywhere (E5).

9Takeaways for our engagement

Three sentences to keep

Classify every artifact by its change path — owned and human-gated (durable), regenerated from a master and never hand-edited (derived), or freely written and pruned on a TTL (disposable) — and the module's remaining machinery has something to govern; the triad is our Report-3 synthesis, but its derived discipline is exactly what the industry already encodes in DO-NOT-EDIT headers, linguist-generated attributes and committed lockfiles. The expensive failures are all classification failures: the hand-edited generated file that forks the truth, the pasted PRD that drifts inside the repo's most-trusted file, the unowned "source of truth", and the plan graveyard that feeds stale context to agents that read whatever is lying around. Movement between categories is a decision, not an accident — promote by validating, owning and gating; demote by admitting nobody owns it — and the first-week classification sweep at a brownfield client surfaces every artifact that would fail the question "who owns this, and what happens when it's wrong?"
