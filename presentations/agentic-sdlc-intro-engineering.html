01Sudo Labs · the software-engineering view

Introducing agentic SDLC.
Verification first, autonomy second.

What exactly we are going to do, why in this order, and what you will hold in your hands at the end of the first phase.

02Why not "just turn the agents on"

An AI agent doesn't improve the system it lands in — it amplifies it. The good and the bad.

Three numbers from three different classes of evidence. We state them plainly, including how much weight each one carries.

DORA 2024 · correlational
−7.2 %

delivery stability, associated with a 25 % rise in AI adoption. The 2025 update: throughput now positive, instability persists — "faster, but not safer".

METR · controlled experiment
−19 %

experienced developers were slower with AI — while believing they were ~20 % faster. Perceived speed is not a measuring instrument.

Faros AI · vendor telemetry
+98 %

merged PRs and +91 % review time — with no org-level improvement. The work moved into the review queue.

Our takeaway: before agents touch your environment, a net must exist that catches them doing something wrong. First we measure that net — then we build.

METR's 2026 follow-up weakens the slowdown for newer developers and lowers its own confidence rating — the durable finding is the perception gap, not the exact magnitude. Point-in-time figures: we re-verify before we rely on them.

03Where we're heading — autonomy levels

The target is robust L3. Not L5.

Autonomy is not a measure of model capability — it is a decision about how much judgment we delegate. Higher is not automatically better.

The L1–L5 scale is our adaptation of published industry autonomy ladders (Swarmia, Tessl and similar) — a working model rather than a formal standard, shown so you can challenge it.

04Shared vocabulary

Agent = model + harness. Seven components, one loop.

The model is increasingly a commodity you swap out. The harness — the scaffolding around it — is the durable asset we build and own. You already work with most of these components; we are wiring them into a deliberate whole.

The harness we set up lives in your repositories, under your ownership — configuration, hooks and tests that are model-agnostic and portable. It is not a black box you rent from us.

05Why verification

The one component that supplies external truth.

Tests, type-checks and CI are a signal the model cannot invent or argue away. Without it, an agent's confidence and the code's correctness are uncorrelated — and delegation is a gamble.

Verification stateSafe ceilingWhy
No / slow / flaky testsL1–L2 (assist only)no trustworthy inner loop; humans must verify everything
Fast, trustworthy test suiteL3 (delegated task → PR)the agent self-corrects against real truth; humans review intent
+ mutation-verified & protected suiterobust L3, selective L4mutation testing (deliberately injected faults) proves the tests have teeth; protection stops quiet weakening

The autonomy ceiling is set by the quality of the verification signal, not the quality of the model. That is why, before extending any harness, we first check whether the signals in today's setup tell the truth at all.

06The other half of the net

Verification tells the truth. Governance sets the boundary.

Verification answers "is the work correct?". Governance answers "was the action allowed at all?" — and the framework's defining distinction is where a rule lives.

The litmus test for every rule: "what must never happen" goes into a hook or a permission — enforced by the harness. "What the model should know" goes into the prompt — advisory.

These are hard boundaries, not shades of maturity — and this phase already checks them: prod credentials unreachable from dev · full-history secrets scan before any large-scale reading of the repos · a verified live credential = STOP, not a ticket. (Replit, July 2025: an agent deleted a production database during a code freeze.)

07What we are probing right now

Can an agent tell, from your existing setup, that it just broke something?

Three probes into the existing environment (CI, tests, delivery data). We measure by experiment, not by questionnaire — CI configurations lie.

Probe 1 · test suite

Baseline suite — 3 runs

The full suite three times from a clean checkout, per repo: runtime, pass rate, flaky tests, one-command invocation. Three runs expose flakiness a single run hides.

Probe 2 · CI signal

A deliberately broken test

We break one test, run CI and watch: non-zero exit code? machine-parsable log? does red block the merge? Then we revert everything.

Probe 3 · baseline

Delivery "before" snapshot

The four DORA delivery metrics (deploy frequency, lead time, change-failure rate, recovery time) plus rework rate, computed retroactively from git/CI/incident data (8–12 weeks). Without a frozen baseline, a pilot's impact can never be proven — or disproven.

The principle of the whole phase: nothing gets written until a net exists to catch it. An agent that cannot read its own failure cannot correct itself.

08Why this is possible now

The economics of tests have flipped.

The reason verification bases have historical gaps was never laziness — it was cost. That cost is exactly what changed.

Until now

Tests = expensive manual work

Writing and maintaining tests competed with feature work for time — and lost. Coverage gaps and flaky tests were a rational consequence of the economics, not a team failure.

→
With agents

Tests = nearly free, under supervision

Agents generate and maintain tests at a fraction of the cost — verification shifts left and tests become the specification the agent converges on. The first thing agents build in your codebase is the net for themselves.

An honest caveat — "under supervision" is not a figure of speech: agents also game tests (weaken an assert, delete a case, hard-code the expected output). So we protect the tests in three layers: a rule → a deterministic hook that blocks test edits → a gate that verifies the suite still has teeth (mutation testing — inject small faults and prove the tests catch them).

09What "supervision" means in practice

Who writes the red.

The division of labour behind "nearly free, under supervision": a human — or a separate agent — writes the failing test, and the coding agent makes it pass without the right to touch it.

10How the level moves over time

Autonomy is a dial, not a ratchet.

The review numbers from slide 02 — +98 % PRs, +91 % review time — are what happens when agent output outruns review capacity. So the level moves in both directions, deliberately asymmetrically.

The rule that caps everything: never scale the number of agents beyond code-review capacity. More agents past that point do not raise throughput — they deepen the review queue. Review capacity, not model capability, is the hard cap on the effective level. It also answers your developers' first question: agents augment them and are capped by their review capacity by design — nobody is automated out of the loop.

11Phase-one plan — assess before build

Trunk → F0 gate → the branch the data picks.

The first steps always run and are read-only. What follows is chosen neither by us nor by you — it is chosen by the measured data, through an F0 gate — the first go/no-go decision point of the phased rollout — signed by your sponsor. We assess one target system — the group of repositories that ship together — not the whole portfolio.

12Outcomes · summary

What you will hold at the end of the phase.

  • Assessment report — the state of verification, governance and documentation, with a recommended autonomy level
  • Frozen delivery baseline — the "before" snapshot the future pilot will be measured against
  • Signed F0 decision — go / no-go / go with conditions, plus STOP criteria with named owners
  • The selected branch's output — remediation plan / system map + pilot shortlist / a running CI loop
  • An agreed next step — what runs in the following phase

In business terms: a delivery baseline that finally makes improvement provable, a shortlisted pilot with an expected-value case, and a signed decision — with exit criteria — that you can hold us to.

Three sentences to remember: The target is robust L3 — a delegated task ending in a pull request a human reviews. The autonomy ceiling is set by the quality of the verification signal, not the model — so we first measure whether your signals can be trusted. And nothing gets written until a net exists to catch it.
