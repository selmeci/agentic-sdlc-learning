02Why this exists — and why now

Everyone is switching agents on. Almost nobody is doing it safely.

An AI agent doesn't improve the system it lands in — it amplifies it, the good and the bad. Three numbers, from three different classes of evidence, stated with how much weight each carries.

DORA 2024 · correlational
−7.2%

delivery stability, associated with a 25% rise in AI adoption. The 2025 update: throughput turns positive, instability persists — "faster, but not safer." AI is an amplifier.

METR · controlled RCT
−19%

experienced developers were slower with AI — while believing they were ~20% faster. Perceived speed is not a measuring instrument.

Faros AI · vendor telemetry
+98%

merged PRs, +91% review time — with no org-level delivery improvement. The work didn't disappear; it moved into the review queue.

The gap in the market is the opening for us. A firm that can make agentic delivery faster and safer — with a method it can defend in front of a client — has something the tool vendors don't sell. That method is what this programme is.

Point-in-time figures (mid-2026), re-verified before any client meeting. METR's 2026 follow-up (arXiv:2507.09089 → ongoing) weakens the slowdown for newer developers and lowers its own confidence to "very weak"; the durable finding is the perception gap, not the exact magnitude.

04The one idea everything hangs on

Agent = Model + Harness. You rent the model. You own the harness.

The model is a commodity you swap out as prices fall and benchmarks move. The harness — the scaffold that gives it state, tools, a verification loop and enforceable limits — is the durable asset we design, own, and are paid to build. This is what turns agentic work into a service, not a licence resale.

Framing after Birgitta Böckeler (ThoughtWorks) and Simon Willison ("an agent runs tools in a loop to achieve a goal"). Memory & orchestration are cross-cutting and conditional — added when the task needs them, not a default eighth component.

05The spine — three ideas in every module

Three principles that make the difference between a demo and production.

01 · where a rule lives

Advisory vs deterministic

Text the model interprets (prompts, CLAUDE.md) is advisory — it shapes behaviour but can be ignored under load or prompt injection. Rules the harness enforces (hooks, permissions, sandbox) are deterministic. Only the deterministic layer holds under pressure.

02 · what caps delegation

Verification sets the ceiling

The strength and speed of the verification loop — not the model — decides how much autonomy is safe. No trustworthy tests means no safe delegation. So the first thing agents build in a codebase is the net that catches them.

03 · how much to delegate

Autonomy is a staffing decision

Not a capability score. Higher is not better; the safe level is set by the weakest enabling component and is always capped by one rule: never scale agents beyond review capacity. Nobody is automated out of the loop.

These three run through all nine modules — the same vocabulary applied to engineering, product, design and security. That shared language is half the value: a Sudo team that agrees on advisory vs deterministic argues about the right things.

06An original lens — autonomy across three disciplines

How much to delegate, made explicit — for engineers, PMs and designers alike.

Three parallel five-point scales. The safe production ceiling today is the middle rung — a task delegated, then reviewed by a human — in every discipline. Higher rungs are narrow or anti-patterns, not aspirations.

09The differentiator most decks skip

Security isn't a module at the end. It re-reads every other one.

The failure mode that matters is not the model being wrong — it's the model being hijacked. The programme carries this from first principles to a per-component threat map with real 2025–26 incidents and CVEs.

real · zero-click

EchoLeak — CVE-2025-32711 (CVSS 9.3): a single crafted email exfiltrates from M365 Copilot's context. No user action.

real · no attacker

Replit, July 2025: an agent deleted a production database during a code freeze, then falsely claimed rollback was impossible.

real · supply chain

postmark-mcp, Sept 2025: a trusted MCP package silently BCC'd every email after 15 clean releases — the first in-the-wild malicious MCP server.

Eight security topics (S1–S8) span prompt injection & blast-radius, the trifecta & Rule of Two, the advisory-vs-deterministic control split, the MCP/tool attack surface, auto-run & sandbox escapes, and memory poisoning across agents. PoCs are labelled as PoCs; in-the-wild incidents as incidents.

11The other half of the field

It covers greenfield too, not just legacy.

The playbook you just saw is the brownfield road. Module M10 maps the same theory onto a blank repo — so we can lead an engagement whether the client hands us a legacy estate or an empty one.

The inversion, in one line: brownfield pays a "bootstrap tax" to retrofit tests and extract conventions from code it didn't write; greenfield skips that tax by authoring both before feature #1 — and inherits the mirror risk. Authorship risk replaces archaeology risk: with no existing code as gravity, the agent overbuilds and drifts instead of regressing.
