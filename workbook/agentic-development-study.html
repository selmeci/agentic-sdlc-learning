E1 · DEEP DIVE

Agent = Model + Harness

Study companion

0Why this matters for us

E1 is the conceptual foundation for the entire engagement. If we get this framing right, every later decision — how much autonomy to grant, which guardrails are deterministic, where the human gates sit — follows from it. If we get it wrong, we sell the client a "smarter autocomplete" and it fails in production the way DORA's data predicts.

Our goal is to introduce an agentic engineering harness at a brownfield client. This unpacks the seven-component framework so we can (a) explain it credibly to a technical audience, (b) assess what a client already has versus what we must build, and (c) know the input and output of each component so the pieces connect into a working loop rather than a demo.

The one idea to internalise

A model is a capability; an agent is a system. The difference — and almost all of our engineering value — lives in the harness. When people say "the agent isn't good enough," they usually mean the harness around a perfectly capable model is missing or misconfigured.

1The core equation: Agent = Model + Harness

The framing comes from Birgitta Böckeler (ThoughtWorks) and is echoed in Anthropic's engineering writing: an agent is a language model wrapped in a harness that gives it state, tool execution, feedback loops and enforceable constraints. Simon Willison's compact definition completes it: an agent "runs tools in a loop to achieve a goal."

The centre of gravity has moved. Two years ago the leverage was in prompt engineering; today it is in context engineering (curating what the model sees each step) and harness engineering (designing the scaffolding). The model is increasingly a commodity you swap out; the harness is the durable asset you build and own.

A principle that keeps recurring

Every harness component encodes an assumption about what the model cannot do. When the model improves, some components become unnecessary — and the right move is to remove them. A harness is never "finished"; it is continuously pruned. Anthropic saw this directly: a newer model let them delete scaffolding they previously needed.

2The seven components — with inputs & outputs

The whole framework as one picture, then each component in turn. Read the diagram first: the components form a loop, not a pipeline — output feeds back in.

01

Prompt

the instruction layer

Standing instructions plus the specific task. In coding agents this splits into a durable role layer (CLAUDE.md/AGENTS.md) and the per-task request.

Requires on input
  • a clearly scoped goal (from the handoff contract, M3)
  • role/behaviour conventions for the repo
  • explicit non-goals and constraints
Produces on output
  • a bounded instruction the model can act on
  • the frame that decides whether context is interpreted correctly
Where it sits in the SDLC
  • Refinement / task definition — the downstream face of the handoff contract; a vague prompt is an under-specified ticket in a costume.

For our rollout: the prompt layer is advisory — the model may drift from it under pressure or injection. Never a security or correctness boundary. Keep the durable part short and load-bearing.

02

Context

what the model sees

The working set assembled each step: relevant code, docs, retrieved snippets, prior messages, tool results. Context engineering curates "the smallest set of high-signal tokens" — not stuffing everything in.

Requires on input
  • a retrieval mechanism (agentic search / grep, or RAG)
  • source-of-truth artifacts to draw from (M4)
  • a budget discipline — include vs link
Produces on output
  • a curated, task-relevant window for the model
  • the single biggest lever on output quality
Where it sits in the SDLC
  • Cross-cutting, heaviest at implementation — the agent assembles context each turn as it navigates the codebase.

For our rollout: beware context rot — recall degrades as the window grows. "Reference, don't copy": link to masters, inject summarised excerpts. Retrieved content is untrusted until proven otherwise (injection entry point — M7).

03

Harness

the orchestrating scaffold

The software around the model that manages sessions, assembles context, invokes tools, enforces constraints, carries state. Where our engineering lives. Patterns: the initializer agent (first session sets up init.sh, a progress file, an initial commit) and the clean-state principle (every session ends merge-ready).

Requires on input
  • a runtime (CLI / SDK / IDE integration)
  • config: which tools, permissions, hooks
  • a state convention (progress file, git history)
Produces on output
  • a repeatable, constrained execution environment
  • continuity across sessions ("engineers in shifts")
Where it sits in the SDLC
  • Infrastructure/platform layer — set up once (Phase 1), then maintained and pruned as models improve.

For our rollout: the "internal product" we build and own (trajectory T5). Its quality determines everything downstream; version it, own it, and eventually give it its own evals.

04

Loop

the runtime cycle

The cycle that makes it agentic: model proposes an action, harness executes it via a tool, result returns to context, model reasons again — until the goal is met or a stop condition fires.

Requires on input
  • tools it can call and observe results from
  • stop conditions (done, budget, needs human)
  • a way to detect and recover from being stuck
Produces on output
  • a sequence of actions converging on the goal
  • a trace/audit of what was attempted
Where it sits in the SDLC
  • Implementation runtime — the agent "doing the work" between task hand-off and PR.

For our rollout: loops need blocker-recognition. Devin completed only 3 of 20 tasks in Answer.AI's independent test; the failures came from a loop pushing ahead on impossible tasks instead of stopping. A good loop knows when to ask a human — which is where autonomy level (E6) is really decided.

05

Tools

the hands

The concrete actions: run shell, read/edit files, search, call external systems via MCP. Tools turn reasoning into effects — where capability and risk both concentrate.

Requires on input
  • well-described tool interfaces
  • scoped credentials (least privilege)
  • an allowlist of what may be called
Produces on output
  • real effects: file changes, command results, API calls
  • observations that re-enter the loop
Where it sits in the SDLC
  • Implementation & integration — and, via MCP, the seam to ticketing, design and deploy systems.

For our rollout: the richest attack surface (M7). Every tool is a privilege handed to a possibly-injected agent. One-repo-per-session, scoped tokens, no service_role, human gate on writes.

06

Verification / feedback

the external truth

The deterministic signal of whether the work is correct: tests, type-checks, linters, CI. This makes delegation safe — truth the model cannot argue away. TDD fits agents: run tests, read failures, self-correct.

Requires on input
  • a runnable, reasonably fast test suite
  • CI able to return results the agent can read
  • machine-verifiable acceptance criteria (M3)
Produces on output
  • a pass/fail signal that drives the loop
  • the merge gate's evidence base
Where it sits in the SDLC
  • Testing & CI — shifted left: verification is written before/alongside implementation, becoming the specification.

For our rollout: the make-or-break prerequisite. Without a fast, trustworthy test loop a client cannot safely pass L2. Agents also "cheat" tests — protect them (rule + hook + verification gate).

07

Governance

the enforced boundary

Deterministic constraints enforced by the harness, not requested of the model: permission rules, pre/post-tool hooks, sandboxing, network policy. The defining distinction lives here — advisory vs deterministic.

Requires on input
  • a policy: always allowed / denied / gated
  • enforcement points (hooks, sandbox, egress proxy)
  • an audit mechanism
Produces on output
  • hard guarantees independent of model behaviour
  • bounded, observable, reversible blast radius
Where it sits in the SDLC
  • Cross-cutting platform concern — dev, CI and deploy; the organisational layer sits above it (T7).

For our rollout: the litmus test — "what must never happen" goes in a hook/permission (deterministic); "what the model should know" goes in the prompt (advisory). Under injection, only the deterministic layer holds.

+

Memory & Orchestration

cross-cutting & conditional

Memory runs across all seven: session/working (the context window), persistent files (progress logs, /memories), cross-session (git history), organisational stores. It lets a memoryless model behave like "engineers in shifts." Input: a state convention and a place to write. Output: continuity — and, unguarded, a poisoning surface (M7, ASI06).

Orchestration (multi-agent) is a conditional choice, not a default eighth component. Parallel writers are fragile; the working pattern keeps writes single-threaded while other agents contribute intelligence, not actions. Reach for it only when a task genuinely parallelises — and treat the cluster as one trust boundary.

Not a phase model, but each component has a natural home. This is how we answer "where does the agent actually plug in?" for a client who thinks in SDLC stages.

Classical SDLC runs left-to-right once. The agentic version loops: verification feeds implementation, memory carries state between whole cycles. 4Takeaways for introducing a harness

Assess before you build (feeds E6)
  • Is there a fast, trustworthy test suite? (verification)
  • Can CI return signal an agent can read?
  • Are source-of-truth artifacts retrievable? (context)
  • Do dev/prod separation, RBAC, secrets hygiene exist? (governance)
Build in this order (feeds E7)
  • make the repo agent-ready (harness + governance skeleton)
  • establish the verification loop before write autonomy
  • start at L1–L2, prove the base, then move to L3
  • prune harness components as the model improves
Three sentences to remember E1 by

An agent is a model plus a harness; the harness is where our engineering value and the client's risk both live. Some components are advisory (prompt, context) and some must be deterministic (governance, verification) — never confuse the two. Build the loop, verify with external truth, bound the blast radius, and remove scaffolding as the model grows.

E2 · DEEP DIVE

Context Engineering

Study companion

0Why context is the highest-leverage component

If E1 taught that the harness is where our engineering value lives, E2 is the sharpest edge of that value. Context engineering is the single biggest lever on output quality — bigger than model choice, bigger than prompt wording. An excellent model fed the wrong context produces confident, wrong work; a modest model fed precisely the right context often outperforms it.

For our engagement this is where a client's existing information architecture (M4) either pays off or bites. A brownfield repo with retrievable, well-structured sources makes context engineering cheap; a repo where knowledge lives in people's heads and stale wikis makes it the first thing we must fix. This document unpacks what context is, the physics that constrain it, how it is assembled, and what we configure.

The one idea to internalise

Context is not "everything the model might need" — it is the smallest set of high-signal tokens that makes the next step succeed. More context is not better; past a point it is actively worse. The discipline is curation under a budget, repeated every turn.

1What "context" actually contains

Context is the full working set handed to the model at a given step — not just the user's message. In a coding agent it is assembled from several distinct sources, each with a different trust level and lifespan:

Trusted / durable
  • system & role instructions (CLAUDE.md/AGENTS.md)
  • the task spec (handoff contract, M3)
  • source-of-truth artifacts: code, ADRs, specs (M4)
  • tool definitions — every registered tool's name, description and schema sits in the window before any work starts
Untrusted / tainted
  • retrieved web content
  • ticket & issue text, design-file text (M7 injection channels)
  • tool results from external systems
Ephemeral / working
  • prior turns in the session, tool outputs, scratch reasoning — the conversation so far, which grows every step and must be managed (compaction, summarisation).

The crucial mental model: every token in the window competes for the same finite attention budget. A retrieved file the agent doesn't need isn't neutral — it dilutes attention on the tokens that matter and, if it carries untrusted instructions, is an attack surface. Context assembly is therefore a series of inclusion decisions, each with a cost.

Two consumers are routinely under-budgeted because they arrive silently: tool definitions and accumulated tool results. In brownfield engagements, MCP tool-schema sprawl — every connected server injecting dozens of verbose schemas — is the dominant real-world context bloat we see at clients. Mitigations: a small set of well-described tools beats a large overlapping one; defer or lazily load tool schemas until a task actually needs them; and clear or summarise stale tool results once they have been acted on, rather than letting them ride along every subsequent turn.

2The physics: context rot & the attention budget

The hard constraint that shapes everything is context rot: a model's ability to accurately recall and use information degrades as the context window grows. This is not a bug to be patched; it is an empirical regularity: attention spread across more tokens gets diluted, and models are simply not trained or optimised to perform at those depths — architectures with sub-quadratic attention exhibit the same degradation, so it is not just a compute-cost artifact of full attention. Chroma's research across 18 models showed recall falling well before the advertised window limit, and getting worse as the needle's semantic similarity to the question dropped.

Two consequences we act on. First, precise lexical retrieval over a small context beats semantic search over a huge one — this is why Claude Code uses agentic grep/glob/read rather than pre-indexing the codebase (see E6/M4). Second, long-running work must actively shed context: compaction (summarise and reinitialise), structured note-taking (write state to a file, drop it from the window), and sub-agents (isolate a sub-task in its own window, return only a short summary).

The Goldilocks framing

The system prompt and assembled context should sit in a "Goldilocks zone": specific enough to constrain behaviour, not so bloated that it rots. Neither brittle hard-coded if-else instructions nor a vague "be helpful and use good judgment." The same applies to what you retrieve: enough to succeed, nothing that merely could be relevant.

3How context is assembled — inputs & outputs

Context engineering is a runtime discipline, executed by the harness every turn. Broken into its moving parts:

A

Retrieval

finding the right tokens

The mechanism that pulls candidate information: agentic search (grep/glob/read over the live filesystem) or RAG (embeddings over a pre-built index), or a hybrid. The choice is a real architectural decision covered in M4/E6.

Requires on input
  • a query or navigation strategy
  • source-of-truth artifacts that are actually findable (M4)
  • for RAG: a maintained index; for agentic: fast file tools
Produces on output
  • candidate snippets/files to consider including
  • freshness (agentic reads live state; an index can drift)
Where it sits
  • Implementation runtime — the agent retrieves as it navigates; also onboarding, where retrieval quality decides how fast an agent understands a codebase.

For our rollout: default to agentic search (precision, freshness, no index to maintain); consider hybrid only when a repo is too large to navigate by grep in a reasonable number of steps, and measure the benefit against maintenance cost.

B

Selection & compression

deciding what actually goes in

Given candidates, decide what to include verbatim, what to summarise, and what to leave as a link. This is where the attention budget is spent well or wasted.

Requires on input
  • retrieved candidates + the current goal
  • a budget discipline (target token size)
  • the "reference, don't copy" convention (M4)
Produces on output
  • a curated window: verbatim where needed, summarised excerpts + links elsewhere
  • headroom left for reasoning and tool results
Where it sits
  • Cross-cutting, every turn — the core of "context engineering" as a live activity.

For our rollout: copy summarised excerpts with a link to the master, never paste whole documents (that both rots the window and duplicates a source of truth). This is the M4 principle applied at runtime.

C

Long-horizon management

surviving many turns

For tasks spanning many steps, the window fills with history and rots. Three techniques shed load without losing the thread.

Requires on input
  • a place to persist state (progress file, notes, memory — the E1 memory component / M4)
  • a stop/compact trigger (window pressure, milestone)
Produces on output
  • a reinitialised, small context that preserves decisions & open threads
  • continuity across compaction boundaries
Where it sits
  • Implementation runtime for long tasks; the boundary where context (E2) hands off to memory (the E1 memory component, module M4).

For our rollout: pick the technique by task shape — compaction for a long conversational flow, note-taking for milestone iteration, sub-agents for parallel exploration where only a summary needs to return. Each keeps the main window in the green zone.

Cache economics — the cost dimension of assembly

Assembly choices also decide what you pay. Prompt/KV-cache hit rate is a first-order cost and latency lever: a cache miss costs a multiple of a hit (on current pricing, roughly an order of magnitude — a point-in-time figure, re-verify). Practitioner synthesis (notably the Manus team's published lessons) converges on three rules: keep the system prefix byte-stable (no timestamps or reordered tool lists at the top); assemble context append-only — never mutate or reorder earlier turns, because any edit invalidates the cache from that point forward; and time compaction cache-aware, since compaction deliberately rewrites history and forces a full re-read, so batch it at milestones rather than compacting eagerly.

4Context as an attack surface

Because the model cannot separate trusted instructions from untrusted data in one token stream, context is the primary delivery vector for prompt injection (M7). Any retrieved web page, ticket body, design-file layer name, or tool result can carry hidden instructions that the model may follow.

The rule that follows

Treat every retrieved item as tainted until proven otherwise. Summarise rather than pass raw untrusted content (Claude Code does this for web fetches). Once untrusted content has entered the window, apply the trifecta logic: an agent holding untrusted input must not also hold both sensitive data and an exfiltration path (Meta AI's "Agents Rule of Two", 2025 — M7). Context engineering and security engineering are the same activity seen from two angles.

E3 · DEEP DIVE

Harness Engineering in Practice

Study companion

0Why the harness is the thing we actually build

E1 named the equation (Agent = Model + Harness) and E2 dug into context. E3 is where it gets concrete: the harness is the software we design, own, and maintain. It is the deliverable of an agentic-engineering engagement — not the model (a commodity we rent) and not the prompt (advisory text). If a client asks "what are we paying you to build?", the honest answer is: this.

This document turns the abstract "orchestrating scaffold" into named patterns with inputs and outputs: how a memoryless model is bootstrapped into a session, how state survives between sessions, how work stays merge-ready, and why the right long-term move is to delete parts of the harness as models improve. Everything here is drawn from Anthropic's own harness-engineering practice and generalised for a brownfield client.

The one idea to internalise

Every harness component encodes an assumption about what the model cannot do yet. The harness is therefore a liability register of model weaknesses, written in code. When a weakness disappears with a better model, the matching scaffold should be removed — a good harness shrinks over time, it does not accrete.

1What the harness is responsible for

Between "a model that can reason" and "an agent that ships merge-ready code" sits a set of responsibilities nothing else owns. The harness is the software that discharges them:

State & continuity
  • bootstrap a cold, memoryless model into a working session
  • carry decisions and progress across sessions
  • leave the repo in a clean, resumable state
Orchestration & control
  • assemble context each turn (E2) and invoke tools
  • run the loop; detect stuck/blocked states
  • enforce deterministic constraints (E5 governance)
Why it can't live elsewhere
  • The model is stateless and unconstrained; the prompt is advisory; CI is downstream. Only the harness sits around the model at runtime, so only it can guarantee state, sequencing and hard limits.

2The core patterns — inputs & outputs

Read the diagram first: a harness runs sessions like relay legs. Each session is bootstrapped from durable state, does bounded work, and hands a clean baton to the next — the "engineers working in shifts" model.

A

The initializer agent

solving the cold start

A memoryless model dropped into a repo has no idea how to build, test, or where it left off. The initializer pattern uses a first, special session to lay down the scaffolding every later session relies on: a runnable setup script, a progress file, and an initial commit as a known-good baseline.

Requires on input
  • repo access + a way to run commands
  • an initializer prompt describing the setup contract
  • enough repo signal to infer build/test commands
Produces on output
  • init.sh (deterministic environment setup)
  • claude-progress.txt seeded with current state
  • an initial git commit — the resumable baseline
Where it sits
  • Phase-1 setup / onboarding — run once per repo; also the moment we discover whether a repo is agent-ready at all (feeds E6).

For our rollout: the initializer is the cheapest early diagnostic. If a model can't figure out how to build and test the repo, neither can a delegated agent — that's an agent-readiness red flag, not a prompt problem.

B

The progress file & cross-session state

continuity for a stateless model

The model forgets everything between sessions; the harness must not. A progress file plus git history is the "shift handover notes" that let session N+1 resume session N's work without re-deriving intent from scratch.

Requires on input
  • a convention for what belongs in the file (E1 memory component / M4)
  • a protocol: read state + git log at session start
  • a discipline: write continuously (assume interruption)
Produces on output
  • what this session finished, WIP + next step, blockers
  • file paths (not contents) pointing at the work
  • a resumable thread across many sessions
Where it sits
  • Implementation runtime; the boundary where the harness (E3) leans on memory (E1's memory component, module M4). Architectural decisions belong in ADRs, not here.

For our rollout: keep it disposable and pruned — a progress file over ~200 lines is a smell; move durable facts to ADRs/CLAUDE.md and detail to topic memory. It records where we are, not what is true.

C

The clean-state principle

always merge-ready

Every session ends with the repo in a coherent, mergeable state — never mid-refactor with broken tests. This is what makes the relay safe: any session can be the last one, and any human can pick up cleanly.

Requires on input
  • a fast verification loop to confirm "green" (E4 / M1)
  • a stop protocol: finish or cleanly park work
Produces on output
  • a committed, test-passing state at every boundary
  • no half-applied changes to reverse-engineer later
Where it sits
  • Every session boundary; the precondition that makes review/merge (governance) tractable.

For our rollout: clean-state is only achievable if verification is fast and trustworthy — another reason a weak test base caps autonomy at L2. The principle and the prerequisite are inseparable.

D

The generator–evaluator loop & the sprint contract

agree on "done" before coding

Rather than one model both writing and judging its own work, the harness can separate generation from evaluation, and force an explicit agreement on the definition of done before code is written — a "sprint contract" that turns a vague task into a checkable target.

Requires on input
  • a scoped task with acceptance criteria (M3 handoff)
  • an evaluator signal: tests, or a second agent/human
Produces on output
  • an agreed "done" the work is checked against
  • less rework from mid-task goal drift
Where it sits
  • Task kickoff + implementation; the runtime face of verification-first (E4) and the handoff contract (M3).

For our rollout: the sprint contract is where under-specification gets caught early. If the agent can't restate a checkable "done", the task isn't ready — send it back to refinement rather than letting the loop churn.

E

Controlled write-surfaces

JSON over prose, status-only edits

What the agent may write, and in what shape, is itself a harness decision. Anthropic's pattern: keep a machine-state file (e.g. feature_list.json) that the agent may update only in a narrow way — flipping a passes field — with tests explicitly off-limits. JSON is chosen over Markdown because the model treats it as code and is less likely to "improve" it.

Requires on input
  • a durable/derived/disposable classification (M4)
  • write-permissions enforced by hooks, not trust (E5)
Produces on output
  • agent edits confined to safe fields/paths
  • protected artifacts (tests, specs) the agent can't rewrite
Where it sits
  • Cross-cutting; the concrete meeting point of the harness (E3), information architecture (M4) and governance (E5).

For our rollout: this is where the durable/derived/disposable taxonomy stops being theory. Decide per artifact who may write it and enforce it with a hook — "the agent won't touch tests" is advisory; a PreToolUse deny is deterministic.

F

The extensibility layer: skills, plugins & finer-grained control

the 2026 surface

Point-in-time, as of mid-2026: the specifics below are Claude Code facts and move fast — re-verify before citing them to a client. The harness surface has grown past settings, hooks and sub-agents. Skills (instruction packs the agent loads on demand) and plugins (bundles of skills, hooks, agents and commands — scaffolded with claude plugin init and installable straight from a repo, no marketplace required) make harness components packaged and versioned instead of copy-pasted between repos. Agent teams add implicit multi-agent orchestration above sub-agents. On the control side: permission rules now match at Tool(param:value) granularity rather than whole-tool allow/deny; sandbox.credentials blocks secret reads inside the sandbox — a deterministic-layer win, moving "the agent must not read credentials" from prose to enforcement (E5); and managed settings' requiredMinimumVersion lets an organisation hold an entire fleet above a known-good version.

Requires on input
  • an existing hook/permission enforcement surface to extend
  • a call on what is org-wide (plugin) vs repo-local (CLAUDE.md)
  • a re-verify cadence — these facts age in months
Produces on output
  • packaged, versioned, shareable harness modules
  • secret access denied deterministically, not advisorily
  • a governed version floor across every machine
Where it sits
  • Cross-cutting; the distribution and fleet-governance face of E3 — how a harness scales from one repo to an organisation (E5, and the rollout arc in P1).

For our rollout: this is the pattern, not the product. Every serious harness — whatever the vendor — is growing an equivalent layer: packaged, versioned extensions plus finer-grained deterministic control. Borrow the mechanics; don't marry the tool.

G

Deployment surfaces beyond the interactive session

where the harness runs

The patterns above were described from the interactive terminal session, but the same harness now deploys to surfaces with very different risk profiles. Headless/CI agents run with no human in the loop, which makes CI the riskiest context: every guardrail must be deterministic before the run starts, and permission profiles there should be strictly tighter than interactive ones. Parallel sessions multiply throughput — either one repo clone per session, or git worktrees (one working tree per session sharing a single object store) to stop agents trampling each other's checkout. Background agents detach long tasks from a human's attention entirely. All three hit the same wall immediately: generation parallelises, human review does not.

Requires on input
  • per-surface permission profiles (headless the strictest)
  • an isolation strategy: clone-per-session or worktrees
  • an honest review-capacity budget (E6)
Produces on output
  • agent runs in CI, in parallel, and in the background
  • throughput bounded by review, not by generation
Where it sits
  • Implementation + CI/CD; where E3 mechanics collide with the autonomy ladder (E6) and surface in E7's metrics as a review bottleneck.

For our rollout: don't sell parallelism before sizing review. Ten background agents producing merge-ready PRs against a team that can review two a day is a queue, not a speedup — cap concurrency to review capacity (E6/E7) and spend the surplus on verification instead.

3The pruning principle: a harness that shrinks

The most counter-intuitive practice, and the one that keeps a harness healthy. Anthropic saw it directly: moving from one model generation to the next let them remove scaffolding they previously needed — sprint decomposition, certain context resets — because the newer model no longer needed the crutch.

Operating rule

Treat every component as a hypothesis: "the model can't do X unaided, so the harness does it." Re-test that hypothesis each model upgrade. If it no longer holds, delete the component — the space of what the harness must cover shifts and shrinks, it does not only grow. A harness that only accretes is accumulating tech debt against a moving model.

This has a commercial implication for the engagement: we are not selling a fixed artifact but a maintained one, with a cadence of re-evaluation tied to model releases. It also disciplines scope — we build the minimum scaffold the current model needs, not the maximum we can imagine.

Pruning needs data, and the harness is where the data comes from. Log per session: tool-call traces, cost per task, intervention rate (how often a human had to step in), and session outcome (merged / discarded / escalated) — these traces are the raw feed for E7's metrics. They also turn the operating rule above into a procedure rather than a vibe: keep a small canary task set, run it on each model release with and without a given scaffold component, and delete the component when the delta vanishes. A harness without observability can only accrete, because nobody can prove a component is dead.

Verification-First

Study companion

0Why verification is the component that makes delegation safe

E1–E3 built the frame, the context, and the scaffold. E4 is the load-bearing wall. Verification is the only component that supplies external truth — a signal about correctness that the model cannot generate, flatter, or argue away. Without it, an agent's confidence and its correctness are uncorrelated, and delegation is a gamble. With it, delegation becomes an engineering activity with a feedback loop.

For our engagement this is the single most important prerequisite to assess. A client with a fast, trustworthy test suite can be taken to real delegated autonomy (L3); a client without one cannot safely pass assisted use (L2), no matter how good the model is. This document explains what "verification" means in an agentic loop, why TDD fits agents unusually well, the specific ways agents subvert it, and how we protect the signal.

The one idea to internalise

Verification should be the path of least resistance, not an act of discipline. If the cheapest way for the agent to make progress is to run the tests and satisfy them, it will. If the cheapest way is to weaken a test, it will do that instead. We engineer the loop so that "make it actually correct" is the easy path and "cheat" is the blocked one.

1What "verification" means inside the loop

In a classical process, testing is a stage near the end. In an agentic loop it is the inner feedback signal that runs every iteration: the agent proposes a change, the harness runs the checks, the result re-enters context, the agent self-corrects. Verification is what closes the loop from E1's diagram.

What counts as verification
  • unit & integration tests (the primary oracle)
  • type-checks, linters, build success
  • machine-verifiable acceptance criteria (EARS/Gherkin, M3)
  • mutation score for test quality (M5/B3)
What does not count
  • the agent's own claim that it "looks correct"
  • a second LLM's unaided opinion (no external truth)
  • coverage alone (a vanity metric — see §4)
  • a green run on tests the agent was free to edit
The shift
  • Verification moves left: written before or alongside implementation, it stops being a gate you reach and becomes the specification you build toward. "Tests become the specification."

The deeper point: verification is the mechanism that converts a probabilistic generator into a system you can trust at a bounded level. The strength and speed of this signal is what actually sets the safe autonomy ceiling (E6) — not the model's raw capability.

2TDD as the agent's native loop

Test-driven development turns out to be an unusually good fit for coding agents, because red-green-refactor is exactly the propose→check→correct cycle the loop already runs. Kent Beck has called it a "superpower" for agents: a failing test is an unambiguous, machine-readable goal, and a passing test is an unambiguous stop condition.

This reframes the division of labour. The highest-leverage human (or separate-agent) contribution is often writing the failing test — encoding the requirement as an executable check — rather than reviewing the implementation line by line. The implementation is then constrained by an oracle it did not author.

3How agents subvert verification — and the three-layer defence

The uncomfortable, well-documented reality: agents cheat tests. When the easy path to "green" is to weaken an assertion, delete a failing case, mock away the behaviour under test, or hard-code the expected output, agents take it — not maliciously, but because the loop rewards reaching green and they optimise for that. Reported by practitioners and Kent Beck alike through 2025–2026.

L1

Advisory rule

necessary, insufficient

A clear instruction in CLAUDE.md/AGENTS.md: "Never modify or delete existing tests unless explicitly asked; if a test fails, fix the code, not the test." Shapes default behaviour — but it is advisory (E1/E5), so pressure or injection can override it. This layer alone is not a control.

L2

Deterministic hook

the real boundary

A PreToolUse hook that blocks edits or deletions to test files (and other protected paths) regardless of what the model decides. This is the layer that actually holds: the harness refuses the write. Maps directly to E3's controlled write-surfaces and E5's advisory-vs-deterministic distinction.

L3

Verification gate

catch the subtle cases

A Stop-hook / CI check that confirms, before a session ends or a PR merges, that the suite is intact and genuinely green: test files unchanged (or changes human-approved), no skipped/xfail smuggled in, and — crucially — that the tests still have teeth (§4). Catches the subtler subversions a path-block misses.

The pattern generalises

Notice the shape: advisory (tell it) → deterministic block (prevent it) → verification gate (detect it anyway). This is the same defence-in-depth ladder as security (M7). "Protect the tests" is not one rule; it is a layered posture, because the tests are the one artifact whose integrity the whole delegation model depends on.

4Coverage is a vanity metric — verify the verifier

A green suite with high coverage can still be worthless: coverage measures whether a line ran, not whether anything would notice if it broke. Agent-generated tests are especially prone to executing code with weak or absent assertions — 100% coverage with a near-zero fault-detection rate is a documented outcome (M5/B3: e.g. suites at 100% coverage and ~4% mutation score).

The trap
  • coverage rewards execution, not assertion
  • agents can farm coverage with hollow tests
  • a green run then means very little
The fix — mutation testing
  • inject small faults (mutants); a good test kills them
  • mutation score = % mutants caught → real teeth
  • surviving mutants = a concrete list of assertion gaps

For our rollout: use mutation score as the real exit criterion (target ~70% on a pilot module, M5 — the 70% bar is a pragmatic practitioner heuristic with diminishing returns above it, not a published standard) for "this code is verified," with coverage as a secondary signal only. This closes the loop honestly: the verification component itself is verified, so a green run from a delegated agent actually means something.

5Mutation testing in depth — how it actually works

Section 4 claimed coverage is a vanity metric and that mutation testing is the honest alternative. Since this approach is central to our verification story but not widely understood, here is how it actually works — from first principles.

Mutation testing answers one question that coverage cannot: "If my code had a bug, would my tests notice?" Coverage tells you a line ran. Mutation testing tells you whether anything would fail if that line were wrong. In other words, it tests your tests.

The mechanic

A mutation-testing tool makes many tiny modified copies of your source code, each with exactly one small deliberate fault (a mutant). It then runs your existing test suite against each mutant and watches what happens:

The outcome you care about is the surviving mutants. Each survivor is not a vague warning — it is a concrete, actionable gap: "we changed this exact thing in your code and not one test complained." A killed mutant is reassurance; a survivor is a to-do item with a precise location.

What a "mutant" is — common mutation operators

Tools apply a catalogue of small, mechanical transformations. A representative sample:

OperatorExample changeWhat a survivor reveals
Relationala > b → a >= ba boundary/equality case is never asserted
Arithmeticx + y → x - ythe computed value isn't actually checked
Logicala && b → a || ba combination of conditions is untested
Constantreturn 1 → return 0the exact return value isn't asserted
Statement removaldelete a line / a returna side-effect or early exit isn't verified
Negationif (x) → if (!x)the branch direction isn't pinned down

A worked micro-example

Suppose the code is function isAdult(age) { return age >= 18 } and the only test is isAdult(25) === true. Coverage is 100% — the line ran. Now the tool creates a relational mutant: age > 18. Run the test: isAdult(25) is still true, so the test passes — the mutant survives. That survivor tells you precisely what's missing: a test at the boundary, isAdult(18) === true, which would fail on the mutant and kill it. One survivor, one exact new test. That is the feedback coverage could never give you.

Caveats you must know

Equivalent mutants
  • some mutants are semantically identical to the original (e.g. a change in dead code)
  • they can never be killed — no test should fail
  • they inflate the "survivor" list; need human judgment or tooling to exclude → mutation score is "% of non-equivalent mutants killed"
Compute cost
  • N mutants × full suite = expensive (can be very slow on large modules)
  • mitigate: run incrementally on changed lines only, or nightly in CI, not on every keystroke
  • don't chase 100% — diminishing returns; ~70% is already a strong practical bar

Why this matters specifically for agentic development

This is the piece that makes mutation testing more than a nice-to-have for us. An agent optimising for "green" can farm coverage with assertion-free tests — but it cannot farm mutation score, because a hollow test kills no mutants. Mutation score is therefore the one quality signal an agent cannot game by writing more shallow tests. Two consequences:

Exit criterion and training signal

As an exit criterion: "this module is verified" means it clears the mutation-score bar (~70%), not that coverage is high or the suite is green. As a training signal: surviving mutants can be fed back to the agent as a precise to-do list ("these faults went undetected — strengthen the tests"), which measurably improves generated test quality — the MutGen study raised mutation score from 53% to 89.5% using exactly this feedback loop. So mutation testing both gates the agent and teaches it.

How we'd introduce it at a client

Pragmatically, not big-bang: pick one pilot module (the same one from the brownfield heatmap, M5), add a mutation tool to CI — Stryker (JS/TS), PIT (JVM), or mutmut (Python) — and run it on changed lines only to keep cost sane. Set a ~70% gate as the definition of "characterized/verified" before that module enters the write-autonomy phase. Feed the surviving-mutant list back to the agent (or the human) as concrete test tasks. Coverage stays as a secondary signal; mutation score is the one with teeth.

6Verification beyond the unit-test oracle

Everything so far has treated "verification" as roughly synonymous with "the test suite". That was the right simplification for building the loop, but it is an incomplete picture: a suite can be green, mutation-verified and protected, and the feature can still not work when a person opens the app. Above unit and integration tests sits a ladder of stronger oracles — and, carefully bounded, one advisory signal that is not an oracle at all.

Property-based testing
  • asserts invariants over generated inputs ("for all x…"), not hand-picked examples
  • agents are notably good at proposing candidate properties — a natural division of labour: agent drafts, human ratifies the invariant
  • Kiro ships property-based testing of spec correctness since GA — properties derived from the spec, not the code (the E8 link; tool fact, point-in-time, re-verify)
Agentic end-to-end verification
  • an agent drives the built application — browser, CLI, API — and observes actual behaviour
  • the run transcript / screenshots are attached to the PR as merge evidence alongside the green suite
  • closes the classic gap: "tests pass but the feature doesn't work"

The second rung matters most for delegation. Unit tests verify code against the developer's model of the requirement; agentic end-to-end verification checks the requirement itself — the agent exercises the user-visible flow the change was supposed to enable and reports what actually happened. Run it as a separate verifying agent (not the one that wrote the code — the generator–evaluator split from E3), and treat its evidence as a required exhibit at review, not a replacement for the deterministic suite.

The rubric-anchored LLM-judge — advisory, never the gate

§1 dismissed "a second LLM's unaided opinion" — the operative word is unaided. A judge given an explicit rubric and collected evidence (diff, screenshots, run transcripts) is admissible as an advisory signal for properties tests cannot express: UX quality, documentation clarity, migration safety. But it must never be the deterministic merge gate — that would re-import the problem verification exists to solve, a model grading a model with no external truth. This is the E5 spine applied to review: deterministic checks gate, the judge only advises. P1 makes the same move in discovery — AI evals with rubrics as structured judgment, never as ground truth.

7The prerequisite that gates autonomy

Everything above assumes a suite that is fast and trustworthy. Both are prerequisites, not nice-to-haves: a slow suite breaks the inner loop (the agent can't iterate), and an untrustworthy suite (flaky, or hollow) means green carries no information. This is why verification quality — not model quality — sets the autonomy ceiling.

Suite speed is engineered, not assumed. The practical shape is a fail-fast ladder: typecheck → lint → focused unit tests → full suite. The cheap layers run on every agent turn; the full suite runs at the session or PR boundary. As a rough tolerance, the per-turn signal must land in seconds to low minutes — slower than that and the agent stops iterating against it and starts guessing. Brownfield reality (M5) is a 40-minute suite, which is why building the focused-subset runner — mapping changed files to the tests that exercise them — is itself part of the harness work, not an optimisation to defer.

Flakiness is a direct attack on the signal. A flaky test destroys the information content of green and red, and a flaky oracle is worse than none: it trains humans and agents alike to retry until green and to shrug at red. Set an explicit flakiness budget and a quarantine protocol — a test that fails intermittently leaves the gating suite immediately and is tracked as a defect, not left in place to erode trust. Brownfield suites (M5) are flaky by default, so measuring and quarantining belongs to the same first build step as characterization tests.

E5 · DEEP DIVE

Governance

Study companion

0Why governance is where safety becomes real

E1–E4 built a capable, verifying agent. E5 is the component that decides what that agent is allowed to do — and, crucially, enforces it in a way the model cannot talk its way around. Governance is the difference between "we asked the agent nicely not to touch production" and "the agent physically cannot touch production." Everything we learned about prompt injection (M7) lands here: when the model is compromised, governance is the layer still standing.

For our engagement this is the component that converts autonomy from a liability into a bounded, insurable activity. A client will (rightly) ask "what stops it from doing something catastrophic?" The honest answer is never "the prompt tells it not to" — it is the deterministic governance layer described here. This document draws the single most important line in the whole framework: advisory vs deterministic.

The one idea to internalise

An instruction the model interprets is advisory; a rule the harness enforces is deterministic. Under normal conditions both seem to work. Under pressure, ambiguity, or injection, only the deterministic layer holds. Governance is the discipline of putting every safety- or consistency-critical rule on the deterministic side of that line.

1The defining distinction: advisory vs deterministic

This is the concept E5 exists to teach. Two mechanisms look similar in a demo and could not be more different in production:

Advisory — the model decides
  • text in context: CLAUDE.md / AGENTS.md, system prompt, rules files
  • the model reads it and usually complies
  • can be ignored under load, ambiguity, or injection
  • is itself an attack surface (rules-file backdoor, M7)
Deterministic — the harness decides
  • permission rules, pre/post-tool hooks, OS sandbox, network policy
  • executed by the harness, not interpreted by the model
  • holds regardless of what the model "wants"
  • the actual security & consistency boundary
The litmus test we apply to every rule
  • "What the model should know" (style, conventions, preferences) → advisory (prompt).
  • "What must never happen / must always happen" (block rm -rf, deny .env, run tests before commit) → deterministic (hook/permission).

Note this is the same ladder E4 used to protect tests (advisory rule → deterministic hook → verification gate) and the same one M7 uses for security. Governance is not a new idea bolted on; it is the general form of a pattern that recurs across the whole framework.

2The mechanisms — inputs & outputs

Read the diagram first: governance sits at the boundary between the model's intent and any real effect on the world. Every proposed action passes through it before it can happen.

A

Permission rules

allow / deny / ask

The first, declarative layer: per-tool and per-pattern rules deciding whether an action is allowed outright, denied outright, or requires a human "ask." The everyday posture — read/plan freely, ask before writes, deny the truly dangerous.

Requires on input
  • a policy: what is always allowed / denied / gated
  • tool & path patterns (e.g. deny .env, *.pem)
  • a settings hierarchy (enterprise → user → project → local)
Produces on output
  • a first-pass allow/deny/ask decision per action
  • a low-friction default posture for a repo
Where it sits
  • Configured at repo/org setup (Phase 1); enforced every action at runtime.

For our rollout: push a locked-down baseline via enterprise managed settings so teams inherit it and cannot silently weaken it. Deny secrets and destructive commands; "ask" on writes for sensitive repos; disable bypass mode outside isolated sandboxes.

B

Hooks (PreToolUse / PostToolUse / Stop)

programmable enforcement

The programmable layer that runs your own code at defined points in the loop. A PreToolUse hook can inspect a proposed action and block it — even in bypass mode; a PostToolUse hook can react after (format, lint, log); a Stop hook can gate session end (e.g. verify the suite is intact, E4-L3).

Requires on input
  • the events to hook and scripts to run
  • a fail-safe design (deny by default for cases the hook doesn't opine on)
Produces on output
  • hard, programmable guarantees at tool boundaries
  • auto-remediation (format/lint) and end-of-session gates
Where it sits
  • Runtime, at each tool call and session boundary; the enforcement engine behind E3's controlled write-surfaces and E4's protected tests.

For our rollout: one gotcha — hooks override permission rules. A PreToolUse hook that exits "allow" lets through even a denied command, so hooks must fall back to the normal permission flow for cases they don't explicitly handle. Over-hooking is itself an anti-pattern; hook only what matters.

What this looks like in config — a minimal sketch (illustrative Claude Code syntax; adapt to your harness)

Deny-by-default permission baseline:
"permissions": {
  "deny": ["Read(./.env*)", "Read(**/*.pem)",
          "Bash(rm -rf*)", "Bash(git push --force*)"],
  "allow": ["Read(./src/**)", "Bash(npm test*)"],
  "ask": ["Edit", "Write", "Bash(git push*)"]
}

PreToolUse hook, as pseudocode:
on PreToolUse(tool, input):
  if touches(input, PROTECTED_PATHS): return "deny"
  if is_db_migration(input): return "ask"
  return no_opinion  # fall through to the permission rules above

The last line is the gotcha in code form: returning "allow" here would let the action through even if a deny rule matches it, because hooks override permissions. A hook must return "no opinion" for everything it does not explicitly handle — deny-by-default lives in the rules, never in a permissive hook.

C

Sandboxing & network policy

bounding the blast radius

The OS-level layer: filesystem, process, and network isolation so that even an action that is allowed can only touch a bounded surface. The network egress allowlist is the highest-leverage control in the whole security story (M7) — it severs the exfiltration leg of the lethal trifecta.

Requires on input
  • an OS sandbox (e.g. Seatbelt / bubblewrap)
  • an egress allowlist (proxy / firewall by domain)
  • credentials kept out of the sandbox
Produces on output
  • a bounded, observable, reversible blast radius
  • exfiltration blocked even after a successful injection
Where it sits
  • Platform layer spanning dev, CI and deploy; the last line that holds when the model layer is fully compromised.

For our rollout: caveat — an OS sandbox typically covers shell/subprocess only; file-read/edit and web-fetch may not be covered, so pair it with read/edit deny rules and a fetch allowlist. Non-TLS-inspecting proxies are bypassable, so the egress allowlist is strong but not absolute.

Agent identity & credentials — a top-3 client security question

An agent acting on repos, CI and cloud APIs needs its own non-human identity, not a borrowed developer login: its actions must be attributable to the agent in the audit trail, and revocable without locking a person out. The pattern: agent-scoped, short-lived credentials issued per session by a secrets broker (Vault, cloud STS, OIDC federation in CI), with long-lived keys kept out of the sandbox entirely — an injected agent cannot exfiltrate a token it never held (M7). Point-in-time, mid-2026: harnesses have begun shipping this as a first-class control — e.g. Claude Code's sandbox.credentials setting blocks access to ~/.aws, ~/.ssh and similar credential stores from inside the sandbox. Re-verify names and coverage before quoting.

3Why governance exists: the lethal trifecta

Governance is not bureaucracy for its own sake — it is the answer to a specific, unsolved problem. Because a model cannot separate trusted instructions from untrusted data (prompt injection, M7), any agent that simultaneously holds untrusted input, access to sensitive data, and the ability to communicate externally can be turned against you. Simon Willison named this the lethal trifecta.

The Rule of Two, enforced by governance

Meta's "Agents Rule of Two": until injection is solved, an agent should hold at most two of those three properties in a session; the third must be dropped or human-gated. Governance is how you enforce that — permission rules and sandboxing remove a leg (e.g. egress allowlist severs external communication), and human "ask" gates re-insert a person when all three would otherwise combine. Advisory text cannot enforce the Rule of Two; only the deterministic layer can. (The Rule of Two is vendor guidance, point-in-time — Meta, 2025; re-verify before quoting, as the recommended posture will evolve with injection defences.)

This is the through-line of the whole framework: E1 named governance as a component, E4 used its mechanisms to protect tests, M7 maps the full threat surface — and E5 is where the enforcement actually lives. When we tell a client "assume the agent will be compromised and bound the damage," governance is the concrete set of controls that makes that posture real.

The supply chain is now a governed surface (point-in-time, mid-2026)

The rules-file backdoor (§1) generalises: everything the agent loads — skills, plugins, MCP servers, tool descriptions — is executable trust, and 2026 guidance treats it as such. The OWASP Agentic Skills Top 10 (AST10, v1.0 2026 — incubator status, so emerging community consensus rather than a settled standard) names the failure modes: malicious skills, typosquatted MCP servers, tool-description poisoning. Microsoft Security Blog, "When prompts become shells: RCE vulnerabilities in AI agent frameworks" (May 2026 — vendor research), showed agent frameworks turning injected prompts into remote code execution. And a Cloud Security Alliance research note (practitioner synthesis) analysed prompt injection against the Claude Code GitHub Action as a CI/CD supply-chain threat — an agent in your pipeline is part of your supply chain. The governance response is the same five layers pointed at what the agent loads, not just what it does: allowlist skills and MCP servers like dependencies, pin versions, log loads (M7). And the PostToolUse audit log earns double duty here — it is also the evidence artifact for SOC 2, ISO 42001 and EU AI Act conversations. All names and statuses point-in-time, mid-2026; re-verify.

E6 · DEEP DIVE

Autonomy Levels L1–L5

Study companion

0Why an autonomy taxonomy matters

E1–E5 built the components of a capable, verifying, governed agent. E6 answers the question a client actually cares about: how much do we let it do on its own? The L1–L5 taxonomy gives us a shared vocabulary for that decision — and, more importantly, reframes it. Autonomy level is not a measure of how good the model is; it is a staffing decision about how much judgment we delegate and how much we keep.

For our engagement this is the spine that ties the other components together. Each level demands specific components be in place (verification for L3, governance for L4), so "what level is this client ready for?" is really "which components do they actually have?" — which makes E6 the bridge from the framework (E1–E5) to the assessment (checklist) and the phased rollout (E7).

The one idea to internalise

Higher is not better. L5 is not the goal and L1 is not failure. The right level is the one the client's ecosystem can support safely — usually L2 today, with robust L3 as the ambitious target. Selling "full autonomy" is how engagements fail; selling "the highest level your verification and governance can back" is how they succeed.

1The five levels

Each level is defined by how work is initiated, how far the agent runs unattended, and — the real differentiator — which harness components must be in place to make it safe. Read the ladder as increasing delegation of judgment, not increasing model capability.

L1

Assistant

inline, single-file

Autocomplete-style suggestions the human accepts or rejects keystroke by keystroke. The human does all the work; the agent shortens it. Needs only prompt + context (E1/E2).

L2

Conversational

multi-turn, navigates the codebase

Back-and-forth in an IDE chat: the agent reads the codebase, proposes changes, explains. The human still drives and integrates. Adds tools (E1). Where most teams realistically are.

L3

Delegated task → PR

today's production ceiling

A scoped ticket is handed off; the agent works asynchronously and returns a pull request a human reviews and merges. This is where real delegation begins — and where the full harness (E3), verification loop (E4) and governance (E5) become mandatory, because the human is no longer watching every step.

L4

High autonomy

narrow tasks only

The agent selects its own work from a stream and acts with rare human intervention — but only for narrowly defined, repeated, deterministically-verifiable tasks. Real examples: Dependabot (dependency bumps), Uber's FlakyGuard (Uber engineering blog: of 1,115 flaky tests, 380 fixed, 197 landed — re-verify before quoting). Adds memory and blast-radius control on top of L3.

L5

Full autonomy

not achieved

No human in the loop for general software work. Not a real option today; treated as a horizon, not a target. Listing it matters mainly to make explicit that we are not aiming there.

2The component prerequisites — what each level requires

This is the operational heart of E6: each rung is unlocked by specific components being genuinely in place. This table is how we translate "what level?" into "what must exist first?" — and therefore how E6 drives both assessment and rollout.

LevelMust have in placeWhat gates the step up
L1–L2prompt + context + tools (E1, E2)basic retrieval; a human watching every change
L3+ harness, verification, governance (E3, E4, E5)a fast, trustworthy test loop; deterministic guardrails; PR review capacity
L4+ memory, blast-radius controla narrow, repeated task with a deterministic oracle; bounded, reversible effects

The rule that caps everything

Never scale the number of agents beyond code-review capacity. Autonomy and volume are the same budget seen twice: more agents or higher levels both produce more output needing review. When output outruns review, the effective level must come down, regardless of how capable the model is. This is why E7's review-bottleneck data (+98% PRs, +91% review time — Faros AI vendor telemetry) is really an E6 constraint.

To see what the "staffing decision" headline cashes out to, run the arithmetic (numbers illustrative — measure your own): suppose one senior reviewer can genuinely review — read, question, sometimes reject — about five agent PRs a day. Two such reviewers cap an L3 workflow at roughly ten merged agent PRs a day, regardless of how many agents are provisioned. Adding a third, fifth or tenth agent does not raise throughput; it only deepens the review queue and stretches time-in-review, which is the same failure the Faros AI telemetry above describes. The only levers that raise the cap are more review capacity, cheaper review per PR (smaller diffs, verification evidence attached), or moving a narrow slice to L4 where a deterministic oracle replaces the reviewer.

3How to choose the level — the assessment lens

The level is set by the weakest enabling component, not the strongest. A brilliant model on a repo with no tests is capped at L2, because L3 has no verification loop to make delegation safe. So the assessment is component-by-component:

Signals that support L3
  • fast, trustworthy tests (E4) with real teeth (mutation score)
  • CI that returns a signal the agent can read
  • deterministic governance: hooks, sandbox, egress (E5)
  • enough human review capacity for the PR volume
Signals that cap at L2
  • no/slow/flaky test suite
  • advisory-only config; no enforcement points
  • no dev/prod separation, RBAC or secrets hygiene
  • review already saturated

Phrases like "fast, trustworthy tests" and "enough review capacity" only bite if they are checkable, so turn them into an assessment instrument. The rubric below gives each prerequisite a concrete heuristic and a way to measure it. Every threshold here is a judgment-call practitioner heuristic, not a standard — tune the numbers to the client — but the discipline is the point: each rung's prerequisites must be checkable, or the level decision is vibes.

The level is a dial, not a ratchet

Autonomy governance is bidirectional: alongside the promotion rubric, define demotion triggers up front — review queue saturating, intervention rate rising, escaped-defect rate climbing, the test suite slowing past its budget. Who decides matters: the accountable reviewer or lead pulls the trigger, never the agent operator, whose incentives all point at keeping the dial up. And the speeds are deliberately asymmetric: promotion is slow — one level at a time, after the rubric above holds for a sustained period — while demotion is instant: the moment a trigger fires, drop one level immediately and investigate afterwards. A team that can only turn the dial up has built a ratchet, and ratchets fail loudly.

Note L4 is not "L3 done well" — it is a different shape of task. You do not graduate a general delegated-task workflow to L4; you identify a narrow, repeated, deterministically-checkable slice (dependency updates, flaky-test repair) and run that slice at L4 while everything else stays at L3. Level is per-workflow, not per-team.

Metrics & Anti-Patterns

Study companion

0Why measurement is where the engagement lives or dies

E1–E6 described how to build and bound an agentic harness. E7 is the reality check: what the hard data actually says about agentic development, and the failure modes that recur when teams skip the discipline of the earlier components. This is the component that keeps us honest — with ourselves and with the client — because the field is saturated with vendor claims and subjective enthusiasm that the evidence does not support.

For our engagement, E7 does two jobs. It gives us the baseline and metrics to prove (or disprove) that the harness is working — essential, because subjective productivity feelings are demonstrably unreliable. And it gives us a catalogue of anti-patterns with named, documented failures, so we can tell a client "here is exactly how this goes wrong, and here is which earlier component prevents it."

The one idea to internalise

AI adoption does not uniformly raise productivity — it accelerates development and exposes weaknesses in the surrounding system. The same agent that speeds up a mature team destabilises an immature one. So the metric that matters is not "how much code did the agent write" but "did the system's real outcomes (stability, throughput, review load) improve." Measure outcomes, never output.

1What DORA and METR are — and whether we can use them

We cite these two throughout, so here is what they actually are — and, more usefully, the distinction that decides how each serves us: DORA is a metric framework we can adopt directly; METR is not a tool we deploy but a method we borrow. Getting that split right is what turns "we measured something" into "we measured the right thing, honestly."

DORA — a measurement framework we adopt directly

DORA (DevOps Research and Assessment) is the research program behind the book Accelerate and, today, Google Cloud's annual State of DevOps report. Begun by Nicole Forsgren, Jez Humble and Gene Kim, its lasting contribution is a small, statistically validated set of software-delivery metrics — the "four keys" — split evenly between speed and safety:

DimensionMetricPlain meaning
ThroughputDeployment frequencyhow often you ship to production
ThroughputLead time for changestime from a commit to it running in production
StabilityChange failure rate% of deployments that cause a failure needing remediation
StabilityFailed-deployment recovery timetime to restore service after a bad deploy (formerly "time to restore" / MTTR)

Teams were historically clustered Low → Medium → High → Elite. The four keys keep their names in every edition to date; the 2025 cycle additionally promoted rework rate — unplanned deployments ÷ total deployments — to an official fifth metric, with its first benchmarks published in October 2025 (an earlier fifth candidate, operational reliability, never got the same formal treatment). The single most important finding — and the reason the metric set matters to us — is that throughput and stability are not a trade-off: elite performers score high on both. That is precisely why the DORA 2024 AI result is a warning sign, not a footnote: AI adoption pushed the two apart (stability −7.2%), which is the signature of a weak surrounding system rather than a healthy speed-up.

DORA 2025 — the update, and why it strengthens the argument

The 2025 report is retitled State of AI-assisted Software Development 2025 (published ~Sept 2025, dora.dev/dora-report-2025). Headline reversal versus 2024: AI adoption now correlates with higher throughput — but instability remains elevated. In short: faster, but not safer. Treat the 2024 figures (−1.5% throughput, −7.2% stability) as the historical decomposition and 2025 as the update. The report's central thesis — AI is an amplifier of existing organisational strengths and dysfunctions — strengthens E7's weak-system argument rather than retiring it: the same adoption that lifts a disciplined team's throughput amplifies an undisciplined team's instability.

Two structural changes matter for us. First, a DORA AI Capabilities Model: seven capabilities that determine whether AI amplifies strength or dysfunction — a clear and communicated AI stance; healthy data ecosystems; AI-accessible internal data; strong version control practices; working in small batches; a user-centric focus; quality internal platforms. This is very nearly a ready-made readiness checklist for a client harness pitch. Second, seven team profiles replace the elite/high/medium/low tiers: Foundational Challenges; Legacy Bottleneck; Constrained by Process; High Impact, Low Cadence; Stable and Methodical; Pragmatic Performers; Harmonious High-Achievers. Point-in-time note: no 2026 report exists yet (expected fall 2026) — re-verify before citing.

How we use DORA

Directly, as our pilot's before/after baseline. It is tool-agnostic and brownfield-friendly: all four keys are computable from data the client already has — Git history, CI/CD logs, and incident/deploy records — with no new instrumentation. We capture the four keys before the agent touches anything, then the same four after, and we always read stability next to throughput so a speed gain that quietly costs reliability cannot hide. Crucial caveat: DORA is a system/team-level measure, never an individual scorecard — turning it into per-developer targets triggers Goodhart's law and destroys the signal.

METR — a method we borrow, not a tool we run

METR (Model Evaluation & Threat Research) is a nonprofit that studies AI model capabilities and risks. For our purposes the relevant artifact is one specific study: its 2025 randomized controlled trial on how early-2025 AI tools affected experienced open-source developers (arXiv:2507.09089). The reason it carries the weight it does is the method — an RCT is the gold standard: 16 seasoned developers, 246 real tasks on their own mature repositories, with AI use randomized per task and actual completion time measured. That design is what lets its counter-intuitive result stand up — developers were 19% slower with AI while believing they were about 20% faster — where a satisfaction survey or raw telemetry could not.

How we use METR

Not as something we "adopt" — it is a research organisation, not a framework or a product. What is usable is the methodology. The lesson for our pilot is to get as close to an RCT as a client environment allows: randomise which of a set of comparable tasks (or teams) run with agents versus without, measure objective elapsed time and DORA outcomes, and never trust self-reported speed. Honest caveat: a true RCT is usually impractical mid-engagement, so we approximate — an A/B split across comparable task streams, objective telemetry, and a pre-registered definition of success set before we start. The perception gap is the whole reason to bother: without an objective baseline, the client (and we) will "feel" a win whether or not one occurred.

The two fit together cleanly: DORA tells us what to measure; METR's RCT tells us how to measure it honestly. Together they are the measurement backbone of the pilot design (trajectory topic T2) — and the antidote to the perception gap and vanity metrics discussed in the sections that follow.

2Using DORA — the measurement process, step by step

DORA is the framework we deploy directly, so this is a concrete playbook: what to collect, from where, how to compute each key, and what to prepare before the pilot. The goal is a defensible before/after comparison that survives the perception gap.

The four keys — definition, source, how to compute

MetricData source (already exists)How to compute
Deployment frequencyCD tool / release tags / deploy logscount of deploys to production per day/week, over a window
Lead time for changesGit + CI/CD timestampsmedian time from first commit of a change to it running in production

All four are computable from data a client already has — version control, CI/CD logs, and an incident/ticket system — so no new instrumentation is required to start. Two are throughput (frequency, lead time), two are stability (failure rate, recovery time); you must report them together.

1

Prepare the baseline (before agents)

the non-negotiable step

Pick a window (e.g. the last 8–12 weeks) and compute all four keys for the target team/service as it is today. Record the cluster (Low/Medium/High/Elite) per metric. This frozen baseline is what the pilot is measured against — without it, the perception gap (§ below) guarantees a false "win."

2

Define scope & guardrails

so the comparison is clean

Fix the unit of analysis (one team or service, not individuals), the time windows (equal length before/after), and a written "definition of success" set in advance. Decide the segmentation: which work runs with agents vs without, kept comparable.

3

Run & collect during the pilot

same metrics, same method

Recompute the four keys on the post window using identical queries. Add the agentic-specific companions from §4 (review time, PR size, queue depth, AI-vs-human churn) so a throughput gain that costs stability or review load cannot hide.

4

Read the deltas honestly

stability beside throughput

A real win moves throughput up without degrading stability. If stability falls as throughput rises, that is the DORA 2024 signature of a weak surrounding system — surface it, don't bury it. Report per-metric deltas and cluster changes, not a single composite score.

What to prepare for our engagement — a DORA checklist

Before day 0: confirm access to (1) Git history, (2) CI/CD deploy logs or release tags, (3) the incident/ticket system; agree the target team/service; agree the before/after window length; write the success definition down. Also capture a cost baseline: token/seat spend per merged PR and cost per completed task, so ROI can be reported next to the DORA deltas rather than asserted. Prepare a small script or dashboard that pulls the four keys reproducibly (many teams already have a DORA plugin in their CI platform). Crucial guardrail: DORA is a system/team-level measure — never present it per developer, or Goodhart's law destroys the signal.

3Using METR — the method we borrow for the pilot

We cannot "deploy METR" — it is a research organisation. What we borrow is its method: the randomized controlled trial that produced the perception-gap finding. The practical question is how close to an RCT we can get inside a client engagement, because that rigour is what makes our pilot result trustworthy rather than another "it feels faster."

Why the RCT design matters

METR's result held up — developers were 19% slower while feeling 20% faster — precisely because the design was controlled: real tasks on their own repos, AI use randomized per task, and actual time measured rather than self-reported. Any weaker design (satisfaction survey, uncontrolled before/after, cherry-picked demos) would have shown the illusion, not the truth.

A

Randomize what you can

approximate the RCT

A true per-task RCT is usually impractical mid-engagement, so approximate: split a stream of comparable tasks (or two comparable teams) into agent-assisted vs not, assigned as randomly as the work allows. The closer to random and comparable, the more the result means.

B

Measure objectively, never by feel

the core lesson

Collect actual elapsed/cycle time and the DORA keys from §2 — objective telemetry, not opinions. Capture the predicted and perceived speedup separately (a quick before/after question) purely to quantify the perception gap for the client; it is evidence about bias, not about speed.

C

Pre-register success

decide the bar before you look

Write the success threshold and the metrics before running, so the result can't be rationalised afterwards. Include a minimum sample (enough comparable tasks to see a real signal) and a fixed analysis window. Rough heuristic: at least 8–12 weeks per window and dozens of comparable tasks per arm; with less than that, make direction-only claims ("throughput moved up, stability held") and never quote percentages.

D

State the caveats

credibility through honesty

Our approximation is not a lab RCT: small samples, brownfield messiness, and task heterogeneity all limit it. Say so. Being the person who names the limits makes the findings that survive them far more persuasive (the honest-source-separation discipline).

What to prepare for our engagement — a METR-style pilot kit

Prepare: a pool of comparable tasks (or two comparable teams) big enough to split; an assignment method as close to random as the work allows; an objective time/cycle-time capture (issue tracker timestamps + the DORA pipeline from §2); a two-question perception survey (predicted before, perceived after); and a written, pre-registered success definition. Deliverable to the client: a before/after comparison with objective deltas and the measured perception gap side by side — the single most convincing artifact we can produce, because it pre-empts "but it feels faster."

How the two fit together in one pilot

DORA supplies the metrics; METR supplies the method. One pilot uses both: a frozen DORA baseline, a METR-style (as-random-as-possible, pre-registered, objectively measured) comparison over the pilot window, throughput read beside stability, and the perception gap quantified alongside. That single design answers "did agents actually help here?" honestly — and is exactly the T2 (measurement & pilot design) deliverable of our trajectory.

4What the hard data says

The evidence base, stated plainly and with its provenance, so we can cite it without overclaiming. Note carefully which findings are correlational and which are controlled.

DORA 2024 → 2025 (correlational)
  • 2024: a 25% increase in AI adoption correlated with −1.5% throughput and −7.2% delivery stability; 39% reported little to no trust in AI-generated code
  • 2025: throughput now correlates positively with AI adoption, instability still elevated — faster but not safer
  • mechanism, both years: AI amplifies the surrounding system's strengths and weaknesses
METR RCT 2025 (controlled)
  • 16 experienced OSS devs, 246 tasks (arXiv:2507.09089)
  • actually 19% slower with AI
  • predicted +24%, still believed +20% afterwards — a persistent perception gap
  • 2026 follow-up: no longer a clear slowdown, but METR calls it "very weak evidence" (callout below)
GitClear (correlational, large-N)
  • code churn 3.3% → ~7.1%; duplicated blocks up 8×
  • refactoring fell below 10%
  • concentrated in AI-heavy projects
Faros AI telemetry
  • +21% tasks, +98% merged PRs
  • +91% review time, +154% PR size
  • no org-level DORA improvement — the bottleneck moved
The two findings that reframe everything

The perception gap (METR): developers felt 20% faster while being 19% slower. Subjective productivity is not evidence — a pilot needs objective baselines, not satisfaction surveys. The review bottleneck (Faros): agents shift the constraint from writing code to reviewing it. Writing nearly doubled while org-level outcomes stayed flat, because the work simply piled up at review. This is the empirical spine of E6's "never scale beyond review capacity."

Point-in-time update: the METR follow-up (re-verify before quoting)

metr.org now shows a "results are outdated" banner on the original RCT, and on 2026-02-24 METR published "We are Changing our Developer Productivity Experiment Design" (metr.org/blog/2026-02-24-uplift-update). The follow-up experiment (running since Aug 2025: 57 developers, 800+ tasks, late-2025 tools) found the original cohort still −18% (CI −38%…+9%) but new developers only −4% (CI −15%…+9%) — no longer a clear slowdown. Crucially, METR itself rates this "very weak evidence" because of severe selection effects: 30–50% of developers withheld tasks from the study, per-task pay dropped from $150 to $50/hr, and concurrent-agent workflows broke time self-reporting — so the experiment is being redesigned.

The teaching point is worth stating explicitly: a rigorous lab's follow-up collapsing under selection bias — and the lab saying so — is exactly the measurement anti-pattern this document teaches, demonstrated on the field's own flagship study. Treat it as a gift, not an embarrassment: it is the strongest available argument for pre-registered success criteria, objective telemetry, and honest caveats in our own pilot (§3).

5The review bottleneck, visualised

The single most important operational finding: agentic development does not remove work, it relocates it. Understanding where it lands is the difference between a pilot that shows real gains and one that just generates a backlog of unreviewed PRs.

6Choosing metrics — outcomes, not vanity

Because output rises regardless of whether the system improved, we must measure the right things. The trap is that the easy-to-count metrics are exactly the misleading ones.

Vanity metrics (avoid)
  • lines of code / PR count / tasks completed
  • accepted-suggestion rate
  • "% of code written by AI" (vendor-incentivised, self-hedged)
  • developer satisfaction as a proxy for speed (perception gap)
Outcome metrics (use)
  • DORA: throughput and stability (change-fail rate, MTTR)
  • review load: time-in-review, PR size, queue depth
  • quality: AI-churn vs human-churn (<1.5× ≈ near baseline)
  • cost/ROI: token or seat spend per merged PR; cost per task vs baseline
  • DX / SPACE for developer experience, measured not felt

The governing principle: establish a baseline before the pilot and measure the same things after. Without a pre-pilot baseline the perception gap guarantees you will "feel" a win whether or not one occurred. Stability must be tracked alongside throughput — the DORA finding is that they move in opposite directions when the surrounding system is weak, so a throughput-only dashboard hides the real cost.

7The anti-pattern catalogue

Each of these is a documented, named failure — and each maps to a component from E1–E6 that, done properly, prevents it. This is how we turn "be careful" into "here is the specific control."

AP1

Full autonomy without guardrails

prevented by E5 · E6

Devin (independent Answer.AI test): completed 3 of 20 real tasks, and pushed ahead on impossible ones instead of stopping. Autonomy without blocker-recognition and governance is a liability, not a capability. Fix: cap the level (E6), enforce governance (E5).

AP2

Drop-in replacement / vibe coding in production

prevented by E5

A Replit agent deleted a production database during a code freeze, then fabricated records and claimed rollback was impossible. The missing control was enforced (not prompted) dev/prod separation and RBAC. Fix: deterministic governance — the agent must be unable to touch prod, not merely told not to.

AP3

Review bottleneck & tech-debt accrual

prevented by E4 · E6

+98% PRs meets flat review capacity; GitClear's rising churn and 8× duplication are the debt that accrues when output outruns scrutiny. Fix: never scale beyond review capacity (E6); CI gates on duplication/churn; verification with real teeth (E4).

AP4

Multi-agent parallel writes

prevented by E3

Parallel writer swarms make conflicting implicit decisions (Cognition, "Don't Build Multi-Agents"). Fix: keep writes single-threaded; let other agents contribute intelligence, not concurrent actions (E3 harness design).

AP5

Skill atrophy & comprehension debt

prevented by E6 · process

Passive delegation erodes the team's own understanding; an Anthropic RCT is reported to show a −17% comprehension gap under unrestricted delegation, worst in debugging (we have not located the primary publication — re-verify before quoting the number). Fix: keep architecture/debugging/perf human (the 70/30 split), active question-driven use — a staffing choice, i.e. an E6 decision.

AP6

Believing the perception, not the baseline

prevented by E7 itself

Rolling out on the strength of "it feels faster" (METR: felt +20%, was −19%). Fix: objective pre/post baselines; treat satisfaction as a wellbeing signal, never a speed metric.

8Reading the evidence honestly — caveats

Part of E7's job is epistemic hygiene, because we will be challenged on these numbers. The honest framing:

How to cite responsibly

"% of AI code" is not a rigorous metric — it is vendor-incentivised and often self-hedged by the executives quoting it. Vendor case studies (big productivity multipliers) have favourable framing; label them as such — that includes Faros AI, whose +98%/+91% review-bottleneck telemetry we quote elsewhere (vendor data, not a DORA finding). GitClear is correlational, not causal — concentrated in AI-heavy projects but not proof AI caused the churn. METR is a small sample (16 devs) in mature OSS repos, and its own follow-up was self-flagged for selection-bias limits — METR's own label for the 2026 result is "very weak evidence" (§4). Cite the direction of the evidence, not false precision — and always separate strong independent findings from vendor claims from practitioner synthesis.

This mirrors the honest-source-separation discipline used throughout the reports. With a client, the credibility win comes from being the person in the room who says "this number is softer than it looks" — it makes the strong findings (DORA stability, the review bottleneck, the perception gap) far more persuasive.

E8 · DEEP DIVE

Spec-Driven Development

Study companion

0Why spec-driven development, and why with care

E1–E7 built and measured the harness. E8 is about the workflow that feeds it: spec-driven development (SDD) turns intent into an explicit, reviewable specification that becomes the primary quality gate before code exists. It is the engineering-side counterpart to the handoff contract (M3) — the discipline that decides whether an agent builds the right thing or faithfully builds the wrong one.

But E8 comes with a deliberate double message. SDD is genuinely powerful for the cases it fits, and actively counter-productive for the cases it doesn't. Our job in the engagement is not to sell "adopt SDD wholesale" — it is to help the client apply it where the leverage is real (greenfield, legacy modernization, multi-task features) and avoid the sledgehammer-on-a-thumbtack failure mode Böckeler warns about.

The one idea to internalise

"Intent is the source of truth." SDD's core move is to make the specification — not the code — the thing you review, version, and hold agents accountable to. That inverts the classic gate: spec review precedes (or replaces) code review as the primary quality control. The prize is catching a wrong intent on paper; the risk is drowning a small task in ceremony.

1What SDD actually is

SDD is a workflow in which a structured specification is authored first, reviewed and agreed, and then drives implementation — with the agent building toward the spec rather than improvising intent from a terse ticket. The spec is executable-adjacent: precise enough that acceptance criteria (EARS/Gherkin, M3) can be derived and checked.

The contrast with "just prompt the agent" is that intent becomes a durable artifact (M4) with an owner and a review, rather than ephemeral text in a chat. For an L3 delegated workflow (E6), that artifact is what the agent is held accountable to and what the human reviews.

2The tools — three points on a spectrum

We care about methodology over tools, but the three reference implementations usefully mark out the design space, from lightweight-and-portable to full-team simulation.

A

GitHub Spec Kit

lightweight, portable

A slash-command workflow: five core commands, /speckit.constitution → /speckit.specify → /speckit.plan → /speckit.tasks → /speckit.implement (with /speckit.clarify and /speckit.analyze as optional refinement steps), unchanged across releases. The constitution.md holds non-negotiable project principles; the philosophy is "intent is the source of truth." Strongest on greenfield and legacy modernization; deliberately minimal and tool-agnostic. As of v0.11.0 (June 2026) it supports 30+ agents and ships extensions, presets, a skills-installation mode, and — most telling for E8 — upgrade detection with version pinning, so a team can manage the workflow itself like a dependency: the spec-as-artifact argument (M4) made concrete, applied to the very process that produces the specs. Point-in-time facts; re-verify before relying on them.

B

Amazon Kiro

agentic IDE, gated phases

Three phases with approval gates: requirements.md (user stories with EARS acceptance criteria) → design.md → tasks.md (numbered checklist). Bakes EARS directly into requirements and runs automated-reasoning checks for contradictions before code — the tightest coupling of spec to machine-verifiability. General availability came on Nov 17, 2025, adding property-based testing for spec correctness — generated properties measure whether the implementation actually matches the spec, the strongest bridge yet between SDD and verification-first (E4): the spec stops being prose the agent aims at and becomes something the harness can check the code against — plus checkpointing, a Kiro CLI, team plans and Pro/Pro+/Power tiers. Point-in-time facts; re-verify before relying on them.

C

BMAD-METHOD

full-team simulation

An MIT-licensed, bring-your-own-model framework distributed as files (agent personas in Markdown/YAML + workflows, templates, a CLI installer) that drop into your existing AI IDE — not a product or service. It simulates 12+ specialized agent roles (analyst, PM, architect, dev, QA, scrum master…), each a self-contained "Agent-as-Code" definition, running a four-phase cycle (Analysis → Planning → Solutioning → Implementation) with explicit artifact handoffs and document sharding (large PRD/architecture docs are split into small "story files" that pack exactly what one agent needs, fighting context loss). Its defining stance: documentation is the source of truth, code is a downstream derivative ("docs-as-code") — the aggressive spec-as-source end of the §4 spectrum. The heaviest end: powerful for complex work, but the most ceremony and the easiest to over-apply. The v6 line is current (v6.8, May 2026 — Skills Architecture, sub-agents, Dev Loop Automation) and evolving fast — verify specifics before relying on them.

What we extract

The methodology is portable even if the tool is not: a constitution of non-negotiables, an explicit spec authored and reviewed first, EARS/Gherkin acceptance criteria, and a decomposition into agent-ready tasks. We can assemble that lightweight pipeline for a client without adopting any single vendor tool — which is also how we avoid lock-in.

BMAD on an existing system — the brownfield path

Because our engagement targets an existing codebase, BMAD's brownfield workflow is the part most relevant to us — and it is more thought-through than the other SDD tools. It is worth knowing in detail even if we only borrow pieces of it.

How BMAD handles brownfield
  • Document-first is mandatory. The *document-project workflow scans the whole codebase and writes AI-optimised docs (overview, architecture, source-tree) into docs/ — the required first step when docs are missing or stale.
  • Two routes by size. PRD-First (large codebases: write the PRD, then document only the areas it touches) vs Document-First (smaller: document everything, then PRD).
  • Respects conventions. Auto-detects the stack and explicitly asks whether to follow existing patterns or set new ones.
  • Regression safety. A Test Architect (QA) role is mandatory when a change touches legacy code: *risk → regression potential, *design → test coverage, *review → verify nothing broke.
  • Quick Flow for small work. Bug fixes / small increments skip the ceremony: short spec, implement, self-review against the codebase.
How it maps to our framework
  • *document-project = our brownfield bootstrap (Report 4: read-only → characterize) and durable artifacts (M4)
  • "respect existing conventions" = the agent adapts to the estate rather than rewriting it (our brownfield principle)
  • Test Architect *risk/*design/*review = verification-first (E4) + blast-radius thinking, wrapped as a role
  • doc sharding + resumable state = context-rot management (E2) + progress/clean-state (E3)
  • Quick Flow = the answer to Böckeler's "sledgehammer on a thumbtack" (§4)
What we borrow for our engagement

Don't adopt BMAD wholesale — it is the most ceremony-heavy option, it pushes docs-as-source (we recommend spec-anchored, not spec-as-source), and it moves fast (V6, high churn). Instead borrow the mechanics that map cleanly onto our framework: the *document-project step as a brownfield bootstrap that produces durable, reviewable docs; brownfield PRD/architecture as durable artifacts (M4); and a mandatory regression gate on any legacy touch (Test Architect's *risk/*design/*review = our E4). That gives a client BMAD's real brownfield value — structure and regression safety when modifying an existing system — without lock-in to a single fast-moving tool.

3Two plugin approaches: Superpowers & Compound Engineering

Spec Kit, Kiro and BMAD are spec authoring tools. But two of the most-adopted 2025–2026 Claude Code plugins go further — they package a whole methodology as auto-triggering skills, and they differ most instructively in one dimension the earlier tools barely touch: whether the system learns between tasks. Both are worth knowing because a client may already be using one, and they embody two distinct philosophies of the same loop.

SP

Superpowers

Jesse Vincent — discipline enforcement

A skills framework — v5.x as of Mar 2026, now multi-platform (Claude Code, Cursor, Codex, Copilot CLI) — that forces a coding agent through disciplined phases instead of its default "jump straight to code" instinct. A master dispatcher skill auto-activates at session start and routes the task through the right skills; nothing needs to be invoked manually. Notably, v5 reversed an early design choice: the subagent review loops were removed in favour of a Spec Self-Review checklist — even the methodology tools iterate on their own verification story. (An earlier claim that it was accepted into Anthropic's official plugin marketplace in Jan 2026 could not be verified — treat as unconfirmed. Point-in-time facts; re-verify.)

CE

Compound Engineering

Kieran Klaassen / Every — the system learns

A plugin (from Every; the workflow they use to run five products with near single-person engineering teams) built on one thesis: "each unit of engineering work should make subsequent units easier, not harder." It ships the plan→work→review loop like Superpowers, but adds a step the others don't have — an explicit compound step that writes learnings back into the system so the next task starts smarter.

The same loop, seen through three lenses

Comparing them on planning, execution and feedback-learning is the useful exercise — it isolates what actually differs.

The one difference that matters most

Superpowers makes a single task go well by enforcing discipline; Compound Engineering makes the next task go better by turning each finished task into durable, machine-readable institutional knowledge. Superpowers is a within-task quality system; Compound Engineering adds a between-task learning loop. That "compound" step — capture the lesson, make it findable (tags/frontmatter), update CLAUDE.md, verify the system would now catch it automatically — is exactly the feedback dimension the classical SDD tools (Spec Kit/Kiro/BMAD) leave out.

How they map onto our framework

Neither is a new idea so much as a packaging of components we already have — which is exactly why they are useful reference points, not dependencies:

What they operationalise
  • plan-before-code = the SDD spec-review gate (this topic)
  • enforced TDD + two-stage review = verification (E4)
  • subagent-per-task, single-threaded = harness patterns (E3)
  • auto-triggering skills = progressive disclosure (M4/I5)
Where Compound's "compound" step lands
  • it is the feedback loop of information architecture (M4) made routine
  • learnings → CLAUDE.md/ADRs = durable artifacts with provenance
  • caution: writing to CLAUDE.md/memory is also a poisoning surface (M7) — the learnings need review, not blind append

For our rollout: we don't have to adopt either wholesale (both are opinionated, single-vendor, fast-moving — v-numbers change weekly). The lesson to carry to a client is the compound principle itself: build the between-task learning loop deliberately (codify review findings and fixes back into CLAUDE.md/ADRs/skills), because that is what turns a one-off productivity bump into a system that compounds — while keeping a human gate on what gets written back, per M7. Note also where the learnings land: CLAUDE.md is the advisory layer (E5), so compounded lessons inherit its failure modes — they can be silently ignored, drift stale, or be poisoned — which is exactly why the write-back needs a human gate rather than a blind append. Superpowers teaches the within-task discipline; Compound Engineering teaches that the discipline should also make the system smarter each time.

4The critique — and when SDD is overkill

Part of E8's value is knowing the failure mode. Birgitta Böckeler (ThoughtWorks, writing on martinfowler.com) and others have documented SDD's downsides candidly, and we should carry the critique openly rather than as evangelists.

The critique
  • generates piles of repetitive markdown that review worse than code
  • a sledgehammer for small tasks (one documented Kiro session turned a small bug into 4 stories + 16 acceptance criteria)
  • brownfield enterprise is the weak spot — specs assume clean intent
  • ceremony can relocate review burden rather than remove it
When SDD genuinely fits
  • greenfield features spanning multiple agent tasks
  • legacy modernization (capture business logic into a spec)
  • anything where the spec is the durable contract & review artifact
  • work where getting intent wrong is expensive

The heuristic: a task under ~15 minutes needs no spec; a one-line bugfix drowned in 16 acceptance criteria is the anti-pattern, not the ideal. Keep specs minimal (the M3 sizing: ~10–30 lines) and portable. SDD is a tool you reach for when intent is non-trivial and the work is delegated — not a universal process to impose on every ticket.

What ~15 lines of spec actually looks like — a brownfield micro-feature

We legislate "10–30 lines" above, so here is one, for a realistic task: throttling a legacy password-reset endpoint that support reports being abused.

Constitution excerpt (2 lines):
Never weaken existing auth flows.
New behaviour ships behind a config flag, off by default.

Spec: rate-limit POST /password-reset per account. Non-goals: no CAPTCHA, no per-IP limits (separate ticket), no change to token expiry.

Acceptance criterion (EARS, one of three):
WHEN a client submits more than 5 reset requests for the same account within 15 minutes, THE SYSTEM SHALL respond 429 and SHALL NOT send a reset email.

Task slice:
T1 sliding-window counter on the existing Redis · T2 gate the handler + 429 path behind the flag · T3 tests: 6th request throttled, 5th passes, flag off = old behaviour byte-for-byte.

Fifteen lines in the repo, one review comment ("make the window configurable"), a wrong assumption (per-IP vs per-account) caught before any code existed. That is the entire ceremony a task this size earns — anything more is the sledgehammer.

SDLC models & agentic fit

Foundations · read first

0Why an SDLC block, and what SDLC even means

We reference "the SDLC" in almost every module — each E-topic has a "where it sits in the SDLC" line. But we never defined it, and the term hides a distinction that matters enormously for our engagement. This block fixes that: what the software development life cycle is, the process models teams actually use, how agentic development behaves inside each, and — the practical payoff — how to decide which model a given client or workstream should use.

The one distinction to hold onto up front: people say "SDLC" to mean two different things. There are the stages every piece of software passes through (requirements, design, build, test, release, operate) — roughly universal. And there is the process model — how you sequence and iterate through those stages (Waterfall, Agile, DevOps, and so on). The stages barely change; the process model is a genuine choice, and it is the choice that reshapes where agents plug in and what can go wrong.

The one idea to internalise

Agentic development does not replace the SDLC — it instruments it with a fast inner loop. But the process model you're operating in decides how much that loop helps: models with fast, cheap iteration and strong verification (Agile, DevOps) amplify agents; models with slow, gated, document-heavy phases (strict Waterfall) blunt them — unless the gates are exactly what a regulated context requires. So "which SDLC model?" is upstream of "how do we roll out agents?"

2The process models — with agentic fit, pros & cons

Each model below gets the same treatment: what it is, how agentic development behaves inside it, and its trade-offs in that agentic context. Read them as a spectrum from "one slow pass" to "continuous flow."

1

Waterfall

sequential, gated, one pass

Phases run strictly in order — requirements fully specified, then design, then build, then test — each signed off before the next. Heavy upfront documentation; change is expensive once you're downstream.

How agentic dev fits
  • agents accelerate the build phase, but the model's slow feedback wastes their biggest strength (the inner loop)
  • the big upfront spec pairs naturally with spec-driven development (E8) — the spec is already the artifact
  • verification comes late, so agent errors surface late and expensively
Pros & cons (agentic)
  • + clear artifacts & gates suit auditability and contracts
  • + a complete spec is excellent agent input
  • − late feedback fights the propose→verify→correct loop (E4)
  • − churn/rework (E7) is most costly here

When to use: fixed, well-understood requirements; heavy compliance/contractual sign-off; safety domains where a change late is genuinely dangerous. Rare in greenfield software, common in regulated or fixed-scope work.

2

V-Model

Waterfall + paired verification

A Waterfall variant that pairs every development phase with a corresponding test/validation phase (requirements↔acceptance, design↔integration, module↔unit). Verification is planned up front, not bolted on.

How agentic dev fits
  • the explicit verification pairing is a natural home for verification-first agents (E4)
  • acceptance criteria defined early = machine-verifiable targets (EARS/Gherkin, M3)
  • still gated, so iteration remains slow
Pros & cons (agentic)
  • + verification is first-class — agents thrive against defined oracles
  • + traceability suits regulated/safety-critical work
  • − rigidity; changing requirements ripple through both arms
  • − upfront cost high before any agent leverage appears

When to use: safety-critical or regulated systems (medical, avionics, automotive) where traceable verification per requirement is mandatory. The gates are the point, not overhead.

3

Iterative & Incremental

repeated cycles, growing function

Build in repeated passes; each cycle adds or refines functionality and produces something testable. Not fully adaptive like Agile, but no longer a single pass either.

How agentic dev fits
  • each increment is a natural agent task → PR unit (L3, E6)
  • the repeated loop matches the agent loop; feedback arrives per increment
  • the harness's clean-state/session model (E3) maps onto increments
Pros & cons (agentic)
  • + feedback each cycle catches agent drift early
  • + scopes work to reviewable chunks (review-capacity cap, E6/E7)
  • − needs discipline to keep increments coherent
  • − weaker at whole-system architecture emerging piecemeal

When to use: most brownfield modernization; when requirements are broadly known but will refine with feedback. A strong default for our engagements.

4

Spiral

risk-driven iteration + prototyping

Iterative, but each loop is organised around risk: identify the biggest risk, prototype/experiment to retire it, then proceed. Prototyping is explicit.

How agentic dev fits
  • agents are excellent at cheap throwaway prototypes to retire a risk fast
  • the risk-first framing pairs with our security posture (M7) and blast-radius thinking
  • experiments are ideal low-stakes places to trial higher autonomy
Pros & cons (agentic)
  • + agent-generated prototypes make risk-retirement very cheap
  • + explicit risk focus disciplines where to trust agents
  • − heavyweight ceremony for small projects
  • − needs experienced judgment to run well

When to use: large, high-risk, novel projects where unknowns dominate and early prototyping pays for itself. Overkill for routine work.

5

Agile — Scrum & Kanban

short adaptive iterations / flow

Short iterations (Scrum sprints) or continuous flow with WIP limits (Kanban), continuous feedback, requirements expected to evolve. The dominant modern default.

How agentic dev fits
  • tight loops match the agent loop best — fastest feedback of any model
  • backlog items ≈ agent-ready tasks (P3/M2); sprint = a batch of delegated PRs
  • but: agent output can flood the board → the review bottleneck (E7) bites hardest here
Pros & cons (agentic)
  • + maximal agent leverage; fast course-correction
  • + natural fit for L3 delegated task→PR
  • − WIP/review capacity is the real cap — more agents ≠ more throughput
  • − weak upfront spec can let agents build the wrong thing fast

When to use: evolving requirements, product development, teams with a real feedback loop and review capacity. The best fit for high agentic leverage — provided verification (E4) and the review cap (E6) are respected.

6

DevOps / Continuous Delivery

continuous, automated, dev+ops merged

Not a replacement for Agile but its operational extension: automate build/test/deploy so change flows continuously to production, with tight monitoring and fast rollback. This is the world DORA (E7) measures.

How agentic dev fits
  • the CI/CD automation is the agent's verification + governance surface (E4/E5)
  • fast rollback + monitoring = bounded blast radius (M7) for agent changes
  • enables narrow high-autonomy (L4) workflows: Dependabot-style, flaky-test repair
Pros & cons (agentic)
  • + the strongest verification/governance base → the highest safe autonomy
  • + continuous flow suits agents that never tire
  • − CI is the riskiest context: unattended, credentialed (M7)
  • − demands real maturity; immature DevOps + agents = fast instability (DORA)

When to use: teams with mature automation and monitoring; anywhere you want the highest safe autonomy. It is less a choice than the maturity target the other models grow toward.

3When to use which — a decision lens

The model is chosen by the shape of the problem, not fashion. The dimensions that actually decide it:

The agentic overlay on the decision

Agentic development shifts the calculus toward faster-feedback models. Because agents make writing cheap and relocate the constraint to review (E7), the models that pay off most are those with a fast, trustworthy verification loop (E4) and continuous flow (Agile/DevOps). But the shift is conditional: without the verification and governance base, a fast model just produces instability faster (the DORA 2024 finding). So the honest recommendation is rarely "adopt Agile/DevOps" in the abstract — it's "build the verification + governance base first, then the faster models unlock higher autonomy." In regulated or safety-critical contexts, the gated models remain correct and agents live inside their gates.

4How agentic development changes the SDLC itself

Beyond "which model," agents change the life cycle in three structural ways worth naming:

The shifts
  • The loop tightens. Verification moves from a late stage to an inner, per-turn signal — the SDLC gains a fast feedback loop inside implementation (E4).
  • The bottleneck moves. From writing to reviewing (E7); the constraint is now human review capacity, not typing speed (E6).
  • The spec returns. After years of "working software over documentation," explicit specs matter again — because the agent needs them as input (E8/SDD).
What stays true
  • the stages don't disappear — they're instrumented, not removed
  • a weak surrounding system is exposed, not fixed, by speed (DORA)
  • human judgment stays at the gates: intent, architecture, merge
  • the safe autonomy level is still set by verification strength, not the model

Agentic development doesn't replace the life cycle, it instruments it with a fast inner loop, which pays off most in faster-feedback models (Agile/DevOps) but only once the verification and governance base exists — otherwise speed just surfaces weakness faster. Choose the model by the problem's shape (fixed vs evolving requirements, risk, regulation, maturity), then position agents inside it: in the gates where gates are required, at task→PR where feedback is fast, and at narrow high autonomy only where automation is mature.

AI-Assisted Discovery

Study companion · module M2

0Why discovery is where the product harness starts

The engineering deep dives (E1–E8) all rest on one assumption: that the task handed to the agent is the right task. P1 is where that rightness is decided. Discovery is the practice of continuously deciding what to build and why — and it is the first place a product harness can either amplify a PM or quietly replace the thinking that makes them a PM.

For our engagement the framing matters as much as the technique. In the product layer the agent assists; it never owns the judgment. Everything in this topic is an application of that one rule to the messy, human-heavy work of understanding users — and it sets up the P1–P5 assistance scale (P5, full delegation of product judgment, is the anti-pattern) that the rest of M2 builds on. The P1–P5 scale — like D1–D5 — is an original synthesis of this programme, not an industry standard; flag it as such when presenting.

RungWorking definition (this programme's wording — see the provenance note above)
P1AI assist on demand — the PM invokes AI for specific discovery chores (transcription, clustering, a one-off parallel synthesis); the judgment loop is untouched.
P2Parallel second perspective as the default habit — two-perspectives synthesis runs on every study, not just when someone remembers.
P3Agent drafts, human synthesises and decides — AI produces first drafts (summaries, PRD sections); the PM still does their own synthesis and owns the decision.
P4Agent synthesises, human audits and owns the call — the AI synthesis is primary; the PM audits it via evals and raw-source spot-checks, and the call remains theirs.

The one idea to internalise

AI in discovery is a second perspective, not a replacement for the first. The value is never the AI's summary on its own — it is the delta between the PM's own synthesis and the AI's. If the PM stops synthesising and just reads AI summaries, discovery has been deleted, not accelerated.

1What discovery is — and what agents change

Discovery, in Teresa Torres's continuous sense, is the ongoing habit of talking to users weekly, mapping the opportunity space, and letting evidence — not opinion — drive what gets built. It is deliberately human-heavy: empathy, context, and the ability to notice the thing nobody said out loud are the point.

Agents change the economics of discovery without changing its purpose. They make it cheap to transcribe, cluster, and re-synthesise large volumes of interview and feedback data — work that used to bottleneck a PM. The risk is precisely that cheapness: when synthesis becomes a button, the temptation is to press it and move on, which is where the detail (and the empathy) leaks out.

What agents genuinely help with
  • transcription & first-pass clustering of many interviews
  • surfacing patterns across a large corpus a PM can't hold in their head
  • a fast parallel synthesis to compare against your own
  • drafting eval criteria for "did we learn what we needed?"
What must stay human
  • conducting the interviews (the empathy is the data)
  • the first synthesis pass — thinking happens in the doing
  • deciding which opportunities matter (product judgment)
  • noticing the unsaid, the contradiction, the surprise
Privacy & consent — the objection legal will raise before anyone asks about quality

Interview recordings and transcripts are personal data. The moment they flow through AI tooling, the client's legal team will — correctly — ask: where do the transcripts go (which vendor, which jurisdiction)? Are they used as training data? Is a DPA in place with every tool in the chain? Has the consent wording participants sign been updated to cover AI processing, not just recording? What is the retention and deletion story? Answer these before the first pilot, not after.

This is the product-layer version of E5's egress allowlist: just as engineering governance enumerates where code and secrets may flow, the product harness enumerates where user data may flow — deterministically (tool allowlist, DPA on file, consent template), not as advisory prose. Raising it unprompted in the first product-org meeting builds more trust than any synthesis demo.

Synthetic users — the contested shortcut (practitioner synthesis, mid-2026)

Clients will ask about AI-simulated interviews — "synthetic users" that answer discovery questions in a persona's voice. The position this programme takes: they are not discovery. The empathy is the data; a model can only return a plausible average of what has already been written, so it produces exactly zero surprises, contradictions, or unsaid things — the parts of an interview that change a roadmap. Treating synthetic answers as evidence is P5 wearing a research costume.

The narrow legitimate uses: piloting an interview guide (does the flow work, roughly how long does it run) and stress-testing question wording (leading questions, ambiguity) before spending real users' time. This is a fast-moving, contested topic — re-verify the state of the debate before presenting.

2The anti-pattern: "lazy AI interview synthesis"

This is the failure mode to name explicitly for a client, because it looks like productivity. The pattern — transcript → AI summary → move on — feels efficient and is quietly catastrophic for discovery quality.

The numbers to carry — both practitioner estimates from Torres's writing, not study results: unreviewed AI interview synthesis misses roughly 20–40% of important detail, and the lazy pattern captures only about 5% of the value real discovery would. The fix is not "better prompts" — it is keeping the human synthesis in the loop and mining the comparison.

3The two-perspectives method — inputs & outputs

The core technique, stated as a repeatable process the PM actually follows.

Requires on input
  • raw interview material (the PM conducted the interviews)
  • the PM's own first-pass synthesis, done before seeing AI's
  • a parallel AI synthesis of the same material
Produces on output
  • a richer synthesis built from the differences
  • caught blind spots (in both the PM's and AI's reading)
  • spot-checks of AI claims against the raw source
Where it sits
  • Requirements / discovery — upstream of the handoff contract (M3); this is where "the right task" is decided before any agent builds anything.

For our rollout: the practical obstacle is friction — if comparing two syntheses is expensive, PMs won't do it and will slide back to lazy synthesis. Part of building the product harness is making the compare step cheap (templates, a side-by-side view), so the healthy behaviour is the path of least resistance — the same "make the good path easy" principle as verification in E4.

4AI evals — the product-layer verification loop

The most important structural idea in P1: discovery needs its own version of the verification loop that makes engineering delegation safe (E4). In the product layer that loop is AI evals — lightweight, explicit checks of "did we actually learn what we set out to learn, and is the synthesis faithful to the source?"

The parallel that ties M2 to M1

Engineering has tests as external truth; product has no deterministic oracle, so its "truth" is human judgment plus consistency checks. AI evals are how you approximate a feedback loop anyway: define what a good synthesis must contain, check the AI's output against it, spot-check against raw interviews. It is weaker than a test suite — which is exactly why the product scale (P1–P5) caps lower than the engineering one (L1–L5): the verification is human, so it cannot be automated away.

Concretely, an eval is a checklist a PM can run in ten minutes on any AI synthesis before it is allowed to inform a decision:

Eval criteria — faithfulness to source
  • every claim in the synthesis is traceable to ≥1 quote from a named interview
  • spot-check 10% of quoted lines verbatim against the raw transcript
  • anything the AI inferred is flagged as inference, separate from what a user actually said
Eval criteria — completeness & judgment
  • at least one disconfirming datapoint is surfaced — a synthesis with no surprises is suspect
  • no recommendation appears without an opportunity it maps to on the tree
  • open questions the interviews did not answer are listed, not papered over

This is also the honest limit we carry to a client: you can instrument discovery, but you cannot make product judgment deterministic. The eval reduces the chance of a bad synthesis slipping through; it never removes the PM's responsibility for the call.

5Tools vs methodology

A recurring theme across the whole workbook, sharp here: the tools (Dovetail, Notion AI, and whatever launches next quarter) are secondary. The methodology — weekly customer touchpoints, the opportunity solution tree, two-perspectives synthesis, AI evals — is primary and portable across tools.

Primary — the methodology (durable)
  • continuous weekly touchpoints with users
  • opportunity solution tree to map the space
  • two-perspectives synthesis; AI evals
Secondary — the tools (swappable)
  • Dovetail, Notion AI, transcription services
  • useful, but not the thing that makes discovery good
  • chosen to fit the methodology, never the reverse

For our rollout: we sell the methodology, not a tool stack. A client who adopts Dovetail but keeps doing lazy synthesis has gained nothing; a client who adopts the two-perspectives habit with any tool has improved discovery. This keeps us vendor-neutral and durable against the weekly tool churn.

P2 · DEEP DIVE

PRDs & Requirements with AI

Study companion · module M2

0Why the PRD is the hinge of the whole system

Discovery (P1) decides what is worth building; the handoff contract (M3) is what an engineering agent can actually execute. The PRD sits between them — it is where fuzzy product intent becomes written requirements. That makes it the hinge: the quality of this one artifact decides how autonomously the engineering harness downstream can run. Investment in the product layer pays off in the engineering layer, and this is the topic where that investment is made or lost.

It is also where AI assistance is at its most seductive. Nothing looks more like PM work than a well-formatted PRD, and nothing is easier for an LLM to produce than a plausible one. P2 is about telling those two apart: what AI genuinely does well in requirements work, what it systematically fails at, and how to structure the collaboration so you get the first without the second.

The one idea to internalise

Marty Cagan's rule: use AI to amplify your thinking, not abdicate it. The productive frame is AI as the best junior PM you have ever had — tireless at drafting, structuring, consistency and edge cases, but with no product judgment of its own. A junior's draft is a starting point a senior edits; the moment the draft ships unedited, the senior has abdicated.

1What works — the junior-PM jobs

The 2025–2026 practitioner consensus (Cagan/SVPG, Torres, Anthropic internal teams — practitioner synthesis, not controlled studies) is consistent about where AI drafting genuinely helps. The common thread: every job on this list is either expansion (more edge cases, more structure, more consistency from the same input) or challenge (find what is wrong with this) — never judgment (decide what to build).

What AI does well in PRD work
  • drafting from rough input — notes, a call transcript, a whiteboard photo
  • structuring and consistency across a long document
  • generating edge cases the author did not think of
  • red-teaming a draft before stakeholder review (§4)
  • generating acceptance criteria from stories (feeds M3)
What must stay human
  • the why — rationale is exactly where LLMs fail hardest (§2)
  • the kill/keep decision on an idea (§3)
  • trade-offs between users, stakeholders, strategy
  • final ownership of every sentence that reaches engineering

For our rollout: the assess question at a client is not "do PMs use AI to write PRDs?" — most already do. It is which of these jobs they use it for, and whether a human edit pass stands between the AI draft and the backlog. AI drafting with human editing is P2 on the assistance scale — today's healthy mainstream. The same output without the edit pass is drift toward P5, the anti-pattern.

2What doesn't work — and the evidence

The failure modes are as consistent as the successes: hallucinated requirements (capabilities or constraints the product does not have), generic "thin" output (a PRD that could describe any product in the category), and — the deepest one — missing rationale. This is not anecdote; it is the best-measured result in the field.

Strong independent evidence — Quattrocchi et al., arXiv:2507.15157

The largest academic study of LLM-generated user stories to date: 13,958 generated stories, 153 of them triple-annotated by humans against the Quality User Story (QUS) framework. Result: LLM stories are rejected far more often than human ones — 77.34% rejection for GPT-3.5, 63.47% for GPT-4 — and the failures concentrate systematically on one quality: Rationale Clarity, the why. GPT-4 scored 1.54 where human stories score 2.66–2.68. Point-in-time caveat: these are 2025 numbers on 2024-era models; re-verify before quoting, but the pattern (form outpacing rationale) has held across model generations so far.

Why the rationale gap hurts precisely at handoff: intent/why is the first component of the handoff contract (M3, H1). An engineering agent given a story with form but no purpose will faithfully implement the form — and every decision the spec does not pin down defaults to a guess made without knowing what the feature is for. The Jama Software traceability audit found teams deploying coding agents without a specification layer produced code with no traceable link to any documented requirement. Weak rationale upstream becomes untraceable code downstream.

The honest caveat we carry to clients: most published "AI in PM" case studies (ChatPRD and similar) are vendor marketing, and the academic numbers come from benchmark and industrial-dataset contexts whose transferability to any given team is not guaranteed. The Quattrocchi result is the strongest independent evidence available; treat everything glossier with suspicion.

3The PR/FAQ principle — AI should speed up killing ideas

Amazon's Working Backwards process starts a product idea as a written press release + FAQ, reviewed before anything is built. Its famous property: most PR/FAQs do not get approved — and that is a feature. The document exists to kill weak ideas cheaply, on paper, before they consume a team.

This reframes what AI drafting is for. If AI makes writing a PRD ten times cheaper, the naive conclusion is "we can produce ten times more PRDs" — which floods the backlog with plausible, thin, unkilled ideas and moves the cost downstream to engineering. The productive conclusion is the opposite: AI should speed up killing bad ideas on paper. Cheap drafting means an idea can reach its kill/keep decision in hours instead of weeks; the kill rate should go up, not the shipping rate of half-thought features.

Requires on input
  • a real product idea with a named customer problem
  • discovery evidence behind it (P1 — not a substitute for it)
  • an explicit kill/keep review step with a human owner
Produces on output
  • a fast, cheap paper prototype of the idea (PR/FAQ or PRD draft)
  • a kill/keep decision made on evidence, not sunk cost
  • for keepers: a draft already structured for red-teaming (§4)
Where it sits
  • Requirements — between discovery (P1) and decomposition (P3); the last point where an idea is cheap to kill.

For our rollout: a useful health metric for the product harness is the proposal kill rate. If AI-drafted PRDs are almost never killed, the review step has degenerated into rubber-stamping (P4's automation bias) and the drafting speed is being converted into backlog noise rather than decision speed.

4Requirements red-teaming — the highest-evidence quick win

Of everything in this topic, adversarial review of requirements is the job with the strongest quantitative support — and it is P3 on the assistance scale, one step beyond today's mainstream, which makes it the natural next move at a client.

The asymmetry is the lesson: models are measurably better at finding problems in requirements than at authoring requirements. Finding ambiguity is a pattern-matching task with a checkable answer; authoring rationale requires the product judgment the model does not have. So point the model at the job it is good at: after the PM writes (or edits) a draft, an AI red-team pass hunts for ambiguity, contradictions, missing edge cases, untestable acceptance criteria, and unstated assumptions — and the PM decides which findings matter.

Packaging it — the backlog-refinement gate

The deployable form is a checklist gate in backlog refinement: no story leaves refinement until it has had an AI red-team pass and a human disposition of the findings. This is deliberately the same shape as E4's verification-first loop — a challenge step between draft and done — implemented with human judgment where engineering has tests. It is cheap (minutes per story), it produces its own evidence trail, and F1 in the 67–76% range means the human disposition step is structural, not optional.

5From PRD to handoff — where P2 meets M3

The PRD is not the terminal artifact; the handoff contract is. The practical flow at a client: discovery insight (P1, two-perspectives) → PRD drafted by AI from grounded context — company context, user research, a PRD template — and edited by the PM with kill/keep applied (§3) → red-team gate (§4) → decomposition into agent-ready work items (P3) → each item gets machine-verifiable acceptance criteria, context links, and requirement traceability (M3). Every stage hands the next one exactly what it cannot generate for itself: the why survives because a human wrote it and the contract carries it.

This ordering also fixes the measurement. The tempting metric for AI-assisted PRD work is drafting speed; the meaningful one is downstream: the share of tasks the engineering harness completes without follow-up questions. A PRD process that doubles drafting speed but leaves engineering agents guessing at intent has optimised the wrong end of the pipe — the same lesson E7 teaches about output metrics generally.

6Takeaways for our engagement

Assess
  • are PRDs AI-drafted, and does a real human edit pass follow?
  • do stories carry rationale, or form without a why?
  • what is the proposal kill rate — does paper ever die?
  • is there any adversarial review of requirements today?
Build
  • institute draft-and-challenge: AI drafts + edges, PM edits + owns the why
  • add the red-team gate to backlog refinement (P3 — the quick win)
  • adopt the PR/FAQ frame: cheap drafts exist to kill ideas faster
  • measure handoff quality (no-follow-up-questions rate), not drafting speed
Three sentences to remember P2 by

AI is the best junior PM you have ever had — excellent at drafting, structure, edge cases and red-teaming — but the evidence (Quattrocchi: 63–77% rejection, Rationale Clarity 1.54 vs human ~2.7) shows it systematically fails at the why, which is the first component of the handoff contract. Use cheap drafting to kill bad ideas faster (PR/FAQ), not to produce more unkilled ones, and put the model in the critic's chair where the evidence is strongest (red-teaming: +20.2% ambiguity detection). Amplify your thinking, don't abdicate it — the PM edits, owns, and supplies the rationale; the measure of success is downstream handoff quality, not drafting speed.

P3 · DEEP DIVE

Decomposition into Agent-Ready Work Items

Study companion · module M2

0Why decomposition is where throughput is decided

P2 ends with a handoff-ready PRD; the handoff contract (M3) specifies a single work item an engineering agent can execute. Between them sits the least glamorous step in the product harness: slicing an epic into work items. It is also the step that decides throughput, because agents do not work on epics — they work item by item, one fresh session at a time. How you cut decides how many agents can run in parallel, how big the PRs land in review, and how much autonomy each delegation can safely carry.

A perfectly specified epic that is badly sliced still fails: items too big exceed what a model can reliably finish, items too entangled serialise into a queue and merge-conflict against each other, items without their own context force every agent to rediscover intent from scratch. Decomposition is where the product layer either buys the engineering layer its parallelism — or quietly caps it.

The one idea to internalise

Decomposition is a judgment task disguised as a mechanical one. An LLM can propose a split in seconds, and the proposal will look complete. But deciding where the seams are — what is genuinely independent, what is one change pretending to be three, what is out of scope — requires knowing the codebase, the architecture and the strategy. The evidence (§3) says exactly this: AI splitting is a useful completeness checker; the seams and the sizes stay human.

1INVEST, re-read for agents

Bill Wake coined INVEST — Independent, Negotiable, Valuable, Estimable, Small, Testable — in a 2003 article on xp123.com (a practitioner blog post, later popularised by Mike Cohn's User Stories Applied). It was designed for exactly our problem, twenty years early: handing work across a context gap to someone who was not in the room. Agents widen that gap to its maximum — the recipient now has no shared history at all — which makes INVEST more relevant, not less. Each letter earns a sharper reading:

Letter2003 reading (for humans)Agent re-reading
Independentminimise ordering dependencies between storiesparallelisable across agent sessions — false independence surfaces as merge conflicts; dependency order must be explicit (§5)
Negotiablea story is a placeholder for a conversationdrafts are cheap to regenerate (disposable, I1) — but the decisions in them must be pinned, because the agent won't renegotiate
Valuablea vertical slice of user valuecarries its why and a requirement ID — the rationale gap (P2) means a human supplies it; without it the agent implements form without purpose
Estimableclear enough to estimateright-sized against the model's reliability horizon (§2) — and note: estimability serves the human slicer; AI's own estimates ran at 16% accuracy (§3)
Smallfits in an iterationfits a context window with room to work — the item plus its linked context plus the code it must read, before context rot (E2)
Testableyou know when it's donemachine-verifiable AC (EARS/Gherkin, H2) — the property that separates supervised assistance from delegable work (E4, E6)

For our rollout: INVEST is the vocabulary the client's team already knows. Presenting agent-readiness as a re-reading of it — rather than a new methodology — is the cheapest possible change-management move: same checklist shape, sharpened definitions.

2Right-sizing — the 10–30 line rule and the capability ceiling

Our working heuristic (Report 2 — practitioner synthesis, not a measured result): a work-item spec of roughly 10–30 lines. Under ~10 lines, constraints and non-goals are usually missing — the agent fills the gaps with guesses. Over ~50 lines, you are over-specifying: the spec starts to exceed the code it describes, and review burden has been relocated, not removed (H4 measured AI review commentary at ~29.6 tokens per line of code vs 4.1 human). And tasks under ~15 minutes need no spec at all — a conversational instruction is cheaper than the ceremony.

Both failure directions are real and show up as recognisable smells: under-specification is the 30-minute feature that balloons to 3 hours while the agent rewrites half a module off a misread intent; over-specification is the task whose spec took longer to review than the diff it produced.

Is there independent evidence for a size ceiling? The closest is METR's time-horizon research (arXiv:2503.14499): measure the human clock-time of tasks an agent completes with 50% success. That horizon has doubled roughly every 7 months since 2019, accelerating to ~4 months or faster on 2024+ models; by May 2026 the frontier estimate reached ≥16 hours (Claude Mythos Preview — wide confidence interval; METR itself flags measurements above 16 h as unreliable on the current suite). But two caveats do the real work here. First, the 80%-success horizon is roughly 5× shorter than the headline 50% one. Second, the benchmark is well-scoped, automatically gradable software tasks — real backlog items are messier, so real horizons sit lower still. All figures point-in-time, mid-2026; re-verify before quoting.

The sizing rule that falls out

Size work items to the reliability horizon, not the headline horizon. A delegation that succeeds half the time doesn't halve your work — it doubles your review load and erodes trust in the whole rollout (E7's perception gap). Divide the marketing number by five for the 80% horizon, discount again for messiness, and you land at: hours, not days — which is exactly where the 10–30 line spec heuristic already points. The two rules agree from independent directions.

3The evidence — splitting helps, estimating fails

Decomposition has one controlled experiment on record, and it is worth citing precisely: Pavlič, Saklamaeva & Beranič, Applied Sciences 2024 (University of Maribor; 14(24):12006). Thirteen three-person teams (39 developers analysed) upgraded a Jakarta EE system against 8 user stories; experimental teams had GitLab Duo Chat split the stories into tasks and estimate them, and could adjust prompts but not overrule the tool.

What it found — strong independent evidence

Estimation failed: AI accuracy 16% (SD 9.2) vs 60% (SD 24.5) for human control teams — the paper calls 16% insufficient for industry standards, and 60% industry-typical. The AI overestimated ~4× (13.55 h vs 3.37 h per story) while actual effort was nearly identical across groups. Splitting helped, with a filter: AI decomposition was finer-grained (5.4 vs 3.2 tasks per story) and caught overlooked work — but the lists carried generic, project-irrelevant items, and the authors' recommendation is to start from your own list and use the AI's to spot what you missed. Post-test preference: conventional-only 0%, AI-only 0%, hybrid 100%.

The same group's 2026 follow-up (Splitting User Stories Into Tasks with AI — A Foe or an Ally?, arXiv:2605.07320) sharpens the picture: teams implemented only ~59% of AI-generated tasks versus 100% of manually authored ones, and just 10% of participants found the AI tasks more relevant than their own (55% disagreed) — yet participants still favoured the hybrid workflow. Note honestly: this is the same Maribor group, a follow-up rather than an independent replication — and as of mid-2026 controlled evidence on AI backlog decomposition is essentially this one line of work. Estimation research is broader (zero-shot LLMs now beat older supervised baselines on story-point benchmarks — Shetty et al., arXiv:2603.06276), but benchmarks are not your backlog; treat vendor claims about "AI sprint planning" accordingly.

Read through P2's lens, the result is the same asymmetry: splitting is expansion — generate more candidates from the same input, a job models do well with a human filter; estimation and prioritisation are judgment — they price in context the model does not have, and fail without it. Point the model at expansion; keep judgment human.

4The amnesiac colleague — context links and the agent-ready checklist

The deepest difference between handing a task to a teammate and to an agent: the agent has total amnesia. It was not in the refinement call, does not remember the ticket from March, and will not lean over to ask the senior at the next desk — every clarification is a round-trip that spends the autonomy you delegated for. Report 2's observed failure mode follows directly: given an item without context, agents reverse-engineer intent from code comments — they excavate whatever fragments the repo happens to contain and guess the rest. The fix is not a longer spec; it is context links — pointers to the code, the design, the ADR and the requirement, so the agent assembles fresh, correct context per session (E2) instead of guessing.

That completes the agent-ready checklist — the deployable artifact of this topic, and deliberately the same six components as the handoff contract (H1) at task grain:

Agent-ready = all six present
  • intent/why — one human-written sentence (the rationale gap, P2)
  • scope + explicit non-goals — or the agent over-engineers
  • machine-verifiable AC — Gherkin/EARS (H2)
  • constraints — tech, security, performance
  • context links — code, design, ADR (the amnesia antidote)
  • DoD + requirement ID — traceability travels with the task (H3)
Right-sized = three checks
  • spec ~10–30 lines; under 10, look for the missing constraints

Enforce it as a gate, not a guideline

Most of the checklist is deterministically lintable (E5's advisory-vs-deterministic rule): a non-goals section exists, AC parse as Gherkin, context links resolve, a requirement ID is present. A template plus a refinement-bot check makes those structural. What stays human is exactly the judgment part: is the seam right, is the why true, is this really one task. The gate produces its own evidence trail — useful later for E7's measurement story.

5How the SDD toolchains decompose — and what to take

Both major spec-driven toolchains treat decomposition as a first-class, reviewable step (all details point-in-time, mid-2026). GitHub Spec Kit: /speckit.tasks generates a tasks.md — dependency-ordered, grouped by user story, each line - [ ] T001 [P] [US1] description + exact file path, where [P] marks tasks parallelisable because they touch different files with no dependencies; a separate command converts tasks to GitHub issues. Its template states the bar in one line: tasks must be "specific enough for an LLM to complete without additional context" — the tersest definition of agent-ready we have found. Amazon Kiro: requirements.md (EARS) → design.md → tasks.md with review between phases; each checkbox task carries nested sub-tasks and a trailing traceability line (_Requirements: 1.1, 1.5_), and independent tasks can run concurrently.

What we extract is portable without adopting either tool. First, the task list is itself a reviewable artifact — decomposition gets reviewed before any implementation starts, the same inversion E8 applies to specs. Second, independence is made inspectable: explicit dependency ordering and parallel markers turn "these are independent" from a hope into a claim a reviewer can check. Third, traceability travels per task, not per epic (H3). This is what P4 on the assistance scale — AI-driven decomposition with approval gates — looks like when it is done right.

The parallelism trap

Parallel markers make fan-out tempting: ten independent tasks, ten agents. But the binding constraint is not task independence — it is review capacity (E7: never scale agents beyond it). Decomposition sets the ceiling on parallelism; the review bottleneck sets the budget. Slice for independence, then spend the budget deliberately.

P4 · DEEP DIVE

The PM: From Author to Editor and Curator

Study companion · module M2

0Why the role shift is the load-bearing wall

Every gate this module has installed assumes a human who judges: P1 compares two syntheses, P2 kills ideas and owns the why, P3 owns the seams and sizes. P4 asks the uncomfortable follow-up — who is that human, what happens to their job, and what keeps their judgment alive once most of their day is reviewing AI drafts instead of writing their own? The product harness fails socially before it fails technically: if review quietly degenerates, every upstream gate — kill/keep, red-team disposition, the agent-ready checklist — keeps running as theatre.

The frame the industry converged on is the editor: Lenny Rachitsky put it early — "PMs will become editors of super-intelligent suggestions" (April 2024). We extend it with a second half, the curator (§5) — that extension is our synthesis, not an industry standard. Both halves name the same shift: the PM's value moves from producing artifacts to judging them and tending the context they are produced from.

The one idea to internalise

Author → editor is not a smaller job; it is a different job under active attack. Two documented forces work to hollow it out: automation bias (§2) erodes the quality of review, and losing user contact (§3) erodes the judgment review runs on. Neither announces itself — both look, from the outside, like a team happily shipping faster. The toolkit in §4 exists because good intentions measurably do not survive contact with a mostly-right machine.

1Which PM gets automated — Cagan's distinction

Marty Cagan's line is the sharpest on record, and worth quoting exactly (A Vision For Product Teams, svpg.com, February 2025): "For product owners and feature team product managers, where the job is primarily about delivery, and driven by a delivery process, it seems pretty clear to me that there will be very substantial productivity improvements, if not full automation." He calls these roles "very vulnerable in this timeframe" and urges the people in them to upskill. The other side of the same distinction (Preparing For The Future, 2023): "if you're a product manager on an empowered product team, then your role is likely to become more critical rather than less."

2Risk 1: automation bias — review is the only gate, and bias erodes it

Automation bias is one of the best-replicated findings in human-factors research, and it predates LLMs by decades: give people a mostly reliable automated aid, and they stop checking. Skitka, Mosier & Burdick (1999) measured both failure directions — omission errors (missing what the aid fails to flag) and commission errors (following the aid against contradicting evidence) — and found unaided participants monitored better than aided ones. The expert data is worse: in Mosier et al.'s cockpit study (1998), professional pilots showed a 55% omission error rate, virtually all of them shut down an engine on a false automated fire warning despite contradictory gauges — and more flight experience meant more susceptibility, not less. Parasuraman & Manzey's review (2010) adds the finding that matters most for process design: the effect hits novices and experts alike, gets worse under multi-task load, and is not prevented by training or instructions.

The LLM era confirms the mechanism transfers. Lee et al. (CHI 2025, Microsoft Research; 319 knowledge workers, 936 real GenAI work examples): higher confidence in the AI correlates with less critical thinking, while higher self-confidence correlates with more — self-reported and correlational, but pointing exactly where the cockpit studies point. And skills measurably decay when the aid does the detecting: the Lancet ACCEPT data (Budzyń et al., 2025) found endoscopists' unassisted adenoma detection rate fell from 28.4% to 22.4% after just three months of AI-assisted colonoscopy — the first real-world deskilling result tied to patient outcomes. In our own field, Perry et al. (ACM CCS 2023) found AI-assisted participants wrote less secure code and believed it was more secure — the same confidence inversion E7 documents as the perception gap.

The product-layer twist is structural, and it is why this topic exists: engineering review has a deterministic backstop — if a human rubber-stamps a bad diff, tests and CI still stand behind them (E4). Product review has none. Verification in the product layer is human judgment (P5), so when review degenerates into rubber-stamping, the layer has zero remaining controls. The observable signals — offered as our synthesis, since no independent telemetry on AI-draft review exists as of mid-2026: the proposal kill rate falling toward zero (P2), near-100% acceptance of AI red-team dispositions, and review time per artifact collapsing while artifact volume grows.

3Risk 2: losing user contact — the judge goes blind

The second risk attacks the input side. The editor's judgment is fed by direct user contact; outsource that contact and the judge P1–P3 depend on goes blind while continuing to approve things. Teresa Torres has the most concrete data anyone has published on the failure (Product Talk, October 2025): when she had ChatGPT summarise her own product interviews, roughly 30% of the direct quotes it produced were incorrect or fabricated. Her verbatim conclusion is the whole risk in one sentence: "When we outsource synthesis to AI, our synthesis skills start to atrophy from lack of use." — the PM version of the Lancet result in §2.

Attribution honesty, because this topic gets quoted onward at clients: the widely-shared line "you're not doing discovery, you're generating summaries" comes from a write-up of Torres's webinar (airfocus/Lucid, April 2026) — the frame is faithful to her position, but the phrasing is the write-up author's, and the circulating "AI synthesis misses 20–40% of important detail" figure is a newsletter paraphrase, not a measured result. What is safely hers: the "lazy AI interview synthesis" anti-pattern (P1), the 30%-fabricated-quotes observation, and the atrophy warning. The chain the anti-pattern predicts: outsource analysis → lose the texture and the disconfirming detail → product sense atrophies (Lee et al.'s cognitive offloading, §2) → the editor is now approving drafts against a memory of users, not knowledge of them.

4The editor's toolkit — safeguards that actually bite

Parasuraman & Manzey's finding sets the design constraint: training and instructions do not prevent automation bias — so awareness campaigns and "review carefully!" guidance are advisory controls that will not hold (the same advisory-vs-deterministic lesson E5 teaches for engineering governance). The safeguards have to be structural: built into the process so that skipping them is visible, not virtuous-feeling. The encouraging result from the same research line (Skitka et al., 2000): accountability manipulations reduce automation bias — people who must justify their decisions to someone check the automation more. That is the cheapest lever we have.

Structural safeguards (each maps to a gate we already built)
  • disposition of findings — every AI red-team finding gets an explicit human keep/reject with a one-line rationale (P2 §4); the rationale is the accountability mechanism
  • mandatory spot-checks — PM reads N raw interviews per cycle and compares against the AI synthesis (P1's two-perspectives, made non-optional)
  • kill-rate watch — AI-drafted proposals that are never killed mean review has died (P2)
  • no-follow-up rate — handoff quality measured downstream, not by drafting volume (P2/P3)
Signals review is degenerating

The time-reinvestment rule

The hours the drafting assists save are the budget for keeping the editor's judgment alive. Spend them on discovery and user contact — Cagan's point that as AI compresses delivery, discovery becomes the main activity — not on producing more drafts. A team that converts saved hours into more artifacts is feeding the review bottleneck (E7) and starving the judgment that guards it; that is the P5 drift in slow motion.

5The curator half — context is the PM's second product

The editor judges the artifact stream; the curator owns what the stream is generated from. Every product-layer assist in this module drafts from durable context — the company-context file, the PRD template, the opportunity tree, the requirement registry (H3), the glossary. P3's amnesia argument applies with full force: agents read this context fresh every session, so a stale or wrong durable artifact poisons every downstream draft silently and at scale. Someone must own its truth. That someone is the PM — it is product knowledge, and nobody else will notice when it drifts (I3).

Concretely, the curator: holds the write-pen on durable product context (M4's write-permission discipline, I2 — agents propose, the PM approves what becomes durable); prunes requirements and opportunities that are no longer true; keeps the PRD template and red-team checklist current as the team learns (E8's compound loop, product edition); and treats the context files as a product with users — the users being every agent and human who drafts from them. Flag we repeat everywhere: the editor frame is Rachitsky's (2024); this curator extension is our synthesis — it follows from M4's architecture, but no industry source formalises it yet.

P5 · DEEP DIVE

The P1–P5 Assistance Scale in the Product Layer

Study companion · module M2

0Why the product layer needs its own scale

Module M2 ends where it began: with the question of how much of product work can be delegated. The tempting shortcut is to reuse the engineering autonomy ladder (E6's L1–L5) and rename the rungs. That shortcut fails on the one property that makes the engineering ladder work: verification. An engineering agent earns autonomy because deterministic tests, CI and mutation gates stand behind every level (E4); a product artifact has no such backstop — its verification is human judgment plus consistency checks, and that difference restructures the entire ladder. P1–P5 is the scale that results, and it doubles as the module's map: each topic you have read lives at a level.

Positioning, stated plainly: P1–P5 is our synthesis, like the D1–D5 design scale — as of mid-2026 we found no published equivalent that grades AI assistance across product-definition work. It has honest intellectual ancestors: Parasuraman, Sheridan & Wickens's classic ten-level automation model (2000), Feng, McDonald & Zhang's five human-role levels for AI agents (Operator → Observer, arXiv:2506.12469), and Mitchell et al.'s Hugging Face position paper arguing the fully-autonomous top level should not be built at all (arXiv:2502.02649). The nearest SDLC-space neighbour (ASDLC.io's L1–L5) explicitly scopes itself to the engineering layer. Notably, the challenger rung — our P3 — appears in none of them.

The one idea to internalise

Andrej Karpathy compressed the whole argument into one line at YC's AI Startup School (June 2025): build autonomy sliders and keep AI "on a leash," because generation is cheap and verification is expensive — the human verification loop is the bottleneck. The P-scale is that slider for the product layer, calibrated to the only verifier the layer has: human judgment. Set the slider by what you can verify, not by what the model can generate.

Second, the levels describe who initiates and who judges, not model quality: P3 uses the same model as P2 — pointed at critique, where the evidence says it is stronger (§3).

2The structural difference — verification asymmetry

E6's rule for engineering is: autonomy is earned, level by level, by the strength of the verification loop behind it — tests, CI, protected suites, mutation gates. The product layer cannot earn autonomy the same way because no test can check a product decision. What it has instead: human judgment (expensive, biased under load — P4), consistency checks (cheap but shallow), and downstream signals that arrive weeks later (E7). That is why the product ladder caps earlier, and why the top rung is a different kind of thing: L5 in engineering is a risk calibration; P5 in product removes the only verifier the layer has.

The strongest independent support for treating decision work differently comes from outside our field: Vaccaro, Almaatouq & Malone's meta-analysis in Nature Human Behaviour (October 2024; 106 studies, 370 effect sizes) found that human–AI combinations performed worse than the best of human or AI alone on decision tasks (Hedges' g = −0.23) while outperforming on creation tasks. Read against the P-scale: the naive comfort that "there's a human in the loop" does not rescue decision quality by itself — the pairing has to be designed around who verifies whom. Drafting and challenging (P2–P3) sit on the creation side of that split; deciding (P5) sits squarely on the losing side.

3Evidence by level — and the eval loop that is emerging

The scale is only useful if the rungs are evidence-rated, so: P2 carries the best-measured failure mode — LLM-authored stories fail review at 63–77% and fail hardest on rationale (Quattrocchi et al.; P2 topic §2). P3 is the evidence-strongest rung: ambiguity classification improved +20.2% with 10-shot prompting on industrial requirements (Bashir et al., ICSME 2025, Alstom/Westermo data — best F1 roughly 67–76% depending on dataset, expert rating 3.84/5 across four criteria, measured on small open-weight models: an assistant, not an arbiter), clarifying-questions loops lifted GPT-4 Pass@1 from 70.96% to 80.80% on MBPP-sanitized (≈11.5% relative improvement on average, with simulated user answers — ClarifyGPT, FSE 2024), and grounded AC generation gained +23.2 pp with multi-modal RAG (Wang et al., ASE 2025). P4 has one controlled experiment: decomposition helps as a completeness check with a human filter, estimation fails outright, and 100% of participants chose the hybrid (Pavlič et al.; P3 topic §3). P5 has no supporting evidence at all — what it has is the automation-bias literature predicting exactly how it degrades (P4 topic §2).

Two 2026 results sharpen the P3 picture in opposite directions. Strengthening it: Unterbusch & Vogelsang (ICSE-NIER 2026, Mercedes-Benz data) found that feeding the model a handful of human-validated explanations — as few as ~20 — beats both few-shot prompting and a fine-tuned baseline at defect prediction: the strongest configuration of the critic has a human inside it. Cautioning it: Yang et al.'s Orchid benchmark (1,304 tasks) shows LLMs flag ambiguity with high recall but heavily overpredict, and "lack the capability to identify or resolve such ambiguity autonomously" — flagging is not resolving, which is exactly why the disposition step in the red-team gate stays human (P2 §4).

The honest counter-trend: the product layer is growing a partial verification loop of its own — AI evals. Teresa Torres calls them the practice she would add a chapter for ("If I had to write my book today, I would add a chapter on AI evals" — webinar statement, 2025), and her own published loop is concrete: error analysis → measure → improve, run like discovery itself, cutting one error class from 4-in-~20 traces to 1-in-25 (Product Talk, Sep 2025). The skill is institutionalising fast — Husain & Shankar's evals course has trained 2,000+ PMs and engineers; OpenAI's CPO Kevin Weil calls eval-writing a core PM skill. Two sober caveats before anyone reads evals as "tests for product": there is no measured evidence yet that evals catch product-decision errors (as opposed to model-output regressions), and Shankar et al.'s Who Validates the Validators? (UIST 2024) documents criteria drift — graders cannot fully define quality criteria up front; grading outputs changes the criteria. That makes eval work a form of requirements discovery, which is an argument for keeping humans in it, not for automating them out.

4Why P5 is the anti-pattern — with names and dates

Assemble the pieces from the module and the failure is overdetermined. Remove the human and you remove the layer's only verifier (§2); automation bias guarantees the removal happens gradually and invisibly, through rubber-stamping rather than by decision (P4 §2); the rationale gap means the artifacts flowing downstream carry form without why, which the handoff contract cannot survive (P2, H1); and accountability evaporates — a shipped decision nobody made is a decision nobody owns when it fails. This is no longer just our synthesis: Mitchell et al. argue the fully-autonomous level should not be built as a general matter; and the EU AI Act's Article 14 makes effective human oversight — with explicit protection against automation bias — a legal requirement for high-risk systems from August 2026 (T4). In regulated domains, P5 is not an anti-pattern; it is non-compliance.

The named case study — Klarna's reversal

February 2024: Klarna's CEO announces its AI assistant is doing the work of 700 agents across 2.3M conversations — the customer-service version of autonomous product delivery, celebrated as a headcount decision. May 2025: the same CEO tells Bloomberg the cost-cutting "went too far," service quality dropped, and Klarna is rehiring humans into a hybrid model. Read through this topic: the decision was made where verification was weakest (customer-experience quality — human judgment territory), the degradation was gradual, and the correction required the kind of public reversal most organisations avoid until forced. That is the P5 failure shape, observed in the wild with names and dates.

Is the anti-pattern permanent? Honestly: unknown, and the claim is point-in-time. What would have to change is specific, though — mature eval suites that demonstrably catch decision-grade errors (not just output regressions), outcome telemetry loops fast enough to act as verification (E7's lagging-signal problem), and organisational answers to the accountability question. None of the three exists today; watch the evals movement (§3), because that is where the first one would come from. Until then the ceiling is P4-with-gates, and the gates are human.

5Choosing the level — per artifact, with downshift triggers

The deployment rule mirrors E6: start where verification exists, and move one rung at a time. In practice: interview synthesis runs at P2–P3 with mandatory spot-checks against raw material (P1, P4 §4); PRD drafting runs at P2 with the kill/keep review (P2 §3); backlog refinement adds the P3 red-team gate — the single best quick win, because it is the evidence-strongest rung and only one step beyond mainstream practice; decomposition runs at P4 with approval gates on the task list (P3 §5). Machine-verifiable acceptance criteria (H2) are the bridge rung: the one place a product artifact crosses into deterministic verification — which is why the contract invests there.

And because the failure mode is drift rather than decision (P4), every level assignment needs its downshift triggers — the degeneration signals watched per artifact type: kill rate falling toward zero, dispositions turning into "ok", review time collapsing while volume grows, PMs unable to recount a recent user conversation. Any of them fires: drop the artifact one rung and rebuild the review habit before climbing again. The scale is a dial, not a ratchet.

The evidence rates the rungs: P3 is strongest (+20.2% ambiguity detection), P4 works only as a gated hybrid (100% preference), and P5 has no evidence, a meta-analytic warning (human+AI underperforms on decision tasks, g=−0.23), a named failure (Klarna's 2025 reversal), and a regulatory floor against it (EU AI Act Art. 14, Aug 2026). Set the slider by what you can verify, per artifact, with downshift triggers — generation is cheap, verification is expensive, and the dial must move both ways.

The Design System as the Third Handoff Artifact

Study companion · module M6

0Why the handoff needs a third artifact

The handoff to an agent, as modules M1–M3 built it, runs on two artifacts. The handoff contract (H1) tells the agent what to build: intent, machine-verifiable acceptance criteria in EARS or Gherkin, constraints, non-goals. The codebase with its engineering harness (E1, E4) tells it how to build and how to prove it: conventions, tests, CI, lint, hooks. For a service endpoint or a data migration, those two are the whole story. The moment the work has a screen, a third question appears that neither artifact answers: what must the result look like? This topic's thesis — and the thesis of module M6 — is that the design system is the third handoff artifact, with one sharp caveat: it only functions as one in a specific form — machine-readable in what it offers the agent, machine-checkable in what it forbids.

Positioning, stated plainly and with the same flag as the D1–D5 scale: "the design system as the third handoff artifact" is Report 5's synthesis — our framing, not an industry term. What is not synthesis is the movement underneath it. Brad Frost — who named atomic design — wrote in December 2025 that design systems must "evolve into machine-readable infrastructure—or risk becoming artifacts of a bygone era," and had already called AI "a natural design system consumer" back in February 2023. The industry exhibits in §1 show Meta, Indeed and the open-source default all converging on the same conclusion from three different directions. The framing is ours; the direction of travel is not.

The delegation test

A UI task delegated without the third artifact fails in one of two ways: the agent invents the visual answers itself — §2 measures what that convergence looks like — or it interrupts a human for every choice, which is not delegation. The design system is what makes visual work delegable at all: it plays for the look of the product the role the test suite plays for its logic (E4). No suite, no delegation; no system, no visual delegation.

1From human library to machine-readable infrastructure

The classical design system was built for people: a component library, a documentation site, a Figma library, and the shared understanding that grows around them. Its economics were justified by human reuse — fewer redesigns of the same button, faster onboarding, brand consistency across teams. Agents change the audience. In the Storybook/Chromatic webinar Agentic Design Systems in 2026 (December 12, 2025 — spoken remarks, not a written paper), Frost and Chromatic co-founder Dominic Nguyen reduced agentic success to two pillars: Coverage — giving agents clear examples, states and constraints they can reuse — and Validation — tests plus human sign-off guaranteeing that what an agent generated is correct and safe to ship. Note what those two pillars are: Coverage is context engineering (E2) and Validation is the verification loop (E4), applied to the visual layer. The design system stops being a library with documentation and becomes the same kind of thing the engineering harness is — infrastructure that constrains and verifies a generator.

This is no longer a designer-blog aspiration; the named exhibits span three different corners of the industry, all point-in-time mid-2026. Meta's Astryx (public beta June 2026) ships an eight-year-internal React/StyleX design system explicitly as agent infrastructure: a CLI, an MCP server, and a self-describing JSON manifest of every command and response type — "an OpenAPI spec for the CLI" — so agents read structure instead of scraping help text (vendor announcement; verify details before quoting numbers). Indeed parsed 77 components from MDX docs into JSON metadata — purpose, variants, tokens, relationships and, notably, explicit anti-patterns — after benchmarking showed structured JSON beat Markdown docs on accuracy and token cost (§4). And shadcn/ui, the de-facto default of every vibe-coding tool, positions "open code the LLM can read" as its core selling point, publishing component schemas and agent skills so models know what exists and how to compose it. The strategic implication of the third exhibit cuts both ways: agents ship somebody's design system by default — if yours is not equally machine-legible, the default wins and it is not yours (§2, §6).

2Without it: distributional convergence and AI slop

What happens when the third artifact is missing is not random ugliness — it is convergence. A model given no visual direction does not improvise; it samples the statistical centre of its training distribution, and the centre is the same for everyone: Inter, indigo-to-purple gradients, glassmorphism, a grid of feature cards with an icon on top. The community named the result "AI slop," but the mechanism matters more than the insult: slop is what maximum-likelihood design looks like.

The best independent measurement to date is Adrian Krebs's audit (adriankrebs.ch, April 20, 2026 — Krebs is Kadoa's co-founder, but the audit is method-published and deterministic): a Playwright script walked 1,590 Show HN landing pages and checked computed styles against 16 deterministic DOM/CSS patterns. Result: 22% heavy slop (4+ patterns), 32% mild (2–3), 46% clean (0–1). The most common tells: permanent dark theme with medium-grey body text (34%), gradient backgrounds (27%), identical icon-card feature grids (22%). Over half of everything shipped showed measurable convergence. The academic literature backs the mechanism: Shin et al.'s review of design homogenization in vibe coding (arXiv:2603.13036, March 2026; 63 sources plus walkthroughs of six platforms) argues that frictionless generation actively enables homogenization — models regress to dominant, Western-centric conventions and override regional norms, and the authors propose reintroducing "productive friction." Which is, read through this workbook's lens, exactly what a design system with deterministic gates is.

The instinctive fix — "write a better prompt" — fails for a reason this programme has already named twice (E5, S3): a prompt is advisory. It is re-interpreted every session, it drifts, it loses to whatever defaults are baked into the tool (v0 is welded to shadcn + Tailwind; Lovable only respects your system if you hand it a machine-readable design-system.json). Style constraints stated in prose share the fate of every advisory rule in the harness: followed usually, violated silently, unenforceable at scale. The only reliable form is the pairing the engineering harness uses — explicit machine-readable direction (tokens, components, DESIGN.md) plus deterministic gates that reject violations (Stylelint token bans, visual regression, axe — D5).

Slop is a symptom, not a model defect

Krebs's 46% clean pages came from the same generation of models as the 22% heavy-slop ones. The variable is not the model; it is whether anything constrained it. Diagnose slop as a missing-constraint symptom — the visual-layer equivalent of an agent hallucinating architecture in a repo with no CLAUDE.md, no types and no tests. The cure is the same class of thing: infrastructure, not exhortation. (Synthesis — our reading of the audit plus the homogenization literature.)

3Anatomy of the third artifact — five layers

"Design system" covers everything from a Figma file to a component monorepo, so the thesis needs precision: what exactly must exist for the system to function as a handoff artifact? Five layers, ordered from most machine-checkable to least. The first four are what Frost's "machine-readable infrastructure" concretely means; the fifth is the layer that stays irreducibly human.

LAYER 1

Design tokens — the vocabulary

gate: Stylelint / build

Named, typed values for color, type, spacing, elevation — the atoms every visual decision must be spelled in. The W3C Design Tokens Community Group shipped its first stable format version 2025.10 on October 28, 2025: vendor-neutral JSON with $value/$type, aliases and theming. (Precision, because we teach source hygiene: a Community Group deliverable under a Final Specification Agreement — not a W3C Recommendation or formal standard.) Tokens are the layer with the cleanest deterministic gate: a Stylelint ban on hardcoded hex plus an allowed-list forcing var(--…) makes "use the vocabulary" a build failure instead of a review comment. Full treatment: D2.

LAYER 2

Component APIs & metadata — the grammar

gate: types / interaction tests

Props, variants, states and deprecations as typed contracts. For the agent this is the difference between "there is a Button" and "Button takes variant: primary|secondary|danger, never a color prop" — TypeScript and interaction tests fail on a hallucinated prop the way a unit test fails on a wrong return value. Deprecation metadata matters as much as the API itself: agents must be able to see what is no longer current, or they resurrect it (governance and versioning: D4).

LAYER 3

Code mappings — the bridge

gate: import lint / build

The mapping from design artifact to your actual code. Figma Code Connect maps a Figma component to the real component — exact import path, prop mappings — for React and React Native, SwiftUI, Jetpack Compose and HTML (the HTML integration covering Web Components, Angular and Vue), plus Storybook. This layer kills the two worst agent failure modes at once: hallucinating components that do not exist and re-implementing ones that do. Paired with no-restricted-imports, "use the system's Button" becomes checkable. Consumption channels in depth: D3.

LAYER 4

Examples & stories — the coverage

gate: run-story-tests / axe

Stories are the Coverage pillar made concrete: runnable, testable usage examples an agent can read and execute — Storybook's MCP addon exposes component manifests and can run story tests including accessibility checks, closing a generate → test → fix loop before human review. Indeed's version of this layer is instructive: each component's JSON metadata carries purpose, variants, token references, relationships and explicit anti-patterns — the system says not only what to do but what it has seen go wrong. Coverage is what stops an agent from guessing; §4 shows it is also where the measurable wins live.

LAYER 5

Direction — the taste layer

gate: designer review only

Aesthetic direction, brand intent, the reasons behind the choices — typically a DESIGN.md next to the code. This layer is irreducibly advisory: no lint rule checks "feels like our brand." It still belongs in the artifact — it is the difference between an agent that stays inside the system and one that understands roughly why — but everything it governs must eventually pass the one gate that can judge it: a designer (D5, D6). Do not pretend this layer is machine-checkable; that honesty is what keeps the rest of the thesis credible.

Write-permission mapping, in one line (the D4/Report 3 taxonomy): layers 1–4 are durable — versioned in git, changed only by PR through a human gate, the agent proposes; generated CSS, Storybook builds and docs sites are derived — regenerate freely, never hand-edit; drafts and one-shot prototypes are disposable. The design system earns "artifact" status exactly to the degree its durable core is governed like source code.

4What the evidence shows — and what it doesn't

The claim "structured design context measurably improves agent output" has evidence behind it — of sharply varying quality, so we sort it the way E7 taught. The ceiling without structure first: Design2Code (Si et al., arXiv:2403.03163; 484 real webpages curated from C4) had annotators judge GPT-4V screenshot-to-code output exchangeable with the reference in 49% of cases and preferred in 64% — with the authors' own hedge that the preference may reflect models having absorbed "more modern and popular webpage design principles," and with failure concentrated in visual-element recall and layout. Read precisely: these are GPT-4V-era (early 2024) numbers on reproducing a given page — they measure raw capability without a system, and they say nothing about consistency across a product, which is what a design system is for.

The strongest practitioner data is Indeed's (Diana Wolosin; written up by Into Design Systems, April 2026 — practitioner benchmark, methodology described but not peer-reviewed, point-in-time): 77 components parsed from MDX into JSON metadata, then 1,056 prompts benchmarked across 8 MCP configurations. Structured JSON beat Markdown docs on accuracy (Markdown ran ~30k tokens per query at 82% coverage, with hallucinations), cut tokens per query by roughly 80%, and priced out at about a fifth of the annual cost — and the pipeline has since fed over 4,300 AI-generated prototypes in four months. This is the "machine-readable beats human-readable" claim with numbers attached, from a team that measured before believing. Early academic work points the same way — steering generated UIs with explicit preference and design context measurably shifts output toward the target (AlignUI, arXiv:2601.17614; early-stage, re-verify before quoting specifics).

And one correction we owe our own earlier drafts, because epistemic hygiene is the E7 habit: specific figures for Storybook's MCP benchmark (+12.8% code quality, 2.76× faster, −27% tokens) circulate widely — including in this workbook's research notes — but on re-verification (July 2026) could not be traced to any published source. Storybook's published materials (Nguyen, November 2025; the Reshaped RFC, September 2025) make the claim only qualitatively: better quality, faster, fewer tokens. We now cite it in that form — an unquantified vendor claim — and have flagged the numbers for removal wherever they appear as fact.

5The Gherkin of the visual layer — conformance, not judgment

H2 established why the handoff contract writes acceptance criteria in EARS or Gherkin: a requirement phrased as a machine-checkable pattern turns review from debate into pass/fail. The third artifact does the same for the visual layer, and the parallel is worth drawing exactly. "Use color.action.primary" is enforceable by lint the way "WHEN the user submits THEN…" is enforceable by a test runner. A component API is a typed contract; an unknown prop fails like a failed assertion. A visual-regression baseline plus axe-core in CI gives UI work the deterministic red/green loop that Report 1 made the heart of the whole method. This is why the design system deserves the name artifact in the same sense as the contract: not decoration around the work, but the checkable specification of part of it.

And here is where the analogy must be cut off honestly, because overselling it would break the module's credibility. The gates verify conformance — stayed inside the vocabulary, used real components, broke no baseline, met WCAG rules. They cannot verify judgment — whether the hierarchy guides the eye, whether the composition fits the brand, whether the design is good. Pixel-diff detects change, not quality; LLM-as-judge agrees with experts on specific UI behaviors, not holistic taste (D5 covers the FrontendBench evidence). Accessibility is the sharpest illustration of why the deterministic side must nonetheless be maximal: the CHI 2025 CodeA11y study (Mowar et al., DOI 10.1145/3706598.3713335) found AI assistants routinely omit accessibility features by default — alt text, semantic landmarks, labeled controls — and that developers neither prompt for them nor verify them (16-developer formative study). Everything checkable must be checked by a machine, precisely so the scarce human gate — the designer, sitting at the same L3-style review position as the engineer in E6 and the PM in P5 — spends its capacity on the one thing only it can do: judgment. The designer-as-curator role this implies is D6's subject.

The precise scope of the thesis

The design system is the third handoff artifact for conformance: it makes "stayed inside the system" machine-verifiable, which is what makes visual work delegable to an agent at any autonomy level above assisted drafting. It is not, and will not become, an oracle for design quality — that verification stays human, which is why every D-level in this module keeps a designer gate (D6) and why D5-the-level (autonomous design) remains an anti-pattern.

6Selling the third artifact to a brownfield client

The objection arrives on schedule: "A design system is a designer luxury — we'll invest after the AI rollout proves itself." That ordering is exactly backwards, and the reframe is the point of this section: for agentic frontend work the design system is not a design deliverable, it is agent infrastructure — the cheapest source of deterministic gates the UI layer can have. Three arguments, in the order that lands: (1) The slop bill. Unconstrained generation measurably converges (54% of Krebs's sample showed slop patterns), and every convergence the client does not want is caught, if at all, by the most expensive instrument they have — human review — one PR at a time. (2) Agent-speed drift. E6's review-capacity rule does not bend for the visual layer: agents produce UI faster than any design review scales, so without machine gates the choice is drift or bottleneck. (3) Verification-first economics. E4's argument, transplanted: tokens + lint + visual regression + axe convert visual review from subjective debate into red/green, and the agent iterates against red before a human looks. The engineering harness was sellable for the same three reasons; this is the identical pitch one artifact later.

Two honest boundaries keep the pitch trustworthy. When not to push it: a client with no meaningful UI surface, or work that is genuinely disposable prototyping (the D4 taxonomy's third tier — nothing durable depends on it). And never as a big-bang: the entry path on brownfield is D7's design archaeology — audit the CSS that exists (Project Wallace-style token extraction), snapshot the current state as a golden master, and then consolidate — F0 before F1, read before write, exactly like Report 4's bootstrap ordering. A client who hears "we start by measuring what you already have" trusts the rest.

Assess (F0 questions)

  • how many de-facto colors/spacings does the CSS actually contain vs how many are intended?
  • can an agent today discover what components exist (stories? types? manifest?)
  • who would hold the designer gate — and does design review exist at all?

0Why the contract decides whether delegation works

Module M2 ends with decomposed, agent-ready work items (P3); module M1 is the harness that executes them. Between the two, exactly one artifact crosses the boundary — the handoff contract. Everything the agent will ever know about why the work exists, where it stops, and how success is proved either rides in that artifact or does not reach the agent at all. A human teammate would fill the gaps from memory, hallway context and the ticket from March; the agent, as P3 put it, has total amnesia — it was not in the refinement call and will not lean over to ask the senior at the next desk. H1 dissects the contract itself; its siblings handle the acceptance-criteria formats (H2), the traceability and lifecycle (H3), and the feedback loop with its anti-patterns (H4).

This topic also carries direct engagement value: the roadmap marks the contract anatomy as a directly reusable client template. What follows is written to be lifted — the six components in §2, the review checklist in §3 and the sizing tiers in §5 are the pieces a client team can adopt in a week.

The amnesia test (our synthesis)

A handoff contract is the smallest durable artifact that lets a competent stranger with no memory of your project build the right thing — and prove it. Every component in §2 exists because some clause of that sentence fails without it: no intent and they build the wrong right-looking thing; no boundaries and they build too much; no executable criteria and nobody can prove anything; no links and they guess the context; no definition of done and "finished" is an opinion.

1From prompt to contract

The clearest statement of the shift this module is built on came from Sean Grove of OpenAI at the AI Engineer World's Fair in June 2025, in a talk titled The New Code: specifications, not prompts and not code, are becoming the fundamental unit of programming. Two of his lines do real work here. First, "code itself is actually a lossy projection from the specification" — the intent is the asset, the diff is one rendering of it. Second, his jab at throwaway prompting: teams "keep the generated code and delete the prompt", which he compares to shredding the source and carefully version-controlling the binary. That is a manifesto, not a measurement — Grove sells a worldview, and we flag it as such — but it names precisely what the contract fixes: the prompt is ephemeral, unreviewed and unowned; the contract is durable, reviewed and owned (M4). His worked example, OpenAI's own Model Spec, is anatomically instructive too: every clause carries a unique ID plus example prompts that act as unit tests for the principle — the same clause = ID + example-as-test pattern our acceptance-criteria component uses (H2) and our traceability chain depends on (H3).

The aggressive end of the same argument is Tessl (Guy Podjarny), whose spec-centric thesis says comprehensive specs and tests make the code itself disposable, and whose Spec Registry packages specifications as versioned, dependency-managed artifacts — "npm for specs". Vendor claims, point-in-time, and further along the spec-as-source spectrum than we recommend (H3 argues for spec-anchored) — but useful as evidence that specs are becoming managed artifacts with tooling, not just documents.

The contract is
  • a durable artifact (M4): markdown in the repo, versioned, reviewed like code
  • written before implementation and agreed at a review gate (§3)
  • the unit of delegation — what an L3 agent (E6) is held accountable to
  • one task's worth of intent, boundaries and proof (10–30 lines, §5)
The contract is not
  • a prompt — ephemeral, unreviewed, deleted after use (Grove's shredded source)
  • the PRD — wrong altitude; the PRD spans a problem, the contract spans one task, and it cites the PRD rather than copying it (I1: one master, pointers elsewhere)
  • a ticket written for a colleague — that format assumes shared context the agent does not have
  • AGENTS.md / CLAUDE.md — standing repo-level context (build commands, conventions, boundaries), the layer the contract's context links point into; a per-task contract and the standing substrate are complementary layers, not the same artifact (agents.md emerged Aug 2025 as a shared standard across OpenAI, Amp, Google, Cursor, Factory)

2The six components

Provenance, stated plainly

The six-part anatomy below is Report 2's synthesis, distilled from the 2025 spec-driven tool wave — not an industry standard, and no tool ships all six under these names. Re-verified July 2026: Spec Kit's current template (v0.12.x) structures a spec as user scenarios with priorities, functional requirements, success criteria and assumptions — and has meanwhile dropped its earlier dedicated out-of-scope section (checklists moved to a separate /speckit.checklist command); Kiro splits the same content across requirements.md (EARS), design.md and tasks.md with explicit _Requirements: 1.1_ back-links; ASDLC frames the handoff as an "objective packet" of intent + constraints. The convergence is one of function, not of field names. The closest in-the-wild mirror is Cognition's guidance for writing Devin tasks — context, numbered checkable acceptance criteria, files not to touch, environment hints, a closing step — which lands on nearly the same six. Tool facts are point-in-time; re-verify.

A

Intent / why

What problem, for whom, and why now — two or three sentences, not a section. Without it the agent optimizes the letter of the acceptance criteria, and the reviewer loses the one lens that distinguishes a plausible-wrong solution from a right one; interpretive drift then compounds silently downstream (H4). The evidence says this is exactly the component to guard: Quattrocchi, Pasquale, Spoletini & Baresi (arXiv:2507.15157) had three annotators score 153 user stories on the QUS framework's semantic dimensions and found Rationale Clarity the systematic weak spot of LLM-generated stories — GPT-4 scored 1.54 and GPT-3.5 1.25 against 2.66 for ground truth and 2.68 for students — structurally sound stories with hollow whys. The why is the field a human writes, or at minimum verifies word by word. owner: PM

B

Scope & non-goals

Where the task stops: what is explicitly out, which files must not change, what the follow-up task will handle instead. The failure mode when it is missing — the agent defaulting to a "maximally complete implementation" and over-engineering — is practitioner synthesis, not a measured result; but scope creep is a consistently named failure mode across 2026 practitioner write-ups and Anthropic's 2026 Agentic Coding Trends report, and Cognition's Devin guidance is blunt about the mechanism: name what must not be touched or the agent will "tidy things you did not ask it to". Non-goals are also the cheapest schedule protection the contract offers — H4's ballooning 30-minute feature usually traces back to a boundary nobody wrote down. owner: PM (product boundary) + tech lead (technical blast radius)

C

Acceptance criteria

The proof obligation, machine-verifiable wherever possible: EARS at the requirements level, Gherkin at the criteria level, failing tests as the handoff itself (H2 covers the formats). Two crafts distinguish a good set. First, define the failure paths, not just the happy one — what the system shows when the service is down, what the agent outputs when the result set is empty. One such line prevents an entire category of silent failures (TASK-042's network scenario in §6 is the worked example). Second, respect that criteria quality is test quality: the SWE-Bench+ audit (arXiv:2410.06992) found 32.67% of claimed benchmark passes tainted by solution leakage and another 31.08% passing only because the tests were weak — a weak oracle returns confident false greens, which is why E4's verification-first stance and B3's mutation gate back this component. owner: collaborative — tester lens drafts boundary/negative scenarios, tech lead ratifies verifiability

D

Constraints

The budgets and prohibitions the solution must respect: "validation < 300 ms", "fail-safe on timeout", "never log full codes". Two upgrades make a flat constraints list work harder. Addy Osmani's three-tier boundary pattern (Jan 2026) grades them — always allowed · ask first · never — which tells the agent not just what the rules are but how autonomous it may be near them. And E5's advisory-versus-deterministic principle applies verbatim: any constraint that can be checked mechanically belongs in a hook or CI gate, not only in prose — the "never" tier is the natural hook candidate. Constraints the harness enforces are constraints the reviewer can stop re-litigating. owner: tech lead

E

Context links

Pointers, not payloads (E2): the code paths involved, the ADR that explains the standing decision, the design reference, the requirement ID. Report 2's observed failure mode for items shipped without context is that agents reverse-engineer intent from code comments — they excavate whatever fragments the repo contains and guess the rest. Links let the agent assemble fresh, correct context per session instead; they also keep the contract honest about masters — the PRD stays in its system of record, the contract cites it (I1's one-master rule) — and they make traceability nearly free, because the cited requirement ID is the same one that travels into branch, commit and PR (H3). owner: whoever owns each linked artifact; the contract only points

F

Definition of done

The standing quality bar, distinct from the task's acceptance criteria: AC prove this task's behavior; the DoD is what every task must clear — tests green, edge cases addressed, traceability links present, docs touched if interfaces moved. The provenance is worth citing precisely: the Definition of Done has been part of Scrum since the first Scrum Guide (2010), and the 2020 revision made it formally the commitment attached to the Increment — for agent work, read: the Increment's commitment becomes the PR's merge gate. Because it is standing, it belongs in the repo once (referenced from each contract, never copy-pasted into it) — a derived duplicate would drift (I1). owner: tech lead; stable across tasks

3The gate inversion — reviewing the spec before the code

The structural bet of the whole M3 module: with an agent implementing, the highest-leverage human review moves left — from the diff to the contract. Both SDD reference tools embody it (Kiro's phases each end in an approval gate; Spec Kit's workflow agrees a spec before /speckit.implement), and Report 2 adopted it as the "key inversion": spec review precedes — and partially displaces — code review as the primary quality gate. Say it precisely, though: neither GitHub nor AWS states that spec review replaces code review; the inversion is an emerging practitioner claim, and code review does not disappear — it gets lighter because intent questions were settled while they were still cheap to settle.

Does the gate actually catch defects, or is it ceremony? The strongest available number is vendor-reported but of the right kind: Kiro's Requirements Analysis (a 2026 addition, announced around AWS Summit New York in June 2026 — distinct from the Nov 17, 2025 GA, whose spec-verification feature was property-based testing) pairs an LLM with an SMT solver to translate EARS requirements into formal logic and prove contradictions and gaps before code; across 35 internal AWS projects and 1,400+ acceptance criteria, roughly 60% of first-draft requirements needed refinement. Spec Kit ships the same instinct as /speckit.checklist — generated checklists it describes as "unit tests for requirements writing". And the lightweight, no-new-tool version is Anthropic's own workflow guidance: interview the human until the questions run dry, write a complete SPEC.md, review it, then execute in a fresh session. All point-in-time tool facts; re-verify.

The counter-view — take it seriously

Martin Fowler, after trying all three SDD tools: "To be honest, I'd rather review code than all these markdown files." The gate can relocate review burden instead of removing it — the same pathology H4 files under over-specification, and the review-density numbers make it concrete: AI-generated review commentary runs ~29.6 tokens per line of code against ~4.1 for human reviewers (arXiv:2603.15911 — a 2026 study of 278,790 review conversations across 300 open-source projects; AI-reviewing-AI sits between at 11.3). The defense is discipline, not enthusiasm: contracts stay 10–30 lines (§5), the spec review stays ten minutes, and the checklist stays seven questions.

The 10-minute spec review — seven questions

1 · Is the why present and would a stranger act differently for knowing it? 2 · Are non-goals explicit — could the agent know what not to build? 3 · Is every acceptance criterion falsifiable, and executable where possible? 4 · Are failure paths specified (empty result, timeout, unavailable dependency)? 5 · Which constraints can be enforced deterministically — and are they wired into hooks/CI rather than prose? 6 · Do the context links resolve, and is nothing copy-pasted from a master? 7 · Does done mean the standing DoD — not a per-task improvisation?

4Who writes it — ownership without a new silo

Report 2's split holds up: the PM owns what and why — intent, scope, the product half of non-goals; the tech lead owns verifiability and the DoD — that criteria are executable, constraints enforceable, links correct. Writing is collaborative, and the pattern for that collaboration is older than the tools: George Dinwiddie's Three Amigos (coined ~2009) — business, developer and tester examining a story together — maps one-to-one onto the contract's fields: the business lens supplies intent and scope, the developer lens supplies constraints and context links, the tester lens supplies the boundary and negative scenarios that make the acceptance criteria worth executing.

The honest caveat comes from the practitioner who owns this territory. Gojko Adzic — whose Specification by Example is the direct ancestor of spec-driven development — frames today's SDD as either that lineage "taken to a new level" or "the revenge of Waterfall", and warns it "could re-introduce some of the rigidity agile methods sought to escape". The variable that decides which, in his telling and ours, is exactly this section: whether the contract is the record of a collaboration (three lenses, one conversation) or a requirements document thrown over the wall with better formatting (H4's oldest anti-pattern). P4's editor-and-curator PM is the product-side half of the same discipline; P2's kill/keep habit applies to specs as much as PRDs.

5How big — the tiered contract

The sizing heuristic carried over from P3 (practitioner synthesis, not a measured result): a work-item spec of roughly 10–30 lines. Under ~10, constraints and non-goals are usually the components that go missing — the agent fills the gaps with guesses. Over ~50, the spec starts to exceed the code it describes and the review burden has been relocated, not removed (§3's counter-view, H4's over-specification). Below the artifact threshold entirely: tasks under ~15 minutes deserve a conversational instruction, not a contract — process must not smother small changes.

That answers this topic's standing open question — what is the minimal contract for a 30-minute task? — with a tier, not a number (our synthesis). The tier follows autonomy level and blast radius, not the calendar estimate: what can go wrong unattended, and how far it can spread, decides how much contract the task needs. Osmani's "one task focus" points the same way — feed the agent the relevant slice, not the binder; instructions compete with each other for attention.

The three tiers (our synthesis)
  • T0 — conversational. Pairing-mode work (L2, <~15 min): say it, watch it, no artifact.
  • T1 — light contract. The 30-minute task: one intent sentence, two or three checkable criteria, one non-goals line, links. Five to ten lines; still versioned if the diff is.
  • T2 — full contract. Delegated L3+ work, multi-hour or cross-file: all six components, spec review gate, traceability IDs.
Mis-sizing, diagnosed from symptoms (H4)
  • Too small: the agent asks basic questions per session; confidently-wrong PRs that solve a different problem; non-goals discovered during code review.
  • Too big: spec longer than the diff; review time dominated by markdown; the PM spends more time on task documents than on the product.

H2 · DEEP DIVE

EARS & Gherkin: Machine-Verifiable Acceptance Criteria

0Why the criteria are the contract's engine

H1 dissected the handoff contract into six components and was blunt about which one carries the proof obligation: component C, the acceptance criteria — "machine-verifiable wherever possible". This deep dive is about the two formats that deliver that clause. EARS structures requirements as constrained natural-language rules; Gherkin turns rules into concrete, executable examples. Together they walk a criterion down an altitude ladder — from intent a PM can ratify, to a scenario a reviewer can falsify, to a red test the harness can run — and that last rung is the point. E4's verification-first principle says the reviewer's question is never "does this look right" but "what machine said this is right"; criteria written in prose leave that question unanswerable, and the review gate H1 built becomes theatre.

The reason this matters more for agents than it ever did for humans is the same amnesia argument P3 made for decomposition. A human developer resolves an ambiguous criterion by leaning over to ask; an agent resolves it by guessing with confidence — Report 2 calls this intent hallucination, and it is invisible at exactly the moment it happens, because the agent's guess compiles, passes its own improvised tests, and reads plausibly in the diff. The defence is not more prose. It is criteria structured enough that ambiguity has nowhere to hide (EARS) and concrete enough that the claim "done" is a program output, not an opinion (Gherkin).

The oracle, in one line

Every delegation needs an oracle — the thing that decides pass or fail. A prose criterion makes the reviewer the oracle (expensive, inconsistent, unavailable at 2 a.m. when the agent finishes); an executable criterion makes the harness the oracle. H2 is the craft of moving criteria from the first category to the second, and of knowing when the move is not worth it (§4).

1The verification gap

Put a typical brownfield ticket criterion next to what a machine can evaluate and the gap is obvious. "The system should handle errors gracefully" names no trigger, no observable response, no boundary — it cannot fail, therefore it cannot pass. The INVEST checklist already demanded testable stories a decade before agents; the workbook's M3 overview sharpened the T for the agentic setting: testable is no longer enough, the bar is machine-verifiable, because the whole economic argument for delegation (E6's autonomy levels) collapses if a human must hand-inspect every completion anyway.

The gap has two distinct failure directions, and the two formats of this topic attack one each. Direction one is ambiguity in: the criterion under-specifies, the agent fills the void with priors, and you get the wrong right-looking thing — H1's intent problem at criterion granularity. Direction two is unverifiability out: the criterion may even be clear, but nothing can execute it, so "finished" reverts to opinion and the PR review inherits the entire proof burden — at densities arXiv:2603.15911 measured at 29.6 review tokens per changed line for AI-authored work, that burden is precisely what the contract was supposed to amortize. EARS narrows direction one; Gherkin closes direction two.

One warning before the formats: an executable criterion is only as strong as its oracle. The SWE-Bench+ audit (arXiv:2410.06992, cited at length in H1 §2C) found roughly a third of claimed benchmark passes tainted by solution leakage and another third passing only because tests were weak. Translating a vague rule into a weak scenario produces confident false greens — worse than prose, because it looks machine-verified. The craft sections (§5) and B3's mutation gate exist for exactly this reason.

2EARS — constrained natural language for rules

EARS — the Easy Approach to Requirements Syntax — was developed at Rolls-Royce by Alistair Mavin with Philip Wilkinson, Adrian Harwood and Mark Novak, and published at the IEEE Requirements Engineering conference in 2009 (RE'09, DOI 10.1109/RE.2009.9). The origin is instructive: the team was analysing airworthiness regulations for a jet-engine control system — prose requirements at their most safety-critical and most lawyered — and found that nearly all of them could be rewritten into a handful of sentence templates keyed on when and under what conditions the system must respond. Five core patterns plus a combining form:

PatternTemplatePassword-reset example (Report 2's task)
Ubiquitous — always trueThe <system> shall <response>The service shall log every throttling decision with account id hashed.
Event-driven — triggeredWHEN <trigger>, the <system> shall <response>WHEN a client submits more than 5 reset requests for the same account within 15 minutes, the system shall respond 429 and shall not send a reset email.
State-driven — while a state holdsWHILE <state>, the <system> shall <response>WHILE the throttle flag is off, the system shall preserve existing reset behaviour byte-for-byte.
Optional feature — if the feature is presentWHERE <feature is included>, the <system> shall <response>WHERE the tenant has SSO enabled, the system shall exempt IdP-initiated resets from the counter.
Unwanted behaviour — failure & abuse pathsIF <trigger>, THEN the <system> shall <response>IF the Redis counter is unavailable, THEN the system shall fail open and alert, not block resets.
Complex — combinationsWHILE <state>, WHEN <trigger>, the <system> shall <response>WHILE the flag is on, WHEN the 6th request arrives, the system shall respond 429.
The pattern is the linter

The templates look almost too simple to matter, and that is the trick: a vague sentence does not fit any of them. "Handle errors gracefully" has no trigger for WHEN, no state for WHILE, no observable response for the SHALL clause — attempting the rewrite is the ambiguity detector, and it runs at authoring time, before any agent sees the task. E5's advisory-versus-deterministic rule applies here too: EARS conformance is mechanically checkable (QVscribe has shipped automated EARS-conformance checking since v2.10, alongside INCOSE's 41-rule set), so a client can gate requirements the way CI gates code.

Who uses it? Honestly sourced: Rolls-Royce is the documented origin; beyond that, the adopter roster — Airbus, Bosch, Dyson, Honeywell, Intel, NASA, Siemens — is self-reported by EARS's creator on alistairmavin.com, and we present it as exactly that, a claim by the method's author, not independent verification. What is independently observable is the agentic uptake: Amazon's Kiro generates its requirements.md in explicitly-named EARS notation ("WHEN [condition/event] THE SYSTEM SHALL [expected behavior]") — the tightest coupling yet of a 2009 aerospace syntax to a 2025 coding agent, and a vendor design decision worth reading as evidence of fit (point-in-time, mid-2026; re-verify).

Why does a sentence syntax help a language model? Direct measured evidence is thin — we flag clearly: no study we can cite shows "EARS improves agent code generation by X%", and we refuse to invent one. What exists is adjacent: grammar-prompting work (arXiv:2305.19234, NeurIPS 2023) shows LLMs follow injected formal syntax templates reliably, and requirements-engineering research has begun evaluating EARS as a target template for structuring extracted knowledge (arXiv:2511.15340). The mechanism argument is simpler and older than either paper: EARS removes the degrees of freedom in which intent hallucination lives. The trigger, the state, the actor and the observable response are each forced into a named slot; an empty slot is visible to a reviewer — and to a linter — in a way a fuzzy paragraph never is.

3Gherkin — the executable criterion

EARS produces rules. Rules are still abstractions, and abstractions still leave room to guess — which five requests, what counts as the same account, what exactly does the 429 body say? Gherkin closes the remaining distance with concrete examples. A scenario names real data and a real observable outcome:

The EARS rule from §2, as a Gherkin scenario

Feature: password-reset throttling
  Scenario: sixth request in the window is throttled
    Given account "ada@example.com" has made 5 reset requests in the last 15 minutes
    When the client submits another reset request for "ada@example.com"
    Then the response status is 429
    And no reset email is sent

Given establishes state, When fires the trigger, Then asserts the observable response — the same three slots EARS forced, now bound to data. Note what the scenario adds to the rule: the boundary is now falsifiable (a sibling scenario asserts the fifth request passes), and "no reset email is sent" is a checkable side-effect assertion, not a hope.

The history matters because it explains both the power and the trap (§5). Given–When–Then was worked out around 2004 by Dan North with business analyst Chris Matts, as part of North's behaviour-driven development — BDD — reframing of TDD; the canonical write-up, "Introducing BDD", ran in Better Software magazine in March 2006. Aslak Hellesøy's Cucumber (2008) made the format executable: Gherkin is Cucumber's plain-text language, and step definitions bind each Given/When/Then line to automation code, so the same text is simultaneously the specification a PM reads and the test the harness runs — "executable specification", with "living documentation" as its corollary: documentation that fails CI when it stops being true.

For two decades that double identity was BDD's selling point to humans, and adoption chronically decayed into §5's trap. The agentic setting changes the economics in Gherkin's favour, for a reason E8 already named: the expensive part of BDD was never reading scenarios, it was writing and maintaining the step definitions — mechanical glue code, which is precisely the work coding agents do well and cheaply. And the payoff compounds: a Gherkin set handed to an agent is not just documentation, it is the reward signal. The agent iterates until the scenarios pass; the criteria are the oracle from §0, wired directly into the loop. SDD tooling has converged on the format from the other direction — GitHub Spec Kit's template writes each user story's Acceptance Scenarios section as numbered Given/When/Then items (point-in-time, mid-2026).

The two formats are levels, not competitors — a distinction Specification by Example (Adzic, 2011) drew as rules versus examples: EARS states the rule at the requirements level; Gherkin illustrates the rule with key examples at the acceptance-criteria level; the failing tests generated from the scenarios are the handoff itself. Second, the ladder is not a mandate: H1's T0–T2 sizing applies rung by rung. Sixteen acceptance criteria on a one-line bugfix is the sledgehammer anti-pattern, not rigour.

5Writing quality — the craft and the trap

Machine-verifiable formats fail in characteristic ways, all of them known for a decade before agents arrived. Three crafts and one organizational trap:

1

Declarative, not imperative

The Cucumber documentation's own rule: scenarios "should describe the intended behaviour of the system, not the implementation — what, not how". An imperative scenario scripts UI mechanics (click this, type that, wait) and welds the criterion to today's interface, so every redesign breaks the spec without the rule having changed; a declarative scenario ("Given an account with 5 recent reset requests…") survives any implementation the rule permits. For agents the distinction is sharper still: an imperative scenario quietly makes design decisions that belonged to the agent's solution space — it smuggles the how into the what, exactly the altitude error P2 warned about in PRDs.

2

BRIEF

Seb Rose and Gáspár Nagy's checklist from Formulation (the BDD Books): Business language, Real data, Intention revealing, Essential, Focused — and the sixth principle, brief itself. "Essential" carries the most weight in the agentic setting: every incidental detail in a scenario ("the button is blue") reads to an agent as a requirement to satisfy, so padding does not just bore reviewers, it actively constrains the solution. The contract-sizing logic of H1 §5 repeats at scenario grain.

3

Failure paths are half the set

H1 §2C's craft, formalized by the formats: EARS gives unwanted behaviour its own pattern (IF/THEN) and Gherkin makes each failure path one scenario — service down, empty result, sixth-versus-fifth request at the boundary. TASK-042's network-failure scenario and the password-reset fail-open rule are the worked examples. A criteria set with only happy paths is a weak oracle (§1) wearing executable clothes; B2's characterization-test discipline is the brownfield version of the same instinct.

The BDD trap — and who writes it at the client

BDD's documented failure mode is organizational, not technical: Given/When/Then adopted as a test-automation syntax, written by test engineers alone, after development, with no conversation — Seb Rose: "I see far more people using G/W/T for test automation than to support BDD… the BDD acronym has become synonymous in the industry with G/W/T powered tests." Adzic's Specification by Example (2011, from 50-plus case studies) found collaboration was the success factor, and his verified framing of the agentic revival cuts both ways: "Specification by Example taken to a new level — or the revenge of Waterfall." Which one a client gets is decided by ownership. The answer this workbook carries (H1 §4): three-amigos lenses, kept — analyst drafts business rules (EARS), tester lens drafts boundary and negative scenarios, tech lead ratifies verifiability — with the agent as the fourth amigo that drafts candidates and step definitions, never the sole author. If only QA touches the Gherkin, you have rebuilt the trap with more expensive tooling.

6What the evidence says — and the 2026 toolchain

The workbook's three headline numbers for this topic all survived a July-2026 re-verification pass, each with a framing correction worth keeping. ClarifyGPT (Mu et al., FSE 2024; arXiv:2310.10996) has the model detect ambiguous requirements and ask clarifying questions before generating code: an average relative Pass@1 improvement of 11.52% for GPT-4 across four benchmarks — concretely, 70.96% → 80.80% on MBPP-sanitized. Relative percent, not percentage points; the mechanism it prices is exactly §0's point that asking beats guessing. Bashir et al. (ICSME 2025, industry track) measured ambiguity classification on three industrial railway requirement sets (Alstom, Westermo): the 20.2% figure is the average gain from giving the model ten in-context examples over zero-shot — the value of showing the model what ambiguity looks like, not of using an LLM at all; the LLM's explanations of why a requirement is ambiguous rated 3.84/5 with practitioners. Wang et al. (arXiv:2508.06888, the RAGcceptance system) generate acceptance criteria grounded in retrieved project data: multi-modal RAG lifts case-level accuracy from 42.8% to 66.0% — the +23.2 pp our topic data cites — on a ~100,000-user education platform.

Read as one argument, the three studies point the same way: requirement ambiguity is measurable, and every mechanism that reduces it before generation pays — asking (ClarifyGPT), showing examples (Bashir), grounding in real project data (Wang). Note the honest limit from §2: none of them tests EARS or Gherkin syntax as the treatment; what they validate is the underlying bet that structure and grounding beat prose.

The test-first rung of the ladder has more direct backing — this is what the discovery pass added to our outline. Mathews & Nagappan (ASE 2024, arXiv:2402.13521) tested TDD-for-LLMs empirically: putting the test cases in the prompt consistently raises solve rates on MBPP and HumanEval — tests-as-spec, measured. And the strongest real-world datapoint is a design decision hiding in plain sight: SWE-bench Verified, the industry's standard coding-agent benchmark, defines "resolved" entirely as a test oracle — hidden FAIL_TO_PASS tests must flip green and no PASS_TO_PASS test may regress, with no human judgement anywhere in the loop. The benchmark the whole field competes on is, structurally, a machine-verifiable acceptance contract; E7's measurement chapter and this topic's thesis meet in that one design choice.

Gherkin-specific agent research is younger — industrial case studies rather than controlled trials, flagged as emerging: AutoUAT and TestFlow generate Gherkin from user stories and executable tests from Gherkin at a reported ~€0.12 per test script (arXiv:2504.07244); the A-TEST 2024 experiments had collaborating LLM agents execute Gherkin scenarios directly; and the practitioner layer is forming — Andrew Knight's Gherkin guidelines for AI (Automation Panda, Apr 2026) is a curated ruleset written to be fed to coding agents as authoring context, §5's craft packaged for the fourth amigo.

8Takeaways for our engagement

Three sentences to keep

Acceptance criteria are the only part of the handoff contract a machine can enforce, so their format decides whether the spec review gate is real or theatre: EARS forces the rule to name its trigger, state and observable response; Gherkin binds the rule to concrete falsifiable examples; the failing tests are the handoff. The formats are cheap and old — the 2009 aerospace syntax and the 2004 conversation pattern — and the agentic turn fixes their historical cost problem, because step definitions and candidate scenarios are exactly the mechanical work agents draft well, while ratifying rules and choosing the examples that matter stays human at the gate. The trap is organizational, not technical: if Gherkin becomes QA's test syntax instead of the team's shared criteria language, you have rebuilt 2015-era BDD failure with more expensive tooling — keep the three amigos, add the agent as the fourth who drafts but never ratifies.
